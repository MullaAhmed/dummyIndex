# How to use `.context/`

This folder is the project's **canonical machine-readable context** for AI coding agents (Claude Code, Cursor, Codex, Aider). It is generated by dummyIndex. Its **deterministic backbone** (map, tree, naming, source-docs) is refreshed by `dummyindex context rebuild --changed`, which **preserves the curated feature docs** — it never re-clusters an enriched index. Its **curated content** (feature `spec.md`/`plan.md`/`concerns.md`, conventions) is updated through the reconcile procedure: `dummyindex context reconcile` reports the delta (read-only, **writes nothing**), then the `/dummyindex` skill's reconcile procedure (`council/65-reconcile.md`) places new files and re-enriches drifted features, then `dummyindex context reconcile-stamp` advances the commit anchor.

## Read this folder *first* for any non-trivial work

Before grepping, reading files, or asking the user for context, consult these in order:

| Question | Look at |
|---|---|
| "What is this project?" | `PROJECT.md` |
| "What's the layout?" | `architecture/overview.md`, then `tree.json` |
| "How does feature `Z` work?" / "What's the flow when…?" | **Start at `features/INDEX.json`** (entries live under a top-level `features` array, each keyed **`feature_id`** — *not* `id` — with `name`/`summary` plus `member_count`/`file_count`/`entry_point_count`/`flow_count`) — then `features/<feature_id>/feature.json` (raw `members`/`files`/`entry_points`/`flow_ids` lists, no `*_count` keys) + `features/<feature_id>/spec.md`. Drill into `plan.md` (implementation) / `concerns.md` (risks). See `features/HOW_TO_NAVIGATE.md`. |
| "Where is `X` defined?" | `map/symbols.json` — every class / function / method has `path` + `range` |
| "What files contain `Y`?" | `map/files.json` — full file inventory with language + size |
| "How does `X` relate to `Y`?" / "Communities / hidden dependencies?" | `features/symbol-graph.json` — NetworkX graph with `contains`, `method`, `inherits`, `imports`, `calls` edges + Leiden communities |
| "What's the naming style?" | `conventions/naming.md` — honor this when adding new code |
| "How is code grouped into folders?" | `conventions/folder-organization.md` |
| "What patterns / idioms does this repo use?" | `conventions/coding-practices.md` |
| "How are tests written here?" | `conventions/testing.md` |
| "How does this repo talk to its database?" | `conventions/data-access.md` |
| "How do I add / change / fix X?" | `playbooks/` — task-specific recipes |
| "How do I retire stale proposals/audits?" | `playbooks/gc-context.md` — the context-hygiene GC sweep (`/dummyindex-gc`) |

## Retrieval principle: walk the tree, don't grep

This index uses **PageIndex-style reasoning-based retrieval**. The procedure for any non-trivial question:

1. **(Optional fast first cut)** Run `dummyindex context query "<your task or question>"`. It scores every feature against the query by token overlap with its name/summary/files/symbols and prints the top-K with cited markdown excerpts — a deterministic *ranked shortlist* (no LLM in the loop). Use it to decide which feature(s) to open; then keep reasoning. Add `--top-k N` to widen, `--json` for machine output.
2. Read `features/INDEX.json` (the table of contents).
3. Reason over the names + summaries. Pick 1–3 features the task touches (the `query` ranking is a hint, not the verdict — reason over the TOC yourself).
4. Read the chosen `feature.json` + `spec.md`.
5. Drill into `plan.md` (implementation) / `concerns.md` (risks).
6. Follow `path:range` citations to source when you need to verify.

**Never grep the source tree first.** The tree has already been mapped; reasoning over the map beats string-matching the source.

## Concrete examples

**Asked to add a new feature?**
Read `playbooks/add-feature.md`. It walks through: check for duplicates in `map/symbols.json`, mirror an existing feature's structure (find via `tree.json`), honor `conventions/naming.md`, write tests first.

**Asked to find or modify something specific?**
Don't grep blindly. `map/symbols.json` has every symbol's `path:range`; load it once, find the match, jump straight to the code.

**Asked an architectural question?**
Start with `PROJECT.md` + `architecture/overview.md`. Most questions answer themselves at that level. Drill into `tree.json` only if you need finer detail.

**Doing a rename or refactor?**
`map/symbols.json` gives you every site to consider. Combine with conventional grep for callers.

## When the index is wrong

The code is the source of truth. If the index disagrees with what you see in a file, trust the file — then fix the index by the path that matches what's stale:

- **Stale deterministic artefacts** (a moved/renamed symbol, a new or deleted file, an out-of-date `map/`/`tree.json`) → run `dummyindex context rebuild --changed`. It re-hashes every source file and refreshes only the changed deterministic bits. On a curated index it **preserves the feature taxonomy + enrichment** — it does **not** touch `features/<id>/*.md` or re-cluster, so it can never undo curated work.
- **Stale or missing *feature* docs** (a `spec.md`/`plan.md`/`concerns.md` that no longer matches how the feature works, or new code that belongs to no feature yet) → `rebuild --changed` will **not** fix these. Update the doc in-session (you may edit it by hand — see "Conventions for agents" below), or run the reconcile procedure: `dummyindex context reconcile` to see the delta (read-only — **writes nothing**; it does **not** re-cluster, so it's safe to run any time), then the `/dummyindex` skill's reconcile procedure (`council/65-reconcile.md`) to place/enrich, then `dummyindex context reconcile-stamp`.
- **When your explicit instruction contradicts a `.context/` spec/plan, you win** — the code and the user's intent are the source of truth. Note the divergence and proceed; flag it so a later reconcile folds it in.

## Feature-oriented navigation

`features/` is the **behavioral** view of the codebase (vs. `tree.json` which is the structural view). Start with `features/INDEX.json` when the user asks "how does X work?" or "what's the flow when…?" — it points at per-feature `feature.json` + `flows/<flow-id>.json` so you can answer without grepping. Folder names get renamed during enrichment but `feature_id` is stable, so always navigate by ID.

## What gets committed and why

Everything in `.context/` except `cache/` and the scratch artefacts listed in `.context/.gitignore` is **meant to be committed** — a fresh clone must be able to navigate this index without running the CLI, and the reconcile loop anchors on the committed state. The heavy machine-layer artefacts (`tree.json`, `map/files.json`, `map/symbols.json`, `features/symbol-graph.json`, `features/graph.json`, `features/graph.html`) are marked `linguist-generated` in the managed `.context/.gitattributes`, so GitHub collapses their large rebuild diffs in PR review while local `git diff` and merges stay untouched.

The committed layout, top-down:

| Path | What it is |
|---|---|
| `PROJECT.md`, `INDEX.md`, `architecture/` | project overview + entry index |
| `map/`, `tree.json`, `source-docs/` | deterministic backbone (regenerated by `rebuild --changed`) |
| `features/` | curated behavioral view (per-feature `spec.md`/`plan.md`/`concerns.md`) |
| `conventions/`, `playbooks/` | authored house style + task recipes |
| `proposals/<slug>/`, `audits/<slug>/` | **generated per-task workspaces** — swept by the hygiene GC (below) |
| `gc/state.json` | **committed** GC commit anchor (`{"anchor": sha}`) — the only file under `gc/`; the per-session nudge memo lives in gitignored `cache/`, never here |
| `cache/`, `_doc_backups/` | per-machine scratch — **gitignored**, never referenced |

## Context-hygiene GC (lifecycle of generated docs)

dummyindex generates per-task workspaces under `.context/proposals/<slug>/` (plan artifacts) and `.context/audits/<slug>/` (argue-and-audit panels). These are **disposable**: once a proposal is superseded, abandoned as an unfleshed scaffold, or an audit's findings are long fixed, the workspace is misleading context for the next agent. The **context-hygiene GC** retires them.

**Generated docs are GC'd — deleted, not archived.** There is no `_archive/` tier going forward (the old `proposals/_archive/` convention is itself a delete candidate). The disposition is decisive deletion: distill nothing, keep nothing around "just in case". A `done`-but-still-relevant plan stays; only the genuinely **stale / superseded / dead / no-longer-useful** is removed.

**How the sweep runs.** It is an LLM council step, `/dummyindex-gc`: subagents walk the candidate docs PageIndex-style (via `dummyindex context query`), judge each against the current code and the work in flight, and propose a delete-list — then **you confirm, and only then does anything get deleted**. Deletion is *always* user-confirmed. Dead **docs** are deleted directly; **trivially-dead code** (a 0-caller private symbol that passes every exclusion guard) goes through the normal implementer+tester path and only if the full suite stays green; anything broader is routed to a new proposal, never auto-removed. GC never edits or removes source code directly.

**When it's nudged.** A commit-count throttle, not a clock: when >= N commits (default 10) have landed since the last GC anchor, the SessionStart hook emits a one-line `N commits since last hygiene sweep — run /dummyindex-gc` nudge. The sweep stamps a fresh anchor to reset the counter. You can also run `/dummyindex-gc` on demand any time.

**The deterministic plumbing** is `dummyindex context gc`, with four verbs (see `playbooks/gc-context.md` for the order to use them in):

| Verb | What it does |
|---|---|
| `gc status [--json]` | read-only sweep report — every candidate doc with its signals, plus the throttle state (`commits_since` / `anchor` / `threshold` / `should_signal`). The skill and the SessionStart hook consume this. |
| `gc delete --kind proposal\|audit (--slug S \| --path P)` | delete **one** workspace dir. A dry-run without `--yes` (deletes nothing); refuses a sentinel/escaping target and an untracked workspace (needs `--allow-untracked`). Never touches source code. |
| `gc stamp [--to SHA]` | advance the committed GC anchor in `gc/state.json` to HEAD (off-git no-op). |
| `gc signal` | the SessionStart throttle probe — prints the nudge iff over threshold and not already signalled this session. |

## CI, linting & secret scanners

- The `sha256` fields in `map/files.json` and `source-docs/INDEX.json` are **content fingerprints of source files**, not credentials. Entropy-based scanners (e.g. detect-secrets' "Hex High Entropy String") flag them as false positives.
- **Don't blanket-exclude `.context/` from every pre-commit hook** — you'd lose JSON validity, merge-conflict, and large-file checks on the index. Scope the exclusion to the entropy scanner only, e.g. for detect-secrets:

  ```yaml
  - id: detect-secrets
    exclude: ^\.context/(map/|source-docs/INDEX\.json)
  ```

  or audit the hits once into your `.secrets.baseline` allowlist.
- Generated files end with exactly one newline and carry no trailing whitespace, so whitespace fixers pass without rewrites.

## Conventions for agents

- **Two layers, two rules.** The **deterministic** artefacts — `map/files.json`, `map/symbols.json`, `tree.json`, `conventions/naming.md`, `source-docs/INDEX.json`, the `features/*` indexes/graphs — are regenerated by `rebuild --changed`; **don't hand-edit them**, your edits are overwritten. The **curated** layer — `features/<id>/spec.md`/`plan.md`/`concerns.md` and the authored `conventions/*.md` (everything except `naming.md`) — is council-owned and **is** edited in-session to clear drift; `rebuild --changed` preserves it. So: hand-edit a feature doc to fix it; never hand-edit a generated map/index.
- **`cache/` is per-machine** — gitignored. Never reference it.
- **Always check `conventions/naming.md`** before naming a new symbol or file.
