# How to use `.context/`

This folder is the project's **canonical machine-readable context** for AI coding agents (Claude Code, Cursor, Codex, Aider). It is generated by dummyIndex. Its **deterministic backbone** (map, tree, naming, source-docs) is refreshed by `dummyindex context rebuild --changed`, which **preserves the curated feature docs** вЂ” it never re-clusters an enriched index. Its **curated content** (feature `spec.md`/`plan.md`/`concerns.md`, conventions) is updated through the reconcile procedure: `dummyindex context reconcile` reports the delta (read-only, **writes nothing**), then the `/dummyindex` skill's reconcile procedure (`council/65-reconcile.md`) places new files and re-enriches drifted features, then `dummyindex context reconcile-stamp` advances the commit anchor.

## Read this folder *first* for any non-trivial work

Before grepping, reading files, or asking the user for context, consult these in order:

| Question | Look at |
|---|---|
| "What is this project?" | `PROJECT.md` |
| "What's the layout?" | `architecture/overview.md`, then `tree.json` |
| "How does feature `Z` work?" / "What's the flow whenвЂ¦?" | **Start at `features/INDEX.json`** (entries live under a top-level `features` array, each keyed **`feature_id`** вЂ” *not* `id` вЂ” with `name`/`summary` plus `member_count`/`file_count`/`entry_point_count`/`flow_count`) вЂ” then `features/<feature_id>/feature.json` (raw `members`/`files`/`entry_points`/`flow_ids` lists, no `*_count` keys) + `features/<feature_id>/spec.md`. Drill into `plan.md` (implementation) / `concerns.md` (risks). See `features/HOW_TO_NAVIGATE.md`. |
| "Where is `X` defined?" | `map/symbols.json` вЂ” every class / function / method has `path` + `range` |
| "What files contain `Y`?" | `map/files.json` вЂ” full file inventory with language + size |
| "How does `X` relate to `Y`?" / "Communities / hidden dependencies?" | `features/symbol-graph.json` вЂ” NetworkX graph with `contains`, `method`, `inherits`, `imports`, `calls` edges + Leiden communities |
| "What's the naming style?" | `conventions/naming.md` вЂ” honor this when adding new code |
| "How is code grouped into folders?" | `conventions/folder-organization.md` |
| "What patterns / idioms does this repo use?" | `conventions/coding-practices.md` |
| "How are tests written here?" | `conventions/testing.md` |
| "How does this repo talk to its database?" | `conventions/data-access.md` |
| "How do I add / change / fix X?" | `playbooks/` вЂ” task-specific recipes |

## Retrieval principle: walk the tree, don't grep

This index uses **PageIndex-style reasoning-based retrieval**. The procedure for any non-trivial question:

1. **(Optional fast first cut)** Run `dummyindex context query "<your task or question>"`. It scores every feature against the query by token overlap with its name/summary/files/symbols and prints the top-K with cited markdown excerpts вЂ” a deterministic *ranked shortlist* (no LLM in the loop). Use it to decide which feature(s) to open; then keep reasoning. Add `--top-k N` to widen, `--json` for machine output.
2. Read `features/INDEX.json` (the table of contents).
3. Reason over the names + summaries. Pick 1вЂ“3 features the task touches (the `query` ranking is a hint, not the verdict вЂ” reason over the TOC yourself).
4. Read the chosen `feature.json` + `spec.md`.
5. Drill into `plan.md` (implementation) / `concerns.md` (risks).
6. Follow `path:range` citations to source when you need to verify.

**Never grep the source tree first.** The tree has already been mapped; reasoning over the map beats string-matching the source.

## Concrete examples

**Asked to add a new feature?**
Read `playbooks/add-feature.md`. It walks through: check for duplicates in `map/symbols.json`, mirror an existing feature's structure (find via `tree.json`), honor `conventions/naming.md`, write tests first.

**Asked to find or modify something specific?**
Don't grep blindly. `map/symbols.json` has every symbol's `path:range`; load it once, find the match, jump straight to the code.

**Asked an architectural question?**
Start with `PROJECT.md` + `architecture/overview.md`. Most questions answer themselves at that level. Drill into `tree.json` only if you need finer detail.

**Doing a rename or refactor?**
`map/symbols.json` gives you every site to consider. Combine with conventional grep for callers.

## When the index is wrong

The code is the source of truth. If the index disagrees with what you see in a file, trust the file вЂ” then fix the index by the path that matches what's stale:

- **Stale deterministic artefacts** (a moved/renamed symbol, a new or deleted file, an out-of-date `map/`/`tree.json`) в†’ run `dummyindex context rebuild --changed`. It re-hashes every source file and refreshes only the changed deterministic bits. On a curated index it **preserves the feature taxonomy + enrichment** вЂ” it does **not** touch `features/<id>/*.md` or re-cluster, so it can never undo curated work.
- **Stale or missing *feature* docs** (a `spec.md`/`plan.md`/`concerns.md` that no longer matches how the feature works, or new code that belongs to no feature yet) в†’ `rebuild --changed` will **not** fix these. Update the doc in-session (you may edit it by hand вЂ” see "Conventions for agents" below), or run the reconcile procedure: `dummyindex context reconcile` to see the delta (read-only вЂ” **writes nothing**; it does **not** re-cluster, so it's safe to run any time), then the `/dummyindex` skill's reconcile procedure (`council/65-reconcile.md`) to place/enrich, then `dummyindex context reconcile-stamp`.
- **When your explicit instruction contradicts a `.context/` spec/plan, you win** вЂ” the code and the user's intent are the source of truth. Note the divergence and proceed; flag it so a later reconcile folds it in.

## Feature-oriented navigation

`features/` is the **behavioral** view of the codebase (vs. `tree.json` which is the structural view). Start with `features/INDEX.json` when the user asks "how does X work?" or "what's the flow whenвЂ¦?" вЂ” it points at per-feature `feature.json` + `flows/<flow-id>.json` so you can answer without grepping. Folder names get renamed during enrichment but `feature_id` is stable, so always navigate by ID.

## What gets committed and why

Everything in `.context/` except `cache/` and the scratch artefacts listed in `.context/.gitignore` is **meant to be committed** вЂ” a fresh clone must be able to navigate this index without running the CLI, and the reconcile loop anchors on the committed state. The heavy machine-layer artefacts (`tree.json`, `map/files.json`, `map/symbols.json`, `features/symbol-graph.json`, `features/graph.json`, `features/graph.html`) are marked `linguist-generated` in the managed `.context/.gitattributes`, so GitHub collapses their large rebuild diffs in PR review while local `git diff` and merges stay untouched.

## CI, linting & secret scanners

- The `sha256` fields in `map/files.json` and `source-docs/INDEX.json` are **content fingerprints of source files**, not credentials. Entropy-based scanners (e.g. detect-secrets' "Hex High Entropy String") flag them as false positives.
- **Don't blanket-exclude `.context/` from every pre-commit hook** вЂ” you'd lose JSON validity, merge-conflict, and large-file checks on the index. Scope the exclusion to the entropy scanner only, e.g. for detect-secrets:

  ```yaml
  - id: detect-secrets
    exclude: ^\.context/(map/|source-docs/INDEX\.json)
  ```

  or audit the hits once into your `.secrets.baseline` allowlist.
- Generated files end with exactly one newline and carry no trailing whitespace, so whitespace fixers pass without rewrites.

## Conventions for agents

- **Two layers, two rules.** The **deterministic** artefacts вЂ” `map/files.json`, `map/symbols.json`, `tree.json`, `conventions/naming.md`, `source-docs/INDEX.json`, the `features/*` indexes/graphs вЂ” are regenerated by `rebuild --changed`; **don't hand-edit them**, your edits are overwritten. The **curated** layer вЂ” `features/<id>/spec.md`/`plan.md`/`concerns.md` and the authored `conventions/*.md` (everything except `naming.md`) вЂ” is council-owned and **is** edited in-session to clear drift; `rebuild --changed` preserves it. So: hand-edit a feature doc to fix it; never hand-edit a generated map/index.
- **`cache/` is per-machine** вЂ” gitignored. Never reference it.
- **Always check `conventions/naming.md`** before naming a new symbol or file.
