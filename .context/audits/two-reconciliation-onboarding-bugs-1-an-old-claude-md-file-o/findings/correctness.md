### Bug 1 — pre-existing CLAUDE.md / `.claude/` not reconciled into the new setup

- `dummyindex/installer/install.py:262-263` (via `runner.py`) — **high** — a pre-existing **root** `CLAUDE.md` is left dangling on a fresh init/install: confirmed real. — evidence: the init path is `install → _auto_init_project → build_all → runner.bootstrap_claude_md(out_root/".claude"/"CLAUDE.md")` (`runner.py:262-263`). It only ever writes `.claude/CLAUDE.md` and never inspects `<root>/CLAUDE.md`. The only code that migrates a legacy root `CLAUDE.md` is `migrate_claude_md_location` (`dummyindex/cli/migrate.py:73`), which is reachable **only** from `migrate_legacy_layout` (`migrate.py:70`), called **only** from `dummyindex context refresh-indexes` (`dummyindex/cli/refresh.py:32`). So `dummyindex install` / `init` / `/dummyindex` never runs it. A repo that already had a root `CLAUDE.md` (the common Claude-Code default) ends up with two files — the user's root one (untouched, with no managed block) and a new `.claude/CLAUDE.md` — and the dummyindex pointer is invisible to a session that reads the root file. — suggested fix: call `migrate_claude_md_location(out_root)` from `runner.py` bootstrap (or from `_auto_init_project`) before/after `bootstrap_claude_md`, so init relocates a legacy root block and preserves user residue exactly as `refresh-indexes` does.
  - — [round1: defend] status=confirmed: C1 is the same root cause as peers A1/A2 (`install.py:271,236` + `runner.py:262-263`) — three names for one missing call. The init/build_all path never invokes any `migrate_claude_md_location`; only `refresh.py:32` does. Concur with A1/A2 that this is one defect surfaced at two entry points (auto-init and first-build); not two distinct bugs.

- `dummyindex/cli/migrate.py:108-117` — **medium** — `migrate_claude_md_location` does NOT fold a legacy root `CLAUDE.md` that has **no managed block** (i.e. a hand-written user `CLAUDE.md`) into `.claude/CLAUDE.md`; it leaves it dangling. — evidence: when `legacy_had_managed_block` is False and `new_path` exists it returns early (`:108-111`); when `new_path` is absent it only writes a fresh managed block to `.claude/CLAUDE.md` (`:114`) and returns (`:119-121`) without ever reading/merging the user's root `CLAUDE.md` content. A user with a plain root `CLAUDE.md` full of their own instructions gets a second file under `.claude/` and their root file is never migrated or even acknowledged. This is exactly the "old CLAUDE.md left dangling / user content not relocated" symptom. — suggested fix: when the legacy root file exists with no managed block, either append the managed pointer to it in place, or carry its user content into `.claude/CLAUDE.md`; do not silently produce two competing files.
  - — [round1: defend] status=confirmed: re-read `migrate.py:108-117` confirms the early-return on `legacy_had_managed_block is False` with `new_path` present, and the no-merge fresh-write when `new_path` absent. Concur with tests T1/T2/T3 — this exact branch (`legacy_had_managed_block==False` dangling scenario) has zero coverage, which is why the bug survived. C2 is a real correctness gap distinct from C1: even when migration IS reached (via refresh), a no-managed-block user file is dropped.

- `dummyindex/installer/install.py:167-181` — **low** — the **global** `~/.claude/CLAUDE.md` skill-registration uses a substring probe (`"dummyindex" in content`) that yields a false "already registered" — evidence: any user `CLAUDE.md` that merely mentions the word "dummyindex" (e.g. a note, or this very repo's docs pasted in) makes the registration block be skipped (`:171-172`), so the skill pointer is never written though it isn't actually present. Narrow blast radius (only the global registration text), but it's an incorrect idempotency check vs. the marker-based check used elsewhere (`BEGIN_MARKER`). — suggested fix: probe for the actual registration sentinel (`_SKILL_REGISTRATION`'s marker), not the bare word.
  - — [round1: defend] status=confirmed: stands as a low. No peer disputed it; the substring vs. marker mismatch is real but narrow.

- `dummyindex/cli/migrate.py:124-130` — **info** — when a legacy root block IS stripped, the residue is written back with `.strip()`+`"\n"` (`migrate.py:106,125`), collapsing all of the user's leading/trailing blank-line structure. Cosmetic, content is preserved — note only.
  - — [round1: defend] status=confirmed (info): unchanged; cosmetic, content preserved.

### Bug 2 — no reconcile triggered/checked after the Stop hook

- `dummyindex/context/reconcile_gate.py:298-346` — **refuted (gate exists)** — a Stop-hook reconcile gate DOES fire: `_STOP_HOOK` wires `dummyindex context reconcile-gate` (`dummyindex/context/hooks.py:128-140`), the CLI calls `decide_block` (`dummyindex/cli/reconcile_gate.py:27`), and a stale-after-substantial-session block reaches Claude Code via stdout. The user's framing ("nothing checks for reconcile after Stop") is wrong. The real defects are in the escape hatches below.
  - — [round1: defend] status=refuted: concur with T9 (Stop hook IS registered and tested via `hooks.py` + `tests/cli/test_reconcile_gate_e2e.py`). Stays refuted — the gate exists. The user's real grievance is the escape hatch, captured by C6.

- `dummyindex/context/reconcile_gate.py:229-238, 293-295` — **high** — drift in an **anchored** repo is silently dropped by the gate even when the session edited owned source: real silent-drift path. — evidence: `_gate_relevant` only blocks on `unassigned_new_files` / `awaiting_enrichment`, or on `report.rows` when `_has_live_anchor` is False (`:291-295`). A normal repo that ran `reconcile-stamp` once HAS a live anchor, so `_has_live_anchor` is True and mtime `rows` are ignored. But `compute_drift` only surfaces `unassigned_new_files` for **added** files and `awaiting_enrichment` for placement markers — a session that **modifies** existing source owned by a feature (the most common case) produces only `rows` and an empty commit-anchored signal at the *drift* layer. So editing a tracked file and committing it without reconciling: `unassigned` empty (not added), `awaiting` empty, `rows` ignored (anchor live) → gate returns None → session ends with the feature doc stale and no prompt. The `ReconcileReport.drifted_features` that WOULD catch this (`build/reconcile.py:174-179`) is never consulted by the gate. — suggested fix: have `_gate_relevant` also block when `compute_reconcile_report(...).drifted_features` is non-empty for an anchored repo (committed modifications to owned files), or feed `drifted_features` into the gate's relevance test.
  - — [round1: defend] status=confirmed (HIGH): VERIFIED end-to-end against source, this is the pivotal finding and it holds. Chain: (1) `compute_reconcile_report` DOES detect committed modifications to owned files → populates `drifted_features` (`build/reconcile.py:175-179`, proven by `tests/context/build/test_reconcile.py:103` where a modified owned file yields `"auth" in report.drifted_features`). (2) BUT `DriftReport` (`context/drift.py:74-77`) has only three fields — `rows`, `unassigned_new_files`, `awaiting_enrichment`; it has NO `drifted_features` field. (3) `compute_drift` constructs `DriftReport(rows=..., unassigned_new_files=reconcile.unassigned_new_files, awaiting_enrichment=reconcile.awaiting_enrichment)` (`context/drift.py:171-175`) — `reconcile.drifted_features` is computed then DISCARDED; never carried into the gate's only data structure. (4) `_gate_relevant` (`reconcile_gate.py:291-294`) tests `unassigned/awaiting` (empty for a pure modification) then `rows and not _has_live_anchor` — on an anchored repo `_has_live_anchor` is True (`reconcile_gate.py:235-238`), suppressing the mtime branch. Net: modify a committed feature-owned file on an anchored repo → gate returns None (`reconcile_gate.py:330`). Silent escape confirmed. `grep drifted_features` across the tree shows it consumed by `cli/reconcile.py`, `cli/rebuild.py`, `cli/status.py` — never by `drift.py` or `reconcile_gate.py`.

- `dummyindex/context/reconcile_gate.py:315-316` — **medium** — the `stop_hook_active` fast-path makes block-once degrade to **never-block** whenever `session_id` is empty. — evidence: with an empty `session_id`, `already_blocked` always returns False and `mark_blocked` is a no-op (`:202-226`), so block-once relies solely on `stop_hook_active`. If a session's transcript yields no `session_id` (cli/memory.py:46-47 falls back to env, which may also be unset) the gate can re-block every turn — OR, conversely, if the harness sets `stop_hook_active` on the very first Stop, the gate returns None and never blocks at all. The correctness hole is the latter: a single `stop_hook_active=true` short-circuits the whole gate before any drift is computed. The docstring frames this as "strong prompt, never a trap", but it also means one early re-entrant Stop suppresses the only reconcile prompt for the session. — suggested fix: only honor `stop_hook_active` as block-once when a memo couldn't be persisted (empty session_id); when `session_id` is present, rely on the memo and let `stop_hook_active` not suppress a first-ever block.
  - — [round1: concede→downgrade] status=disputed: my Round-0 "a single `stop_hook_active=true` short-circuits the whole gate → never-block" overstates the harness contract. Claude Code sets `stop_hook_active=true` ONLY on the re-entrant Stop that fires *because a previous Stop hook already blocked* — it is never true on the first Stop of a turn. So `stop_hook_active=true` cannot cause a never-block on its own; by the time it is true, a block has already fired (or the agent stopped for another reason). The CLI passes it through verbatim (`cli/reconcile_gate.py:26-30`); the early return (`reconcile_gate.py:315`) is the intended re-entry guard, validated by `tests/cli/test_reconcile_gate_e2e.py:103-110`. I CONCEDE the never-block claim. What remains real and narrow: the empty-`session_id` degradation — with no id, `mark_blocked`/`already_blocked` are no-ops (`reconcile_gate.py:202-226`) so block-once leans entirely on `stop_hook_active`; benign for a single re-block but the memo contract silently disappears. Downgrade C7 to LOW, scoped to the empty-session_id case only. Concur with T12 that this fallback is untested.

- `dummyindex/context/reconcile_gate.py:341-346` — **medium** — in the multi-index (submodule) case, `mark_blocked(root, session_id)` records the block only under the **session root's** `.context/cache` (`:341`, and `mark_blocked` hard-codes `root.resolve()/".context"` at `:216`). — evidence: the memo is keyed per session_id, not per ctx_root, so the block-once is shared across all roots — correct for "block once per session" — but `already_blocked` is likewise only checked against the root (`:317`). This is internally consistent, so not a bug; the latent risk is that a future per-root re-block would need per-root memos. Downgrade to info if the panel agrees one-block-per-session is the intended contract. — suggested fix: none (document the single-memo contract).
  - — [round1: concede→downgrade] status=disputed: re-read confirms `already_blocked` (`reconcile_gate.py:209`) and `mark_blocked` (`reconcile_gate.py:216`) both hard-code `root.resolve()/".context"` and key on `session_id`, not `ctx_root`. Since `decide_block` blocks at most once per session regardless of how many indexes are stale (`reconcile_gate.py:341` marks, `:344-346` renders one combined message), the single root-scoped memo IS the correct contract — there is no per-root re-block path to break. CONCEDE this as a real defect; it's working as intended. Downgrade to INFO (document the single-memo contract). Concur with peer A6's separate concern that the memo is set BEFORE the agent reconciles (`reconcile_gate.py:341`) — that is the genuine block-once weakness, not the root-scoping.

- `dummyindex/context/reconcile_gate.py:332-333` — **medium** — the gate silently allows the stop whenever the main transcript is missing/unreadable, even when an index is genuinely stale. — evidence: `if main_transcript is None or not main_transcript.exists(): return None` (`:332-333`). A real edited-source session whose transcript path the hook couldn't resolve (the `_resolve_transcript` fallback at cli/memory.py:51 can return None) escapes the gate entirely — drift ships with no prompt. Defensible (no proof of work), but it's a silent-drift path triggered by transcript-resolution failure, not by a trivial session. — suggested fix: when an index is gate-relevant but the transcript is unresolvable, fall back to a conservative block (or at least an `additionalContext` advisory) rather than a hard allow.
  - — [round1: defend] status=confirmed (medium): re-read confirms the hard allow at `reconcile_gate.py:332-333` precedes any significance check. Real but bounded — it only bites when `_resolve_transcript` can't find the transcript, a genuine edge. Holds at medium. Concur with A8's framing that the gate is over-coupled to transcript internals, which is the upstream cause of this fragility.

- `dummyindex/context/reconcile_gate.py:241-270` — **low** — `_session_drifted_source` treats ANY subagent file (`subagent_file_count > 0`) as source drift (`:257-258`), so a session whose only subagents did read-only research (Explore/Plan agents that never wrote) is counted as having drifted source and can be blocked spuriously. — evidence: `subagent_file_count` is just the count of `agent-*.jsonl` files (transcript.py:79-81), not a count of subagent *edits*; a research-only fan-out trips it.   - — [round1: defend] status=confirmed (low): `reconcile_gate.py:257-258` confirms `subagent_file_count > 0` short-circuits to True with no edit-vs-read distinction. - `dummyindex/context/build/reconcile.py:277-283, 264-267` — **info** — `stamp_reconciled` advances the anchor using only the report computed from the very anchor it rewrites, with no second source of truth (acknowledged in the docstring at `:264-267`).
