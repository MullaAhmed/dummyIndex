## performance findings

- `dummyindex/pipeline/build/references.py:54-79` — **high** — O(F²) textual-reference scan: for every effective source file it loops over *every other* file's rel-path doing `text.find(tgt_rel)` (F outer × F inner `str.find`), and the inner `_find_reference` falls through to a second `text.find(tgt_name)` on a basename miss. On a ~35k-LOC repo this is the dominant non-AST pass of a full build and grows quadratically with file count. Evidence — `build_structure` calls `_derive_textual_references(effective_files, …)` unconditionally at structure.py:120 on *every* `build_all` and every enriched `refresh_deterministic_artifacts`, even with `include_extras=False`. Suggested fix — build one combined regex/Aho-Corasick automaton of all rel-paths+long basenames and scan each file's text once (O(F·text) instead of O(F²·text)), or at minimum skip files whose text length is below the shortest target path.

- `dummyindex/pipeline/build/references.py:54-79` + `dummyindex/pipeline/extract/generic.py:42` — **medium** — the same source files are read from disk and decoded up to three times per build: once for SHA256 in `incremental._hash_files` (incremental.py:330-344), once inside `extract`→`load_cached`/`_extract_generic` (`path.read_bytes()` at generic.py:42, plus `file_hash` re-reads in cache.py:38), and a third full `path.read_text` in `_read_reference`/`_derive_textual_references` (references.py:99-111). Evidence — three independent read+decode passes over the whole code set on one `rebuild --changed`. Suggested fix — thread the bytes already read during extraction (or a shared content cache keyed by path) into the textual-reference pass; or hash from the bytes extraction already loaded instead of re-reading in `_hash_files`.

- `dummyindex/context/domains/query.py:386-466` — **medium** — `dummyindex context query` re-reads and re-parses `map/symbols.json` from disk once per top-K match: `_index_symbols_by_feature` already parses the whole symbols file once (query.py:339-350), then `_build_match`→`_symbol_paths` parses the *entire* `map/symbols.json` again for each of the up-to-K matches (query.py:444-451), iterating all symbols each time. The in-code comment at query.py:428-432 acknowledges this and waves it off, but symbols.json is the largest map artefact. Evidence — K full JSON loads + K full symbol scans per query on the hot retrieval path. Suggested fix — parse symbols.json once in `query()` and pass the parsed symbol→(path,range) map down to `_build_match`; the `by_member` dict from `_index_symbols_by_feature` already holds most of it.

- `dummyindex/context/domains/query.py:296-305` — **low** — `_score_feature`'s symbol-hit loop is O(features × symbols-per-feature × query-tokens) substring scans (`tok in sym_l`). Fine for the modest feature/symbol counts here, but scales with both feature count and query length. Evidence — nested loop over every member symbol of every feature for every query token, run once per `query` call (a cold per-invocation CLI command). Suggested fix — none (cold path, current magnitudes small); revisit only if feature/symbol counts grow an order of magnitude.

- `dummyindex/context/build/tree.py:139-146` — **low** — `_build_dir_or_project` recomputes `subdirs` by filtering the *entire* `dir_paths` frozenset and `files_here` by filtering the *entire* `file_id_by_path` dict at every directory node, so tree assembly is ~O(dirs × (dirs + files)). For dummyindex's own tree (tens of dirs) this is negligible, but it's accidental quadratic structure that grows with directory count on large repos. Evidence — full-set comprehension scans inside the recursive per-directory builder, run once per `build_all`/refresh (cold path). Suggested fix — pre-bucket files and subdirs by immediate parent (one pass over `file_id_by_path` and `dir_paths`) and index into that map per node; low priority given it's run-once and the constant is tiny.

- `dummyindex/context/build/incremental.py:330-344` — **low** — `_hash_files` reads + SHA256s every detected source and in-repo doc file on *every* `rebuild --changed`, even the common no-op case where nothing changed and the function returns `skipped=True` (incremental.py:135-136). This is the unavoidable cost of content-addressed change detection, but it's a full-repo read on the most frequently-run verb. Evidence — `current_by_path = _hash_files(current_files, root)` before the `has_changes` check. Suggested fix — short-circuit with a cheap stat/mtime+size pre-filter against the manifest and only SHA256 files whose mtime/size moved (hashing remains the tiebreaker for mtime-only touches). Low because it's I/O-bound sequential reads, not algorithmic blowup.

- `dummyindex/pipeline/extract/generic.py:617-618,365-615` — **info** — the call-graph pass re-walks the full subtree of every function body (`walk_calls` recurses all children) after the structural `walk` already traversed the whole tree once, so each file is effectively AST-walked ~twice. This is inherent to the two-phase design and fully cache-shielded (`load_cached` skips unchanged files at extract.py:163-166), so it only costs on first-index or changed files. Evidence — second traversal loop at generic.py:617. Suggested fix — none; per-file extraction cache already amortizes it to near-zero on re-runs.

- `dummyindex/context/build/git_delta.py:160-212` — **info** — reconcile/delta detection shells out to `git` (2 subprocesses: `diff` + `status`) rather than re-hashing the tree — this is the *correct* cheap choice and degrades to None gracefully. No quadratic or per-file cost. Noted to pre-empt a "subprocess in the build" concern: it's two calls per reconcile, not per-file. Suggested fix — none.

## Round 1 — performance rebuttals
- **P1** → confirmed (downgraded HIGH→MEDIUM) — O(F²) is real: outer loop over `effective_files` (references.py:54) × inner loop over `rel_to_id` which holds one entry per file (references.py:62), str.find per pair. BUT it does NOT run on the frequent `--changed` no-op path: incremental.py:135-136 returns `skipped=True` before any `build_structure`/`refresh` call. It runs only on actual changes or full rebuild/ingest (runner.py:118, enriched_refresh.py:106), both rare run-once paths. F can reach thousands on a large target repo, but a once-per-rebuild quadratic on a CLI cold path is MEDIUM, not HIGH. Fix (combined regex / Aho-Corasick single pass) still worthwhile.
- **P2** → confirmed (MEDIUM stands, cold-path) — same bytes read by three distinct sites on the change path: `file_hash` (cache.py:38) for SHA256, `generic.py:42` `read_bytes()` for AST, `references.py:109` `read_text()` for the scan. Warm extract cache (extract/__init__.py:163) skips generic.py:42 but still hashes for the cache key, so 2-3 reads of each changed file. Real redundancy, but only on the cold build/change path.
- **P3** → confirmed (downgraded MEDIUM→LOW) — `_symbol_paths` (query.py:424-466) re-reads + re-parses the whole `map/symbols.json` (query.py:448) once per match, called per match in `_build_match` (query.py:386). Default `top_k=3` (query.py:152), so symbols.json is parsed 3× per query. The docstring (query.py:429-431) is NOT stale — it's a deliberate "wins are small at K=3" tradeoff. At K=3 on a run-once CLI query path this is LOW.
- **P4** → confirmed (LOW stands) — score loop O(features×symbols×tokens) (query.py:297-305) is a cold per-invocation CLI path with small token counts; honestly LOW.
- **P5** → confirmed (LOW stands) — per-dir-node filtering of the full `dir_paths` frozenset + `file_id_by_path` dict (tree.py:139-146), recursive → ~O(dirs×(dirs+files)). Build-time-only, run-once, low-thousands scale. LOW.
- **P6** → confirmed (LOW stands) — `_hash_files` (incremental.py:330-344) SHA256s the whole repo before the no-op early return (incremental.py:116 before 135). Unavoidable for content-addressed change detection; I/O-bound, OS-cache-warm, not algorithmic. LOW. mtime/size pre-filter is the right fix.
- **P7** → confirmed (INFO stands) — AST double-walk at generic.py:617-618 over already-collected function bodies; cache-shielded (extract/__init__.py:163). INFO.
- **P8** → confirmed (INFO stands) — git_delta.py:160-212 shells out to git twice per reconcile, the correct cheap choice. INFO.
