- `dummyindex/context/domains/equip/errors.py:33` — **medium** (dead code) — `WireError(EquipError)` is defined and re-exported in `equip/__init__.py:42,161` but is never raised or caught anywhere in `dummyindex/` or `tests/` — verified by `grep -rn "raise WireError\|WireError(" dummyindex/ tests/` returning only the class definition and the re-export lines. — evidence: a typed exception that no producer raises is unreachable; it signals an error path that was planned and removed (or never wired), so a maintainer reading `errors.py` will wrongly assume wiring failures surface as `WireError` and write `except WireError` handlers that never fire. — suggested fix: delete `WireError` and its `__init__.py` re-export, or wire it into the `wiring/` failure path it was meant to cover (`equip/wiring/*` currently raises no error at all).

- `dummyindex/context/domains/doc_reorg/safety.py:57-78` — **low** (dead wrapper / duplicated logic) — `require_clean_tree(root, *, allow_dirty=False)` is exported and described in the module docstring as THE refuse-on-dirty gate, but the only caller in production is none: the CLI `guard` action re-implements the check inline with `git_is_clean(root)` (`cli/doc_reorg.py:51-60`); `require_clean_tree` is referenced solely by `tests/context/domains/test_doc_reorg.py:60-78`. — evidence: the advertised safety contract and the actually-wired safety check are two separate code paths over the same `git_is_clean` primitive — they can drift (e.g. the `allow_dirty` / "unknown → refuse" semantics live only in the unused wrapper), and a maintainer will edit the wrong one. — suggested fix: have `cli/doc_reorg.py` GUARD call `require_clean_tree` (collapsing to one gate), or drop the wrapper and keep the inline check as the single source of truth.

- `dummyindex/pipeline/io/git.py:87-105` — **low** (test-only public function) — `superproject_root` is fully implemented and has 6 tests (`tests/pipeline/io/test_git.py:233-276`) but zero production callers — `grep -rn superproject_root dummyindex/` (excluding its own def) is empty. — evidence: a public, tested helper with no internal consumer reads as live API; the next maintainer can't tell whether it's a planned extension point or abandoned, and it carries maintenance + test weight for no shipped behaviour. — suggested fix: wire it into the submodule-detection path it was built for, or remove it (and its tests) if the submodule case was dropped.

- `dummyindex/context/domains/equip/generate/adopt.py:147` (`adopt_existing`), `dummyindex/context/domains/equip/generate/adopt.py:216` (`registry_capabilities`), `dummyindex/context/domains/equip/plugins/vendor.py:23` (`vendored_item`) — **low** (test-only public API) — each is re-exported in `equip/__init__.py` and exercised only from `tests/context/domains/equip/*`; no module under `dummyindex/` calls them (`adopt_existing` is even self-described as a "back-compat thin wrapper" at `adopt.py:23`). — evidence: back-compat/public surface with no internal consumer accumulates as a maintenance liability — it must keep passing tests and stay signature-stable for callers that may not exist. — suggested fix: confirm whether the equip skill layer calls these dynamically (if so, leave a one-line "consumed by skills/equip/*" note next to each); otherwise demote `adopt_existing` to the non-wrapper form it wraps and drop the dead wrapper.

- `dummyindex/context/domains/audit/log.py:166-172` and `dummyindex/context/domains/council.py:332-340` — **low** (duplicated logic that will drift) — both define `latest_status(...)` as the identical "iterate `read_log(...)`, keep the last entry matching a (key, agent) pair, return its `.status` or None" loop, differing only in the key fields (round/persona vs stage/agent). — evidence: two copies of a last-write-wins log scan; a fix to one (e.g. handling an empty log, or a tie-break change) silently won't reach the other, and the council copy is load-bearing for resumption (`skills/council/19-resume.md:28-76`). — suggested fix: none required if intentional (different log shapes), but consider a shared `last_matching(entries, predicate, attr)` helper in a common module so the scan semantics stay identical.
