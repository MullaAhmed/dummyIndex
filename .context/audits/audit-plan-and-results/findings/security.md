## security findings

> Subject: the unimplemented plan at `.context/proposals/outstanding-audit-fixes/` vs. the REAL post-split source under `dummyindex/`. Verified against `c574d41` (split `reality_check.py` → package) + `d7d2e74` (recouncil). Severity = impact ON PLAN QUALITY.
>
> **Attacker model (calibrated, stated honestly):** this is a *local, single-user developer tool that indexes a repo on disk*. There is no network, no server, no multi-tenant boundary, no subprocess/shell/eval, and deserialization is uniformly `json.loads` (never pickle/eval) — confirmed `cli-dispatch/concerns.md:21`, `extraction-pipeline/concerns.md:14`. The realistic adversary is **untrusted repo content**: a repo you clone and index ships (a) tracked `.context/` artefacts — `meta.json` IS committed (`git ls-files .context/meta.json` hits; not in `.context/.gitignore`), feature docs are LLM/council-authored — and (b) source files whose comments can prompt-inject the council that writes those docs. The blast radius is **graph/index corruption + an existence/line-count read-oracle over extension-bearing host files + a constrained out-of-`features/` write**, NOT code execution. With that model the trust-boundary themes are real but several are **lower severity than "critical"**, and two of the plan's own proof strings are mis-specified. Findings below are ordered by impact on the plan.

### BLOCK — the plan's load-bearing fix is built on a false model of the code

- `dummyindex/context/build/common.py:130-141` + `dummyindex/pipeline/io/cache.py:51-55` — **HIGH→BLOCK — the cache "trust-channel split" (Task 3a) is architecturally inverted; the fix as specified BREAKS the channel it claims to preserve, and leaves a residual hole it claims to confine.** The plan (plan.md:15, spec.md:30) asserts: "confine only the `DUMMYINDEX_CACHE_DIR` env-var read; the trusted `cache_root` threaded through `extract()` is not confined, so `cache_dir_override` still works with an out-of-repo target by passing its target through the trusted `cache_root` channel." **This is false in both directions.** (1) `cache_dir_override` does NOT use the `cache_root` channel — it sets the **env var** `DUMMYINDEX_CACHE_DIR` directly (`common.py:134`, `os.environ[key] = str(target.resolve())`). (2) Inside `cache_dir()` the **env var takes precedence over** the threaded `root`: `if override: d = Path(override).resolve()` returns *before* `root` is ever consulted (`cache.py:51-55`). So `cache_root` reaches `cache_dir(root)` only as the `root` arg, which is **discarded whenever the env var is set** — and the env var is set at *every* production extraction call site that wraps in `cache_dir_override` (`runner.py:111`, `enriched_refresh.py:102`). Consequences the plan didn't see: **(a)** confining the env-var read rejects exactly the `cache_dir_override(<out-of-repo>)` case the plan promises to keep working — the documented "legitimate out-of-repo temp dir" use is killed, not preserved; **(b)** the "still-works trusted `cache_root`" bypass the plan relies on does not actually carry the cache-dir decision at those call sites (env var wins), so the plan's claimed escape hatch is dead code there. **Mitigation:** none — the plan's stated mechanism is non-existent in the code. **Fix:** name the real trusted channel correctly: either (i) confine the env-var value *and* re-route `cache_dir_override` to set a separate already-trusted in-repo target (note its only two real callers point it at `<out_root>/.context/cache/`, which IS in-repo — `runner.py:109`, `enriched_refresh.py:100` — so confinement actually wouldn't break the real callers, contradicting the plan's own justification for leaving it open), or (ii) confine inside `cache_dir()` itself and drop the false "trusted `cache_root`" framing. Re-derive the acceptance bullet (spec.md:48), whose premise ("`cache_dir_override` with an out-of-repo target still caches there") is the inverted model.

### MEDIUM — placement / residual-hole nuances

- `dummyindex/pipeline/io/cache.py:60-77` + `dummyindex/pipeline/extract/__init__.py:163-178` — **MEDIUM — cache-poisoning is real (self-poisoning only); the accept-superset schema (Task 3b) blocks the *crude* payload but NOT a well-formed poisoned graph.** `load_cached` returns the parsed `{hash}.json` verbatim, `extract` does `per_file.append(cached)` → `all_nodes.extend(result.get("nodes"))` with zero validation. Content-addressing means an attacker can only poison the result *of a file whose SHA256 they control* (drop the file + a `<that-hash>.json`), not forge another file's entry — so this is **self-poisoning**, correctly scoped by the concerns file (`extraction-pipeline/concerns.md:14`). The plan's schema (dict; list `nodes`+`edges`; every node has a string `id`; ignore unknown keys — plan.md:15) **does** reject a non-dict / non-list / id-less payload and **does** avoid mass-invalidation (a real `save_cached` entry still hits) — both acceptance bullets (spec.md:49) are constructible and correct. But calibrate honestly: a payload that *is* a dict with list `nodes`/`edges` and string `id`s — i.e. an attacker-shaped-but-well-formed graph — passes the schema and still injects arbitrary `label`/`source_file`/edge `relation` into `symbol-graph.json`. The schema is an **anti-corruption** guard (blocks crash-shaped junk), not an **anti-poisoning** guard. **Fix:** the plan should state this limit explicitly (it implies more than it delivers); the genuine mitigation is the cache-dir confinement (BLOCK above) preventing a *shared* `DUMMYINDEX_CACHE_DIR`, since same-repo self-poisoning only corrupts your own index from your own files — low real impact for a local tool.

- `dummyindex/pipeline/extract/__init__.py:268-293` (`collect_files`, `follow_symlinks=True`) + `:163,169` (read sites) — **MEDIUM — symlink-escape read is real but gated on caller opt-in; Task 4a read-time containment is the right placement.** The walk's only guard (`:279-283`) rejects self-referential loops (`parent_real == real` / prefix), not symlinks whose *target* is outside `root` — so `evil -> /etc` is walked and leaf files handed to extractors/`_read_text_safely`. Confirmed `follow_symlinks` defaults to `False` (rglob path, `:268-276`, does not chase symlinked dirs), so this needs a caller passing `follow_symlinks=True`. The plan's **read-time** `resolve_under_root` check (just before `read_bytes`, plan.md:17) is correctly TOCTOU-safer than a walk-time check (a post-enumeration symlink swap can't redirect the read past a check that happens at the read). One placement note: the plan says "at read time (just before `read_bytes`/`_read_text_safely`)" — but `read_bytes` happens *inside* the per-file extractors (`generic.py:42` etc.) and inside `cache.file_hash` (`cache.py:38`), not in `extract/__init__.py` directly; the containment check must wrap the path **before `load_cached`/`extractor(path)` at `:163,167`**, or be pushed into `collect_files` leaf emission, else the cache-hash read at `:163` touches the out-of-tree file before any check. Verify the implementer puts the gate at the right seam.

- `dummyindex/context/domains/reality_check/confidence.py:56-60,91-93,108-111` — **MEDIUM — the cross-file mirror non-atomicity is real; Task 10's fix is correctly scoped to best-effort (no overclaim).** Confirmed: `feature.json` is `_atomic_write`-replaced first (`:56`), then `_mirror_confidence_to_index` does a *separate* read-modify-write of `INDEX.json` (`:111`); a raise between leaves the two stores disagreeing, and on promote `del payload[DEMOTED_FROM_KEY]` (`:90`) commits before the INDEX mirror so a failure after `:90` loses the stash irrecoverably. Not a security finding per se (trust-signal integrity, local tool), recorded as a **confirmation the fix is correctly bounded**. Caveat for the implementer: `write_text_atomic` (`atomic_io.py:11`) and the retiring `render.py:_atomic_write` (`render.py:62-66`) are byte-for-byte equivalent (both `tmp = path.with_suffix(suffix + ".tmp")` → `tmp.replace`), so the Wave-1 retire-and-repoint (Task 2) is safe and the back-to-back-replace in Task 10 can reuse it unchanged.
