

--- critic-database (stage 3) ---

## Data integrity

Read-only retrieval over `.context/` JSON+markdown; no writes, so no write-integrity/atomicity exposure (all reads via `read_text` + `json.loads`, every parse guarded). Concerns below are read-path correctness: staleness handling, scoring/citation determinism, tokenization, and unbounded params.

- **Citation join is silently lossy under a stale index — no flag (`query.py:451-465`, `query.py:386-392`).** `_symbol_paths` keys citations on member ids present in `feature.json` that also resolve in `map/symbols.json`. If the index is stale (member id removed/renamed by a rebuild that didn't refresh `feature.json`, or vice-versa), the join just drops that symbol — the match still renders with fewer/zero ranged citations and nothing marks the result as degraded. Plan claims "tolerates a stale index (the code wins)" (`plan.md:40`) and "citations are non-negotiable" (`plan.md:53`), but a stale member→symbol mismatch silently weakens exactly the navigation guarantee. Consider surfacing a `stale`/`unresolved_members` signal rather than dropping silently.

- **Symbol-name collision is last-writer-wins → citation non-determinism across index reorderings (`query.py:451-466`).** `_symbol_paths` builds `out[name] = (path, range)` keyed by `symbol.name`, but `symbols.json` keys are `symbol_id` (unique), not `name` (not unique — e.g. `run`, `__init__`, `to_dict` recur across files; the live index has 2453 symbols where dupes are expected). When two owned members share a `name`, the citation path/range emitted depends on iteration order of the `symbols` array. A rebuild that re-emits symbols in a different order (sorting is asserted for maps in `data-access.md:13`, but query does not re-sort on read) would flip which `path:range` a colliding symbol cites — a determinism hole the retrieval-eval gate (`test_retrieval_eval.py`) won't catch because `SAMPLE_REPO` is small. Same collision exists in `_index_symbols_by_feature` via `by_member[sid]` — there it is keyed by `sid` (safe), so the two parallel joins disagree on dedup key.

- **Substring token matching over-matches short/common tokens, inflating scores (`query.py:269,272,275,290,299`).** Matching is `tok in fid_l` / `tok in name_l` / `tok in basename` / `tok in sym_l` (substring, not whole-word). A 2-char token like `id` (passes the `len(p) < 2` floor at `query.py:59`) hits `validate`, `build`, `grid`, etc.; `app` substring-hits `mapper`/`happen`. Scores are deterministic but semantically noisy, and `feature_id` substring (`query.py:269`, weight 1) compounds it. This is the documented design (`plan.md:22`, docstring `query.py:19-21`) so flagging as a known precision tradeoff, not a defect — but the "mean tokens-to-answer" eval metric is the only guard, and it only watches budget, not precision.

- **`top_k` and `budget_tokens` are unbounded/unvalidated at the boundary (`cli/query.py:32-59`, `query.py:225-237`).** CLI parses `--top-k`/`--budget` as bare `int(...)` with no lower bound. `--top-k 0` or negative → `scored[:top_k]` yields empty/odd slices (negative slices from the end), and `--budget 0`/negative → `remaining = max(0, …) < 80` trips `truncated=True` with zero matches immediately. No "must be positive" validation, unlike the typed error for non-int (`cli/query.py:35-37`). Per common/coding-style input-validation rule (fail fast at boundaries), these should reject `<= 0`.

- **Budget thresholds are inline magic numbers, weakening the auditable budget contract (`query.py:227`, `query.py:409`, `query.py:488`, `query.py:525`).** The min-block cutoff `80` (`query.py:227`, duplicated at `query.py:488`) and shrink floor `40` (`query.py:409`, `query.py:525`) are literals while sibling tunables are named constants (`query.py:151-160`). The `80` at `query.py:227` (token budget) and `80` at `query.py:488` (char floor) are numerically equal but semantically different units — easy to mis-edit one and silently desync the budget invariant. Already noted as an open question (`plan.md:60`); promoting to named constants would make the truncation contract self-documenting. No correctness bug today.

- **`estimated_tokens` is a `len//4` heuristic, so the budget cap is approximate, not guaranteed (`query.py:163-165`, `query.py:406-414`).** `estimate_tokens` can under- or over-count real token cost; the `used >= budget_tokens` stop (`query.py:235`) and per-block shrink (`query.py:407-409`) therefore bound an *estimate*, not actual emitted tokens. Acceptable for a navigation tool and the eval "mean tokens-to-answer" metric is the drift early-warning (`plan.md:62`), but consumers must not treat `total_estimated_tokens` as an exact ceiling.

- **Citation range trusts a malformed-tolerant parse but emits no diagnostic on bad ranges (`query.py:457-464`).** `range` is accepted only as a 2-element list coercible to ints, else `range_tuple=None` — defensively correct against a stale/corrupt `symbols.json`. The match then renders as a bare-path citation with no line number and no signal that a range was expected-but-dropped. Fine for read-only navigation; noting that range loss is silent, consistent with the staleness concern above.


--- critic-security (stage 3) ---

## Security

Adversarial lens: user-supplied query string (ReDoS), reading the index, info exposure in cited excerpts. Local single-user READ-ONLY CLI over `.context/` — no network, no auth, no multi-tenancy, no writes. Realistic threat model is a malformed/hostile `.context/` index (e.g. cloned from an untrusted repo) and a pathological query string. Verdict: **no exploitable vulnerability; 3 low-severity hardening notes.**

- **ReDoS — CLEAN. Query tokenization is linear-time, no catastrophic backtracking (`query.py:44`, `query.py:77-79`, `query.py:505`).** All three regexes over attacker-influenced text are backtracking-safe: `_TOKEN_RE = [A-Za-z][A-Za-z0-9_]*` (`query.py:44`) is a single bounded char-class run; `_split_camel_and_snake`'s `[A-Z][a-z]+\d*|[a-z]+\d*|[A-Z]+\d*(?![a-z])|\d+` (`query.py:77-79`) is an alternation of disjoint simple runs with no nested/overlapping quantifiers; `_PARA_BREAK = \n\s*\n` (`query.py:505`) is trivial. None has the `(a+)+` / overlapping-alternation shape that drives exponential backtracking. Recording the negative result so a future regex edit (backreference / nested quantifier) is re-reviewed against this baseline.

- **Index-data drives filesystem paths — `feature_id`/`fid` from `INDEX.json` is concatenated into read paths without traversal containment (`query.py:357`, `query.py:434`, `query.py:381`).** `_index_symbols_by_feature` builds `context_dir / "features" / fid / "feature.json"` (`query.py:357`) and `_symbol_paths` builds `context_dir / "features" / feature_id / "feature.json"` (`query.py:434`) directly from the `feature_id` in `features/INDEX.json`, with no check that the result stays under `context_dir`. A hostile/corrupt index carrying `feature_id: "../../../../etc/passwd"` (or an absolute path — `Path("a") / "/etc/passwd"` → `/etc/passwd`) would walk outside `.context/`. Impact is bounded to read-only `.exists()` + `read_text` of a file that must still `json.loads` (non-JSON → `JSONDecodeError`, caught), and the source is the user's own local `.context/` — low severity, not classic untrusted-input traversal, but not first-party once a repo/index is shared or cloned. Recommend a containment guard: assert `resolved.is_relative_to(context_dir)` before any read and reject `feature_id` containing `/`, `\`, or `..`. `query()` resolves `context_dir` (`query.py:180`) but never re-validates per-feature children.

- **Cited excerpts surface raw markdown verbatim, no redaction layer (`query.py:494`, `query.py:500`, `query.py:517-518`).** `_excerpt_from_feature` reads feature markdowns with `errors="ignore"` (`query.py:494`) and returns the first paragraph containing a query token (`query.py:517-518`), truncated to budget (`query.py:500`). A crafted query can pull any secret an index author embedded into stdout/JSON. Info-exposure-by-design, not a leak the feature introduces — the data already lives in plaintext under `.context/` the same user can `cat`; no trust boundary is crossed. Severity: informational. Mitigation is upstream (no-secrets-in-`.context/` rule); query does not log, cache, or transmit excerpts off-process.

- **No info-exposure via error paths — CLEAN (`cli/query.py:84-89`, `query.py:439`, `query.py:449`, `query.py:362`).** Missing index → generic "run ingest first" scoped to `features_index` (`cli/query.py:84-89`); every index `json.loads` is wrapped in `except (OSError, json.JSONDecodeError)` that degrades to an empty result, no stack trace / internal IDs / secrets. Consistent with the read-only navigation contract.


--- critic-product (stage 3) ---

## Product surface

User lens: a developer running `dummyindex context query "..."` to find which feature to read and where in source to look. Verdict: results are useful when they hit, but the surface is opaque about *why* a result ranked, silent when the index is stale, and offers no recovery on a near-miss query.

- **No-match returns tokens but zero recovery guidance — scenario: developer queries `"authentication"` but the index calls it `auth` — observed: `render_markdown` prints `_No feature matched the query tokens ['authentication'] across N feature(s)._` (`query.py:543-548`) and CLI exits 1 — gap: no "did you mean" / nearest-feature / "try fewer or broader terms" hint, and no list of available feature ids the user could have matched. The user is told *what* failed but given no next step; the tool knows all N feature names but surfaces none of them.**

- **Zero-token reduction is a silent dead-end — scenario: developer queries `"the and is"` (all stopwords) or `"a"` (below the `len < 2` floor, `query.py:59`) — observed: `render_markdown` prints `_(query reduced to zero tokens after stopword filtering — nothing to look up.)_` (`query.py:537-542`), exit 1 — gap: message explains the mechanism but doesn't echo what *was* stripped or hint that the query was too short/common; a user who typed a real-looking query gets an explanation that reads like the tool's fault, not theirs, with no example of a query shape that works.**

- **Score is an unbounded raw integer with no scale — scenario: developer sees `## Auth · auth (score 13)` and the next match `(score 4)` (`query.py:557`) — observed: score is the raw weighted sum (name×5 + summary×3 + …, `query.py:267-305`) printed verbatim — gap: no max, no normalization, no confidence band, so "13" is meaningless in isolation (is that a strong or weak hit?) and incomparable across queries of different token counts. `matched_tokens` is shown (`query.py:562-566`) which helps, but the *weighting* — why name-hits dominate — is invisible, so a user can't tell why a feature whose summary merely contains a substring outranks one whose name is an exact match.**

- **Substring matches are not visually distinguished from whole-identifier matches — scenario: developer queries `"id"`, gets features matched because `id` is a substring of `validate`/`build`/`grid` (substring matching, `query.py:269,272,275,290`) — observed: those land in `matched_tokens` identically to a real hit (`query.py:562-566`) — gap: the renderer can't tell the user a "match" was a noisy substring hit vs. a meaningful whole-word hit, so spurious top results look as authoritative as real ones. The data-integrity section already flags the scoring noise; the *product* gap is that the output gives the user no way to discount it.**

- **Stale-index results render identically to fresh ones — no freshness signal — scenario: developer edited source hours ago; the SessionStart hook reported drift but they query anyway — observed: `query()` reads `.context/` as-is, tolerates staleness silently (`plan.md:40`), and the markdown/JSON carry no `as_of`, index mtime, or staleness flag — gap: a user navigating off a stale result has no in-band warning that the cited `path:range` may have moved; the citation drop under a stale member→symbol join (data-integrity §, `query.py:451-465`) compounds this — fewer citations appear with no "results may be stale, run rebuild" banner. The product never tells the user the map might be out of date at the moment they're trusting it.**

- **`--top-k` / `--budget` defaults are undiscoverable and silently clamp output — scenario: developer expects all matches, gets exactly 3 — observed: `_DEFAULT_TOP_K=3` and `_DEFAULT_BUDGET_TOKENS=2000` (`query.py:151-160`) apply with no echo of the active cap in the output — gap: there is no `--help` for the subcommand (only a one-line usage on empty input, `cli/query.py:64-70`), so the defaults are invisible; a 4th relevant feature is dropped with no `…and M more below top-k` hint, and a budget-truncated run only appends `(truncated by budget)` (`query.py:553`) without telling the user *which* knob to raise.**

- **Truncation hides matches with no count of what was cut — scenario: budget exhausts after 2 of 3 top-k matches — observed: `truncated=True` and the summary line shows `(truncated by budget)` (`query.py:235-237,553`) — gap: the user isn't told how many matches were suppressed or that raising `--budget` would reveal them; a dropped match (header+citations would have survived per Decision, `plan.md:53`, but the block is skipped entirely when `remaining < 80`, `query.py:227-229`) vanishes with only a generic flag, so the user can't tell a complete result from a clipped one without re-running.**

- **JSON output is contract-stable but the human default buries the actionable field — scenario: a script consumes `--json`, a human reads the default markdown — observed: JSON carries `schema_version`, `truncated`, `feature_count_considered`, full citations (`to_dict`, `query.py` model) — gap: parity is fine for machines, but the markdown default leads with prose and puts citations last (`query.py:571-578`); for a navigation tool the `path:range` *is* the answer, yet a user scanning output hits summary/excerpt first and the citations (the thing they act on) are at the bottom of each block. Minor, but the highest-value output is the least prominent.**

## Cross-review notes

- **Amplifying critic-database's "citation join is silently lossy under a stale index" (10-critiques §Data integrity, `query.py:451-465`):** the data-integrity lens flags the *mechanism* (dropped symbols); the product impact is that the user has **no signal at all** that navigation was degraded — fewer citations render with the same confident formatting as a complete result. The fix they propose (surface `stale`/`unresolved_members`) is exactly the freshness-signal product gap above; both point to one missing field on `QueryResult` (a staleness/degraded flag) that would close the data-integrity *and* the product hole simultaneously. Strongly endorse promoting it from "consider" to a contract addition.

- **Amplifying critic-database's unbounded-`top_k`/`budget` finding (`cli/query.py:32-59`):** they correctly call for `<= 0` rejection on correctness grounds (`--budget 0` → instant empty `truncated`). The product reinforcement: the *reason* a user reaches for these flags at all is undiscoverable (no `--help`, defaults never echoed), so a user experimenting with `--top-k 0` to "see everything" gets a silent empty result with no error — the validation gap and the discoverability gap are the same wound. Reject `<= 0` *and* echo the active caps.
