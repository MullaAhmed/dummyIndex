## Data integrity

- **Concurrent appends to one feature's log can lose entries — read-modify-write is not inter-process atomic.** `append_log` reads the whole JSON, mutates `entries` in memory, then `tmp.replace`s (`council.py:124-159`). Only the final rename is atomic; the read→append→write window is not guarded. Deep mode dispatches 3 critic units per feature (`CRITIC_ROSTER`, `council_batch.py:53-58`; `_units_for_feature` critique branch, `council_batch.py:236-245`) as **parallel Task agents**, each appending `started` then `complete` to the same `features/<id>/council/_council-log.json`. Interleaved read-modify-writes drop entries (last-writer-wins on the in-memory list). A dropped `complete` leaves a phantom `started`, so `is_stage_complete` (`council.py:186-203`) returns False forever and the stage never converges.
- **Fixed `.tmp` path makes the atomic-write itself racy under concurrency.** Every append writes to the single shared name `log_path.suffix + ".tmp"` (`council.py:156-158`), not a per-process unique temp. Two concurrent appenders to the same feature both write the same `_council-log.json.tmp` and both `replace` it — the second clobbers the first's payload before/after rename, compounding the lost-update above. The data-access convention mandates tmp+replace (`data-access.md:5-9`) but assumes a single writer; the council's parallel-dispatch model (`plan.md:30`, `plan.md:45`) violates that assumption for same-feature critic fan-out.
- **Frontier recomputation is correctly resumption-idempotent and monotone for the partial/failed-log cases — no concern there.** `earliest_incomplete_stage` (`council_batch.py:91-112`) + `is_stage_complete` (`council.py:186-203`) are pure log-derived functions: a lone `started`/`failed` re-surfaces the stage and `_feature_ready_for` (`council_batch.py:149`) re-dispatches only unfinished agents. Re-running `--next` against a larger log converges (`plan.md:51-52`). This is sound **as long as** appends are not concurrent on the same feature — i.e. the bug is the write path, not the read/frontier path.
- **`force_recouncil` mid-run idempotency holds, but inherits the same concurrent-append exposure.** `force_recouncil` only resets features whose `earliest_incomplete_stage` is None (`council_batch.py:114-124`), so firing it mid-run is a no-op for in-flight features — monotone and idempotent as designed (`plan.md:53`). However, `append_reset_marker` (`council.py:209-227`) routes through the same non-locked `append_log`; a reset marker appended while a critic's `complete` is being written can be lost or clobber it, corrupting the post-marker window that `is_stage_complete` counts (`council.py:194-202`).
- **Corrupt/partial log silently degrades to "empty" instead of failing — masks integrity loss.** Both `read_log` and `append_log` swallow `json.JSONDecodeError` and treat the payload as `{}` (`council.py:163-170`, `council.py:146-150`). A truncated write (e.g. from the `.tmp` race above, or a crash mid-`write_text`) is read as no-entries: the frontier reschedules the feature from stage 1 (`earliest_incomplete_stage`, `council_batch.py:106-111`) and clobbers completed work — the exact failure `backfill_log_from_artifacts` was built to prevent (`council.py:251-292`), now reachable via corruption rather than pre-v0.20 absence. Contrast the schema-version guard other artifacts enforce (`data-access.md:11-13`); the council log validates `status`/`stage`/`agent` on write (`council.py:138-145`) but never validates `schema_version` on read, so a future-version or malformed log is indistinguishable from empty.
- **Stage monotonicity is sound; backfill never regresses a stage with any entry.** `backfill_log_from_artifacts` skips any stage that already has an entry, even pre-reset (`council.py:264-271`, docstring `council.py:251-262`), so it cannot overwrite real progress. Stages 1–4 only (`_ARTIFACT_STAGE_DOCS`, `council.py:73-77` + `_FLOW_STAGE`); stage 5 TREE is dev-authored but not backfillable (open question, `plan.md:61`) — not a data-integrity defect, but means a tree-enrich run resumed against a v0.20-backfilled log will (correctly) re-dispatch stage 5, never silently skip it.

## Security

- **Feature-id path traversal in `write_section` — agent-supplied `--feature` is joined into a filesystem path with no slug validation, only an `is_dir()` gate.** `run_section_write` reads `--feature` raw (`cli/features.py:260`) and passes it to `write_section`, which does `feat_dir = features_dir / feature_id` then guards solely on `feat_dir.is_dir()` (`context/domains/features/ops.py:472-477`) before writing the section file via tmp+rename (`ops.py:489-490`). The `section` arg IS hardened (rejects `/` and leading `.`, `ops.py:479-481`; CLI `_validate_section_name`, `cli/features.py:223-239`), but `feature_id` is never run through `_validate_feature_id` (`features/helpers.py:18-30`) — the strict `[a-z0-9_-]` slug validator that `merge_feature` (`ops.py:206`), `rename_feature` (`ops.py:48`), and placement (`placement.py:295,397`) all DO apply. Attack path: a critic Task agent (or any caller of `dummyindex context section-write`) passes `--feature ../../../../etc/skel/x` (or any existing dir reachable from `.context/features/`); `is_dir()` succeeds for a real directory and `concerns.md` is written outside the feature tree. Concretely the council fans out untrusted-prompt-shaped agents that call this exact verb, so the input is attacker-influenceable. Severity HIGH (arbitrary in-tree-relative file write/overwrite of `<dir>/<section>.md`). Fix: call `_validate_feature_id(feature_id)` at the top of `write_section` and at the `section-write` CLI boundary, mirroring `merge_feature`; reject any id that isn't a bare slug.
- **Same traversal class in `append_log` (the `council-log` verb) — `--feature` reaches a path join with no slug check.** `run_council_log` (`cli/council.py:29`) forwards `--feature` straight to `append_log`, which builds `feat_dir = features_dir / feature_id` and only checks `feat_dir.is_dir()` (`context/domains/council.py:118-121`) before `mkdir(parents=True)` of `<feat_dir>/council/` and writing `_council-log.json` (`council.py:132-159`). `agent` IS guarded against `/` (`council.py:129-130`) but `feature_id` is not. Contrast `council-log backfill`, which DOES validate membership (`feature_id not in indexed` against `features/INDEX.json`, `cli/council.py:116-124`) — the plain `council-log` verb skips that check entirely. Attack path: `dummyindex context council-log --feature ../../../tmp/evil --stage 1 --agent x --status started` creates `../../../tmp/evil/council/_council-log.json` if `../../../tmp/evil` exists, planting a JSON file (and a `council/` dir) outside `.context/features/`. Severity MEDIUM (constrained to a fixed `council/_council-log.json` leaf, attacker-controlled directory). Fix: validate `feature_id` as a slug (reuse `_validate_feature_id`) and/or assert `feat_dir.resolve()` is `relative_to(features_dir)` before any write.
- **`remove_flow` is an unbounded `unlink()` keyed on unvalidated `flow_id` + `feature_id` — a deletion primitive, not just a write.** `remove_flow` joins both ids raw: `feat_dir = features_dir / feature_id` (guarded only by `is_dir()`, `ops.py:372-376`) then `flow_json = feat_dir / "flows" / f"{flow_id}.json"` / `.md` and calls `p.unlink()` for any path that `.exists()` (`ops.py:381-388`). Neither id is slug-validated. `flow_id="../../<target>"` (no extension control needed if the file already ends in `.json`/`.md`) resolves the unlink outside `flows/` — e.g. `--flow ../../authentication/concerns` would delete another feature's `concerns.md`/`.json`. The CLI `run_flow_remove` passes both straight through (`cli/features.py:179,196`). Severity HIGH (out-of-tree file deletion / cross-feature artifact destruction; idempotent no-op masks it as benign). Fix: `_validate_feature_id(feature_id)` and a matching slug guard on `flow_id`, plus a `relative_to(feat_dir)` containment assert before `unlink`.
- **`from-file` source is read with no containment check — `section-write` is also an arbitrary-file READ-into-`.context` primitive.** `write_section` does `content = source_file.read_text()` on the raw `--from-file` path (`ops.py:486-489`), wrapped as `Path(from_file)` at the CLI with no bound (`cli/features.py:262,290`). A caller can point `--from-file /etc/passwd` (or any readable file) and exfiltrate its contents into a committed `concerns.md` inside `.context/`, where the next agent reads it as trusted context. Severity MEDIUM (local read-into-index; meaningful because `.context/` is treated as trusted by every downstream agent and may be committed). This is by-design for a local CLI but worth a note: the source is unbounded while the destination is access-controlled by section name only. Mitigation: acceptable for a single-user local tool, but if council agents are ever sandboxed, bound `--from-file` to the repo/workspace root.
- **Agent-supplied `note` is persisted but NOT a log-injection vector — flagged as a NON-issue after checking the write path.** `note` flows from `--note` straight into `LogEntry.note` and is serialized via `json.dumps(payload, indent=2)` (`council.py:158`), so newlines/control chars are JSON-escaped — no line-oriented log forging, no breakout of the JSON structure. `status`/`stage`/`agent` are all validated on write (`council.py:123-130`). The only soft concern is that `note` is unbounded in length and is later read back and matched with `.startswith(STANDALONE_NOTE_PREFIX)` (`council.py:245`) — an agent that sets `--note standalone...` on a stage-0 `complete` entry could self-exempt a feature from rescheduling (`is_standalone_complete`, `council.py:230-248`). That's a trust-the-agent integrity quirk, not a memory-safety/injection bug; severity LOW/INFORMATIONAL. No fix required beyond noting that `note` semantics are load-bearing and agents are trusted to use the `standalone` prefix honestly.
- **Corrupt-log-swallowing has a security framing too: silent `{}` on `JSONDecodeError` means a planted/truncated `_council-log.json` is indistinguishable from a fresh feature.** `read_log` and `append_log` both treat any `JSONDecodeError` as empty (`council.py:139-140`, `council.py:168-171`) and never check `schema_version` on read. Combined with the traversal write above, an attacker who can drop a file at a feature's log path can either erase audit history (truncate → reads as empty → frontier reschedules from stage 1, clobbering completed council work) or, via a crafted valid-JSON log, forge `complete`/`standalone` entries to make `is_stage_complete`/`is_standalone_complete` short-circuit a feature as done. Severity MEDIUM (audit-integrity / state-machine forgery; overlaps the Data-integrity section's corruption finding but with an adversarial rather than crash-induced trigger). Fix: validate `schema_version` on read and fail loud (or quarantine) on decode error rather than silently substituting `{}`.

## Product surface

- **`--force` on a mid-run feature is a silent no-op — the operator gets zero feedback that the reset was refused.** `force_recouncil` only appends a reset marker to features whose `earliest_incomplete_stage` is None (`council_batch.py:131-136`); an in-flight feature is skipped and its id is absent from the returned tuple. The CLI prints the "forced re-council for: …" line ONLY when `forced` is non-empty (`cli/council_batch.py:175-176`), so `dummyindex context council-batch --next --feature X --force` against a half-completed X prints the ordinary next-batch and nothing about X. Observed: operator believes they forced a fresh run; nothing happened — gap is no "feature X is mid-run, not reset (finish or let it converge first)" notice, neither in text nor in the JSON `forced: []` payload (`cli/council_batch.py:170`).
- **Frontier is a hard global barrier — one slow/stuck feature freezes every other feature's progress, with no output explaining why.** `earliest_incomplete_stage` returns the single lowest stage incomplete for ANY pipeline feature (`council_batch.py:104-111`) and `next_batch` dispatches only that stage (`council_batch.py:268-272`). A feature still at SPECIFY pins the whole batch to stage 1; features whose plan/critique are ready cannot advance until it clears. The text output says `stage N — K parallel units` (`cli/council_batch.py:181-187`) but never "M other features are blocked waiting on this stage", so an operator watching a 20-feature run stall on one feature has no surfaced reason — gap is no per-feature frontier/blocked-on visibility.
- **`failed` is invisible in the dispatch surface — a retry is indistinguishable from a first attempt.** A `failed` entry makes `is_stage_complete` return False (`council.py:201-203`), so `_feature_ready_for` re-surfaces the stage (`council_batch.py:158-163`) and the feature is re-emitted as a plain `DispatchUnit` with no failure flag (`DispatchUnit` fields, `council_batch.py:166-183`). Neither the text nor JSON output marks a unit as a retry or carries the prior `failed` note. Observed: an operator re-running `--next` after a crash sees the same `feature: role → subagent` line whether it's attempt 1 or attempt 5 — gap is no retry/attempt-count or "previously failed: <note>" signal, so a persistently-failing agent loops without escalation.
- **Cap silently overshoots on the first oversized feature, and the overshoot is never reported.** The cap guard is `if collected and len(collected)+len(feature_units) > cap` (`council_batch.py:286`) — when the FIRST ready feature alone exceeds cap (e.g. `--cap 2` in deep mode where one feature expands to 3 critic units, `CRITIC_ROSTER[DEEP]`, `council_batch.py:56-61`), `collected` is empty, the guard is skipped, and all 3 units ship. Intended (never split a feature, `plan.md:55`,`plan.md:62`), but the output prints `K parallel units` (`cli/council_batch.py:180-184`) with no "exceeds your --cap of N" note, so an operator who set a cap to bound concurrency silently gets more agents than requested — gap is no overshoot disclosure.
- **Fairness across features is INDEX.json order with no operator lever and no disclosure.** When ready features at a stage exceed cap, `next_batch` iterates `pipeline_ids` in INDEX.json order and `break`s at the first overflow (`council_batch.py:276-288`), so the same low-INDEX features win every batch and a high-INDEX feature can starve across many `--next` calls. The output gives no hint that features were deferred for cap, nor any way to prioritise — gap is the deferred set is invisible (operator can't tell "3 more features are ready but capped out" from "only these 3 are ready").
- **A corrupt/truncated `_council-log.json` mid-council degrades to a silent stage-1 reschedule with NO warning when the feature has no enriched docs yet.** `read_log` swallows `JSONDecodeError` → `[]` (`council.py:168-171`), so the frontier treats the feature as fresh and re-dispatches from SPECIFY. The only guard, `_warn_if_backfill_needed`, fires only when `needs_artifact_backfill` is true — i.e. empty log AND enriched artifacts on disk (`cli/council_batch.py:41-51`; `council.py:295-306`). A feature corrupted while still at stage 1-2 (no enriched spec/plan yet) trips no warning and silently restarts; even the artifact case only warns about backfill, never "this log failed to parse". Observed: operator sees a feature inexplicably back at stage 1 — gap is the malformed-log path is indistinguishable from a fresh feature in the operator-facing output.
- **The `--next` text output has no "what's left" summary — completion is the only terminal signal.** Output is either `all features complete for this mode.` (`cli/council_batch.py:177-178`) or the current stage's units; there is no count of remaining stages/features, no progress fraction. An operator driving the loop by hand can't tell from one `--next` call whether they're near done or just starting — gap is no progress/remaining surface across the multi-call loop.

## Cross-review notes

- The Security section's two path-traversal findings (`write_section` feature-id, `council.py:118-121`; `append_log` feature-id, `cli/council.py:29`) ALSO have a product-surface tail: the `council-log` verb's own success line echoes the raw `feature_id` back verbatim (`cli/council.py:69-72`) and `--json` re-emits unvalidated ids, so a traversal id (`../../tmp/evil`) is reflected to the operator as if it were a legitimate feature. The Security fix (`_validate_feature_id` at the CLI boundary) is also the UX fix — rejecting the bad id loudly at parse time is strictly better than writing out of tree and printing a misleading success row. Concur with Security severity; flagging that the missing validation degrades operator trust in the success output, not just file-system containment.
