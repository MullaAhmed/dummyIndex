## Data integrity

- `enums.py` + `__init__.py` — VERIFIED clean: 44 enum members, `_HANDLERS` total over all 44 (no missing/extra), no duplicate enum values; `test_every_enum_member_has_a_handler` keeps this true. The plan's "44-member" claim and table-totality claim are correct — no action.
- `common.py:64-75` — `_FLAGS_TAKING_VALUE` covers only the *legacy* flag alphabet; newer value-flags (`--model`, `--describe`, `--scope`, `--scope-path`, `--top-k`, `--budget`, `--reason`, `--proposal`, `--slug`, `--title`, `--round`, `--persona`, `--repo`, `--usage-doc`, `--for-proposal`, `--specialist`, `--item`, `--doc`) are absent. This is by design (those handlers own their parsing, e.g. `propose.py:83-118`), but the plan's "single source of truth for value-flags" framing overstates it — the global set is authoritative only for `_wants_help` and the shared `parse_path_and_root`/`parse_kv_flags`, not for self-parsing handlers. Suggest the plan note this scope explicitly so a future reader doesn't add a new shared-parser handler assuming the set is complete.
- `__init__.py:75-77` — consequence of the above: for a subcommand whose value-flag is NOT in the global set, a value literally `-h`/`--help` (e.g. `propose --slug -h`) trips `_wants_help` and prints usage instead of the handler's "needs a value" error. This is read-only and is the documented "help wins" bias direction — benign, not a corruption. No fix; just confirming it stays harmless because no handler mutates before the help guard.
- `common.py:183-204` — `parse_kv_flags` is last-wins on repeated keys, which would silently drop earlier values of a repeatable flag; the one repeatable consumer (`council-batch`'s `--feature`) correctly extracts via `pull_repeatable_flag` BEFORE `parse_kv_flags` (`council_batch.py:80,86`), so no data loss today. Worth a one-line guard-rail note in the plan: any future repeatable flag MUST be pulled before `parse_kv_flags`, never read from its dict.
- State surface is genuinely thin — the dispatcher reads no persistent state and writes none; `resolve_context_root` (`common.py:13-45`) is pure path arithmetic over its args (no env/global reads beyond an injectable `cwd`). Nothing else to flag on persistence.

## Security

- `cli/common.py:30-45` (`resolve_context_root`) — an absolute `--root`/scope arg is `.resolve()`d and returned **unconfined**: nothing checks it stays inside cwd, so `dummyindex context hooks install --root /` or `--root ~` makes every downstream filesystem op (settings.json write, `.context/` read/write) target an arbitrary tree. This is by-design (the docstring says "the user typed a full path on purpose") and the caller is a *local* CLI the user already controls, so the realistic attack is footgun/path-confusion (a relative scope that escapes cwd via `../../..` falls to the `except ValueError` branch and is honoured as-is), not privilege escalation — load-bearing mitigation: **none**, accepted by trust model (local single-user CLI, no remote arg source).

- `cli/hooks.py:51,56-57` → `context/hooks.py:340,360,382` — the `hooks install` verb writes to whatever `settings.json` the resolved root points at (`--global` → `~/.claude/settings.json`); a wrong/escaped `--root` could mutate an unintended repo's `.claude/settings.json`. Concrete blast radius is bounded: the written command bodies are **static module constants** (`_SESSION_START_HOOK`…`_PRE_COMPACT_HOOK`, `context/hooks.py:89-158`) with **no interpolation of any user/arg/stdin value**, the only runtime expansion is `"$CLAUDE_PROJECT_DIR"` (Claude-Code-provided, quoted), and the merge is sentinel-gated (`install_hook_entry(..., sentinel=SENTINEL)`, `:382-384`) so user-authored hooks lacking the sentinel are preserved, not clobbered — load-bearing mitigation: no-interpolation static bodies + sentinel-gated merge confine the write to dummyindex's own entries; **no command-injection path exists** from CLI args into the hook command string.

- `cli/memory.py:22-38` (`read_hook_stdin`) — untrusted hook JSON on stdin is parsed; malformed/oversized input is the only vector and it is fully defended: non-dict, JSON errors, OSError, and TTY all collapse to `{}` (`:26-38`), and every hook-fed handler returns 0 regardless (`memory nudge/breadcrumb/session-start` `:94,101,107`) so a crafted payload cannot fail the user's turn — load-bearing mitigation: total swallow-to-`{}` + invariant-0 exit. (Note: no size cap on `sys.stdin.read()` at `:29` — a giant payload is an OOM/DoS footgun, but the producer is Claude Code itself, not an attacker, so not load-bearing.)

- `cli/memory.py:48-51` (`resolve_transcript`) — `hook["transcript_path"]` (attacker-influenced if stdin is spoofed) is taken as a `Path` and flows into `find_main_transcript`/downstream read. It is type-guarded (`isinstance(tp, str) and tp`, `:49-50`) but **not confined** — a spoofed `transcript_path` could point the breadcrumb/nudge reader at an arbitrary file (e.g. `/etc/passwd`) for *reading*, not writing. Impact is read-only and the output goes into the user's own session context, so disclosure is to the user themselves on a local CLI — load-bearing mitigation: **none**; relies on stdin being Claude-Code-supplied (trust model), flag if this layer ever ingests remote hook payloads.

- `cli/query.py:32-68`, `cli/__init__.py:75-77` — int-flag parsing (`--top-k`/`--budget`) and the help-wins interceptor are correctly fail-closed (non-int → exit 2; `--help` short-circuits before any handler side effect, killing the documented "bare-equip-mutates" hazard). No injection or path escape here. Nothing to flag.

- Overall: this is a thin arg→exit-code translation sink with **no subprocess execution, no shell, no eval, no secrets handling, and no network**. The only genuine residual is unconfined path/root resolution (first two bullets), which is a deliberate trust-model decision for a local single-user CLI, not a defect. Little to flag.

## Product surface

- All 44 enum members documented — every `ContextSubcommand` value has a `  <verb>` line in `USAGE`; `gc`, `migrate-docs`, and `guard-doc-write` are present and dispatchable alongside the older `hooks`/`wire` verbs. No undocumented-but-dispatchable verb. The hunted gap does not exist.
- `query "text" --top-k` (trailing, no value) — `error: --top-k requires an integer value`, exit 2 (`query.py:60-68`). Reachable and consistent with the `--top-k must be an integer` arms; but the message omits the `--help` hint that `usage_error` (`common.py:47-61`) adds elsewhere — query hand-rolls its errors and never calls `usage_error`, so its four exit-2 paths (`query.py:36,43,67,78`) point nowhere, unlike most other verbs.
- `query` with a typo'd flag, e.g. `query "text" --topk 5` — `--topk` and `5` are unknown, fall to `leftover` (`query.py:69-71`) and are silently joined into the query string `"text --topk 5"`. No error; the user gets a (probably empty) match set and a misleading exit 1. Surprise: a misspelled flag becomes search text.
- `query` no-match — exit 1 (`query.py:102`), distinct from exit 2 (usage) and exit 0 (hit). ✅ RESOLVED (proposal `outstanding-audit-fixes`, "cli-dispatch UX"): the `query` USAGE block now documents both — "Exits 1 (no error output) when nothing matches, so shell scripts can detect 'no hit'; exits 2 on a usage error" (`help.py:206-208`). A caller scripting `&&` on query now sees the no-match exit in the help text.
- `--top-k`/`--budget` absent from `_FLAGS_TAKING_VALUE` (`common.py:64-75`) — so for `query --top-k --help` the help-bias still fires (`__init__.py:73` matches `--help` directly before reaching `--top-k`), help wins. No gap, but the safety property here is incidental (the flag isn't value-skipped), not the documented "help wins over a value-flag's value" path (`__init__.py:75-77`); query's value flags simply aren't in that set.
- `config show` is the only `config` verb wired (`__init__.py:113` → single `config.run`); `get`/`set` are advertised-as-reserved in USAGE (`help.py:253-255`) and flagged as planned-not-built in `plan.md:57`. A user reading "(get/set reserved)" who runs `config get x` hits the generic unknown-subcommand path? No — `config` is the enum token, so `config get x` dispatches to `config.run` with `["get","x"]` as args; behavior depends on `config.run`'s leftover handling, not the dispatcher. Advertised-but-unbuilt sub-verb, handler-owned.
- `ingest` referenced in query's FileNotFound message — `error: ... run \`dummyindex ingest\` first` (`query.py:95`) is correct: `ingest` is a real top-level command aliasing `context init` (`__main__.py:277-280`). Not a stale reference. Code confirms; no conflict.
- Help interceptor reaches every verb uniformly — `_wants_help` runs before `_HANDLERS[sub]`, so `<verb> --help` prints `usage_for(sub)` for all 44 with no per-handler opt-out. Consistent; the only declared cost is the literal-`--help`-as-value non-use-case (`plan.md:48`).
- Exit-code inversion for hook-fed verbs — `memory` and `reconcile-gate` always return 0 (`plan.md:52`) regardless of internal failure, by design (a failing Stop/SessionStart hook would break the session). A user invoking these manually gets no failure signal on stderr-only errors; intentional, documented, but a surprise to anyone scripting them outside a hook.
- **(reconcile 2026-06-23) Depth-error clarity fix verified** — `init` and `reconcile` now validate `--depth` against `CouncilMode` up front (`init.py:43-48`, `reconcile.py:57-62`), so the surviving `except ConfigError` (`init.py:51-56`, `reconcile.py:65-68`) can only be a malformed `config.json`, printed verbatim instead of the old misleading `--depth must be light|standard|deep, got None`. Lands as documented in `spec.md` / `plan.md`; no gap.
- **(reconcile 2026-06-23) `init` pays the heavy import before the cheap `--depth` validation can short-circuit it** — `build_all` is imported at `run()` entry (`init.py:15`) *before* the depth guard at `init.py:43`, so `dummyindex context init --depth bogus` still eats the `build.runner` import cost before exiting 2. Minor avoidable startup cost on the error path; ordering, not correctness.
