## Data integrity

- `dummyindex/cli/init.py:90-154` + `dummyindex/context/domains/config.py:458-494` — without an explicit `--depth`, `resolve_depth` fully deserializes `config.json` before the `--no-default-plugins` and Codex-only gates take effect, so a malformed plugin-only field can block the deterministic index build even though the caller explicitly disabled or cannot use Claude plugins; this contradicts the plan's pre-config opt-out boundary — read only depth fields at this stage or defer full plugin-config validation until `_wire_default_plugins_step`.
- `dummyindex/cli/init.py:51-84` + `dummyindex/cli/init.py:225-231` — default reconciliation can commit `config.wired`/`default_plugins_enabled` before settings declaration or materialisation fails, but `_wire_default_plugins_step` returns no aggregate and `init.run` unconditionally exits 0, leaving automation unable to distinguish a fully applied state from config-only intent — return a structured partial-state result and surface a nonzero or persisted pending marker until the settings/machine state is healed.
- `dummyindex/cli/init.py:51-78` + `dummyindex/cli/wire.py:88-123,221-244` + `dummyindex/context/default_plugins.py:448-524,645-699` — headless init passes every custom `config.wired` plugin and interactive wire passes each approved custom plugin through the reviewed-default declaration seam, which writes `enabledPlugins=true` without a marketplace source while materialisation selects zero targets; the next `wire` run then treats the enabled key as satisfied and never offers the interactive repair path — route custom targets through source-resolving `equip install` or persist source/declaration/materialisation as separate states instead of equating an enabled key with completion.
- `dummyindex/cli/wire.py:109-123` + `dummyindex/cli/wire.py:156-173` + `dummyindex/cli/wire.py:221-244` — when declaration writes `enabledPlugins=true` and materialisation then errors, the command reports needs-user but the next run classifies the entry as satisfied solely from the settings key and never retries the failed install — roll back the enablement on install failure or track declaration and materialisation as separate persisted states during classification.
- `dummyindex/cli/wire.py:239-244` + `dummyindex/context/default_plugins.py:677-711` — `_wire_plugin` treats `deferred` and `skipped` install results as success because it checks only `errors`, collapsing “installed,” “CLI unavailable,” and “not eligible” into one `wired` state — require `installed` for synchronous success and render deferred/skipped explicitly without removing the entry from needs-user tracking.
- `dummyindex/cli/wire.py:96-166` — `wire` snapshots the entire ledger before potentially blocking on multiple prompts and never re-reads it, so a concurrent config edit can remove or change an entry while the stale approved copy is still enabled — after each affirmative prompt, re-read under a per-config lock and verify that the exact entry is still declared before mutating settings.
- `dummyindex/cli/common.py:13-46` + `dummyindex/cli/init.py:109-119` + `dummyindex/cli/wire.py:46-63` + `dummyindex/context/default_plugins.py:448-524` — the selected project root is not enforced as the settings-store boundary: a symlinked `.claude` directory redirects declaration writes and later effective-state reads into a sibling/user tree while `config.wired` remains under the selected root, splitting desired and effective state across two repositories that both appear locally satisfied — resolve and validate the settings destination beneath the selected root immediately before every read/write, rejecting symlinked parents or using no-follow directory handles.

## Security

1. **High — repository-controlled symlinks can redirect Claude settings mutation outside the selected root.** `init` and `wire` trust the root returned by `resolve_context_root`, then construct `.claude/settings.json` without binding it back to that root (`dummyindex/cli/common.py:13-46`, `dummyindex/cli/init.py:109-119`, `dummyindex/cli/wire.py:46-63`, `dummyindex/context/default_plugins.py:448-524`). Attack scenario: a malicious clone supplies `.claude` as a directory symlink to a sibling/user tree, or supplies `.claude/settings.json.tmp` as a symlink to any user-writable file; declaration follows the redirected directory, and `write_settings` follows the predictable temporary symlink during `write_text` before its rename (`dummyindex/context/claude_settings.py:175-179`). The selected repo's `config.wired` remains local while effective-state reads follow the external settings, so both the overwrite and the split-store state appear satisfied on later runs. **Mitigation today: none at the dispatch/integration boundary.** Config validation and atomic rename constrain content/commit shape only after the unsafe destination is selected. Resolve and enforce the settings destination beneath the selected root, reject symlinked parents, and create temporary files with exclusive no-follow directory-handle operations.

2. **High — headless `context init` bypasses and then suppresses the CLI's untrusted-plugin confirmation model.** Repository config accepts any non-empty `WiredEntry.target`, and init passes the entire post-build `cfg.wired` tuple into the headless declaration service (`dummyindex/context/default_plugins.py:62-108`, `dummyindex/cli/init.py:51-78`). Attack scenario: a committed `attacker-plugin@known-marketplace` entry is written as `enabledPlugins=true` during ordinary init without the warning/prompt reserved for `context wire`; the reviewed-default materialiser filters the custom target out, but the next `wire` run sees the enabled key as `SATISFIED` and never offers confirmation or repair (`dummyindex/context/default_plugins.py:430-445,487-524,645-699`, `dummyindex/cli/wire.py:88-123,193-218`). **Load-bearing mitigation today:** dummyindex's immediate subprocess pass does not fetch the custom target because it intersects with `DEFAULT_PLUGINS`; the persistent Claude declaration can still materialise later, and the only approval surface has been disabled by classification. Restrict unattended declaration to reviewed targets and require verified marketplace provenance/materialisation—not mere key presence—before classifying a custom entry as satisfied.

3. **Medium — plugin-capable CLI handlers execute whichever `claude` binary wins inherited `PATH`.** Both init's default pass and wire's approved path reach `install_default_plugins` (`dummyindex/cli/init.py:70-78`, `dummyindex/cli/wire.py:221-244`). Its availability check executes bare `claude --version`, then marketplace/install commands reuse the unresolved name; `default_runner` changes into the selected repository and inherits the caller's environment (`dummyindex/context/default_plugins.py:565-581`, `dummyindex/context/default_plugins.py:602-642`). A malicious repository combined with a launcher, direnv rule, or user PATH containing the repository first gains code execution at the availability probe. **Load-bearing mitigation today:** subprocess uses list argv without a shell, has a timeout, allowlists immediate plugin targets, pins non-official marketplace refs to full SHAs, and init discloses reviewed code-running surfaces before the probe (`dummyindex/context/default_plugins.py:149-223`, `dummyindex/cli/init.py:46-49`). None authenticates the executable. Resolve an approved absolute Claude CLI path before entering repository context and execute with a minimal explicit environment/PATH.

## Product surface

- Recovery run with `context init --no-default-plugins` or `--platform codex` against a config whose plugin fields are malformed — `init` removes the opt-out flag but, without an explicit `--depth`, calls `resolve_depth`, which fully deserializes `config.json` and exits 2 before the build or the later Claude/plugin gates (`dummyindex/cli/init.py:90-154`, `dummyindex/context/domains/config.py:458-494`) — the documented one-run opt-out cannot isolate indexing from irrelevant plugin corruption; read only the depth fields here or defer full plugin validation until the plugin step.
- Successful `context init --no-default-plugins` — `_wire_default_plugins_step` returns before printing anything, and Codex-only init bypasses the same step entirely (`dummyindex/cli/init.py:17-27,222-231`) — the caller cannot distinguish an honored explicit opt-out from a plugin phase that was unreachable or omitted; emit one stable `plugins -> skipped for this run` line.
- `context init` or approved `context wire` in a cloned repository whose `.claude` directory or `settings.json.tmp` is a symlink — declaration follows the external destination, while config remains under the selected root and the command reports plugin outcomes without revealing which settings file was actually mutated (`dummyindex/cli/init.py:51-78,109-119`, `dummyindex/cli/wire.py:46-63,156-173`, `dummyindex/context/default_plugins.py:448-524`, `dummyindex/context/claude_settings.py:175-179`) — the caller receives no destination preview, containment error, or split-store warning before a sibling/user file is changed and later treated as this project's effective state.
- Ordinary headless `context init` in a repository whose `config.wired` contains an arbitrary valid custom plugin target — init writes `enabledPlugins=true` without a prompt, the reviewed-default installer silently filters that target out, and a later `context wire` classifies the key as satisfied so it never presents the promised approval/repair surface (`dummyindex/cli/init.py:51-78`, `dummyindex/cli/wire.py:88-123`, `dummyindex/context/default_plugins.py:430-445,448-524,645-699`) — current help and guide say the headless reconciler only classifies/reports needs-user and that `wire` is the escalation surface (`dummyindex/cli/help.py:462-474`, `docs/guide/07-cli.md:586-590`), so code wins and the public trust boundary is both stale and unrecoverable through the advertised command; restrict unattended mutation to reviewed defaults.
- `context init` with a durable per-target `enabledPlugins=false` tombstone — reconciliation correctly preserves the opt-out but puts the target in `already`, which renders as `<target> already enabled (left as-is)` (`dummyindex/context/default_plugins.py:293-313,478-524`) — the success transcript states the opposite of the effective state; render `already disabled (left as-is)` separately.
- `context init` when config reconciliation succeeds but settings declaration or Claude materialization fails — warnings are printed, `_wire_default_plugins_step` returns no aggregate, and `init.run` still exits 0 (`dummyindex/cli/init.py:51-84,225-231`) — automation cannot distinguish a fully ready repo from config-only intent or a partially applied machine state without scraping prose; expose a structured final summary and a documented partial-success signal.
- `context wire --yes` for a custom `plugin@marketplace` target — declaration writes a dangling enabled-plugin key without a marketplace source, the reviewed-default installer selects zero targets, `_wire_plugin` sees no error, and the command prints `wired: <target>`; on the next run that key is classified as satisfied and omitted from repair (`dummyindex/cli/wire.py:88-123,156-173,221-244`, `dummyindex/context/default_plugins.py:430-445,448-524,670-699`) — the affirmative success message promises resolution for a plugin that cannot be loaded and makes the false completion sticky; reject unresolved custom sources or route them through source-aware installation.
- `context wire` for a reviewed default when the Claude CLI is unavailable — materialization returns `deferred`, but `_wire_plugin` tests only `errors` and prints the same `wired` success used for a completed install (`dummyindex/cli/wire.py:156-173,221-244`, `dummyindex/context/default_plugins.py:701-729`) — interactive output hides that installation is pending until a future session; render installed, deferred, and skipped as distinct outcomes.
- `context wire --yes` when declaration or materialization actually errors — the command prints `could not wire` and leaves needs-user state, yet the handler always returns 0 (`dummyindex/cli/wire.py:156-173,221-244`) — shell callers receive the same process success as a fully wired run; reserve exit 0 for complete resolution or provide a machine-readable result mode with explicit partial/failure status.
- Interactive `context wire` for one of the pinned reviewed defaults advertised by init — every syntactically valid absent plugin is labeled and prompted as `untrusted plugin`, with no distinction for the allowlisted, full-SHA-reviewed defaults (`dummyindex/cli/wire.py:193-218`) — the CLI presents two contradictory trust models for the same target; reuse the registry trust metadata so reviewed defaults and genuinely custom plugins receive different disclosure.
- Interactive `context wire` pauses at a plugin prompt while another process removes or changes that entry in `config.wired` — the handler keeps the startup snapshot and, after affirmative input, enables the stale target without rereading or checking that it is still declared (`dummyindex/cli/wire.py:96-166`) — the confirmation appears current but is not bound to a config revision, and the final summary gives no indication that the persisted intent changed during the prompt; reread and compare the exact entry before mutation.
