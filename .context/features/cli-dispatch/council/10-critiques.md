## critic-database (data-integrity lens)

## Data integrity

- **No persistence owned here — clean.** cli-dispatch is the wire-only layer between `argv` and the domains; it holds no frozen dataclasses, no JSON/markdown artifacts, no cache. All real models and writers live in the `context/domains/<x>/` modules each handler lazy-imports (`.context/features/cli-dispatch/plan.md:53-62`). There is therefore no atomic-write, schema-version, or drift surface to violate in this cluster — those disciplines belong to `build/*` + `domains/*` (`.context/conventions/data-access.md:23-28`). Nothing to flag.
- **I/O is correctly confined to the boundary — the convention's intended shape.** `print` lives only in `cli/*`; domains return/raise and the dispatcher maps typed exceptions to exit codes `0/1/2` (`.context/features/cli-dispatch/plan.md:92-96`). This matches data-access's "keep I/O at the boundary — `cli/*` prints/formats and calls writers; don't scatter `json.dump` into business logic" rule (`.context/conventions/data-access.md:25`) — verified: no handler serializes or persists directly. Compliant, not a concern.
- **argv path parsing does not corrupt downstream artifacts (no abs-path leakage from here).** `parse_path_and_root` extracts a raw positional scope + optional `--root` and `resolve_context_root` decides the repo root (`dummyindex/cli/common.py:103-148,13-45`; both symbols present in `map/symbols.json`). The scope it passes downstream is the user's *raw* `Path`, deliberately un-`.resolve()`-d (`common.py:26-28`). Any absolute path that reaches a committed JSON is normalized to repo-relative POSIX by the writer side via `_rel_posix`, which drops out-of-tree files (`.context/conventions/data-access.md:15-17`) — so a malformed/absolute scope from argv cannot leak an absolute path into a persisted artifact; the boundary that matters is on the writer, not here. No integrity concern in dispatch.
- **Value-flag parsing is purely structural — cannot silently mutate persisted data.** `_FLAGS_TAKING_VALUE` and `pull_repeatable_flag` only re-order/forward tokens to the subcommand parser (`dummyindex/cli/common.py:64-100`); they perform no type coercion, null-defaulting, or record construction, so there is no place for implicit null propagation or schema drift to originate in this layer. The known limitation (one global value-flag set can't tell council-log's `--status` value from build's boolean) is a help/UX bias question, not a data-integrity one (`.context/features/cli-dispatch/plan.md:115-119`). Noted, out of lens.
- **Exit-code / stream discipline is intact.** Mandatory-flag failures route through `usage_error` → stderr + exit 2 (`dummyindex/cli/common.py:47-61`, symbol present), unknown subcommands raise `ValueError` from `ContextSubcommand(...)`, and help is read-only and fires before any side effect (`.context/features/cli-dispatch/plan.md:97-102`). Errors go to stderr, results to stdout, codes are `0/1/2` — no stream crossover that would let a failed command's partial output be mistaken for valid data. Compliant.

**Verdict: 0 data-integrity concerns.** This is the expected outcome for a persistence-free routing layer; the layer actively upholds the data-access convention (I/O at the boundary, no direct serialization) rather than violating it.

## critic-security (adversarial lens)

## Security

Adversarial lens: argv passing unvalidated paths/values downstream, privileged I/O in cli modules, abs-path leakage in errors. Trust model is a **local, single-user CLI** — the invoking operator already owns the filesystem, so "attacker == argv" is mostly self-harm, not privilege escalation. Findings are scoped accordingly; none rise above LOW for the stated threat model, but two are worth a guard against accidental footguns and any future non-local caller (e.g. a SessionStart hook or agent forwarding untrusted feature ids).

- **LOW — `feature_id` flows from argv into path construction with no segment guard (mild traversal footgun).** `run_section_write` reads `--feature` straight from `parse_kv_flags` and hands it to `_validate_section_name`, which builds `(features_dir / feature_id / f"{name}.md")` (`dummyindex/cli/features.py:260,226`), and to the domain writer `write_section`, which joins `features_dir.resolve() / feature_id` (`dummyindex/context/domains/features/ops.py:472-477`). The `section` name *is* guarded against traversal (`"/" in section or section.startswith(".")` → reject, `ops.py:480`), but `feature_id` is **not** — a value like `../../some-dir` is only constrained by the subsequent `feat_dir.is_dir()` existence check (`ops.py:474`). That check bounds the blast radius (target must be an existing directory) and the read source is the operator's own `--from-file`, so on a local CLI this is self-inflicted at most. Still asymmetric with the `section` guard and worth a `"/" not in feature_id and not feature_id.startswith(".")` check at the same boundary, both here and in the sibling `scaffold_feature` / `assign-files` handlers that take `--id`/`--feature` the same way (`features.py:312,362,408`). `write_section`, `parse_kv_flags`, `_validate_section_name` (via `write_section`) all present in `map/symbols.json`. **Remediation:** reject `feature_id` containing `/`, `\`, or a leading `.` at the CLI boundary (mirror the `section` guard), failing with `usage_error` → exit 2.

- **LOW / informational — error messages echo absolute filesystem paths to stderr.** Multiple handlers interpolate resolved absolute paths into operator-facing errors: `run_section_write` prints `f"error: {features_dir} not found…"` (`dummyindex/cli/features.py:272-275`) and surfaces `FeatureRenameError` text that itself embeds `feat_dir` / `source_file` absolute paths (`ops.py:476,481,488`), and `resolve_doc_paths` warns `f"--docs path not found, skipping: {p}"` with a resolved abs path (`dummyindex/cli/common.py:174-177`). For a local CLI this is **desirable** diagnostics, not leakage — the user owns those paths. Flagging only because the prompt named it: there is **no untrusted remote consumer** of this stderr, so no real disclosure boundary is crossed. No change recommended unless this output is ever forwarded into a shared/agent log. Not a concern under the stated trust model.

- **No privileged I/O in the dispatch layer — clean.** No `cli/*` module shells out, sets perms, touches network, reads secrets/env for auth, or escalates. `statusline.run` is the only always-on path and is read-only + fail-closed: a single broad `except Exception: return 0` swallows everything to empty stdout, and it never recomputes or writes (`dummyindex/cli/statusline.py:37-66`). The dispatcher itself only constructs a closed `ContextSubcommand` enum and indexes `_HANDLERS` — an unknown token is rejected with exit 2 before any handler runs, and help fires before side effects (`dummyindex/cli/__init__.py:127-145`). No command-injection, deserialization, or eval surface in this cluster.

- **Value/flag parsing is injection-inert.** `_FLAGS_TAKING_VALUE`, `pull_repeatable_flag`, `parse_kv_flags`, `parse_path_and_root` only tokenize/reorder argv — no `shell=True`, no `eval`, no format-string-from-input, no SQL (`dummyindex/cli/common.py:64-203`, symbols present). Values are forwarded verbatim to typed domain functions; the only sink that matters (path construction) is the `feature_id` LOW above. Nothing else to flag.

**Verdict: 2 LOW findings (1 actionable path-segment guard on `feature_id`; 1 informational abs-path-in-stderr, no remediation under the local-CLI trust model), 0 medium/high/critical.** Expected for a wire-only router — the real validation and write discipline correctly live one layer down in `domains/features/ops.py`, which already guards the `section` axis; the `feature_id` axis is the one asymmetry worth closing.

## critic-product (product-surface lens)

## Product surface

Lens: the developer's lived experience at the `dummyindex context <sub>` boundary — discoverability, unknown-input UX, flag/error consistency, and `--json`/`--help` coverage. Trust model is a single local operator driving a 39-verb CLI, much of it agent-invoked. Evidence is the live code (it wins); plan.md citations are cross-checked against `map/symbols.json`-present symbols. Net: the wire layer is mechanically sound (help-before-side-effect, closed alphabet, exit 0/1/2) but the *surface* has real rough edges — no "did you mean" on a 39-verb alphabet, structurally inconsistent unknown-flag handling across handlers, and `--json` that silently degrades to plain-text on the error path.

- **Unknown subcommand on a 39-verb alphabet — dumps the entire USAGE block, offers no suggestion — gap.** `dispatch` rejects an unknown token with `error: unknown context subcommand '{subcmd}'` then prints the *whole* `USAGE` to stderr and exits 2 (`dummyindex/cli/__init__.py:134-137`). With 39 members (`context/enums.py:47-86`) and many near-twins (`reconcile`/`reconcile-stamp`/`reconcile-gate`, `features-rename`/`features-merge`/`features-remove`, `assign-files`/`unassign-files`, `enrich-plan`/`enrich-apply`, `audit`/`audit-log`), a fat-fingered `recouncil` or `feature-remove` gets a ~370-line wall instead of "did you mean `reconcile`?". The codebase already proves it knows the pattern — `equip` does exactly this one level down (`"did you mean `equip apply`?"`, `dummyindex/cli/equip/dispatch.py:138`) — so the absence at the top level is an inconsistency, not a missing capability. Gap: add `difflib.get_close_matches(subcmd, [m.value for m in ContextSubcommand])` to the rejection message; the full-USAGE dump is overkill for a typo.

- **Unknown-flag / leftover-arg handling is structurally inconsistent across handlers — same mistake, different UX — gap.** Three different shapes coexist: `status` rejects leftovers as `error: unknown argument(s) for `status`: {rest}` exit 2 (`dummyindex/cli/status.py:24-25`); `config show` does the same but only after a manual `parse_path_and_root` (`dummyindex/cli/config.py:28-30`); `debt` rejects per-token in a loop (`dummyindex/cli/debt.py:43-53`); but `query` *silently absorbs* every unrecognized token into the query string — `query --tpo-k 3 auth` treats `--tpo-k`, `3`, and `auth` all as search words and returns a confusing match instead of flagging the typo (`dummyindex/cli/query.py:60-62`). A user who mistypes a flag gets exit 2 + a message on `status`/`debt`/`config` but a wrong-but-exit-0/1 result on `query`. Gap: there is no shared "reject unknown flags" helper (`cli/common.py` has `usage_error` at `:47-61` but no leftover-checker), so each handler reinvents the policy and one forgets it. The plan names the shared-helper seam as the intended discipline (`plan.md:46-50`) yet this cross-cutting concern was not extracted.

- **`--json` contract breaks on the error path — `query --json` on a missing index prints plain text to stderr — gap.** `query` accepts `--json` and renders JSON on success (`dummyindex/cli/query.py:29-31,87`), but when the index is missing it prints `error: {exc} not found — run `dummyindex ingest` first.` as bare text to stderr and exits 2 (`:84-90`) — a script that set `--json` and pipes stdout to a JSON parser now has no machine-readable error and an empty stdout. Same shape in `config show` (`--json` is not even offered, but `reconcile`/`debt`/`status`/`reality-check`/`preflight` all advertise `--json` in USAGE at `help.py:156,207,216,249,378`) — none of them wrap the *failure* in a JSON envelope. For an agent-driven CLI (the stated primary consumer) a `--json` flag that emits non-JSON on error is a latent foot-gun. Gap: when `--json` is set, errors should also serialize (e.g. `{"error": "..."}` to stdout or stderr) — the success/error format contract should be symmetric.

- **`query` exit code 1 on "no matches" is undocumented surface behavior — agents can't distinguish "empty result" from "failure" — gap.** `query.run` returns `1` when `result.matches` is empty (`dummyindex/cli/query.py:91-92`, comment: "shell scripts can detect no-hit"). This is a deliberate and reasonable feature, but the USAGE entry for `query` (`help.py:201-206`) never mentions it, and exit 1 is *also* the generic runtime-error code used everywhere else (`config` read failure `config.py:39`, `reconcile-stamp` refusal, etc.). A caller seeing exit 1 cannot tell "valid query, zero hits" from "query blew up". Gap: document the no-hit exit semantics in the `query` help block; consider that overloading exit 1 for "successful empty result" collides with the layer's own "1 = runtime error" convention (`plan.md:94-95`).

- **`config get`/`config set` are advertised-as-reserved but rejected with exit 2 — the help promises a future the runtime denies — minor gap.** USAGE says `config show … (get/set reserved for a future release.)` (`help.py:246-248`) and the handler rejects any non-`show` action with `error: unknown config sub-action '{action}' (expected: show)` exit 2 (`dummyindex/cli/config.py:23-25`). A user who reads "reserved for a future release" and tries `config get model` gets "unknown sub-action" rather than "not yet implemented". The error text and the help text disagree on the *reason*. Minor, but a curated CLI should make reserved-vs-unknown legible: reject `get`/`set` with a distinct "reserved for a future release" message, not the generic unknown-action path.

- **Help discoverability has no `--help` list of subcommands distinct from the firehose, and no per-family index — gap (mild).** A bare `dummyindex context` (or `-h`) prints the full `USAGE` (`dummyindex/cli/__init__.py:128-130`) — the same ~370-line block. There is no terse "here are the 39 verbs, one line each" mode; `usage_for` slices a *single* verb family on `<sub> --help` (`help.py:427-447`, symbol present in `map/symbols.json`), which is good, but the entry point to *discover* verb names is the firehose. For `equip` (11 sub-verbs) the family slice is itself large. Gap: a one-line-per-verb summary (name + first sentence) would make the 39-verb surface scannable; the structure to derive it already exists (the two-space-indented subcommand lines `help.py:405-411`).

- **`--root` / `[path]` scope duality is powerful but unexplained at the surface — gap (informational).** Every verb takes both an optional positional `[path]` and `--root DIR`, with non-obvious resolution (`resolve_context_root` at `cli/common.py:13-45`, symbol present; "default root: cwd if scope is a subdir of cwd, else scope itself", `help.py:21-26`). This rule is stated *once* in the `init` block and never repeated, yet it governs all 39 verbs. A user invoking `status ../other-repo` vs `status --root ../other-repo` gets materially different root resolution with no per-verb hint. Informational: the scope/root model deserves a top-of-USAGE "Common flags" preamble rather than being buried in `init`'s prose.

**Verdict: 5 actionable product-surface gaps (unknown-verb suggestion, inconsistent unknown-flag handling, `--json` error asymmetry, undocumented `query` exit-1, reserved-vs-unknown `config` messaging) + 2 informational (firehose help discoverability, unexplained scope/root duality), 0 correctness defects.** The dispatch *mechanics* are clean — the gaps are all about a 39-verb surface that grew faster than its discoverability and consistency affordances. The single highest-leverage fix is a shared "reject unknown flags + suggest" helper in `cli/common.py`, which closes the inconsistency gap and the suggestion gap together and matches the plan's own shared-helper-seam intent (`plan.md:46-50`).
