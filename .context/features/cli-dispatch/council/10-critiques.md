## Security

- `cli/common.py:30-45` (`resolve_context_root`) — an absolute `--root`/scope arg is `.resolve()`d and returned **unconfined**: nothing checks it stays inside cwd, so `dummyindex context hooks install --root /` or `--root ~` makes every downstream filesystem op (settings.json write, `.context/` read/write) target an arbitrary tree. This is by-design (the docstring says "the user typed a full path on purpose") and the caller is a *local* CLI the user already controls, so the realistic attack is footgun/path-confusion (a relative scope that escapes cwd via `../../..` falls to the `except ValueError` branch and is honoured as-is), not privilege escalation — load-bearing mitigation: **none**, accepted by trust model (local single-user CLI, no remote arg source).

- `cli/hooks.py:51,56-57` → `context/hooks.py:340,360,382` — the `hooks install` verb writes to whatever `settings.json` the resolved root points at (`--global` → `~/.claude/settings.json`); a wrong/escaped `--root` could mutate an unintended repo's `.claude/settings.json`. Concrete blast radius is bounded: the written command bodies are **static module constants** (`_SESSION_START_HOOK`…`_PRE_COMPACT_HOOK`, `context/hooks.py:89-158`) with **no interpolation of any user/arg/stdin value**, the only runtime expansion is `"$CLAUDE_PROJECT_DIR"` (Claude-Code-provided, quoted), and the merge is sentinel-gated (`install_hook_entry(..., sentinel=SENTINEL)`, `:382-384`) so user-authored hooks lacking the sentinel are preserved, not clobbered — load-bearing mitigation: no-interpolation static bodies + sentinel-gated merge confine the write to dummyindex's own entries; **no command-injection path exists** from CLI args into the hook command string.

- `cli/memory.py:22-38` (`read_hook_stdin`) — untrusted hook JSON on stdin is parsed; malformed/oversized input is the only vector and it is fully defended: non-dict, JSON errors, OSError, and TTY all collapse to `{}` (`:26-38`), and every hook-fed handler returns 0 regardless (`memory nudge/breadcrumb/session-start` `:94,101,107`) so a crafted payload cannot fail the user's turn — load-bearing mitigation: total swallow-to-`{}` + invariant-0 exit. (Note: no size cap on `sys.stdin.read()` at `:29` — a giant payload is an OOM/DoS footgun, but the producer is Claude Code itself, not an attacker, so not load-bearing.)

- `cli/memory.py:48-51` (`resolve_transcript`) — `hook["transcript_path"]` (attacker-influenced if stdin is spoofed) is taken as a `Path` and flows into `find_main_transcript`/downstream read. It is type-guarded (`isinstance(tp, str) and tp`, `:49-50`) but **not confined** — a spoofed `transcript_path` could point the breadcrumb/nudge reader at an arbitrary file (e.g. `/etc/passwd`) for *reading*, not writing. Impact is read-only and the output goes into the user's own session context, so disclosure is to the user themselves on a local CLI — load-bearing mitigation: **none**; relies on stdin being Claude-Code-supplied (trust model), flag if this layer ever ingests remote hook payloads.

- `cli/query.py:32-68`, `cli/__init__.py:75-77` — int-flag parsing (`--top-k`/`--budget`) and the help-wins interceptor are correctly fail-closed (non-int → exit 2; `--help` short-circuits before any handler side effect, killing the documented "bare-equip-mutates" hazard). No injection or path escape here. Nothing to flag.

- Overall: this is a thin arg→exit-code translation sink with **no subprocess execution, no shell, no eval, no secrets handling, and no network**. The only genuine residual is unconfined path/root resolution (first two bullets), which is a deliberate trust-model decision for a local single-user CLI, not a defect. Little to flag.

# critic-security — cli-dispatch stage 3 first pass

## Security

1. **High — CLI-selected project state can redirect Claude settings mutation outside the selected root.** `resolve_context_root` deliberately accepts absolute and escaping paths, while `init` and `wire` trust the resulting root (`dummyindex/cli/common.py:13-46`, `dummyindex/cli/init.py:109-119`, `dummyindex/cli/wire.py:46-63`). Their plugin path then constructs `.claude/settings.json` without resolving it back under that root and follows normal filesystem semantics during declaration (`dummyindex/context/default_plugins.py:448-524`). A malicious clone with `.claude` as a directory symlink can therefore redirect `context init` or an approved `context wire` write into a sibling repository or user configuration; marketplace and enabled-plugin mutations follow that external target. **Mitigation today: none at the dispatch/integration boundary.** Config existence checks, malformed-JSON refusal, and marketplace conflict preservation constrain content but not destination. Require project integration targets to resolve beneath the selected root and reject symlinked parents, or use no-follow directory descriptors for settings reads and replaces.

2. **High — headless `context init` bypasses the CLI's own untrusted-plugin confirmation model.** Repository config accepts any non-empty `WiredEntry.target`, and init passes the entire post-build `cfg.wired` tuple into the headless declaration service (`dummyindex/context/default_plugins.py:62-108`, `dummyindex/cli/init.py:51-78`). That service writes `enabledPlugins=true` for every syntactically valid custom plugin, not only reviewed defaults (`dummyindex/context/default_plugins.py:487-524`). This means a committed `attacker-plugin@known-marketplace` entry is promoted to Claude execution state during ordinary init without the warning/prompt that `context wire` reserves for “untrusted plugin” targets (`dummyindex/cli/wire.py:193-218`). **Load-bearing mitigation today:** the immediate subprocess materialiser intersects selections with `DEFAULT_PLUGINS`, so it does not fetch the custom target during init (`dummyindex/context/default_plugins.py:645-697`); however Claude can later materialise the enabled declaration. Restrict headless declaration to reviewed targets and leave custom entries needs-user, or make init invoke the same explicit approval/provenance gate as wire.

3. **Medium — plugin-capable CLI handlers execute whichever `claude` binary wins inherited `PATH`.** Both init's default pass and wire's approved path reach `install_default_plugins` (`dummyindex/cli/init.py:70-78`, `dummyindex/cli/wire.py:221-244`). Its availability check executes bare `claude --version`, then marketplace/install commands reuse the unresolved name; `default_runner` changes into the selected repository and inherits the caller's environment (`dummyindex/context/default_plugins.py:565-581`, `dummyindex/context/default_plugins.py:602-642`). A malicious repository combined with a launcher, direnv rule, or user PATH containing the repository first gains code execution at the availability probe. **Load-bearing mitigation today:** subprocess uses list argv without a shell, has a timeout, allowlists immediate plugin targets, pins non-official marketplace refs to full SHAs, and init discloses reviewed code-running surfaces before the probe (`dummyindex/context/default_plugins.py:149-223`, `dummyindex/cli/init.py:46-49`). None authenticates the executable. Resolve an approved absolute Claude CLI path before entering repository context and execute with a minimal explicit environment/PATH.
