## Data integrity

- **Concurrent writers race on a fixed `.tmp` name.** `write_text_atomic` derives the tmp as `path.with_suffix(path.suffix + ".tmp")` (`context/domains/atomic_io.py:22`), i.e. a single deterministic `freshness-badge.tmp` — no PID/uuid uniqueness. Two overlapping `plan-update` runs (e.g. two SessionStart hooks firing near-simultaneously, or a manual `dummyindex context plan-update` overlapping a hook) both write the same tmp then `replace`. On POSIX the final `replace` is atomic per-call so the *published* badge is never torn, but the two `tmp.write_text` calls can interleave and one `replace` can act on bytes the other writer is mid-truncating; the surviving badge is one writer's value (last-replace-wins), which is acceptable for a best-effort projection but is an unbounded last-writer-wins, not a documented guarantee. Plan claims only "concurrent reader never sees a half-written badge" (`plan.md:31`, `cli/plan_update.py:44-46`) — true — but says nothing about writer/writer. Low-severity given the value is idempotent-ish (same `DriftReport` → same badge), but worth one line.
- **No reader/writer atomicity gap, confirmed.** Both readers target the exact final name (`badge_cache_path` → `.context/cache/freshness-badge`, `cli/plan_update.py:34-36`; wrapper `cat .context/cache/freshness-badge`, `statusline.sh:16`); the transient `freshness-badge.tmp` is a different filename, so a concurrent statusline read either opens the old badge or the new one, never the tmp. POSIX `rename` over an open read fd is safe. Clean.
- **Read-verbatim vs. byte-faithful write — no corruption, but a latent encoding asymmetry.** Writer is byte-faithful (`atomic_io.py:11-19`, no normalization); Python reader does `read_text(encoding="utf-8")` then `print(badge, end="")` (`cli/statusline.py:60-62`) and the shell wrapper does a raw `cat` (`statusline.sh:16`). The badge contains a non-ASCII glyph (`✓`, `drift.py:103`). The two readers can disagree if the host shell's locale isn't UTF-8: `cat` emits raw bytes (host renders per its own encoding) while the CLI forces utf-8 decode then re-encodes via `print` to `sys.stdout`'s encoding — on a non-UTF-8 stdout the CLI path can `UnicodeEncodeError`, which is swallowed (`cli/statusline.py:64-66`) → empty badge, while `cat` would still show garbled-but-present bytes. Not corruption (degrade-to-silent holds), but the two "identical" readers are not byte-identical under a non-UTF-8 host. Unmentioned in `plan.md:31` ("readers echo the file's exact bytes").
- **Best-effort write cannot corrupt or fail the hook — verified.** `_write_badge` is severed from the drift report (receives an already-computed `DriftReport`, only adds a side effect) and the caller wraps it in a bare `try/except: pass` (`cli/plan_update.py:71-74`) that also covers the `mkdir` (`:49`) and any `compute_badge` raise. An unwritable/read-only `.context/cache/` degrades to "no badge," never a non-zero exit or perturbed stdout (stdout is only touched after, `:76-78`). Matches the load-bearing decision (`plan.md:47`). No Windows `PermissionError` copy-then-delete fallback as in `pipeline/io/cache.py:98-105`, but the outer swallow makes that unnecessary here — consistent, not a gap.
- **Cache-staleness is by-design drift, not data corruption — but the badge has no freshness stamp.** The badge refreshes only at SessionStart `plan-update` (`plan.md:51`, `cli/plan_update.py:66-74`); a mid-session source edit leaves `[ctx ✓]` cached while the real state has drifted. The cached file carries *no* mtime/anchor/version of its own — a reader (CLI or wrapper) cannot tell a fresh badge from a stale one, and there's no TTL or "stale" sentinel. This is the intended trade-off (hot path = zero recompute), but it means the badge can confidently assert "no drift" when drift exists, with no in-band way to detect it. The cache layer convention fingerprints staleness via `manifest.json` mtime/size compare (`conventions/data-access.md:19-21`); the badge deliberately opts out of that, so consumers must treat the badge as advisory-only. Worth a one-line user note (already flagged as open question `plan.md:56`).
- **Single-source path holds on the Python side; shell wrapper is the lone divergence, cwd-fragile.** Writer and CLI reader both resolve through `badge_cache_path` (`cli/plan_update.py:34-36`, imported at `cli/statusline.py:28`) so they cannot diverge. The wrapper hard-codes the relative literal `.context/cache/freshness-badge` (`statusline.sh:16`) and assumes cwd == project root; a non-root cwd silently reads nothing → empty badge (degrade-to-silent, `plan.md:50,55`). Not a corruption risk (read-only, swallowed), but the relative literal is a second source of truth that will silently rot if `BADGE_CACHE_NAME` (`cli/plan_update.py:31`) ever changes — the wrapper won't be caught by the import coupling. A test asserting the wrapper's literal equals `badge_cache_path(...)`'s tail would close the divergence.

## Security

- **No command-injection sink: badge bytes are echoed, never evaluated.** `compute_badge` emits only `[ctx ✓]` or `[ctx: N drift]` with `N` an int (`context/drift.py:91-109`); wrappers `cat` it (`statusline.sh:16`) / `Write-Host -NoNewline $badge` (`statusline.ps1:17-18`) and the CLI `print(badge, end="")` (`cli/statusline.py:62`) — none use `eval`/`Invoke-Expression`. Tampered content can't run code. Keep it that way: a future `eval "$(cat …badge)"` would turn a writable gitignored cache into per-prompt RCE.
- **Terminal-escape / statusline spoofing via verbatim echo.** Badge is printed unsanitized; anything that can write `.context/cache/freshness-badge` (hostile dep, cloned repo, co-tenant) can plant ANSI/OSC escapes rendered every prompt (`statusline.sh:16`, `cli/statusline.py:62`). Display tampering, not RCE. Strip C0/C1 control bytes in the Python reader before `print`.
- **Attacker-plantable trust input, no integrity check.** Gitignore (`.gitignore:19`, `.context/.gitignore:3`) excludes only *your* commits — a cloned hostile repo can ship its own badge; no signature/owner/freshness check, reader trusts the bytes (`cli/statusline.py:60`). `git clone evil && cd evil` → first prompt renders the planted badge (can fake `[ctx ✓]`). Advisory only, never a security signal.
- **cwd-relative read widens surface.** Wrappers read the relative literal against process cwd (`statusline.sh:16`, `statusline.ps1:17`), no root confinement/canonicalization; `cd` into an attacker tree renders its planted badge. Python cold path gates on resolved root (`cli/statusline.py:47-53`) but isn't the per-prompt path.
- **Emit-only summary → session prompt, not shell — bounded.** `render_drift_summary` (`context/drift.py:178-201`) feeds `additionalContext`; malicious feature/path names are generic SessionStart prompt-injection, never a shell. Badge writer severed + swallowed (`cli/plan_update.py:39-50,71-74`); write failure can't fail the hook or touch stdout.
- **Swallow-all blinds tamper detection.** Every error → silent-empty (`statusline.sh:16` `|| true`; `statusline.ps1:15`; `cli/statusline.py:64-66`); a planted/flipped badge gives no signal. Doc the statusline as non-authoritative — verify drift via `dummyindex context plan-update`.

## Product surface

- **Setup discoverability hinges on one transient `print` during `hooks install`** — the nudge is surfaced only in the `for nudge in result.nudges: print(f"  {nudge}")` loop of `cli/hooks.py:69-70`, which runs during `dummyindex context hooks install`. A user who installed hooks before this feature shipped, or who scrolled past that one install line, never sees the wiring tip again — `statusline_nudge` is *emit-only* and re-checked only on the next install (`context/hooks.py:225-241`, `:400-403`). Gap: no `status`/`doctor` surface re-emits the nudge, so the badge is effectively undiscoverable post-install except by reading `plan.md`. The cold-path command `dummyindex context statusline` itself prints nothing when unwired (silent by contract, `cli/statusline.py:50-56`), so a curious user running it gets no "you haven't wired this" hint.

- **The nudge snippet hard-codes the Python cold path, never mentions the shipped wrappers** — `_STATUSLINE_NUDGE` advises `"command": "dummyindex context statusline"` (`context/hooks.py:201-205`), but `plan.md:11,21,49` says that command is only the *cold-path fallback* and the shell wrappers (`statusline.sh`/`.ps1` under `SCRIPT_DIR`, `cli/statusline.py:34`) are "the deliverable / the real per-prompt hot path." Gap: the one discoverability surface points users at the slow path and never tells them the fast `cat` wrappers exist or where `SCRIPT_DIR` is — a user can't find the wrapper they're supposedly meant to wire. The wrappers' own header comments give wiring JSON (`statusline.sh:14-15`, `statusline.ps1:13-14`) but you'd only see those if you already located the file.

- **Badge meaning is undocumented for users — `[ctx ✓]` vs `[ctx: N drift]` is only explained in docstrings** — the semantics live in `compute_badge`'s docstring (`context/drift.py:92-101`: N = distinct drifted files + unassigned-new + awaiting-enrichment) and the cold-path module docstring (`cli/statusline.py:14`). Gap: nothing user-facing (no README/help text) tells a wirer what `[ctx ✓]` asserts or what the integer in `[ctx: 3 drift]` counts, so the badge reads as opaque. Worse, `N` is a *composite* of three different signals (mtime drift + two commit-anchored signals, `drift.py:104-108`) — a user seeing `[ctx: 3 drift]` cannot tell whether that's 3 edited files or 1 file + 2 unindexed, and the badge offers no drill-down path (the real detail is in `render_drift_summary`, only shown at SessionStart, `drift.py:178-201`).

- **Stale-cache badge is actively misleading on the hot path, with no in-band tell** — confirmed by-design (`plan.md:51`, Decisions): badge refreshes only at SessionStart `plan-update`, so a mid-session edit leaves `[ctx ✓]` rendered every prompt while drift actually exists (`cli/plan_update.py:66-74`). From the *user's* seat this is the sharpest product gap: the statusline confidently shows a green check for an entire session after you've edited source, and there is zero affordance (no dimming, no age, no `[ctx ✓?]`) to signal "this is a cached snapshot from session start." `plan.md:56` flags it as an open question needing "a user-facing note," but no such note exists in any shipped surface — the wrappers' header comments (`statusline.sh:1-15`) don't mention it. A user will reasonably trust the green check and be wrong.

- **"What if not wired" degrades to total invisibility, indistinguishable from "wired but no drift"** — if the user never wires `statusLine`, there is simply no badge anywhere; and if they wire it but no `plan-update` has run yet, the readers print nothing (`cli/statusline.py:54-56`, `statusline.sh:16` empty `cat`). Gap: "not wired," "wired but cache cold," and "wired, cache present, genuinely no drift but badge empty due to a decode error" (`cli/statusline.py:64-66`) all collapse to the same empty render. A user cannot self-diagnose which state they're in from the statusline alone; the only signal that wiring *worked* is a non-empty badge, which itself requires a `plan-update` to have fired first. The first-run ordering (wire → start a session so `plan-update` writes the cache → see badge) is never spelled out, so a user who wires it and sees nothing will reasonably conclude it's broken.

- **Cross-platform parity is structurally good but the nudge/docs are sh-only** — both `statusline.sh` (POSIX) and `statusline.ps1` (pwsh) ship with matching contracts (`statusline.ps1:1-20` mirrors `statusline.sh:1-16`), and the ps1 header even gives Windows wiring JSON with `pwsh -File` (`statusline.ps1:14`). Good. Gap: the single discoverability nudge (`context/hooks.py:201-205`) is platform-agnostic only because it points at the Python command, not the wrappers — there is no guidance for a Windows user on whether to wire `statusline.ps1` vs the Python command, and `plan.md` never states the wrappers require `cwd == project root` on *either* platform. The cwd-relative `.context/cache/freshness-badge` literal (`statusline.sh:16`, `statusline.ps1:17`) means a Windows user wiring the absolute ps1 path from a non-root cwd gets a silently empty badge with no error — same trap as POSIX, undocumented on both.

- **No uninstall / un-wire story for the badge** — `statusline_nudge` deliberately never writes `statusLine` because a scalar has no sentinel to safely un-clobber (`plan.md:48`, `context/hooks.py:198-200`), which is correct. But the inverse is also unaddressed: nothing tells a user how to *remove* the badge once wired, and `hooks uninstall` (`cli/hooks.py:74-82`) only touches the hooks block, leaving the user's `statusLine` (and the gitignored cache) in place. Minor, but "how do I turn this off" has no documented answer, consistent with the emit-only design but worth one line.
