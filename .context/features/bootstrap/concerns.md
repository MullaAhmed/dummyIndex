## Data integrity

There is no relational database in this feature. Integrity depends on preserving
user-authored guidance across multiple filesystem snapshots and atomic replaces.

- **High — consolidation can overwrite and then delete a concurrently edited root file.** `reconcile_claude_md` snapshots existence and both files, derives a complete replacement from those strings, and performs no lock, inode check, mtime check, or content revalidation (`dummyindex/context/output/claude_md.py:126-209`). It then replaces canonical and unlinks the root path based only on the earlier snapshot (`dummyindex/context/output/claude_md.py:221-269`). If another process edits or replaces root after the read, the stale bytes are committed to canonical and the newer root is deleted; a concurrent canonical edit is likewise overwritten. The fixed-name temporary used by `write_text_atomic` adds collision risk between concurrent reconcilers (`dummyindex/context/domains/atomic_io.py:12-25`). Lock the root/canonical pair for the whole transaction, use a unique sibling temp, and revalidate source identities/content immediately before replace and unlink.

- **High — the inode-shared repair path breaks hardlink identity and leaves a stale second file.** `_same_file` intentionally detects root and canonical aliases and routes them to `_refresh_single_file` (`dummyindex/context/output/claude_md.py:129-133`, `dummyindex/context/output/claude_md.py:272-289`). On a changed block, that helper calls `write_text_atomic` on only the canonical pathname (`dummyindex/context/output/claude_md.py:290-335`); its temp-and-replace creates a new inode for that directory entry (`dummyindex/context/domains/atomic_io.py:12-25`). For hardlinks, root remains attached to the old stale inode while canonical receives the refreshed inode, contradicting the single-file branch's premise. A canonical symlink can similarly be replaced as a regular file. Preserve alias semantics explicitly, or refuse/warn instead of claiming a successful shared-file refresh when an atomic replacement cannot retain the relationship.

- **Medium — legacy consolidation is not byte-preserving for user residue.** `_user_residue` calls `.strip()` after removing managed spans (`dummyindex/context/output/claude_md.py:75-93`), the two reads use universal-newline text mode (`dummyindex/context/output/claude_md.py:135-178`), and the merge reconstructs separators with `\n` (`dummyindex/context/output/claude_md.py:190-209`). This drops leading/trailing whitespace and normalizes CRLF/mixed endings; leading indentation can be semantic Markdown, such as an indented code block. Preserve exact residue segments and original newline/whitespace policy, or document and narrowly validate any intentional normalization before deleting the source file.

- **Medium — byte-idempotent direct bootstrap is not filesystem-state idempotent.** Even when assembly reproduces the existing string, `bootstrap_claude_md` always calls `_atomic_write` (`dummyindex/context/output/bootstrap.py:106-132`). That writer replaces the inode and restores only POSIX mode bits (`dummyindex/context/output/bootstrap.py:396-415`), so a no-content-change run can sever hardlinks and discard extended attributes, ACLs, ownership nuances, and timestamps. Skip the write when `new_content == existing`; when content changes, define and preserve the metadata contract beyond mode bits where the platform supports it.

- **Medium — `--platform both` can commit only one host file despite successful preflight.** The CLI preflights Claude and Codex, writes Claude, and only then recomputes and writes the active Codex target (`dummyindex/cli/bootstrap.py:41-69`; `dummyindex/context/output/agents_md.py:110-139`). A post-preflight race, changed fallback selection, or Codex write failure returns exit 3 after Claude has already committed, with no rollback. Either stage both final payloads and commit with a recoverable journal/rollback strategy, or return a structured partial-success result that names the file already changed.

- **Medium — direct Claude bootstrap can create the two-location state that reconciliation otherwise treats as legacy drift.** The managed marker tells users to regenerate with `dummyindex context bootstrap` (`dummyindex/context/output/bootstrap.py:14-18`), but that CLI always writes `<root>/.claude/CLAUDE.md` through the single-file renderer and never inspects `<root>/CLAUDE.md` (`dummyindex/cli/bootstrap.py:33-61`). If a legacy root block exists, the command succeeds with a fresh canonical block while leaving the old root block in place; only a later build/install/migration path through `reconcile_claude_md` restores the single-canonical-file invariant (`dummyindex/context/output/claude_md.py:96-110`, `dummyindex/context/output/claude_md.py:211-269`). Route direct Claude bootstrap through the reconciler, or reject and report a legacy root file before creating a second managed region.

## Security

- **[HIGH] Legacy reconciliation reintroduces a predictable-temp symlink clobber outside the validated project boundary** — `dummyindex/context/output/claude_md.py:112-115,221-224,321-323` and `dummyindex/context/domains/atomic_io.py:12-25`. Attack scenario: an attacker pre-creates `.claude/CLAUDE.md.tmp` as a symlink to any file the bootstrap-running user can write; `reconcile_claude_md` validates only the final `CLAUDE.md` path, then `write_text_atomic` follows the fixed temp symlink and truncates the external victim before renaming that symlink over the canonical guidance path. Mitigation: the direct renderer's unique `O_EXCL` temp allocator (`dummyindex/context/output/bootstrap.py:396-415,428-442`) closes this attack on `context bootstrap`, but the build/install reconciliation path bypasses it. Replace the shared fixed-temp helper with the same unique non-following primitive and validate the opened directory/target identity before publication.

- **[HIGH] Resolved-path containment is check-then-use, so a parent-symlink swap can redirect the atomic write after approval** — `dummyindex/context/output/bootstrap.py:48-64,82-112,396-410`, `dummyindex/cli/bootstrap.py:41-57`, and `dummyindex/context/output/agents_md.py:142-161,255-270`. Attack scenario: in a shared workspace, `.claude` or a configured nested Codex fallback parent initially resolves inside the project and passes preflight; a concurrent attacker atomically repoints that parent symlink outside the project before `_open_atomic_temp`, causing temp creation and `replace` to operate in the attacker-selected external directory. Mitigation: resolving the full path rejects stable out-of-scope links, and random exclusive temp names prevent pre-creation, but neither binds the checked directory chain to the write. Open the project and descendant directories through retained dirfds with no-follow/beneath semantics (for example `openat2` where available), then create and rename the temp relative to the verified final parent; otherwise detect inode changes and fail closed.

- **[MED] Direct bootstrap silently drops access-control metadata, so a normal or byte-identical refresh can broaden who may read guidance** — `dummyindex/context/output/bootstrap.py:100-132,396-415` and `dummyindex/context/output/agents_md.py:110-129`. Attack scenario: a shared repository protects `CLAUDE.md` or `AGENTS.md` with a named-user ACL or security xattr stricter than its ordinary mode bits; bootstrap records only `stat.S_IMODE`, creates a new inode, reapplies those mode bits, and replaces the target without copying the ACL, xattrs, ownership, or security label. A collaborator previously denied by the named ACL can then fall through to permissive group/other mode bits and read the guidance, and this happens even when the rendered bytes were already current because the writer never skips an identical payload. Mitigation: skip publication when bytes match; for changed files, copy and verify supported ownership, ACL, xattr, and security-label metadata before replacement, or fail closed when the preservation contract cannot be met.

## Product surface

- **A repository still has a legacy root `CLAUDE.md` and the user follows that managed marker’s `dummyindex context bootstrap` regeneration instruction — the command writes only `<root>/.claude/CLAUDE.md` and never invokes the root-to-canonical reconciler — it can report success while leaving the stale block that directed the user untouched.** The instruction is embedded in the stable Claude begin marker (`dummyindex/context/output/bootstrap.py:14-18`), whereas the CLI always selects the canonical path and calls the single-file renderer (`dummyindex/cli/bootstrap.py:33-61`). Root folding is a separate `reconcile_claude_md` operation used by build/install/migration paths (`dummyindex/context/output/claude_md.py:96-110`, `dummyindex/context/output/claude_md.py:211-269`), so direct bootstrap can leave managed blocks in both locations.

- **A user or editor saves the legacy root `CLAUDE.md` while install, build, or refresh is consolidating it — reconciliation commits its earlier snapshot and deletes the newer root file — the operation can still present this as successful guidance setup.** The reconciler reads both files, performs no identity/content revalidation, then writes canonical and unlinks root (`dummyindex/context/output/claude_md.py:126-209`, `dummyindex/context/output/claude_md.py:221-269`). That race produces no warning: build marks guidance ready (`dummyindex/context/build/runner.py:280-297`), while install and migration print the successful consolidation message (`dummyindex/installer/install.py:404-409`, `dummyindex/cli/migrate.py:74-89`). The surface needs conflict detection that leaves both copies intact and names the changed path instead of reporting a clean consolidation.

- **A user runs `context bootstrap --platform both`, both preflights pass, and the Codex target then changes or becomes unwritable before its write — Claude guidance is committed and announced before the Codex failure returns exit 3 — the single “both” operation can leave host guidance split with no rollback or final partial-success summary.** Deterministic Claude and Codex conflicts are correctly checked before mutation, but the implementation then writes Claude first and independently recomputes/writes the Codex target (`dummyindex/cli/bootstrap.py:41-69`, `dummyindex/context/output/agents_md.py:110-139`). The preflight boundary therefore prevents known conflicts, not post-check filesystem or target-selection changes.

- **A user wants to stop using dummyindex’s Claude project guidance without hand-editing its managed markers — the renderer has a validated `remove_managed_block` operation, but `context bootstrap` exposes only create/update and Claude uninstall explicitly leaves project guidance intact — there is no supported public inverse for the Claude block.** The remover safely preserves surrounding content and handles empty files and symlinks (`dummyindex/context/output/bootstrap.py:135-180`), while the bootstrap CLI accepts only root/platform selection (`dummyindex/cli/bootstrap.py:10-71`) and uninstall documents the retained Claude guidance (`dummyindex/installer/args.py:32-46`). This conflicts with the marker’s instruction not to hand-edit the region without giving the user a removal command.

- **A user reruns direct bootstrap to verify that guidance is current — identical final bytes are still published through an atomic replacement and the CLI prints the same `managed block written` line used for creation or update — the apparent check can silently change filesystem security state.** `bootstrap_claude_md` always calls `_atomic_write` after assembly even when `new_content == existing` (`dummyindex/context/output/bootstrap.py:106-132`); that writer creates a new inode and restores only mode bits, discarding hardlink identity, ACLs, security xattrs, ownership nuances, and timestamps (`dummyindex/context/output/bootstrap.py:396-415`). Neither the renderer’s string return nor the CLI’s fixed success message distinguishes create/update/no-op or tells the user that bytes were current while metadata changed (`dummyindex/cli/bootstrap.py:54-69`).

- **A cautious user wants to inspect the exact Claude and Codex guidance changes before applying them — public bootstrap help offers only path, root, and platform, and every successful command path writes — there is no read-only preview, diff, or check mode.** Help is safely intercepted before side effects, but its complete bootstrap surface contains no dry-run option (`dummyindex/cli/help.py:54-58`, `dummyindex/cli/__init__.py:136-154`), and the handler’s preflight functions are internal to the `both` mutation path rather than a user-visible preview (`dummyindex/cli/bootstrap.py:17-69`).

The message promises preservation the code can't guarantee, so the user is actively told their content is safe while it may have been mis-cut.
