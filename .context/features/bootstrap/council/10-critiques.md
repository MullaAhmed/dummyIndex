## Data integrity

- **Atomic write diverges from the mandated canonical helper.** `_atomic_write` (`bootstrap.py:86-89`) is a private reimplementation of the "one mandatory rule" in `data-access.md:5-9`. It is *not* wrong, but it drifts from `write_text_atomic` (`atomic_io.py:11-24`) in two ways: (a) it does **not** `mkdir(parents=True)` itself — it relies on the create-branch caller doing `path.parent.mkdir(...)` at `bootstrap.py:42`, while the replace/append branches assume the parent already exists (true for an existing file). Net effect is correct, but the invariant is split across two call sites instead of owned by the writer. (b) The convention explicitly warns "Don't bypass `write_text_atomic` … you lose atomicity *and* break hash-baselining" (`data-access.md:27`). CLAUDE.md is a user-edited file, not a hash-baselined generated artifact, so the baselining concern does not bite here — but the divergence is undocumented and means a future fix to the canonical helper won't reach this writer. Flag, not block.

- **tmp-sibling path is correct for `CLAUDE.md` but fragile by construction.** `path.with_suffix(path.suffix + ".tmp")` (`bootstrap.py:87`) yields `CLAUDE.md.tmp` (verified: `.md` → `.md.tmp`). Covered by `test_bootstrap.py:114-118`. Edge case: if the target ever had **no** suffix (e.g. a bare `CLAUDE`), `with_suffix(".tmp")` still works; but a multi-dot or trailing-dot name would behave differently than a plain `str(path) + ".tmp"` append. The CLI fixes the name to `.claude/CLAUDE.md` (`cli/bootstrap.py:18`), so this is theoretical today — but the shared `bootstrap_claude_md(path)` contract accepts an arbitrary `path` and five callers pass their own (`installer/install.py`, `cli/migrate.py:114`, `build/runner.py:263`, etc.), so the tmp-naming assumption is part of the public contract and untested for non-`.md` targets.

- **Idempotent regeneration: body is `rstrip()`-normalized, surrounding bytes are not — correct.** `bootstrap_claude_md` rstrips only the managed *body* (`bootstrap.py:38`) and `_replace_block` stitches exact `existing[:begin_idx] + managed + existing[end_idx:]` (`bootstrap.py:80-83`), so user content on either side is byte-preserved. Re-running on already-current content reproduces identical bytes (managed region is deterministic; `_V0_BLOCK_BODY` is a constant). No row-duplication analogue. Clean.

- **Marker-count dispatch is a genuine integrity guard, but substring-based — a real corruption vector.** `begin_count`/`end_count` via `str.count` (`bootstrap.py:48-49`) and `str.index` (`bootstrap.py:81-82`) match the marker *anywhere*, including inside a fenced code block in surrounding user prose. A user who pastes the literal `BEGIN_MARKER` into their own CLAUDE.md prose (plausible — the marker text instructs "regenerate with `dummyindex context bootstrap`", exactly the kind of thing a user copies into docs) makes `begin_count == 2` and triggers `UnbalancedMarkersError` (`bootstrap.py:56-59`) on an otherwise valid file, OR worse, with one stray marker, `_replace_block` will splice from the *first* `BEGIN_MARKER` to the *first* `END_MARKER` and **silently delete the user text between a prose-mention and the real block**. This is the plan's own "open question" (`plan.md:49`) but it is under-rated: it is not just a miscount, it is a silent destructive replace. The fail-loud guard only fires when counts are unequal; a balanced stray pair passes the guard and corrupts. Three callers share the blind spot (`migrate.py:102-106` uses the same `index`-based strip).

- **Legacy migration replace is correct and depends on body-agnostic anchoring.** `migrate.py:102-106` strips the legacy block by the same begin/end `index` slice and re-bootstraps; because `_replace_block` keys on markers not body, an over-long legacy block migrates to the short pointer in place (`plan.md:31,40`). Correct — with the same substring caveat above: a legacy file with a marker mentioned in prose would mis-strip. Same vector, second site.

- **No content-integrity verification after write.** Neither `_atomic_write` nor `bootstrap_claude_md` reads back / verifies the post-write content matches the returned string. `os.replace` is atomic so a torn file is impossible, but a silent encoding round-trip issue (non-UTF-8 user content already in the file is read with `encoding="utf-8"` at `bootstrap.py:47` and would raise `UnicodeDecodeError` *un-typed* — not mapped to `UnbalancedMarkersError`, so it escapes the CLI's exit-3 handler at `cli/bootstrap.py:21` and surfaces as a raw traceback / exit 1). User-facing error handling for a non-UTF-8 existing CLAUDE.md is missing.

#### Deep first pass — current source

## Data integrity

There is no relational database in this feature. Integrity depends on preserving
user-authored guidance across multiple filesystem snapshots and atomic replaces.

- **High — consolidation can overwrite and then delete a concurrently edited root file.** `reconcile_claude_md` snapshots existence and both files, derives a complete replacement from those strings, and performs no lock, inode check, mtime check, or content revalidation (`dummyindex/context/output/claude_md.py:126-209`). It then replaces canonical and unlinks the root path based only on the earlier snapshot (`dummyindex/context/output/claude_md.py:221-269`). If another process edits or replaces root after the read, the stale bytes are committed to canonical and the newer root is deleted; a concurrent canonical edit is likewise overwritten. The fixed-name temporary used by `write_text_atomic` adds collision risk between concurrent reconcilers (`dummyindex/context/domains/atomic_io.py:12-25`). Lock the root/canonical pair for the whole transaction, use a unique sibling temp, and revalidate source identities/content immediately before replace and unlink.

- **High — the inode-shared repair path breaks hardlink identity and leaves a stale second file.** `_same_file` intentionally detects root and canonical aliases and routes them to `_refresh_single_file` (`dummyindex/context/output/claude_md.py:129-133`, `dummyindex/context/output/claude_md.py:272-289`). On a changed block, that helper calls `write_text_atomic` on only the canonical pathname (`dummyindex/context/output/claude_md.py:290-335`); its temp-and-replace creates a new inode for that directory entry (`dummyindex/context/domains/atomic_io.py:12-25`). For hardlinks, root remains attached to the old stale inode while canonical receives the refreshed inode, contradicting the single-file branch's premise. A canonical symlink can similarly be replaced as a regular file. Preserve alias semantics explicitly, or refuse/warn instead of claiming a successful shared-file refresh when an atomic replacement cannot retain the relationship.

- **Medium — legacy consolidation is not byte-preserving for user residue.** `_user_residue` calls `.strip()` after removing managed spans (`dummyindex/context/output/claude_md.py:75-93`), the two reads use universal-newline text mode (`dummyindex/context/output/claude_md.py:135-178`), and the merge reconstructs separators with `\n` (`dummyindex/context/output/claude_md.py:190-209`). This drops leading/trailing whitespace and normalizes CRLF/mixed endings; leading indentation can be semantic Markdown, such as an indented code block. Preserve exact residue segments and original newline/whitespace policy, or document and narrowly validate any intentional normalization before deleting the source file.

- **Medium — byte-idempotent direct bootstrap is not filesystem-state idempotent.** Even when assembly reproduces the existing string, `bootstrap_claude_md` always calls `_atomic_write` (`dummyindex/context/output/bootstrap.py:106-132`). That writer replaces the inode and restores only POSIX mode bits (`dummyindex/context/output/bootstrap.py:396-415`), so a no-content-change run can sever hardlinks and discard extended attributes, ACLs, ownership nuances, and timestamps. Skip the write when `new_content == existing`; when content changes, define and preserve the metadata contract beyond mode bits where the platform supports it.

- **Medium — `--platform both` can commit only one host file despite successful preflight.** The CLI preflights Claude and Codex, writes Claude, and only then recomputes and writes the active Codex target (`dummyindex/cli/bootstrap.py:41-69`; `dummyindex/context/output/agents_md.py:110-139`). A post-preflight race, changed fallback selection, or Codex write failure returns exit 3 after Claude has already committed, with no rollback. Either stage both final payloads and commit with a recoverable journal/rollback strategy, or return a structured partial-success result that names the file already changed.

--- critic-security (stage 3) ---

## Security

- **Marker injection → silent destructive splice (highest-impact vector).** `_replace_block` keys on the **first** `BEGIN_MARKER` and **first** `END_MARKER` via `str.index` (`bootstrap.py:81-82`), and the only pre-dispatch guard is a *count* check (`bootstrap.py:48-59`). Concrete attack path: any actor that can write to `CLAUDE.md` before bootstrap runs (a malicious repo template, a cooperating editor plugin, a PR that adds onboarding prose, or a user copy-pasting the marker text the marker itself advertises — `"regenerate with \`dummyindex context bootstrap\`"`, `bootstrap.py:11-14`) lands **one balanced decoy pair** of the literal markers *above* the real managed block. `begin_count == end_count == 1` passes both guards (`bootstrap.py:51,56`), then `_replace_block` slices `existing[:decoy_begin] + managed + existing[decoy_end:]` — **silently deleting every byte between the decoy `END_MARKER` and the real block, including the real managed region and any user instructions in between.** No error, exit 0, atomic write commits the truncation. Because CLAUDE.md is the agent's instruction file, this is also an *instruction-poisoning* primitive: the attacker fully controls where the managed (trusted) region lands relative to their prose. The fail-loud `UnbalancedMarkersError` (`bootstrap.py:18-19,52-59`) never fires on a position attack — it only catches count skew. Remediation: anchor on a **unique full-line** match (require markers to occupy their own line and reject >1 *line-anchored* occurrence), or after `index`, assert exactly one begin/end and that `begin_idx < end_idx` with no second occurrence between them; treat any decoy as `UnbalancedMarkersError` (fail closed) rather than splicing. Same defect at the second call site `migrate.py:102-106`.

- **No symlink guard on the read/replace path — TOCTOU + inode confusion.** `bootstrap_claude_md` does `path.exists()` (`bootstrap.py:41`, follows links) then `path.read_text(...)` (`bootstrap.py:47`, follows links) with no `path.is_symlink()` / `O_NOFOLLOW` check. Concrete attack path: an attacker pre-seeds `.claude/CLAUDE.md` as a symlink to a file the bootstrap-running user can read (e.g. another repo's CLAUDE.md, or any readable file). bootstrap then **reads marker state from the link target**, makes its replace/append decision on the *wrong* file's content, and the subsequent `tmp.replace(path)` (`bootstrap.py:89`) — because the tmp sibling is created in the *link's* directory via `with_suffix` (`bootstrap.py:87`) and `os.replace` swaps the link itself — silently destroys the symlink and writes a regular file, but only *after* the decision logic has trusted foreign content. The write does not follow the link (good, no arbitrary-overwrite primitive), but the **read/decision does**, which is enough to drive the marker-injection splice above using content the attacker chose. Remediation: `if path.is_symlink(): raise` (fail closed) before reading, or open with `os.O_NOFOLLOW` and resolve+validate the parent dir is not attacker-writable.

- **tmp sibling write is non-exclusive and predictable — local symlink-in-tmp clobber.** `tmp.write_text(...)` (`bootstrap.py:88`) opens the fixed, predictable path `CLAUDE.md.tmp` with truncation semantics and **no `O_EXCL`**. In a world-writable or shared `.claude/` dir, an attacker who pre-creates `CLAUDE.md.tmp` as a symlink to an arbitrary file the user can write causes the user's `write_text` to **follow the link and overwrite the attacker-chosen target** with managed-block content (an arbitrary-write / clobber primitive scoped to whatever the bootstrap user can write). The canonical `write_text_atomic` (`atomic_io.py:11-24`) is the mandated helper precisely to centralize this; the private reimplementation drops the protection. Remediation: route through the canonical helper, or create the tmp via `tempfile.mkstemp(dir=path.parent)` (O_EXCL, 0600) and `os.replace` from there.

- **Non-UTF-8 existing CLAUDE.md → unhandled crash, not a managed-region failure (availability + info-leak).** `existing = path.read_text(encoding="utf-8")` (`bootstrap.py:47`) raises a bare `UnicodeDecodeError`, which is **not** caught by the CLI's only handler (`except UnbalancedMarkersError`, `cli/bootstrap.py:19-23`). An attacker who plants non-UTF-8 bytes in CLAUDE.md (or simply a user with a latin-1/UTF-16 file) gets a raw traceback to stderr (exit 1), leaking the absolute path and internal frames, and bootstrap cannot proceed (DoS on the managed-block update). Remediation: catch decode errors at the read boundary and re-raise as a typed, user-facing error mapped to exit 3, mirroring the marker handling.

- **Marker text is self-advertising, which *lowers* the bar for the injection vector above.** `BEGIN_MARKER` literally embeds the regeneration command (`bootstrap.py:11-14`), making it likely to be quoted verbatim in onboarding docs / READMEs that get pasted into CLAUDE.md. This is not a vuln on its own but it materially raises the probability of the balanced-decoy-pair condition arising *organically* (no malicious actor needed), turning the silent-splice integrity bug into a realistic data-loss path. Pairs with remediation #1 (line-anchored, fail-closed matching).

--- critic-product (stage 3) ---

## Product surface

- **First contact: the agent's instruction file gets a surprise managed block, never announced before the fact** — On the install/auto-init path the user runs an *index* command and `bootstrap_claude_md(project_root/".claude"/"CLAUDE.md")` is fired as a side-effect (`installer/install.py:236-271`, the `bootstrap_claude_md(...)` call ~`:258`); the only surface is an after-the-fact one-liner `"  CLAUDE.md (proj) ->  managed block written"`. Same for the build path — `if bootstrap: bootstrap_claude_md(...)` runs as the final step with **zero output** on success (`build/runner.py:262-263`). — A user whose CLAUDE.md gets a managed block has no pre-write notice, no diff, no "I am going to edit your CLAUDE.md" prompt, and (on build) often no confirmation at all; the file edit is the most surprising thing the tool does and is the least surfaced.

- **No dry-run / no preview anywhere** — `dummyindex context bootstrap` only writes; there is no `--dry-run`/`--check` to print *what would change* (`cli/bootstrap.py:7-25` has only the write path). — A cautious user cannot ask "show me the block you'd insert" without running it for real; the only way to inspect the managed body before it lands in their file is to read source (`generate_managed_block`, `bootstrap.py:28-30`).

- **Idempotency is real but invisible — the success message is identical whether it created, appended, replaced, or no-op'd** — All four branches (`bootstrap.py:41-64`) funnel to the same `"managed block written: <path>"` print (`cli/bootstrap.py:24`); the CLI discards the returned content and never reports *which* action occurred. — A user re-running cannot tell from output whether bootstrap touched their file or was a clean no-op; "written" reads as "I just modified your file" even when bytes are unchanged. The reassuring idempotency property (re-run = identical bytes, `test_bootstrap.py:52-57`) is never communicated.

- **Block-already-exists is handled silently and correctly, but the replace is mute about what it overwrote** — On an existing single block the body is swapped in place (`_replace_block`, `bootstrap.py:63-64,80-83`); if the user hand-edited *inside* the managed region (despite "do not hand-edit", `bootstrap.py:12`), their edits are discarded with no warning, no backup, no diff. — The marker says "do not hand-edit" but nothing detects or warns when someone did; a user who tweaked the managed prose loses it silently on the next index/build with no recovery path (no `.bak`, atomic replace destroys the prior bytes — `bootstrap.py:86-89`).

- **Hand-edited / malformed markers fail loud only on the direct CLI; the auto-init and migrate paths swallow the same failure** — The direct verb maps `UnbalancedMarkersError` → exit 3 with a clear stderr message (`cli/bootstrap.py:19-23`). But auto-init wraps the call in `except Exception` and prints `"  CLAUDE.md (proj) ->  skipped ({exc})"` then *continues* (`installer/install.py` ~`:259-263`), and migrate wraps it in `except Exception` → `"migration warning: …"` and returns (`cli/migrate.py:113-117`). — A user whose CLAUDE.md has unbalanced/duplicate markers gets a clear actionable error if they happen to run `context bootstrap` directly, but on the far more common install/update/build path the same condition degrades to a buried one-line "skipped" warning amid other output — easy to miss, and the managed block silently stays stale while every other artefact updated.

- **Legacy migration (root → .claude/) is reasonably visible, but only via stderr/stdout prints with no summary or undo** — `migrate_claude_md_location` prints `"migration: relocated CLAUDE.md managed block … (user content preserved at <legacy>)"` or `"removed empty root file"` (`cli/migrate.py:124-136`). Good: it tells the user their root `CLAUDE.md` was rewritten or deleted. Gaps: (a) the legacy strip uses the same first-`index` splice (`migrate.py:104-106`) flagged in Security/Data-integrity, so a prose-mentioned marker mis-strips with the *same* reassuring "user content preserved" message — the message can lie; (b) deletion of the root file (`legacy_path.unlink()`, `migrate.py:132`) is irreversible with no backup; (c) the migration runs inside `refresh-indexes` as a side-effect, so a user refreshing indexes can have their root `CLAUDE.md` deleted without ever asking to migrate.

- **No way to opt out, disable, or remove the managed block via the tool** — There is no `--no-bootstrap` on the index/build paths surfaced to the user (the `bootstrap` flag exists internally at `runner.py:262` but isn't a user-facing off switch on auto-init), and no `context bootstrap --remove` to cleanly excise the block. — A user who does not want dummyindex editing their CLAUDE.md has no supported escape hatch; the only removal is manual deletion of the marker region, which they're told not to hand-edit. Every index/refresh/build re-asserts the block.

- **"do not hand-edit … regenerate with `dummyindex context bootstrap`" points users at the verb that is the *narrowest* surface** — The marker advertises the standalone verb (`bootstrap.py:12-13`), but in practice the block is (re)written by install/refresh/build far more often than by anyone typing `context bootstrap`. — A user following the marker's own instruction runs a command that does exactly one thing in isolation, which is fine — but the marker doesn't tell them the block also silently regenerates on every index/build, so the mental model the surface gives ("it changes only when I run this verb") is wrong.
