## Data integrity

- **No cross-file atomicity ÔÇö only per-file atomicity.** Each op persists `feature.json`, `INDEX.json`, `INDEX.md`, `graph.{json,html}` as *independent* tmp+`replace` writes (`helpers.py:131-140`); `assign_files` does `_write_json(feature_json)` Ôćĺ `_write_pending_marker` Ôćĺ `_update_index_counts` Ôćĺ `_refresh_index_artifacts` as four separate replaces (`placement.py:170-182`). Validate-before-write makes a *rejected* op a clean no-op, but a crash/SIGKILL *mid-sequence* (e.g. after `feature.json` updates but before `INDEX.json`) leaves the canonical machine map permanently out of sync with the feature folder, with no journal/rollback. The plan's "atomic persistence" claim (`plan.md:23,43`) holds per-artifact, **not** across the 3-5 artifacts a single op mutates. Material for `merge_feature`, which spans section.md Ôćĺ feature.json Ôćĺ folder delete Ôćĺ INDEX Ôćĺ graph Ôćĺ council-log (`ops.py:226-342`) ÔÇö a crash after `_rmtree(src)` (`ops.py:263`) but before the INDEX drop (`ops.py:266-299`) strands an INDEX entry pointing at a deleted folder.

- **Member derivation is deterministic but silently empties on absent/stale symbols map.** `_members_for_files` returns `()` when `map/symbols.json` is missing (`placement.py:466-468`, `indexes.py:115-116`) ÔÇö so a `scaffold`/`assign`/`unassign` run before/without a maps rebuild writes `members: []` *and* `member_count: 0` into both `feature.json` and `INDEX.json`, looking authoritative. Determinism holds only relative to the symbols-map snapshot at op time; nothing re-derives members when `symbols.json` later changes (`rebuild --changed` rebuilds the map but no placement re-runs), so `members` can silently drift from the symbol set the files actually contain. Plan flags the absent-map tolerance as deliberate (`plan.md:38,55`) but not the resulting count drift in INDEX.

- **`merge_feature` does NOT re-derive members ÔÇö it set-unions raw lists.** Unlike placement, merge merges target+source `members`/`files`/`entry_points` by dedup union (`ops.py:253-257`) rather than re-running `_members_for_files` over the combined file set. If source `members` were stale (derived against an older symbols map) they propagate into the target unchecked, violating the "members follow files" invariant (`plan.md:38`) for the one op that crosses two features' member sets.

- **Duplicate-id rejection covers scaffold but the INDEX can hold duplicates from a partial merge/rename.** `scaffold_feature` rejects an existing folder id (`placement.py:78-82`) and `_validate_placement_id` blocks reserved `community-*` (`placement.py:395-403`) ÔÇö folder-level dedup is sound. But `_append_index_entry` blindly appends with no scan for an existing `feature_id` entry (`placement.py:489-509`); it relies on the folder-existence guard upstream. If a prior op crashed leaving a stale INDEX entry (see cross-file point), or `rename` partially completed, a subsequent scaffold of a *new* folder whose id collides with a stranded INDEX row yields two entries for one id. `_update_index_counts`/`_drop_index_entry` then update/drop *all* matching rows ambiguously (`placement.py:521-524`, `placement.py:537`).

- **`rename_feature` referential update is best-effort per-artifact, not all-or-nothing.** It updates `feature.json`, every `flows/*.json` `feature_id`, INDEX.json, INDEX.md, and graph.json as guarded independent writes (`ops.py:65-149`). Each guard is `if <file>.exists()`, so a missing graph.json silently skips graph rewiring while INDEX is updated ÔÇö the folder is renamed but `graph.json` nodes/edges still carry `from_id` if graph.json was absent at a prior step then regenerated stale elsewhere. No post-condition verifies all four references converged on `to_id`.

- **Idempotency of `assign`/`unassign` is real for the file set but the pending-enrichment marker is re-dropped every call.** Re-assigning an already-owned file is a clean no-op on `files` (set union, `placement.py:163`) ÔÇö good. But every call unconditionally re-writes `.pending-enrichment` (`placement.py:173`, `placement.py:248`), so a no-op-effect assign still re-flags the feature for enrichment and blocks `reconcile-stamp` (`plan.md:39,61`). Idempotent on payload, **not** on enrichment state.

- **`graph.json` regenerated from disk while `INDEX.json` is hand-edited ÔÇö two divergent sync paths.** Post-mutation, `INDEX.json` is patched in place (`_append`/`_update`/`_drop`, `placement.py:489-548`) but `graph.json` is fully rebuilt by walking `features/*/feature.json` (`indexes.py:51-104`). These can disagree if a `feature.json` on disk is itself stale vs. its INDEX row (e.g. after the partial-write window above): INDEX shows old counts, graph shows current file contents. `_refresh_index_artifacts` swallows `FileNotFoundError` (`placement.py:562-563,567-568`) so a missing INDEX.md/graph leaves the op "successful" with navigation artifacts absent and no warning.

- **`_drop_index_entry` flow_count decrement trusts the INDEX row's own `flow_count`, not the folder.** It subtracts the dropped entry's recorded `flow_count` from the top-level total (`placement.py:540-546`); if that per-entry count had drifted from the actual `flows/*.json` on disk, the top-level `flow_count` becomes wrong and there is no rebuild-from-disk reconciliation for it (the plan notes "no disk-rebuild helper" for INDEX, `plan.md:36`). Same trust-the-stored-count pattern in `merge_feature` (`ops.py:275`) and `remove_flow` (`ops.py:413,418`).

## Security

- **Stored XSS in `graph.html`: the confidence chip is the ONE feature field interpolated raw into `innerHTML`.** Grid-card chips are built by string-concat and assigned via `sub.innerHTML = chips` (`viewer.py:298-304`). Every sibling field that reaches `innerHTML` is wrapped in `escapeHtml` (`viewer.py:517-583`, e.g. `escapeHtml(feat.label)`, `escapeHtml(feat.summary)`), but the confidence chip is `f.confidence ? \`<span class="chip">${f.confidence}</span>\` : ""` (`viewer.py:302`) ÔÇö **no escaping**. `f.confidence` is copied verbatim from `feature.json`'s `confidence` with no enum check (`indexes.py:66` `fp.get("confidence", ConfidenceLevel.EXTRACTED)` accepts any string) into graph.json, which `loadFeatures` reads into `f.confidence`. Attack path: a `feature.json` whose `confidence` is `"<img src=x onerror=alert(document.cookie)>"` (planted by `scaffold`/`assign`/reconcile from an untrusted index, or a poisoned-PR diff) Ôćĺ next `_refresh_index_artifacts` Ôćĺ `rebuild_features_graph` (`placement.py:565`) copies it into graph.json Ôćĺ anyone who opens `.context/features/graph.html` executes it. Same raw-interpolation pattern in the count chips (`viewer.py:299-301`) but those are `int`-coerced (`len(...)`) upstream so not attacker-string-controlled. **Fix:** wrap with `escapeHtml(f.confidence)` at `viewer.py:302`; defensively validate `confidence Ôłł ConfidenceLevel` when materializing graph nodes.

- **`graph.html` runs with `file://` origin ÔÇö XSS here is not sandboxed to a null page.** The viewer is opened directly (or via `python3 -m http.server` per the error banner, `viewer.py:225`) and loads `d3` from `cdn.jsdelivr.net` (`viewer.py:215`) with no CSP/SRI. An injected payload (above) executes with the page's privileges: under `file://` it can read sibling `.context/` files via `fetch`; under the suggested local `http.server` rooted at `.context/features/`, it can exfiltrate any served file. There is no `Content-Security-Policy` meta and no `<meta http-equiv>` lockdown in `VIEWER_HTML` (`viewer.py:25-214`). **Fix:** add a strict CSP meta (`script-src 'self' cdn.jsdelivr.net; default-src 'none'`), pin d3 with SRI, and treat all `feature.json`-sourced strings as untrusted at render time.

- **id-slug path-traversal is correctly closed ÔÇö keep it that way.** `_validate_feature_id` is a strict whitelist (`[a-z0-9-_]`, no `.` or `/`, no leading/trailing `-`) (`helpers.py:18-29`), and `_validate_placement_id` layers the reserved `community-*` block on top (`placement.py:395-403`). Crucially `remove_feature` validates the slug BEFORE deriving `feat_dir` so a `../sibling` id cannot reach `_rmtree` of an out-of-tree dir (`placement.py:292-296`). No traversal via id on scaffold/assign/remove ÔÇö this is a correct control, flagged here so a future "allow richer ids" change doesn't silently regress it (e.g. permitting `.` would reopen `_rmtree`/folder-creation traversal).

- **`assign_files` path containment is enforced, but `.resolve()` follows symlinks ÔÇö a symlinked file inside the repo can point its canonical path out of tree (rejected) OR be used to make the feature claim a path that later resolves elsewhere.** `_normalize_files` does `(repo_root / f).resolve()` then `relative_to(repo_root)`, erroring on out-of-repo (`placement.py:419-427`) ÔÇö this correctly blocks `--file ../../etc/passwd`. But `.resolve()` canonicalizes through symlinks: a repo-internal symlink whose target is outside the repo resolves outside and is rejected (good), yet a symlink whose target is *inside* the repo stores the canonical inner path ÔÇö acceptable. The residual risk is TOCTOU/semantic: the stored repo-relative string is later fed to `graph.html`/docs as a file path and to `(repo_root / f).is_file()` re-checks in `_live_files` (`placement.py:348`); a path that was a file at assign time but is later replaced by a symlink changes what `remove_feature`'s live-guard sees. Low severity (local, single-user CLI) but worth a note: containment is path-string-based, not inode-pinned.

- **`unassign_files`/`_normalize_for_removal` skips the `is_file()` stat ÔÇö containment relies solely on `relative_to(repo_root)`.** By design (removed files no longer exist) it does not stat, only rejects out-of-repo via `relative_to` after `.resolve()` (`placement.py:447-453`). `Path.resolve()` on a non-existent path still normalizes `..` without touching disk, so `--file ../../x` is rejected ÔÇö containment holds. No write happens to the named path (it only edits the feature's `files` list), so even a containment miss here cannot write outside the repo; the blast radius is a bad entry in `feature.json`, not arbitrary FS write. Documented as correct-but-load-bearing: the security of removal depends entirely on `relative_to` since the existence gate is intentionally absent.

- **Feature `name`/`summary` from `feature.json` are attacker-influenceable text that reach both Markdown (`INDEX.md`) and HTML (`graph.html`).** In `graph.html` they ARE escaped (`escapeHtml(feat.label)` / `escapeHtml(feat.summary)`, `viewer.py:517,521`; grid `name`/`summary` use `.textContent`, `viewer.py:294,312`) ÔÇö safe there. The unescaped sink is confined to `confidence` (above). In `INDEX.md`, `name`/`summary` are written as raw Markdown (no HTML context), so injection there is content-spoofing only, not script execution. Net: the only code-execution XSS is the `confidence` chip; `name`/`summary` are defense-in-depth-clean in the viewer. Flagged so the fix focuses on the real sink rather than over-escaping the already-safe fields.

## Product surface

- **Merge silently discards source flows and never merges flow accounting ÔÇö operator runs `features-merge from=A into=B` and B is left with fewer flows than the union ÔÇö gap: `merge_feature` set-unions only `members`/`files`/`entry_points` (`ops.py:253-257`); `flow_ids` is NOT in that key tuple, and the source's flow `.json/.md` are deleted with the folder via `_rmtree(src)` (`ops.py:263`). The graph step then drops the source flow nodes entirely (`ops.py:309-314`). So flows that documented behavior in the absorbed feature vanish ÔÇö they are neither re-parented to `into_id` nor folded into the section prose. Contrast `rename`, which DOES re-point every `flows/*.json` `feature_id` (`ops.py:86-93`). An operator consolidating a "trivial" feature that happens to own a real flow loses that flow with no warning; the success line only reports a file-touch count (`cli/features.py:163-166`).

- **Merge is `supporting`-only and the error names no path to widen it from the CLI ÔÇö operator who wants `as-section=noise-absorbed` gets exit 2 with an allowlist they can't extend without editing source ÔÇö gap: `as_section` is gated against `_VALID_MERGE_SECTIONS` = `{"supporting"}` (`ops.py:211-215`, `constants.py:48`) and the CLI defaults to `supporting` (`cli/features.py:132`). There is no `--allow-new-section` escape hatch on `features-merge` (unlike `section-write`, which has one ÔÇö `cli/features.py:19-29`). The docstring says broadening "requires updating the procedure in `18-filter-trivial.md`" (`ops.py:198-200`), i.e. a code/skill edit, not an operator flag. Confusing because the same repo offers `--allow-new-section` elsewhere, so an operator reasonably expects it here and the rejection message doesn't say it's source-only.

- **No `split` op exists despite the taxonomy domain claiming to let the council "reshape that taxonomy in place" ÔÇö operator who needs to break one over-broad feature into two has no command ÔÇö gap: `plan.md:23` advertises a "family of atomic CRUD ops" to reshape the taxonomy, but the delivered verbs are rename/merge/scaffold/assign/unassign/remove/remove_flow/write_section (`plan.md:10-11`). The only way to split is the multi-step dance: `scaffold-feature` a new id with the carved-out files, then `unassign-files` them from the original ÔÇö two ops, two `.pending-enrichment` drops, two INDEX edits, and the curated prose does NOT follow the moved files (scaffold writes a fresh stub, `placement.py:107`). Merge is the documented inverse; its dual (split) is hinted by the symmetric "reshape" framing but never delivered.

- **`assign-files` silently swallows already-owned and missing-from-symbols files ÔÇö operator can't tell which of the `--file` args actually landed ÔÇö gap: already-assigned paths are dropped by the set-union (`placement.py:163`) with no count of how many were new vs. skipped; the `PlacementResult` reports the full merged set, not the delta. Separately, if `symbols.json` is absent or a file contributes no symbols, `members` is silently `()` / unchanged (`placement.py:466-468`) so the feature gains a file but zero members and the operator sees `member_count` not move ÔÇö easily mistaken for "assign didn't work." No CLI line distinguishes "0 new files" from "files added but no symbols matched."

- **`rename from=A to=B` where a stale `B` INDEX row already exists writes TWO rows both keyed `B` ÔÇö operator gets a duplicated/ambiguous INDEX after a rename onto a previously-stranded id ÔÇö gap: the INDEX refresh loop matches `entry.feature_id == from_id OR == to_id` (`ops.py:101`) and rewrites every match to `to_id` (`ops.py:102`). If a prior crashed/partial op left a row for `to_id` (the cross-file-atomicity window in `## Data integrity`), rename now stamps `to_id` onto both the A-row and the pre-existing B-row, yielding two `feature_id=B` entries. Downstream `_update_index_counts`/`_drop_index_entry` then touch *all* matching rows (`placement.py:521-524`, `:537`), so counts double-apply. Pure-metadata rename (`from==B, to==B`) is the documented idempotent path (`ops.py:43`), but the OR-match makes a *real* rename onto an occupied id non-idempotent and corrupting.

- **`unassign` refusing to empty a feature forces a two-command detour that the error names, but the inverse asymmetry is a trap ÔÇö operator removing the last file of a feature is told to use `features-remove`, yet `assign` happily creates single-file features ÔÇö gap: `unassign_files` raises if removal leaves zero files (`placement.py:234-238`) ÔÇö reasonable, but it means the reconcile flow "file deleted from repo" must branch: unassign if other files remain, else `features-remove`. The operator/skill must pre-check the feature's file count before choosing the verb; there is no single "remove file, and if it was the last one delete the feature" op. Easy to script wrong and hit exit 2 mid-reconcile.

- **Graph/INDEX regeneration after every mutation re-derives `members` only from a possibly-stale `symbols.json`, so a rename/merge can publish a graph that disagrees with the just-edited `feature.json` ÔÇö operator opens `graph.html` and sees old counts ÔÇö gap: placement ops re-derive members at op time (`placement.py:164`), but `rebuild_features_graph` walks `feature.json` on disk and re-reads `symbols.json` for class/method nodes (`indexes.py:100`); if `symbols.json` predates the file moves, graph member/touches edges reflect the old symbol set while INDEX counts reflect the op's snapshot. The success messages never warn that a maps rebuild is owed. Plan flags the absent-map tolerance (`plan.md:38,55`) but not that an operator must run `rebuild --changed` BEFORE placement for counts/graph to be trustworthy.

## Cross-review notes

- **The stored-XSS in `graph.html` (Security `concerns.md:21`) is directly operator-reachable through the product verbs reviewed here.** An operator running `merge_feature` or any placement op bumps `confidence` to `INFERRED` (`ops.py:258`, `ops.py:286`) ÔÇö but `confidence` is copied verbatim from `feature.json` when materializing graph nodes (`indexes.py:66`) with no enum check, and rendered unescaped at `viewer.py:302`. So the very ops in this critique (`scaffold`/`assign`/`merge`/`rename` all call `_refresh_index_artifacts` Ôćĺ `rebuild_features_graph`, `placement.py:565`) are the regeneration trigger that promotes a poisoned `confidence` string from a feature.json into `graph.json`. Operator impact: an operator who runs `features-merge`/`scaffold-feature` against a `.context/` folded in from an untrusted PR diff is the person who unknowingly arms the payload ÔÇö the next `graph.html` open executes it. The product-surface mitigation (beyond the Security fix at `viewer.py:302`) is to validate `confidence Ôłł ConfidenceLevel` at op time, since every reviewed verb already rewrites that field.
