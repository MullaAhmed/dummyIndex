## Data integrity

- `lifecycle/manifest.py:34-46` + `models.py:236-244` — forward-compat gap: `read_manifest` validates enum/kind values but **never rejects a `schema_version` newer than the build supports**, unlike the repo's own contract (`build/meta.py:60-66` raises on a too-new version, per `conventions/data-access.md:13`). A future v5 manifest loads silently, unknown top-level keys are dropped by `from_dict`, and `_apply_write`/`discover` then rewrite it at the hard-coded `SCHEMA_VERSION=4` (`dispatch.py:519`, `discover.py:586`) — a silent downgrade with field loss on any `apply`/`install`. Fix: in `read_manifest`, after `from_dict`, raise `EquipError` when `manifest.schema_version > SCHEMA_VERSION`.
- `cli/equip/dispatch.py:519` + `discover.py:586` — schema_version is written as a constant regardless of the version read, so a manifest is rebranded to v4 even when its records were authored under an older/newer schema. Combined with the missing too-new guard above, this is the actual data-loss vector. Fix: preserve `max(prior.schema_version, SCHEMA_VERSION)` only after the too-new guard rejects truly-newer files; never lower it.
- `lifecycle/manifest.py:58-59` — manifest JSON is emitted via `json.dumps(..., indent=2)` **without `sort_keys=True`**, while `conventions/data-access.md:13,28` mandates sorted/stable emission for byte-identical re-runs (`meta.py:96` uses `sort_keys=True`). Key order here is insertion-order of `to_dict` (stable today) but `items` order is whatever the merge produced in `_apply_write` (rendered → adopted → hooks → carried-forward), so two runs that re-derive items in a different order yield non-byte-identical manifests, defeating the hash-stable-artifact goal. Fix: add `sort_keys=True` and emit `items` in a deterministic order (e.g. sorted by `name`).
- `lifecycle/manifest.py:56` / `evolve.py:61` / `dispatch.py:575` / `atomic_io.py:22` — every atomic writer derives the tmp path as `path + ".tmp"` (a fixed name). Two concurrent equip runs (e.g. a SessionStart-triggered apply racing a manual `apply`) write the **same** `equipment.json.tmp` / `<agent>.md.tmp`, and the interleaved `write_text`→`replace` can publish a torn or wrong-run payload — the `replace` is atomic but the shared tmp is not isolated. No lockfile guards the manifest. Fix: use a per-process unique tmp suffix (pid/uuid) or an O_EXCL flock on the manifest; at minimum document the no-concurrent-run assumption.
- `lifecycle/manifest.py:38-39` — `read_manifest` rejects a non-dict top level but `from_dict` (`models.py:237-239`) silently **filters out any `items` entry that is not a dict** (`if isinstance(i, dict)`). A manifest corrupted to `items: [null, "x", {...}]` loads as a partial item set with no error, and the next `apply` merge then *persists* the silently-shrunken set — losing the dropped records (their generated files become orphaned, untracked, and re-classified as foreign user files). Fix: raise `EquipError` when any `items` element is not a dict rather than dropping it.
- `lifecycle/evolve.py:60-69` — patch is **not transactional across file + manifest**: `write_text_atomic(target, patched)` (line 61) commits the file, then `write_manifest` (line 69) commits the new `origin_hash`. If the process dies between the two, the file holds patched bytes whose hash no longer matches the stored (pre-patch) `origin_hash`, so the next `classify_item` reports the just-patched item as `USER_MODIFIED` — the evolution is silently frozen and never refreshable. Fix: write the manifest first (or stage both tmps and `replace` file→manifest last), and on the post-file `write_manifest` failure re-raise so the caller knows the baseline is stale.
- `models.py:113-132` + `lifecycle/manifest.py:58` — `to_dict` always emits the four `origin_*`/`marketplace`/`mechanism` keys as explicit `null` for every item (generated, adopted, hook), but `invariants` is conditionally omitted to stay "byte-identical to v3". This is internally inconsistent: any future addition that toggles a key's presence (the `invariants` pattern) will silently change the serialized bytes of unrelated v4 manifests on the next rewrite. Low-risk today, but flag: the byte-identical-re-run guarantee rests on every optional key following one rule. Prefer always-emit (with `null`) or always-omit-when-empty, not a mix.

## Security

- **[HIGH] TOCTOU + attacker-controlled blast-radius disclosure on untrusted install** — `discover.py:489` wires the marketplace with `add_marketplace(..., ref=None)` and `_record_native` leaves `origin_ref=None` deliberately (`discover.py:580-583`), so Claude Code's native fetcher pulls the marketplace repo at **moving HEAD**, never a pinned sha. The blast radius the user approves is computed purely from the *marketplace.json `declared_surfaces`* (`marketplace.py:105-106` `_surfaces`, `blast_radius.py:32-35` `analyze_blast_radius`) — attacker-controlled metadata fetched at discovery time. Attack path: a malicious untrusted repo publishes a `marketplace.json` that omits `hooks`/`mcpServers`/`bin` keys → `analyze_blast_radius` returns `runs_code=False`, and the dry-run prints `inert` (`discover.py:361-367`) and emits `"runs_code": false` in `--json` (`discover.py:338`); the user (or an automated `--for-proposal` flow) sees a "safe" disclosure, approves with `--yes`, and `claude plugin` then fetches the repo's *actual* HEAD payload which ships a hook/bin that runs on the next session. The `--yes` gate (`install_plan.py:42`, correctly `not candidate.trusted`) does still fire, but the disclosure it gates on is forgeable and the fetched bytes are never the bytes that were disclosed — there is no content pin and no post-fetch surface re-check. Fix: pin `ref` to a resolved commit sha at install time and record it in `origin_ref`; treat any untrusted plugin as `runs_code=True` for *disclosure* regardless of declared surfaces (the code comments at `install_plan.py:5-9` already concede the claim is unreliable — the print path doesn't reflect that).

- **[HIGH] Hash baseline is not tamper-evident against an adversary with `.claude/` write** — `content_hash` (`hashing.py:16-18`) is a bare `sha256` of file bytes, and the baseline is stored alongside the file in the same writer transaction (`_record_native` → `write_manifest`, `discover.py:585-586`; `apply_patch` re-baselines `origin_hash=content_hash(patched)` then persists, `evolve.py:60-69`). The ownership oracle (`status.py classify_item`, cited `plan.md:63-66`) decides PRISTINE vs USER_MODIFIED by comparing disk-hash to the stored hash — but anyone able to write a generated `.claude/<agent>.md` can also rewrite the matching `equipment.json` `origin_hash`, so a trojaned generated agent re-classifies as **PRISTINE and is silently trusted and eligible for overwrite-refresh**. The baseline defends against accidental user edits, not against the actual threat surface (filesystem writes into `.claude/`). The `is_safe_to_write` sentinel (`safety.py:18-33`) is explicitly "a human marker only" (`plan.md:139-140`) and is trivially copyable, so it adds no adversarial protection either. Fix: the baseline must derive from a trust root the writer of the artifact cannot also forge (e.g. signed manifest, or hash chained to a value not co-located with the artifact); at minimum document that equipment.json integrity depends on `.context/` being no more writable than the code itself.

- **[MED] Impersonation guard only reserves TRUSTED seed names; untrusted-marketplace identity is repointable without re-gate** — `_RESERVED_NAME_REPOS` is built `for seed in SEED_MARKETPLACES if seed.trusted` (`discover.py:57-59`), so untrusted seed names (`superpowers`, `ecc`, `wshobson-agents`, `obsidian-skills`, `marketplace.py:55-63`) are NOT reserved. `enable_plugin` keys settings by the user-typed `@marketplace` *name* token (`discover.py:490`), and `_already_enabled_in` treats that name being present in settings as **prior approval that waives the `--yes` re-gate** (`discover.py:454`, `:540-556`). Attack path: a GitHub-discovered repo claims `name: "superpowers"` in its `marketplace.json`; `_admit` accepts it (no reserved-name block for untrusted names — `discover.py:107-114` only guards reserved ones), and although first-wins ordering lets the real seed win *if it was fetched*, the name→repo binding written to `extraKnownMarketplaces` (`claude_plugins.py:102-121`) carries no integrity check and can be repointed by a later `add_marketplace` of the same name to a different repo (`add_marketplace` happily overwrites `block[name]`, `:117-120`). Because re-approval keys on the name (`_already_enabled_in`), the repointed (attacker) repo inherits the prior approval and re-installs without `--yes`. Fix: reserve ALL seed names (drop the `if seed.trusted` filter, or maintain a separate untrusted-seed reservation), and bind `_already_enabled_in` prior-approval to the (name, repo) pair, not the name alone.

## Product surface

- **`equip discover` (no query, fresh repo) — silently narrow auto-match** — `run_discover` with no positional query falls to `_needed_caps` (`discover.py:267-276`), which emits at most `("test","implement")` and only when a test_runner / non-generic label is detected. A user who runs bare `equip discover` to "see what's available" gets candidates filtered to those two caps with no on-screen statement that the result is auto-scoped — the printed plan (`discover.py:356-371`) never says "showing matches for capabilities: test, implement". Gap: the headline result silently omits security/db/perf/docs plugins the user might be hunting for; nothing tells them to pass a query to widen it.

- **`equip discover` hidden network cost / latency** — every `discover` (and every `install`, which re-runs `_collect_catalogs` on the plugin name, `discover.py:410`) fans out: a `gh`-gated `fetch_catalog` per seed marketplace (`discover.py:136-142`), a fetch per `--repo`, per declared marketplace, plus a `search_github` when a query is present (`discover.py:174-190`). That is N+ sequential GitHub API round-trips on the foreground path with no progress output, no `--offline` flag, and no timeout surfaced to the user. Gap: on a slow link or rate-limited token the command hangs with zero feedback; the only network caveat printed is the post-hoc degraded-search warning (`discover.py:177-183`).

- **`install` runs the full discovery fan-out just to resolve one exact target** — `run_install` calls `_collect_catalogs(plugin_name, …)` (`discover.py:410-412`) which triggers `search_github` on the plugin name and fetches every seed + declared marketplace, then throws all of it away except the exact `name==marketplace && name==plugin` match (`discover.py:419-433`). Gap: installing a known `plugin@marketplace` pays the same multi-fetch latency + GitHub-search cost as open-ended discovery, surprising for what reads like a direct install.

- **`install` success is a half-truth — "enabled" ≠ "working"** — `run_install` prints `equip install: enabled <target> (native)` (`discover.py:511`) on exit 0, but the plugin is only *declared* in settings; it does not load until a Claude Code restart. The follow-up `note:` (`discover.py:512-515`) and the separate `equip verify` (`plugin_state.py:78-116`) exist precisely because exit 0 here does not mean the plugin runs. Gap: a scripted/`--for-proposal` caller that keys on rc 0 will treat a not-yet-loaded plugin as ready; the success line leads with "enabled" before the caveat.

- **`install` usage-doc gate blocks the obvious one-liner** — a plain `equip install foo@bar` exits rc 2 with "a plugin install needs a usage playbook" (`discover.py:239-246`, via `_validate_usage_doc`). The first-time user who just wants to install a plugin must learn `--skip-usage-doc` or author a doc. Gap: a mandatory flag with no default surprises the direct-CLI user; the message names the `/dummyindex-equip` council as the writer, which a bare-CLI user has no context for.

- **`install --scope user` silently drops manifest tracking** — at `discover.py:499` only `None/project/local` scopes record into `equipment.json`; a `--scope user` install wires settings but writes nothing to the manifest (by design, `discover.py:495-498`). The success line still prints the same "enabled … (native)" with no hint that this install is untracked and therefore invisible to `equip status` / `refresh` / `remove`. Gap: a user who later runs `equip remove foo@bar` or `equip status` will not see their user-scoped plugin and may conclude the install failed.

- **`install` ambiguity refusal gives no resolution path** — when a `plugin@marketplace` matches multiple repos, `run_install` prints `… is ambiguous across repos [...]; refusing — disambiguate the marketplace` (`discover.py:435-441`) and exits 1. But the only disambiguator offered downstream is `--repo`, and the message says "disambiguate the marketplace" without telling the user *how* (there is no `--repo` mentioned here, unlike the not-found branch at `discover.py:444-449`). Gap: the error states the problem but the named remedy ("disambiguate the marketplace") is not an actual flag.

- **`equip` verbless — `equip <path>` positional is silently inert** — `_split_verb` (`dispatch.py:146-161`) treats any non-verb first token (including a path) as "no verb", and a verbless non-`--dry-run` call just prints the verb-required error (`dispatch.py:105-109`). The plan's verb table (`dispatch.py:9`) and skill docs advertise `equip apply [path]`, so a user typing `equip ./sub` expecting it to apply to a subdir instead gets a usage error with no acknowledgement that they passed a path. Gap: a path-shaped argument is dropped into the generic "requires a verb" message rather than "did you mean `equip apply ./sub`?".

- **`apply` summary line is a dense counter dump** — the non-JSON apply footer (`dispatch.py:539-546`) concatenates seven counts ("wrote N, adopted N (manifest-only), wired N hook event(s), kept N prior record(s), skipped N, preserved N user-modified, kept N evolved"). Two different "kept" counts (prior records vs evolved) appear in one line. Gap: the at-a-glance result is hard to parse and ambiguous about which "kept" is which; a user can't quickly tell whether anything they cared about was skipped.

- **`refresh` reads `RefreshReport` as an unimported name** — `_print_refresh_report` is annotated `report: RefreshReport` (`verbs.py:35`) but `RefreshReport` is never imported in `verbs.py` (imports end at line 24). It survives only because `from __future__ import annotations` (`verbs.py:8`) makes the annotation a string. Gap (latent): any tooling that evaluates annotations (typing.get_type_hints, a future runtime-check) raises `NameError`; this is a real broken reference even though the happy path runs.

- **`status` "incomplete (no usage playbook)" is opaque** — `run_status` lists items with `missing_playbook` as `incomplete  <name>  (no usage playbook)` (`verbs.py:101-102`). For a plugin a user installed via `--skip-usage-doc`, this surfaces later as a permanent "incomplete" with no hint of how to clear it (no "run equip patch / add --usage-doc" remedy). Gap: a state label that looks like an error but has no offered fix; the user installed it deliberately and now sees a nag with no resolution.

- **`reset` / `patch` rc 1 vs rc 2 boundary is invisible to the user** — `reset` of an item not in the current catalog exits 1 (`verbs.py:146-151`), while an unknown specialist on `apply --specialist` exits 2 (`dispatch.py:180-182`); `patch` validation errors exit 2 but a domain PatchError exits 1 (`verbs.py:252-268`). These are internally principled (usage vs runtime) but the messages don't telegraph the distinction. Gap: a scripting user can't predict from the message whether they passed a bad flag (fixable) or hit a real domain failure (state-dependent).

- **`verify` "loaded: NO" can't distinguish never-installed from not-yet-loaded** — `run_verify` reports `loaded: NO — declared but not registered` and exits 1 (`plugin_state.py:111-116`) whenever the target is absent from `installed_plugins.json`. But it reaches this same branch whether the user just installed-and-hasn't-restarted, or never installed at all, or typo'd the marketplace name. The marketplace-clone line (`plugin_state.py:103`) is the only differentiator and it's easy to miss. Gap: the actionable "restart Claude Code" next-step is printed even when the real cause is a wrong target name, sending the user to restart fruitlessly.
