## Data integrity

- `dummyindex/context/domains/config.py:378-405` + `dummyindex/context/domains/config.py:622-658` — every non-empty v2/v3 `wired` ledger becomes `default_plugins_enabled=True`, after which reconciliation appends every missing reviewed default; a legacy custom-only ledger or one from which a user removed defaults therefore changes meaning on migration — split “reconcile declared wired entries” from “backfill reviewed defaults,” or persist a distinct migration state based on actual reviewed-target membership.
- `dummyindex/context/domains/config.py:403-421` + `dummyindex/context/domains/config.py:641-658` — the v2/v3 empty-ledger migration identifies a Codex baseline only by coincidental field values, so an intentionally empty Claude config with the same model/hook/profile values is converted from opt-out to `None` and then enabled on the next Claude/both pass — persist host provenance for new configs and treat legacy matches as ambiguous rather than auto-opting in.
- `dummyindex/cli/equip/install.py:171-210` + `dummyindex/cli/equip/install.py:449-489` — native install commits settings, equipment manifest, and `config.wired` sequentially, suppresses failures in the last two stores, and still returns success; this can leave settings+config without lifecycle ownership or settings+manifest without declared intent — use a staged transaction/compensating rollback and return a non-zero partial-state result until all in-repo ledgers commit.
- `dummyindex/cli/equip/install.py:391-441` — vendoring commits `SKILL.md` before `_record_vendored`, so a manifest write failure leaves an unmanaged file carrying dummyindex's sentinel and the uncaught `EquipError` does not restore the prior bytes — stage both writes and publish the manifest last with rollback of the file on failure.
- `dummyindex/context/domains/equip/lifecycle/manifest.py:25-51` + `dummyindex/context/domains/equip/models.py:236-245` + `dummyindex/cli/equip/install.py:258-332` — the equipment loader accepts any numeric `schema_version`, drops unknown top-level fields, and subsequent install recorders rewrite the result as the current schema, allowing a future manifest to be silently downgraded — reject versions newer than `SCHEMA_VERSION` before deserializing or rewriting.
- `dummyindex/context/domains/equip/models.py:236-245` — `EquipmentManifest.from_dict` silently filters non-object `items`, so a partially corrupt ledger loads as a smaller valid manifest and the next write permanently loses the rejected ownership records — require `items` to be a list of objects and fail closed on the first invalid element.
- `dummyindex/context/claude_settings.py:175-179` + `dummyindex/context/domains/equip/lifecycle/manifest.py:54-69` + `dummyindex/context/domains/config.py:662-681` + `dummyindex/cli/equip/dispatch.py:596-600` — settings, manifest, config, and generated equipment files all use shared deterministic `.tmp` names and unlocked read-modify-write cycles, so concurrent SessionStart/manual equip runs can collide on a temp file, publish the wrong payload, or lose another run's plugin/manifest entry — use unique same-directory temp files plus a per-target interprocess lock or compare-and-swap around each full read/merge/replace operation.
- `dummyindex/context/default_plugins.py:62-108,149-202,392-406,645-729` + `dummyindex/cli/equip/install.py:131-145,171-210,258-286` + `dummyindex/context/claude_plugins.py:105-124` + `dummyindex/cli/equip/plugin_state.py:107-145` — native plugin identity is not durably bound to the bytes that were approved: `config.wired` and prior-approval checks key only on `<plugin>@<marketplace>`, marketplace settings can overwrite that name with another repo, dynamic manifest records always leave `origin_ref=None`, the official default has no repo/ref record, and `verify` ignores the registry's `gitCommitSha`; re-registration can therefore replace source lineage under the same logical target and no later read can prove the loaded revision matches approval — persist and compare `(target, repo, commit)` across config, settings, manifest, and registry verification, requiring reapproval before any identity change.

## Security

- **[HIGH] Repository-controlled fixed temporary filenames are followed as symlinks before the atomic rename** — `dummyindex/context/claude_settings.py:175-179`, `dummyindex/context/domains/equip/lifecycle/manifest.py:54-69`, `dummyindex/context/domains/config.py:662-681`, `dummyindex/cli/equip/dispatch.py:596-605`, and `dummyindex/context/domains/atomic_io.py:12-25`. Attack scenario: a malicious repository commits `.claude/settings.json.tmp`, `.context/equipment.json.tmp`, `.context/config.json.tmp`, or a generated `<tool>.md.tmp` as a symlink to any user-writable external file; a normal `equip install`/`apply` write calls `Path.write_text` on that symlink, truncating and replacing the external target's contents before `replace()` moves the symlink over the intended in-repo destination. Mitigation today: none; parent creation, exception cleanup, and atomic rename occur after the unsafe followed write and do not confine it. Create a unique same-directory temporary file with exclusive/no-follow semantics, validate its parent beneath the selected root, and rename the already-open inode rather than writing a predictable path.

- **[HIGH] Untrusted native installs approve attacker-authored metadata, then execute different moving-HEAD bytes** — `dummyindex/context/domains/equip/plugins/marketplace.py:17-24,123-140`, `dummyindex/context/domains/equip/plugins/blast_radius.py:33-37`, `dummyindex/cli/equip/discover.py:333-378`, `dummyindex/cli/equip/install.py:158-178,258-286`, and `dummyindex/cli/equip/plugin_state.py:107-145`. Attack scenario: a malicious marketplace omits `hooks`, `mcpServers`, `lspServers`, and `bin`, so discovery labels the plugin `inert`; after the user supplies the load-bearing `--yes` approval required by `dummyindex/context/domains/equip/plugins/install_plan.py:36-48`, the attacker changes HEAD and the enabled native marketplace later supplies an executable hook/bin because settings and the ownership record contain no commit ref, while `equip verify` returns success for the first registry record without comparing its available `gitCommitSha`. Mitigation: the untrusted-source approval gate prevents silent first install, but it neither authenticates the disclosure, binds fetched bytes, nor detects later revision drift; resolve and persist a full commit SHA before approval, wire and inspect that exact payload, conservatively disclose untrusted native plugins as code-running when inspection is unavailable, and compare the loaded registry SHA during verification.

- **[HIGH] Marketplace-name repointing lets a new untrusted repo inherit an old approval and erase the prior source lineage** — `dummyindex/cli/equip/discover.py:49-57,105-149,164-207`, `dummyindex/cli/equip/install.py:131-145,171-178,239-286`, `dummyindex/context/claude_plugins.py:105-124`, and `dummyindex/context/default_plugins.py:62-108`. Attack scenario: after `skill@superpowers` was approved, the real untrusted seed fetch fails and an explicit or already-declared attacker repo publishes the same unreserved marketplace name; `_admit` accepts it, `_already_enabled_in` sees only the unchanged target string and waives `--yes`, `add_marketplace` overwrites the name-to-repo binding, and `_record_native` replaces the manifest record with the attacker's repo while keeping `origin_ref=None`, leaving `config.wired`'s target unchanged. Mitigation: trusted seed names and same-pass duplicate names are guarded, but untrusted seed identities, prior approval, and durable intent are not source-bound; reserve every seed name and require reapproval whenever persisted `(marketplace, repo, commit)` differs from the candidate, retaining an auditable identity transition rather than overwriting lineage.

## Product surface

- **Misspelled or extra `equip install` arguments** — `run_install` consumes known flags, selects the first remaining token containing `@`, and never rejects the rest (`dummyindex/cli/equip/install.py:62-87`); a typo such as `--scpoe user` is ignored and the default project settings path is mutated (`dummyindex/cli/equip/install.py:54-59,171-178`) — gap: unlike the other equip verbs, a mutating install can succeed at the wrong scope instead of returning a usage error for unknown input.

- **A caller runs `equip install` or `equip apply` in a cloned repository containing a predictable `.tmp` symlink** — settings, config, manifest, and generated-file writers call `write_text` on the repository-controlled symlink before renaming it, so the command can overwrite an external user-writable file and then report only the intended in-repo destination (`dummyindex/context/claude_settings.py:175-179`, `dummyindex/context/domains/equip/lifecycle/manifest.py:54-69`, `dummyindex/context/domains/config.py:662-681`, `dummyindex/cli/equip/dispatch.py:596-605`) — gap: there is no pre-write target preview, containment failure, or confirmation that reveals the actual followed destination.

- **A legacy v2/v3 repository with a custom-only non-empty `wired` ledger, or an intentionally empty Claude ledger matching the old Codex baseline fields, crosses a Claude/both reconciliation** — the loader infers the former as `default_plugins_enabled=True` and the latter as `None`; reconciliation then appends every missing reviewed default and persists `True` (`dummyindex/context/domains/config.py:378-421,622-658`) — gap: migration guesses host/plugin intent from ledger shape and coincidental field values, so an upgrade can opt the project into unrelated defaults without a preview, prompt, or migration-specific output.

- **A caller uses `equip verify` for the documented supply-chain check after installation or re-registration** — the handler discards `--root`, reads only the first `installed_plugins.json` record, and reports clone presence plus loaded scope/version without comparing its available `gitCommitSha` to any approved repository/ref (`dummyindex/cli/equip/plugin_state.py:87-145`, `dummyindex/cli/equip/install.py:258-286`) — gap: `docs/COMMANDS.md:172` and `docs/guide/07-cli.md:468-470` promise pinned-SHA drift verification, but the command can return success for changed upstream bytes, a repointed marketplace, or a record loaded in a different project; the code wins and those public descriptions are stale.

- **A caller re-runs `equip install <plugin>@<marketplace>` after discovery resolves that same target string to a different repository** — `_already_enabled_in` treats the old enabled key as approval, prints `already enabled ... re-registering`, skips the `--yes` gate, overwrites the marketplace binding, and replaces the manifest origin repository while leaving `origin_ref=None` (`dummyindex/cli/equip/install.py:131-145,171-210,239-286`, `dummyindex/context/claude_plugins.py:105-124`) — gap: no source-change diff, confirmation, or retained history tells the caller that “re-registering” changed providers rather than refreshed the approved one.

- **A plugin is installed with `--scope user`** — settings are written under `~/.claude`, but manifest and `config.wired` recording are skipped while the same `enabled ... (native)` success shape is printed (`dummyindex/cli/equip/install.py:54-59,171-215`) — gap: the install is invisible to project `status`, `remove`, and lifecycle reconciliation, with no output warning; `docs/guide/07-cli.md:433-437` incorrectly says the native mechanism records a marketplace item without documenting this scope exception.

- **A project explicitly disables one reviewed default with `enabledPlugins[target]=false`** — `wire_default_plugins` correctly preserves the tombstone and the materializer skips it, but the result is put in `already` and rendered as `<target> already enabled (left as-is)` (`dummyindex/context/default_plugins.py:293-313,478-529,680-699`) — gap: install/init output states the opposite of the effective default-plugin decision, so users cannot confirm that their per-target opt-out was honored.

- **A native plugin is installed with the documented `--skip-usage-doc` opt-out** — validation accepts the opt-out and records no grounding, then every later `status` classifies that marketplace item as `incomplete (no usage playbook)` (`dummyindex/cli/equip/install.py:258-286,492-518`, `dummyindex/context/domains/equip/lifecycle/status.py:198-211`, `dummyindex/cli/equip/verbs.py:96-118`) — gap: “skip” suppresses the install gate but not the permanent incomplete state, and status gives no command for attaching a playbook afterward.

- **Settings wiring succeeds but manifest or config write-back fails** — the native installer prints a warning, continues to its normal enabled message, and returns `0` (`dummyindex/cli/equip/install.py:171-215,449-489`) — gap: automation and the final success line cannot distinguish fully tracked installation from the partial states identified by the data-integrity critic, even though later lifecycle commands may be unable to see or remove the plugin.

- **A vendored-skill install writes `.claude/skills/<name>/SKILL.md` and then its manifest update fails** — the file write commits before `_record_vendored`, the resulting `EquipError` is not rolled back at the install boundary, and the skill remains on disk without a lifecycle record (`dummyindex/cli/equip/install.py:391-446`, `dummyindex/context/domains/equip/lifecycle/manifest.py:54-69`) — gap: the failed command leaves usable-looking but unmanaged residue with no repair/resume instruction, so `status`, refresh, and removal cannot explain or own what was just written.

- **A caller omits the mandatory usage-playbook flag on a known target** — install performs tool probing, sequential seed/declared catalog fetches and a GitHub search before `_validate_usage_doc` returns the deterministic usage error (`dummyindex/cli/equip/install.py:89-148`, `dummyindex/cli/equip/discover.py:105-225`, `dummyindex/context/domains/equip/plugins/sources.py:34-49,99-112`) — gap: invalid syntax can incur many foreground network calls, each with a 30-second timeout, and may first surface a missing-approval error; validation should precede discovery I/O.

- **An untrusted catalog declares an inert plugin** — install planning still requires `--yes` for every untrusted source, and discovery prints that requirement alongside `inert` (`dummyindex/context/domains/equip/plugins/install_plan.py:36-48`, `dummyindex/cli/equip/discover.py:333-378`) — gap: `docs/COMMANDS.md:171` and `docs/guide/07-cli.md:433-439` say the gate applies to untrusted code-running plugins, so the conservative code policy is correct but the documented boundary is narrower and surprises callers; the code wins.

- **A fresh repository runs `equip status`** — the empty-state message says `run \`equip\` first`, while a bare equip invocation intentionally exits `2` and only suggests `equip apply` (`dummyindex/cli/equip/verbs.py:107-109`, `dummyindex/cli/equip/dispatch.py:102-152`) — gap: the only next step printed by the empty state is a command that cannot create tracked items.

## Cross-review notes

- **Amplifies the Security `origin_ref=None` / moving-HEAD finding (concerns.md:13) from a UX angle** — the product cost of that pin gap is that `install` shows the user a blast-radius disclosure (`discover.py:356-371`, `:455-462`) computed from a `marketplace.json` fetched at discovery time, while the *actual* fetch happens later at Claude Code session start against moving HEAD. The user approves bytes they will never receive, and the only thing standing between approval and execution is a restart they're explicitly told to perform. The disclosure UX actively builds false confidence — a clean reason to pin the ref beyond the security argument.

- **Amplifies the Data-integrity concurrency finding (concerns.md:6) — apply uses the same shared-tmp pattern** — `_write_file` (`dispatch.py:573-583`) derives `tmp = target.with_suffix(suffix + ".tmp")`, the same fixed-name tmp the integrity reviewer flagged for the manifest. A SessionStart-triggered apply racing a manual `equip apply` can publish a torn `.claude/<agent>.md`, and the user sees no lock contention error — just a silently corrupt agent file on next session. Confirms the no-concurrent-run assumption is undocumented at the CLI surface, not only in the manifest writer.
