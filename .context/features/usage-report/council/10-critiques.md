## Data integrity

- **Non-numeric `usage` field crashes the whole read, defeating best-effort.** `_turn_from_line` does `int(usage.get("input_tokens") or 0)` etc. (`dummyindex/usage/transcripts.py:138-141`). `int()` on a float-string (`"12.5"`) or non-coercible str raises `ValueError`, but `_read_file`'s try/except only guards `json.loads` (`transcripts.py:165-168`) and the `open` (`OSError`, `156-159`) — the `_turn_from_line` call at `174-176` is unguarded. One malformed numeric value aborts the generator mid-file, silently truncating every later turn in that transcript (and, for `load_session`, the whole report stops). Contradicts the plan's "a single bad line shouldn't sink a read" claim (`plan.md:107-109`). Coerce defensively (per-field try/except → 0) or wrap the `_turn_from_line` call.

- **Dedup keeps FIRST line per `message.id`, but streaming usage is cumulative on the LAST line — risks undercount.** `_dedup_key` = `message.id|requestId` (`transcripts.py:100-111`); `_read_file` adds the key to `seen` on first yield and skips all later lines with that key (`172-181`). Claude Code rewrites the same assistant message across lines as the stream completes; if the authoritative/final token usage lands on a *later* line for the same `message.id`, first-wins undercounts. The dedup is correct for "count once" but assumes every line for a key carries identical usage — that invariant is unstated and unverified. Confirm against a real multi-line message, or switch to last/max-per-key.

- **Global seen-set in `iter_all_turns` can collapse distinct turns sharing a fallback key across projects.** `iter_all_turns` uses one `seen` set spanning every main + subagent file (`transcripts.py:243-262`), whereas `load_session` scopes main/sub independently (`204-227`). For lines lacking `message.id`, the key falls back to line `uuid` (`_dedup_key:110-111`) — `uuid` is globally unique so that's safe — but any two turns that legitimately share a `message.id` across *different sessions* (forked/resumed transcripts re-emit ids) are counted once globally. That is the intended dedup, yet it means a genuinely re-issued turn with *new* usage is dropped. The correctness of cross-project totals hinges on "same `message.id` ⇒ same tokens," which is asserted nowhere in code or tests.

- **`find_main_transcript` tie-breaks only on `st_mtime`, so equal-mtime matches are non-deterministic.** `transcripts.py:70-74` (and the no-session fallback `77-79`) sort by `st_mtime` reverse and take `[0]`. Two transcripts with identical mtime (same `session_id` under two project slugs, or copied files) yield an OS/filesystem-order-dependent winner — the report silently picks a different session run-to-run. Add a stable secondary sort key (e.g. path) for determinism.

- **`load_session` indexes `main_turns[-1]`/`min`/`max` over file order, not timestamp order.** `chat_report` takes `window_now = window_tokens(main_turns[-1])` (`aggregate.py:95`) — "most recent main turn" — but `main_turns` is in *file line order*, not timestamp-sorted (`load_session:205-213`). If a transcript is appended out of chronological order (resume/fork interleaving), `[-1]` is not the latest turn and `window_now` is wrong. `started`/`last` correctly use `min`/`max` over timestamps (`aggregate.py:107-108`), but `window_now` assumes monotonic file order — an unguarded assumption.

- **Timestamp parse normalises `Z` via blind `replace`, corrupting any literal `Z` mid-string.** `_parse_timestamp` does `raw.replace("Z", "+00:00")` (`transcripts.py:92`) — replaces *every* `Z`, not just a trailing offset marker. A pathological timestamp like `2024Z01...` would be mangled; in practice Claude emits clean ISO-8601 so impact is low, but the transform is not anchored to the end. Prefer `raw.removesuffix("Z") + "+00:00"` (3.9+) or slice the final char.

- **`chat_report` reads `main_turns[-1]` after an `if main_turns` guard but `infer_context_limit(peak_window=0)` returns 200K for an empty session** — a session with only subagent turns and zero main turns reports `context_limit=200000` and `window_now=0` (`aggregate.py:95-100`), which is a sane floor, not a bug; noting for completeness that the empty-main path is handled.

## Security

Threat model — the usage feature ingests `~/.claude/projects/**/*.jsonl`. Content (model strings, usage numbers, timestamps) is partly assistant/tool-output-derived, i.e. influenceable by anything that appeared in a chat. Treat every parsed field as untrusted at the render boundary.

- **[HIGH] Terminal-escape injection via the `model` field into rendered report.** `model = message.get("model") or ""` is taken verbatim from transcript JSON (`transcripts.py:127`), carried into `TurnUsage.model` (`:137`), and printed *raw* as the first table cell — `_totals_row(usage.model, …)` (`render.py:84`) → `_table` `.ljust`+join (`render.py:31-33`) → `print(text)` (`__main__.py:97`). Nothing strips control characters. Attack: a transcript line `{"type":"assistant","message":{"model":"\x1b[2J\x1b[1;1H FAKE…","usage":{…}}}` (writable by a malicious MCP/tool whose output Claude Code persists, or a hand-edited/synced transcript) renders ANSI escapes to the terminal on `dummyindex usage`. Clear-screen / cursor-move / OSC-8 hyperlink / `\r`-overwrite let an attacker forge a usage table, hide rows, or impersonate a `TOTAL` line — masking unexpected token spend. `_c()` numbers are safe; `model`/`session_id`/`project` are not. Fix: strip/escape C0/C1 controls at the render boundary before any transcript-derived string enters a cell.

- **[MED] `project` column rendered unsanitized.** `render_sessions` prints `_shorten(bucket.project, 28)` (`render.py:179`); `project = main_transcript.parent.name` (`transcripts.py:203,246`). `encode_project_slug` cleans slugs this tool writes (`:49-51`), but the glob trusts pre-existing directory names — a dir name with embedded escapes/newlines (planted by another tool/archive/synced `.claude`) flows into the table. Same sanitization fix.

- **[MED] Unbounded single-line read → memory-exhaustion DoS.** Module claims streaming "never slurps" (`transcripts.py:4-5`) and iterates `for line in handle` (`:161`) — but with no per-line length cap. A transcript with one multi-GB line (no `\n`) is read whole into one `str`, then `line.strip()` (`:162`) + `json.loads` (`:166`). The per-project kinds run this over *every* file via `iter_all_turns` (`:244-262`), so one planted oversized file OOM-kills the report. Fix: bound line length / skip over-long lines, consistent with the existing skip-unparseable stance (`:167-168`).

- **[LOW/MED] `json.loads` recursion DoS.** Same untrusted line feeds `json.loads` (`:166`), guarded only by `JSONDecodeError`/`ValueError` (`:167`). Deeply-nested JSON raises `RecursionError`, which is *not* a `ValueError` subclass → **uncaught**, aborting the whole `iter_all_turns`/`load_session` generator (same best-effort-defeating shape as the `int()` Data-integrity issue). Fix: cap nesting / broaden the parse guard to catch `RecursionError`.

- **[LOW] Arbitrary-file-read via "transcript path" largely N/A.** The CLI exposes no path arg — `_run_usage` parses only a report-kind and rejects extra args (`__main__.py:67-84`); paths are always glob-derived under `projects_root` (`find_main_transcript:69-80`, `iter_all_turns:244`). No direct traversal sink. Only seam is `CLAUDE_CONFIG_DIR`/`$HOME` → `default_projects_root` (`:36-40`): controlling that env repoints the scan, but that needs prior environment control (informational). `session_id` is used only inside a glob pattern (`:71`); glob doesn't traverse `..`/absolute out of root here. No fix; flagged so a future path-accepting variant adds `Path.resolve()` + containment.

## Product surface

- **No-session error is correct but the message blames the user for an env var Claude Code owns.** `dummyindex usage` (chat) with no `$CLAUDE_CODE_SESSION_ID` and no cwd transcript exits 1 with `could not identify the current session: $CLAUDE_CODE_SESSION_ID is unset and no transcript exists under <slug>` (`report.py:69-73`). A human running `dummyindex usage` directly in a terminal (not via the `/tokens` slash command, which injects the env var) hits this every time — the message reads like a bug report about an internal variable, not an actionable instruction. Gap: no hint that this command is meant to run inside a Claude Code session, nor a fallback suggestion (e.g. "try `dummyindex usage session` for cross-session history"). The non-chat kinds work fine without the env var, but the error doesn't say so.

- **Brand-new-session "empty" chat renders a confusing all-zero table with no explanation.** When `session_id` is known but its transcript isn't flushed, `_chat` builds `chat_report(session_id, (), (), subagent_count=0)` (`report.py:62-68`) → `render_chat` prints `Context window now 0 tokens (≈0% of 200K …)`, a bare `TOTAL  0 0 0 0 0` row (no per-model rows since `by_model` is empty), `started`/duration omitted, and `subagents: none` (`render.py:81-128`, `aggregate.py:95-108`). Gap: a user sees a zero-filled report with no "session not flushed yet / no turns recorded" note — indistinguishable from a real session that genuinely spent zero tokens, and the design intent ("renders empty, never a sibling's numbers" — `spec.md:30-33`) is invisible to the reader.

- **`(deduplicated; cache re-reads counted each turn)` caption is the only on-screen explanation of dedup — and it conflates two different ideas users will misread.** The chat table header is `Session cumulative — by model (deduplicated; cache re-reads counted each turn)` (`render.py:122-123`). "Deduplicated" (counting one logical turn once across rewritten/forked lines — `spec.md:9-11`) and "cache re-reads counted each turn" (a billing/accounting choice about `cache_read_tokens`) are unrelated, but the parenthetical glues them together with a semicolon, implying they're the same caveat. Gap: a user trying to reconcile these numbers against Anthropic billing has no way to learn *what* was deduplicated or *why* their cumulative total may exceed naive line-summed totals; daily/session/monthly/blocks tables carry **no** dedup caption at all (`render.py:131-203`), so the accuracy caveat silently disappears on 4 of 5 report kinds.

- **`(matches /context)` / `(matches /context)` window claim is an unverifiable promise the tool can't keep on inferred limits.** The window line asserts `(≈{pct}% of {limit} · main thread · matches /context)` (`render.py:92-95`). The denominator is *inferred* — smallest of `(200K, 1M)` holding peak window (`aggregate.py:100`, `infer_context_limit`) — not the model's real advertised window. A 1M-context session whose peak never exceeded 200K reports `% of 200K`, and `window_now` is `window_tokens(main_turns[-1])` over **file order, not timestamp order** (flagged in Data integrity, `aggregate.py:95`). Gap: the literal "matches /context" claim will be wrong whenever the inferred tier differs from the live model's window or file order isn't chronological, yet the text states it as fact — a user who spots the mismatch will distrust the whole report.

- **`>100%` window overflow has no clamp — a >1M session prints a nonsensical percentage.** `pct = round(window_now / context_limit * 100)` (`render.py:87-91`) with `context_limit` capped at the largest tier 1M (`infer_context_limit`, `aggregate.py`). The plan itself flags this open question (`plan.md:131-133`): a session exceeding 1M reports `≈>100%` with no clamp. Gap: the percentage — the single headline number of the `/tokens` view — can read e.g. `≈137%`, which looks like a bug to a user and undermines confidence in the figure that's supposed to "match /context."

- **Help text and the doc-hinted capability set list five report kinds with zero hint of what each shows or that 4 need no session.** `dummyindex usage -h` prints exactly `Usage: dummyindex usage [chat|daily|session|monthly|blocks]` (`__main__.py:70`) — kind names only, no one-line description, no note that `chat` requires a live session while the other four scan all projects. Gap: a user discovering the command via `--help` cannot tell `blocks` (5-hour billing windows) from `session` (per-chat rows) without trial and error; the richer guidance exists only in the `/tokens` skill prose (`skills/commands/tokens.md:14-15`), invisible to anyone using the bare CLI. The spec's per-kind descriptions (`spec.md:27-39`) are not surfaced anywhere the end user sees them.

- **Truncated 8-char session ids and trailing-anchored `…`-truncated project names can collide / mislead in multi-session tables.** `render_sessions` shows `bucket.session_id[:8]` (`render.py:178`) and `_shorten(bucket.project, 28)` which keeps the *tail* and prepends `…` (`render.py:210-211`). Gap: two sessions sharing an 8-char prefix render as the same id with no disambiguator, and two projects differing only in a long common suffix (e.g. `…/projects/app-frontend` vs `…/app-backend` truncated to a shared tail) become visually indistinguishable — a user cannot reliably map a row back to a real session/project, and there's no `--full`/`--no-truncate` escape hatch.
