## Data integrity

- **Non-numeric `usage` field crashes the whole read, defeating best-effort.** `_turn_from_line` does `int(usage.get("input_tokens") or 0)` etc. (`dummyindex/usage/transcripts.py:138-141`). `int()` on a float-string (`"12.5"`) or non-coercible str raises `ValueError`, but `_read_file`'s try/except only guards `json.loads` (`transcripts.py:165-168`) and the `open` (`OSError`, `156-159`) â€” the `_turn_from_line` call at `174-176` is unguarded. One malformed numeric value aborts the generator mid-file, silently truncating every later turn in that transcript (and, for `load_session`, the whole report stops). Contradicts the plan's "a single bad line shouldn't sink a read" claim (`plan.md:107-109`). Coerce defensively (per-field try/except â†’ 0) or wrap the `_turn_from_line` call.

- **Dedup keeps FIRST line per `message.id`, but streaming usage is cumulative on the LAST line â€” risks undercount.** `_dedup_key` = `message.id|requestId` (`transcripts.py:100-111`); `_read_file` adds the key to `seen` on first yield and skips all later lines with that key (`172-181`). Claude Code rewrites the same assistant message across lines as the stream completes; if the authoritative/final token usage lands on a *later* line for the same `message.id`, first-wins undercounts. The dedup is correct for "count once" but assumes every line for a key carries identical usage â€” that invariant is unstated and unverified. Confirm against a real multi-line message, or switch to last/max-per-key.

- **Global seen-set in `iter_all_turns` can collapse distinct turns sharing a fallback key across projects.** `iter_all_turns` uses one `seen` set spanning every main + subagent file (`transcripts.py:243-262`), whereas `load_session` scopes main/sub independently (`204-227`). For lines lacking `message.id`, the key falls back to line `uuid` (`_dedup_key:110-111`) â€” `uuid` is globally unique so that's safe â€” but any two turns that legitimately share a `message.id` across *different sessions* (forked/resumed transcripts re-emit ids) are counted once globally. That is the intended dedup, yet it means a genuinely re-issued turn with *new* usage is dropped. The correctness of cross-project totals hinges on "same `message.id` â‡’ same tokens," which is asserted nowhere in code or tests.

- **`find_main_transcript` tie-breaks only on `st_mtime`, so equal-mtime matches are non-deterministic.** `transcripts.py:70-74` (and the no-session fallback `77-79`) sort by `st_mtime` reverse and take `[0]`. Two transcripts with identical mtime (same `session_id` under two project slugs, or copied files) yield an OS/filesystem-order-dependent winner â€” the report silently picks a different session run-to-run. Add a stable secondary sort key (e.g. path) for determinism.

- **`load_session` indexes `main_turns[-1]`/`min`/`max` over file order, not timestamp order.** `chat_report` takes `window_now = window_tokens(main_turns[-1])` (`aggregate.py:95`) â€” "most recent main turn" â€” but `main_turns` is in *file line order*, not timestamp-sorted (`load_session:205-213`). If a transcript is appended out of chronological order (resume/fork interleaving), `[-1]` is not the latest turn and `window_now` is wrong. `started`/`last` correctly use `min`/`max` over timestamps (`aggregate.py:107-108`), but `window_now` assumes monotonic file order â€” an unguarded assumption.

- **Timestamp parse normalises `Z` via blind `replace`, corrupting any literal `Z` mid-string.** `_parse_timestamp` does `raw.replace("Z", "+00:00")` (`transcripts.py:92`) â€” replaces *every* `Z`, not just a trailing offset marker. A pathological timestamp like `2024Z01...` would be mangled; in practice Claude emits clean ISO-8601 so impact is low, but the transform is not anchored to the end. Prefer `raw.removesuffix("Z") + "+00:00"` (3.9+) or slice the final char.

- **`chat_report` reads `main_turns[-1]` after an `if main_turns` guard but `infer_context_limit(peak_window=0)` returns 200K for an empty session** â€” a session with only subagent turns and zero main turns reports `context_limit=200000` and `window_now=0` (`aggregate.py:95-100`), which is a sane floor, not a bug; noting for completeness that the empty-main path is handled.

## Security

Threat model â€” the usage feature ingests `~/.claude/projects/**/*.jsonl`. Those transcripts are written by Claude Code, but their *content* (model strings, usage numbers, timestamps) is partly assistant/tool-output-derived, i.e. influenceable by anything that ever appeared in a chat. Treat every parsed field as untrusted at the render boundary. Severity tags below are HIGH/MED/LOW.

- **[HIGH] Terminal-escape injection via the `model` field into rendered report (info-leak / spoofing).** `model = message.get("model") or ""` is taken verbatim from transcript JSON (`transcripts.py:127`) and carried unsanitized into `TurnUsage.model` (`:137`), grouped by `by_model`, and printed *raw* as the first table cell â€” `_totals_row(usage.model, â€¦)` (`render.py:84`) â†’ `_table` left-justifies it with `.ljust` (`render.py:31`) and joins (`:33`), then `__main__` does `print(text)` (`__main__.py:97`). Nothing strips control characters. Attack path: any transcript line with `{"type":"assistant","message":{"model":"[2J[1;1H FAKE-BILLINGâ€¦","usage":{â€¦}}}` â€” writable by a malicious MCP/tool that can emit content Claude Code persists, or by a hand-edited/synced transcript file â€” renders ANSI escapes straight to the user's terminal when they run `dummyindex usage`. Impact: clear-screen / cursor-move / hyperlink (`OSE 8`) / color sequences let an attacker forge a benign-looking usage table, hide rows, or (with `\r`/overwrite) rewrite earlier output â€” a credible vector for masking unexpected token spend or impersonating a "TOTAL" line. `_c()` numbers are safe (int-formatted), but the `model` and the session/project columns are not. Fix: at the render boundary strip/escape C0/C1 controls (e.g. reject anything matching `[\x00-\x1f\x7f-\x9f]`, or `repr`-quote) before placing any transcript-derived string in a cell; do it for `model`, `project`, `session_id`.

- **[MED] `project` column is filesystem-derived but still rendered unsanitized.** `render_sessions` prints `_shorten(bucket.project, 28)` (`render.py:179`); `project` is `main_transcript.parent.name` (`transcripts.py:203`, `246`) â€” the project-slug directory name. `encode_project_slug` maps non-alphanumerics to `-` (`:49-51`), so slugs *this* tool creates are clean, but the glob (`*/*.jsonl`, `*/<sid>.jsonl`) trusts whatever directory names already exist under `projects_root`. A directory named with embedded escapes/newlines (planted by another tool, an archive extraction, or a synced `.claude` dir) flows straight into the table. Same sanitization fix as above covers it; lower severity because creating such a dir name needs filesystem write under `~/.claude/projects`.

- **[MED] Unbounded single-line read enables memory-exhaustion DoS on a malformed transcript.** The module advertises streaming "never slurps â€” some are tens of MB" (`transcripts.py:4-5`), and it does iterate `for line in handle` (`:161`) â€” but that yields *one line at a time with no length cap*. A transcript containing a single multi-gigabyte line with no `\n` (a corrupted/oversized JSON blob, or a deliberately crafted file) is read entirely into one `str` by the iterator, then handed to `line.strip()` (`:162`) and `json.loads` (`:166`) â€” peak memory â‰ˆ a fewÃ— the line size. For the per-project kinds (`daily/monthly/session/blocks`) this runs over *every* transcript in `projects_root` via `iter_all_turns` (`:244-262`), so one planted oversized file OOM-kills the whole report. Fix: bound line length (read in chunks / enforce a max bytes-per-line and skip over-long lines, consistent with the existing best-effort "skip unparseable" stance at `:167-168`).

- **[LOW/MED] `json.loads` on attacker-sized lines â€” CPU/recursion DoS, not just memory.** Same untrusted line feeds `json.loads` (`transcripts.py:166`), guarded only by `JSONDecodeError`/`ValueError` (`:167`). A deeply-nested JSON line can hit Python's recursion limit and raise `RecursionError` â€” which is *not* a subclass of `ValueError`, so it is **uncaught**, propagating out of the generator and aborting the entire `iter_all_turns`/`load_session` (same best-effort-defeating shape as the `int()` issue in Data integrity above). A pathological huge-but-valid line also burns CPU. Fix: pair the line-length cap with broadening the parse guard to also catch `RecursionError` (or cap nesting) so one crafted line is skipped, not fatal.

- **[LOW] Arbitrary-file-read via "transcript path" is largely N/A â€” but note the env-var seam.** The CLI exposes **no** path argument: `_run_usage` parses only a report-kind token and rejects extra args (`__main__.py:67-84`); the file path is always derived by globbing under `projects_root` (`find_main_transcript:69-80`, `iter_all_turns:244`). So there is no direct path-traversal sink. The one influence seam is `CLAUDE_CONFIG_DIR` â†’ `default_projects_root` (`transcripts.py:36-40`): an attacker who controls that env var (or `$HOME`) can repoint the scan at any directory and have its `*/*.jsonl` contents read and rendered â€” but that requires already controlling the victim's environment, so it is informational rather than a vuln in this code. `session_id` is used only inside a glob pattern (`f"*/{session_id}.jsonl"`, `:71`); `glob` does not interpret `..`/absolute paths as traversal here, so a hostile `$CLAUDE_CODE_SESSION_ID` cannot escape `projects_root`. No fix required; documented so a future path-accepting variant gets `Path.resolve()` + containment checks before it is added.

Net: the single material finding is **terminal-escape injection of transcript-derived strings into the rendered report** (HIGH) plus the **unbounded-line / uncaught-`RecursionError` DoS pair** (MED). Both are remediable purely at the trust boundary (render-time sanitization; per-line size + recursion guard) without changing the dedup/aggregation logic.

## Product surface

- **No-session error is correct but the message blames the user for an env var Claude Code owns.** `dummyindex usage` (chat) with no `$CLAUDE_CODE_SESSION_ID` and no cwd transcript exits 1 with `could not identify the current session: $CLAUDE_CODE_SESSION_ID is unset and no transcript exists under <slug>` (`report.py:69-73`). A human running `dummyindex usage` directly in a terminal (not via the `/tokens` slash command, which injects the env var) hits this every time â€” the message reads like a bug report about an internal variable, not an actionable instruction. Gap: no hint that this command is meant to run inside a Claude Code session, nor a fallback suggestion (e.g. "try `dummyindex usage session` for cross-session history"). The non-chat kinds work fine without the env var, but the error doesn't say so.

- **Brand-new-session "empty" chat renders a confusing all-zero table with no explanation.** When `session_id` is known but its transcript isn't flushed, `_chat` builds `chat_report(session_id, (), (), subagent_count=0)` (`report.py:62-68`) â†’ `render_chat` prints `Context window now 0 tokens (â‰ˆ0% of 200K â€¦)`, a bare `TOTAL  0 0 0 0 0` row (no per-model rows since `by_model` is empty), `started`/duration omitted, and `subagents: none` (`render.py:81-128`, `aggregate.py:95-108`). Gap: a user sees a zero-filled report with no "session not flushed yet / no turns recorded" note â€” indistinguishable from a real session that genuinely spent zero tokens, and the design intent ("renders empty, never a sibling's numbers" â€” `spec.md:30-33`) is invisible to the reader.

- **`(deduplicated; cache re-reads counted each turn)` caption is the only on-screen explanation of dedup â€” and it conflates two different ideas users will misread.** The chat table header is `Session cumulative â€” by model (deduplicated; cache re-reads counted each turn)` (`render.py:122-123`). "Deduplicated" (counting one logical turn once across rewritten/forked lines â€” `spec.md:9-11`) and "cache re-reads counted each turn" (a billing/accounting choice about `cache_read_tokens`) are unrelated, but the parenthetical glues them together with a semicolon, implying they're the same caveat. Gap: a user trying to reconcile these numbers against Anthropic billing has no way to learn *what* was deduplicated or *why* their cumulative total may exceed naive line-summed totals; daily/session/monthly/blocks tables carry **no** dedup caption at all (`render.py:131-203`), so the accuracy caveat silently disappears on 4 of 5 report kinds.

- **`(matches /context)` / `(matches /context)` window claim is an unverifiable promise the tool can't keep on inferred limits.** The window line asserts `(â‰ˆ{pct}% of {limit} Â· main thread Â· matches /context)` (`render.py:92-95`). The denominator is *inferred* â€” smallest of `(200K, 1M)` holding peak window (`aggregate.py:100`, `infer_context_limit`) â€” not the model's real advertised window. A 1M-context session whose peak never exceeded 200K reports `% of 200K`, and `window_now` is `window_tokens(main_turns[-1])` over **file order, not timestamp order** (flagged in Data integrity, `aggregate.py:95`). Gap: the literal "matches /context" claim will be wrong whenever the inferred tier differs from the live model's window or file order isn't chronological, yet the text states it as fact â€” a user who spots the mismatch will distrust the whole report.

- **`>100%` window overflow has no clamp â€” a >1M session prints a nonsensical percentage.** `pct = round(window_now / context_limit * 100)` (`render.py:87-91`) with `context_limit` capped at the largest tier 1M (`infer_context_limit`, `aggregate.py`). The plan itself flags this open question (`plan.md:131-133`): a session exceeding 1M reports `â‰ˆ>100%` with no clamp. Gap: the percentage â€” the single headline number of the `/tokens` view â€” can read e.g. `â‰ˆ137%`, which looks like a bug to a user and undermines confidence in the figure that's supposed to "match /context."

- **Help text and the doc-hinted capability set list five report kinds with zero hint of what each shows or that 4 need no session.** `dummyindex usage -h` prints exactly `Usage: dummyindex usage [chat|daily|session|monthly|blocks]` (`__main__.py:70`) â€” kind names only, no one-line description, no note that `chat` requires a live session while the other four scan all projects. Gap: a user discovering the command via `--help` cannot tell `blocks` (5-hour billing windows) from `session` (per-chat rows) without trial and error; the richer guidance exists only in the `/tokens` skill prose (`skills/commands/tokens.md:14-15`), invisible to anyone using the bare CLI. The spec's per-kind descriptions (`spec.md:27-39`) are not surfaced anywhere the end user sees them.

- **Truncated 8-char session ids and trailing-anchored `â€¦`-truncated project names can collide / mislead in multi-session tables.** `render_sessions` shows `bucket.session_id[:8]` (`render.py:178`) and `_shorten(bucket.project, 28)` which keeps the *tail* and prepends `â€¦` (`render.py:210-211`). Gap: two sessions sharing an 8-char prefix render as the same id with no disambiguator, and two projects differing only in a long common suffix (e.g. `â€¦/projects/app-frontend` vs `â€¦/app-backend` truncated to a shared tail) become visually indistinguishable â€” a user cannot reliably map a row back to a real session/project, and there's no `--full`/`--no-truncate` escape hatch.
