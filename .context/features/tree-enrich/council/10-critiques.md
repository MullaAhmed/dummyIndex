## Data integrity

- **Atomic writers bypass the one mandatory helper.** Both `write_plan` (`dummyindex/context/domains/enrich.py:182-185`) and the tree rewrite in `apply_updates` (`dummyindex/context/domains/enrich.py:219-221`) hand-roll `tmp + replace` instead of calling `write_text_atomic` (`dummyindex/context/domains/atomic_io.py:11-24`). The convention forbids this verbatim â€” "Don't bypass `write_text_atomic` with a plain `open(...,"w")` â€” you lose atomicity *and* break hash-baselining" (`.context/conventions/data-access.md:27`, doc HIGH; identifier `write_text_atomic` verified in `map/symbols.json` at `atomic_io.py:11`). The inline copies also omit the cache writer's Windows `PermissionError` copy-then-delete fallback (`data-access.md:7`), so a concurrent reader/AV lock on Windows can make `tmp.replace(tree.json)` raise mid-apply.

- **Non-deterministic serialization of the 2MB+ `tree.json`.** `apply_updates` emits `json.dumps(tree, indent=2)` with no `sort_keys=True` (`dummyindex/context/domains/enrich.py:220`). The convention requires "deterministic â€¦ `sort_keys=True`" and "Don't leak â€¦ unsorted dicts into a committed JSON; both break byte-identical re-runs" (`.context/conventions/data-access.md:13,28`, doc HIGH). `tree.json` is committed; key order here is whatever the deterministic ingest emitted, so apply happens to preserve it today, but any dict-key churn upstream yields a non-byte-stable rewrite and noisy diffs on a large file.

- **No `schema_version` validation on the `tree.json` read.** `apply_updates` does a bare `json.loads(tree_path.read_text(...))` (`dummyindex/context/domains/enrich.py:211`) and immediately walks `tree["root"]` with no version/required-field check. The convention states "Readers validate it: â€¦ raises if `schema_version` is newer than the build supports and rejects payloads missing required fields" (`.context/conventions/data-access.md:13`, doc HIGH). A `tree.json` written by a newer/older build, or one missing `root`, raises a raw `KeyError`/`TypeError` rather than a typed schema error â€” and the partial-write protection is the only guard against corruption, with no schema gate.

- **In-place mutation of the loaded tree (immutability + partial-state risk).** `_apply` mutates `node["abstract"]`/`node["confidence"]` directly on the parsed dict (`dummyindex/context/domains/enrich.py:270-271`), against the global immutability rule (new objects, never mutate). The single atomic `replace` (`:221`) means a crash leaves the *prior file* intact, so on-disk integrity holds; but if `json.dumps` raises after `_apply` returns (e.g. a non-serializable value injected upstream), the in-memory tree is already half-mutated â€” there is no all-or-nothing staging of the merged document before serialization.

- **`_enrich_plan.json` staleness â€” apply trusts tree membership, not plan membership.** `apply_updates` validates incoming ids only against `_collect_ids(tree["root"])` (`dummyindex/context/domains/enrich.py:213-215`), never against the plan that produced them. The plan's own Open question flags this (`.context/features/tree-enrich/plan.md:104-107`). Failure mode: after `dummyindex context rebuild --changed` regenerates `tree.json` (new/renamed node_ids), a stale `cache/_enrich_plan.json` from before the rebuild feeds node_ids that no longer exist â†’ silently routed to `ApplyResult.unknown` and dropped, losing authored abstracts; or worse, a reused-but-reassigned node_id applies an abstract to the wrong node with no signal. The plan is gitignored/transient (`cli/enrich.py:34-43`) with no generation stamp tying it to the current tree, so no freshness check is possible.

- **Idempotency holds; confidence promotion is correctly one-way.** `_apply` is a no-op when both `abstract` and `confidence==INFERRED` already match (`dummyindex/context/domains/enrich.py:266-272`), and only ever sets `INFERRED`, never demotes â€” re-running a partial/interrupted apply converges without corrupting prior `INFERRED` work. `ConfidenceLevel` enum values verified (`dummyindex/pipeline/enums.py:16-24`, cited in plan `:56-58`). This part of the EXTRACTEDâ†’INFERRED contract is sound; no concern.

## Security

Threat surface = a local CLI that merges agent-/file-supplied JSON into a committed structural artifact (`tree.json`). No network, no shell-out, no DB at this seam. Adversarial inputs: the `--from-json` mapping (LLM-authored or attacker-substituted), the `tree.json` if pre-corrupted, and `node_id` keys. node_ids are dict lookups, not filesystem paths â€” classic path traversal does not apply here.

- **[MEDIUM] Unsanitized abstract content merged verbatim into `tree.json` â€” stored-injection into the retrieval/prompt path.** `run_apply` validates only `isinstance(v, str)` (`dummyindex/cli/enrich.py:102-104`), then `_apply` writes it straight into `node["abstract"]` (`dummyindex/context/domains/enrich.py:270`) with no length cap, no control-char strip, no escaping. `tree.json` is committed and feeds the PageIndex walk future sessions read as trusted prose. Attack path: a poisoned/jailbroken authoring step or a hand-substituted `/tmp/abstracts.json` supplies a prompt-injection abstract ("ignore prior instructionsâ€¦") or a multi-MB blob; it persists and is replayed into every downstream session's context, INFERRED confidence laundering it as model-authored. The feature's core risk. Fix: cap length, strip control/NUL chars, escape/delimit `abstract` wherever the walk renders it; validate at the `run_apply` boundary (`cli/enrich.py:101-109`).

- **[MEDIUM] Unbounded JSON parse of an external file â€” local DoS.** `json.loads(from_json.read_text(...))` (`dummyindex/cli/enrich.py:101`) and `json.loads(tree_path.read_text(...))` (`dummyindex/context/domains/enrich.py:211`) read/parse arbitrary files with no size or depth guard. A multi-GB or pathologically nested `--from-json` OOMs/`RecursionError`s the process. Local self-DoS, low real severity, but an unvalidated-input boundary. Fix: stat-and-reject above a ceiling before `read_text`; fail fast with a typed error.

- **[MEDIUM] No structural validation of `tree.json` before walk.** `tree["root"]` is accessed (`dummyindex/context/domains/enrich.py:214,217`) with no dict/`root`-present check, and `run_apply` has no try/except around `apply_updates` (`cli/enrich.py:111`). A corrupt/truncated/swapped tree raises an opaque `KeyError`/`TypeError` stack trace instead of exit 2. Couples with the missing `schema_version` gate. Weak attack path (needs `.context/` write access), robustness-leaning. Fix: validate shape + `schema_version` and raise the typed schema error before walking.

- **[LOW] `node_id` keys are unvalidated but structurally inert.** Adversarial keys are matched only by set membership (`dummyindex/context/domains/enrich.py:213-215`); unmatched ids go to `ApplyResult.unknown` and exit 1 (`cli/enrich.py:116-123`). The id never reaches a path join, `eval`, or format-string sink â€” it cannot create nodes or touch the filesystem. No action; recorded as reviewed-and-bounded.

- **[INFORMATIONAL] No path confinement on `--from-json` / `--root`.** `Path(rest[i+1])` (`cli/enrich.py:70`) and `--root` (`cli/common.py:129-133`) accept any path the invoking user can read/write; `resolve_context_root` `.resolve()`s an absolute scope and operates there (`cli/common.py:32-33`). By design for an owner-run local tool â€” no privilege boundary to cross. Recorded only because if this seam were ever driven from a less-trusted context (a hook / a CI bot consuming an untrusted PR's `--from-json`), the lack of confinement would matter. Keep invocation owner-only.

## Product surface

- **Empty / fully-enriched tree â€” `enrich-plan` prints `stubs: 0  batches: 0` and exits 0 â€” no signal that there is nothing to do.** When every node is already `INFERRED` (re-running after a complete enrich, or a tree with no `EXTRACTED` nodes), `build_plan` yields zero stub nodes and zero batches (`dummyindex/context/domains/enrich.py:142-158,160-164`) yet `run_plan` still writes `cache/_enrich_plan.json` and exits 0 (`dummyindex/cli/enrich.py:43,55`). The operator sees `stubs: 0  batches: 0` but no "nothing to enrich â€” tree already fully enriched" message, and a stale empty plan is left on disk. Gap: success exit with an empty work-list is indistinguishable from a real plan; the session/skill must inspect `stubs` to know there is no work.

- **Huge tree â€” no progress, no streaming, whole `tree.json` re-serialized on every apply.** `apply_updates` loads the entire tree, mutates in place, and rewrites the whole file for *any* batch, however small (`dummyindex/context/domains/enrich.py:211,217-221`). On a large repo the per-file batch loop means each `enrich-apply` call re-reads and re-writes the full multi-MB `tree.json`. `enrich-plan` likewise has no node cap and emits every stub into one plan file (`:127,176`). Gap: no pagination/progress output, no "applied N of M" running tally across batches â€” the operator gets one line per apply with a count but no sense of overall completion against the plan's `stub_nodes`.

- **Partial enrich is invisible â€” no `enrich-status` verb, no completion percentage.** The plan ships `stats.stub_nodes` (`dummyindex/context/domains/enrich.py:160-164`) but there is no verb to ask "how many of the planned stubs are still `EXTRACTED`?" after a few apply rounds. The operator must re-run `enrich-plan` (regenerating the whole plan) to see the shrinking stub count. Gap: the natural "where am I in the enrich loop" query is only answerable by side effect of re-planning, which also overwrites the scratch plan the session may still be working from.

- **`enrich-apply` re-applying an old plan after a tree rebuild silently drops abstracts.** Operator flow: run `enrich-plan`, author abstracts into a JSON, then `dummyindex context rebuild --changed` renames/re-ids nodes, then `enrich-apply --from-json`. Ids that no longer exist route to `ApplyResult.unknown` and `run_apply` exits 1 listing them on stderr (`dummyindex/cli/enrich.py:116-123`) â€” but the authored prose for those ids is gone, never written, with the same "not found" wording whether the id was a typo or a victim of a legitimate rebuild. Gap: no distinct signal for "this id existed in the plan you generated but the tree moved underneath you" vs "you fat-fingered an id" â€” the operator cannot tell a recoverable rename from a real mistake. (Data-integrity peer already flags the underlying staleness; the UX gap is the indistinguishable error message.)

- **`enrich-apply` has no dry-run / preview â€” apply is the only way to see what would change.** There is no `--dry-run` to print "would update N, M unknown" before committing the rewrite. The mode is planâ†’authorâ†’apply, but the apply step is irreversible-by-rewrite (atomic replace overwrites `tree.json`) with no preview gate. Gap: an operator who wants to validate a hand-authored `--from-json` against the current tree before mutating the committed artifact has to just run it and inspect the diff afterward.

- **No machine-readable output on either verb â€” `--json` hinted by the plan's own data shape but not delivered at the CLI.** `run_plan` prints human prose (`stubs: â€¦  by_kind: {...}`, `dummyindex/cli/enrich.py:50-54`) and `run_apply` prints `updated N abstract(s)` (`:112-115`); neither offers `--json` for a script/skill to consume the counts structurally. The skill must parse the plan file directly (fine) but cannot get apply results except by exit code + scraping stderr lines. Gap: the `unknown` list â€” the one thing a caller must act on â€” is only emitted as `    - <id>` stderr lines (`:121-122`), forcing brittle text parsing instead of a structured result.

- **`--from-json` parsed before validation ordering is slightly surprising but benign; the real UX gap is the all-or-nothing apply.** `run_apply` validates every value is a `str` (`dummyindex/cli/enrich.py:102-104`) and then applies the *entire* mapping; there is no partial apply of the valid subset when some ids are unknown â€” known ids are applied AND unknown ids are reported together, exit 1 (`:111,116-123`). So a 100-id batch with one typo applies 99 and exits 1; the non-zero exit may lead a script to treat the whole batch as failed and retry, re-applying 99 no-ops plus the still-bad id. Gap: exit 1 conflates "some abstracts were dropped" with "the batch failed" â€” idempotency saves correctness but the exit-code contract misleads automation.

- **`enrich-plan` overwrites the prior scratch plan with no warning if a session is mid-enrich.** `run_plan` unconditionally writes `cache/_enrich_plan.json` (`dummyindex/cli/enrich.py:42-43`) via atomic replace, clobbering any in-progress plan the operator/skill is still authoring against. Re-running `enrich-plan` to "check status" (see above) silently discards the work-list the session was driving from. Gap: no `--force`/overwrite guard and no notice that an existing plan was replaced; the transient-scratch design (plan `:71-75`) makes this intentional, but it is a footgun when the only way to re-check stub counts is to regenerate.
