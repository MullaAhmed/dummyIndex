## Data integrity

- **Read-only by construction — no write-integrity surface.** `build_preflight_report` resolves the root and assembles one frozen `PreflightReport` from value-returning probes; nothing writes (`inventory.py:28-62`, `inventory.py:1-7`). The atomic-write / schema-version disciplines in `conventions/data-access.md:5-13` do not apply here — this feature persists nothing. No concern.

- **Ownership computation is correct and conservative.** `context_ownership` treats missing/empty `.context/` as ABSENT, a `meta.json` JSON-object carrying `dummyindex_version` as OURS, and everything else (including unreadable/malformed) as FOREIGN, never raising (`ownership.py:34-53`, `ownership.py:64-70`). `_owned_flag` collapses this to the tri-state `Optional[bool]` correctly — `None` for ABSENT, else `is OURS` (`inventory.py:65-74`). Deliberately bypassing strict `read_meta` so a newer-`schema_version` index still reads OURS is sound (`ownership.py:9-15`). No concern.

- **Settings parse-failure handling is sound, with one minor inventory-accuracy nit.** OSError / JSONDecodeError and a non-`dict` top-level both yield `parseable=False` and an empty inventory rather than a crash (`inventory.py:86-101`) — correct "won't touch" semantics per `models.py:19`. Nit: at `inventory.py:117-119` a non-string `command` is coerced to `""`, and since `SENTINEL not in ""`, that entry falls to the `else` branch and flips `event_has_user=True` (`inventory.py:127-130`). A malformed/null-command hook entry under an event therefore inflates `user_hook_events`, over-reporting a "user hook." Read-only and conservative-leaning (it errs toward "leave alone"), so low severity — but the reported inventory is not strictly faithful to that entry. Code wins; flag for awareness only.

- **`_inspect_settings` empty-`inner` traversal is safe.** Non-dict entries (`inner=None`) and non-list `inner` both short-circuit the inner loop via the `isinstance(inner, list)` guard (`inventory.py:115-116`); arbitrary hook shapes (nulls, non-list entries at `:111`) are skipped without raising. Matches the "never-raises" decision (`plan.md:48`). No concern.

- **Empty-dir ownership probe: unreadable reads as non-empty ⇒ FOREIGN.** `_is_empty_dir` returns `False` on `OSError` (`ownership.py:56-61`), so an unreadable `.context/` is classified FOREIGN, not ABSENT — the safe (don't-clobber) direction. Correct. No concern.

- **Git-clean detection edge cases are handled defensively.** `_git_clean` returns `None` on subprocess OSError/ValueError and on any non-zero return code, and only `True` on empty `--porcelain` output (`inventory.py:181-194`). `git_clean` is gated behind `is_git_repo(project_root)` so a non-repo yields `None`, never a spurious `True` (`inventory.py:45-46`). `GIT_OPTIONAL_LOCKS=0` keeps the read from stranding `index.lock` (`inventory.py:181-189`). Untracked-only working trees correctly read as not-clean (porcelain emits `??` lines, output non-empty). No concern.

## Security

Adversarial lens: this feature ingests untrusted, repo-local config (`.claude/settings.json`, `.claude/**`, `.context/meta.json`) and renders a report. The threat model is "I cloned/opened a hostile repo, then ran `dummyindex context preflight`." Surface is low (read-only, local CLI, no network, no eval). Verified clean on the highest-value vector; the rest are low-severity hardening notes. Code wins where it disagrees with prose.

- **No sensitive settings content leaks into the report — verified clean (the headline finding).** This was the primary worry, and the design forecloses it. `_inspect_settings` reads each hook `command` only to test `SENTINEL in command`, then **discards the string** — it is never stored (`inventory.py:117-130`). `SettingsState` carries only `exists`/`parseable`/`dummyindex_hook_present` booleans plus `user_hook_events` (event-name keys), and `to_dict()` serialises exactly those (`models.py:14-29`). `PreflightReport.to_dict()` likewise emits only booleans, the resolved root path, and the rules/agents *path* lists — never file bodies (`models.py:48-60`). So secrets, env values, tokens, or full command lines living in `settings.json` cannot reach either the markdown (`render.py:138-146` prints only a state label) or the `--json` output. Low/no concern by construction.

- **LOW — untrusted hook-event keys are echoed verbatim into the rendered report.** `user_hook_events` is built from `hooks_block.items()` keys (`inventory.py:110`), i.e. arbitrary attacker-controlled JSON object keys, with no character validation. They are `_join`-ed and printed straight into the markdown under "your hooks under:" (`render.py:44-47`, `render.py:113-117`). A hostile `settings.json` can name an event with embedded newlines/markdown/ANSI escapes that render into the terminal or a downstream markdown viewer (log/format injection). Impact is cosmetic on a local CLI — no code execution, the values are `sorted()` not interpreted — but it is an untrusted-input→output path with zero sanitisation. Mitigation: sanitise/escape control chars (and cap length) before rendering. Severity low.

- **LOW — symlinked `.claude/rules/` can disclose absolute out-of-tree paths.** `_list_rule_files` enumerates with `rules_dir.glob("**/*.md")` and `is_file()`, both of which follow symlinks (`inventory.py:145-147`). On Python ≤3.12 `**` also descends *into* symlinked directories; a malicious `.claude/rules/link → /etc` (or any out-of-repo target) lets the glob surface files outside the repo. `relative_to(project_root)` then raises `ValueError`, and the `except` branch falls back to the **absolute** path (`inventory.py:150-151`), which is rendered into the report (`render.py:41`). No file *content* is read — only the path is disclosed — so this is path-disclosure / inventory-pollution, not data exfiltration. Same symlink-following applies to `_list_agent_names` (`inventory.py:159`). Mitigation: skip entries whose resolved real path escapes `project_root`, or pass `follow_symlinks=False` where supported. Severity low (read-only, local, path-only).

- **LOW — unbounded full-file reads on attacker-sized inputs (local DoS).** `_has_managed_block` reads all of `CLAUDE.md` (`inventory.py:162-166`) and `_has_dummyindex_marker` reads all of `meta.json` (`ownership.py:64-70`) with `read_text()` and no size cap. A `.claude/CLAUDE.md` or `.context/meta.json` that is a huge regular file (or a symlink to `/dev/zero`-style growth) is read whole into memory before the substring/JSON check. Both catch `OSError` so they won't crash, but a multi-GB file can exhaust memory. Local-only, requires a hostile repo, no privilege gain — pure availability nuisance. Mitigation: bounded read (e.g. read a capped prefix; `BEGIN_MARKER` and the `meta.json` marker both appear early). Severity low.

- **`_git_clean` subprocess invocation is not injectable.** The argv is a fixed list `["git", "status", "--porcelain"]` with no shell and no interpolation of repo-controlled data; only `cwd` and a hardened `env` (`GIT_OPTIONAL_LOCKS=0`) vary (`inventory.py:181-189`). `is_git_repo` gates it (`inventory.py:45-46`). No command-injection surface. Note in passing: `git` is resolved via `PATH`, so a poisoned `PATH` could shadow it — but that is an ambient-environment trust assumption shared by the whole tool, not specific to preflight. No concern here.

- **Parse-failure handling is robust against malformed/hostile JSON.** Every external read is wrapped: settings JSON catches `OSError`/`JSONDecodeError` and a non-`dict` top level (`inventory.py:86-101`); `meta.json` additionally catches `UnicodeDecodeError` (`ownership.py:67-68`); the hook-shape walk tolerates nulls, non-dict entries, non-list values, and non-string commands without raising (`inventory.py:109-130`). A hostile config degrades to a conservative "unparseable / hands-off" classification, never an exception or a write. This is the right failure mode for a preflight gate. No concern.

## Product surface

Lens: a developer reading `dummyindex context preflight` output before letting the index build run. Does the report tell them clearly what gets written, what is safe, and what to do about warnings — without false alarms or blind spots? Identifiers checked against `map/symbols.json`.

- **Headline clarity is strong — three-section shape reads well.** "Will write / manage" → "Will leave untouched" → "⚠ Warnings" → "State" is exactly the mental model a cautious user wants, and each managed line spells the *scope* of the write inline ("managed block only — surrounding content preserved", "one additive SessionStart hook") (`render.py:21-24,35-63`). No gap.

- **`.context/**` is over-claimed for the empty/absent case but harmlessly.** The managed-paths line shows bare `.context/**` whenever `context_owned is not False` (`render.py:68-73`), so an absent or empty `.context/` reads identically to a dummyindex-owned one in the "Will write / manage" block — the *nuance* (create vs refresh-in-place) only appears far below in the State line (`render.py:120-127`). A user skimming the top block can't tell a fresh build from an in-place overwrite. Gap: minor — the State section disambiguates, but the two highest-signal lines (what will be written) lose the create-vs-modify distinction.

- **Dirty-tree warning fires on untracked-only trees — false-alarm risk during normal first-run.** `_git_clean` returns `False` whenever `git status --porcelain` is non-empty, which includes untracked files (`??` lines) (`inventory.py:192-194`); the warning then says "working tree has uncommitted changes — commit or stash first" (`render.py:91-95`). The canonical first run is on a repo where the user just added files (or has normal untracked build artifacts) — they'll see a scary "commit or stash first" with no path to satisfy it (you can't stash untracked-only without `-u`). SKILL.md itself downgrades this to "advisory… mention it" (`SKILL.md:126-130`), so the *report's* imperative phrasing ("commit or stash first") is harsher than the documented intent. Gap: the warning text overstates severity for the most common case; an untracked-only tree is effectively reversible (writes land in new files), yet the report demands stashing.

- **No warning when a user hook already occupies the SessionStart event dummyindex installs into — the one collision that matters is silent.** `user_hook_events` is surfaced only as a neutral "Will leave untouched" line (`render.py:43-47`); there is no signal when one of those events is `SessionStart` — i.e. the exact event dummyindex adds its own hook into (`inventory.py:125`, `CURRENT_CLAUDE_EVENTS`). The user's existing SessionStart hook is preserved (the install is additive), but the report gives no heads-up that two SessionStart hooks will now coexist and both fire. This is the highest-value actionable signal a preflight could give about hook interaction, and it's absent — the plan's own Open Question flags it unresolved (`plan.md:56`). Gap: missing signal on the only event where dummyindex and the user overlap.

- **Settings-unparseable warning is actionable but omits the downstream consequence the skill knows.** The warning says "dummyindex will refuse to touch it. Fix it by hand first." (`render.py:86-90`) — correct, but it doesn't tell the user that the *rest* of the build still proceeds and only the SessionStart hook is skipped this run (`SKILL.md:123-125`). A user reading just the report can't tell whether this is a hard stop or a partial-skip. Gap: warning is truthful but under-informs on blast radius (hook-only skip vs whole-build abort).

- **Foreign-`.context/` warning is the model the others should follow — clear, scoped, with a concrete remedy.** It states the verdict, the *reason* (no `meta.json` with `dummyindex_version`), the guarantee (won't be touched), and an exact shell remedy (`mv .context .context.other`) (`render.py:78-85`). This is the gold standard for actionability; the dirty-tree and unparseable warnings are terser by comparison. No gap — cited as the bar.

- **"every source file and prose doc in the repo" is reassuring but unverified-by-probe.** The leave-untouched block asserts source/prose are untouched as a static string (`render.py:50`), which is true of the ingest/council pipeline but NOT of `/dummyindex-equip` (run separately, "writes more into `.claude/`", `SKILL.md:128`). A user who reads "will leave untouched: every source file" and then runs equip could be surprised. Gap: the report scopes its promise to the standard flow but doesn't say so; the equip caveat lives only in SKILL.md (`SKILL.md:128`), not in the user-facing report.

- **Truncation at 8 items hides the tail without a total mismatch.** `_join` shows the first 8 and appends "+N more" (`render.py:113-117`), and `_leave_line` prefixes the true count (`render.py:107-110`), so a 30-rule repo reads "30 (a, b, …, +22 more)" — count is honest. No gap; noted as correct.
