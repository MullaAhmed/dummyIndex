# Stage 3 — critiques


## critic-database (data-integrity lens)

## Data integrity

- **Determinism holds; sort is total and stable.** Rows sort on `(rel_path, line)` (`harvest.py:56`) before the ledger freezes; `rel_path` is repo-relative POSIX and `line` is unique-per-file, so the key is a total order with no ties needing a stable-sort fallback. Re-running on an unchanged tree is byte-identical, and `to_dict()` emits a fixed key order (`models.py:37`, `models.py:62`) — no dict-ordering or absolute-path nondeterminism leaks into JSON. Clean against the `conventions/data-access.md` "no absolute paths / sorted-stable emission" rule.
- **No absolute-path leak — even from out-of-tree code roots.** `detect()` can accept `extra_doc_roots` outside `root` and *accepts code files under them* (`detect.py:360-361`); those paths would be absolute and outside the repo. `_rel_or_none` resolves `path.relative_to(root)` and returns `None` on `ValueError`/`OSError` (`drift.py:372-377`), and the harvester skips on `None` (`harvest.py:46`). So an out-of-tree `.py` is silently dropped, never rendered as an absolute path. Correct, but note the *silent drop*: if a future caller passes `--docs` at an external code root expecting its debt counted, it vanishes with no signal (`harvest.py:46`). Acceptable for v1 (harvest_debt takes no extra roots), flag if that surface widens.
- **`.py` filter is suffix-only — `detect()` already gates on real code classification.** `raw.endswith(".py")` (`harvest.py:42`) runs over `files["code"]` which `detect()` populates by its own classification, so a `.pyc`/binary blob is excluded upstream; the suffix check is a cheap narrowing, not the sole guard. A non-UTF-8 file mis-suffixed `.py` that slips through is caught fail-soft (`harvest.py:48-53`). No silent corruption path.
- **Marker-parse: true-comment-only guard is correct; one residual false-positive class.** `stripped.startswith(prefix)` (`harvest.py:81`) means a `# TODO:` *inside* a string body or trailing inline comment (`x = 1  # TODO: ...`) never matches — confirmed against the plan's stated scope (inline trailing is a known open question, plan.md:40). The residual: a line that is genuinely a comment but whose author wrote prose like `# TODO: handle in a string "with quotes"` is parsed verbatim into `ceiling`/`trigger` — that's intended capture, not corruption. No regex, no multi-line state, so no cross-line bleed. Clean.
- **`marker` token derivation is fragile to prefix-set drift.** `marker = prefix.strip("# :").upper()` (`harvest.py:94`) derives the bare token by stripping the leading `#`/space/colon chars off the matched `DEBT_PREFIXES` entry. `str.strip(chars)` strips *every* leading/trailing char in the set, so `"# TODO:"` → `"TODO"` correctly today — but it strips from BOTH ends and treats the arg as a char-set, so a future prefix containing those chars internally (none today) or with a trailing space would mis-token. Low risk given the frozen tuple at `python_rationale.py:15`, but the derivation is implicit; an explicit map prefix→token would be byte-safer. Flag, not block.
- **`; upgrade:` split is first-occurrence and null-safe.** `_UPGRADE_SEP in body` then `body.partition(_UPGRADE_SEP)` (`harvest.py:97-98`) splits on the FIRST separator, so a trigger containing `"; upgrade:"` again keeps the remainder intact (`harvest.py:30` comment matches code). Empty trigger after strip degrades to `no_trigger` (`harvest.py:101-111`) rather than emitting an empty-string trigger — preserves the `no_trigger == (trigger is None)` invariant (`models.py:27`). Only `# DEBT:` carries triggers; `TODO/FIXME/HACK` always `no_trigger` (`harvest.py:97` gate on `marker == "DEBT"`). Tally `M == no_trigger_count` (`cli/debt.py:101`, `models.py:59`) is consistent.
- **Write path is atomic and fail-loud, per the one mandatory data-access rule.** `_persist` routes through `write_text_atomic` (`cli/debt.py:135-138`) — tmp+replace, no half-file observable — and `mkdir(parents=True, exist_ok=True)` first. Unlike fail-soft harvest, a user-requested `--write` surfaces its `OSError` rather than swallowing (matches plan.md:35 and `conventions/data-access.md` atomic-write rule). `--json` to stdout still persists the *markdown* view (`cli/debt.py:62-65`) — one on-disk format, no dual-format drift. Clean.
- **Empty-ledger and empty-ceiling rendering never produce a malformed row.** Empty ledger short-circuits to a fixed no-debt message (`cli/debt.py:82-86`); an empty `ceiling` renders `(no detail)` so the dash never dangles (`cli/debt.py:113`). Trailing `.rstrip() + "\n"` (`cli/debt.py:103`) gives a single deterministic EOF newline. No data-shape edge leaks a broken line.


## critic-security (adversarial security lens)

## Security

- **Verdict: clean — low surface, read-only harvest, no untrusted input crosses a privileged sink.** Pure read over the host's own trusted `.py` source plus an opt-in `--write` to a fixed in-repo path. No network, shell, eval, or deserialization. None of the four hypothesised threats is an exploitable bug; residual hardening notes below.
- **Symlink traversal — NOT reachable.** `detect(project_root)` (`harvest.py:37`) passes no `follow_symlinks`, defaulting to `False` (`detect.py:351`); `os.walk(..., followlinks=False)` (`detect.py:414`) never descends symlinked dirs, and `_rel_or_none`'s `p.resolve().relative_to(root)` (`drift.py:375`) drops any symlinked file resolving outside `root` (`harvest.py:46`). Residual: safety is one keyword-arg away — if a refactor threads `follow_symlinks=True` into `harvest_debt`, symlinked dirs get walked. Pin the default explicitly. Low.
- **Absolute-path leak — closed.** Rows only ever interpolate `row.rel_path` (repo-relative POSIX from `drift.py:372-377`); a `None` rel is skipped pre-row (`harvest.py:45-46`). Neither stdout nor `.context/debt.md` leaks a machine path. Residual: `{leftover}` raw-argv echo to stderr (`cli/debt.py:52-55`) is flags-not-paths, stderr-not-ledger — informational.
- **Marker content written to `debt.md` — content-injection in-scope, impact bounded.** `ceiling`/`trigger` (`harvest.py:99-100`) interpolated into markdown unescaped (`cli/debt.py:114-117`); a comment author can inject markdown/HTML/`<script>` into `debt.md`. Stays **low not high**: write path is fixed (`cli/debt.py:138`, marker text can't influence it — no traversal/overwrite), the actor already has source-commit access (stronger primitive), and it's inert on disk. Real exposure is downstream if `debt.md` is later rendered by an HTML-executing viewer (stored-XSS-style) — HTML-escape or inline-code-wrap in `_render_row` as defense-in-depth. Not a v1 blocker.
- **Huge-file DoS — present, low.** `read_text()` (`harvest.py:49`) slurps whole files with no size cap; O(n) single-pass `splitlines` (`harvest.py:63`), `startswith` only — no ReDoS (`harvest.py:81`). Low because input is the host's own trusted tree on a short-lived local CLI. Note: a true `MemoryError` is neither `OSError` nor `UnicodeDecodeError`, so it propagates past the fail-soft catch (`harvest.py:50-53`). Optional: size guard using sizes `detect()` already has.
- **Unreadable / path-set drift — fail-soft.** Unreadable/non-UTF-8 skipped without raising (`harvest.py:48-53`); suffix gate runs over `detect()`'s already sensitive/ignore-filtered `files["code"]` (`harvest.py:42`). No secret-scanning in scope, none claimed.


## critic-product (product-surface lens)

## Product surface

- **Clean-repo run — prints `_No debt markers found in the repo's Python source._` under a `# debt ledger` heading (`cli/debt.py:82-86`); exits 0 — clear and correctly scoped to Python.** Gap: the message is the *only* output, with no `0 markers, 0 with no trigger.` tally line that every non-empty run ends on (`cli/debt.py:100-102`). A CI gate keyed on `grep "with no trigger"` matches on a debt-laden repo but gets an empty grep on a clean one — the success-signal *shape* differs between states, so a clean repo passes by absence rather than by a `0` value.
- **`--write` gives no on-success feedback — silent persist (`cli/debt.py:64-67`).** `--write` writes `.context/debt.md` then prints the same ledger to stdout as a no-flag run; nothing names the written file or confirms it. A user refreshing the on-disk ledger has no `wrote .context/debt.md` confirmation and must `ls`/`cat` independently. Contrast the fail-loud write contract (`plan.md:35`): failure surfaces loudly, success is mute.
- **`--write` + `--json` is a quiet semantic trap — stdout JSON, disk markdown (`cli/debt.py:62-67`).** A deliberate "on-disk ledger is always human-readable" decision (`spec.md:16`), but undiscoverable at the CLI: `debt --write --json` expecting JSON in `debt.md` gets markdown, no warning. No flag to persist JSON, no stderr note that `--json` was ignored for the written artifact. Help (`help.py:213-215`) lists both flags without resolving which format `--write` writes when both are passed.
- **No-trigger tally `N markers, M with no trigger.` (`cli/debt.py:100-102`) under-explains the feature's reason to exist.** The rot-risk framing lives only in `spec.md:9`/`plan.md:32`, never in user-facing output. `12 markers, 9 with no trigger.` gives no in-band cue that the 9 are the actionable set or that `# DEBT: ...; upgrade: ...` adds a trigger. No legend, and no-trigger rows interleave with triggered ones in line order (`cli/debt.py:90-97`) — the rot-risk subset can't be skimmed.
- **Python-only scope is a silent surprise — no JS/TS markers, CLI never says so on a populated ledger (`harvest.py:42`).** A polyglot user runs `context debt`, sees a count, assumes `.ts`/`.js` `// TODO:` are covered — they aren't (`plan.md:31`). The populated-ledger header `# debt ledger` (`cli/debt.py:88`) carries no "(Python only)" qualifier, so a low count on a JS-heavy repo reads as "clean" when it's "unscanned". The clean-repo message *does* say "Python source" (`cli/debt.py:85`); the populated header does not.
- **Discoverability rests on `context --help` alone (`help.py:207-215`).** `debt` is listed with flags, but there's no `debt --help` sub-help — an unknown flag just errors exit 2 dumping the raw `leftover` list (`cli/debt.py:51-56`) rather than a usage line. And the feature isn't cross-referenced from the drift signal it complements (`community-8`, `plan.md:27`): drift output gives no "see also: context debt" pointer, so the intent-vs-mtime pairing the design intends (`spec.md:5`) is undiscoverable from the drift side.
- **`[path]` positional scope is unsurfaced in output (`cli/debt.py:38`, `help.py:207`).** `debt [path] [--root DIR]` scopes the harvest, but the help gives no subdir example and the `# debt ledger` header never echoes which root/path was scanned. `debt src/` vs `debt` produce differently-scoped ledgers with identical headers — a persisted `.context/debt.md` from a scoped run is indistinguishable from a full-repo one.
