## Data integrity

- **Determinism holds; sort is total and stable.** Rows sort on `(rel_path, line)` (`harvest.py:56`) before the ledger freezes; `rel_path` is repo-relative POSIX and `line` is unique-per-file, so the key is a total order with no ties needing a stable-sort fallback. Re-running on an unchanged tree is byte-identical, and `to_dict()` emits a fixed key order (`models.py:37`, `models.py:62`) — no dict-ordering or absolute-path nondeterminism leaks into JSON. Clean against the `conventions/data-access.md` "no absolute paths / sorted-stable emission" rule.
- **No absolute-path leak — even from out-of-tree code roots.** `detect()` can accept `extra_doc_roots` outside `root` and *accepts code files under them* (`detect.py:360-361`); those paths would be absolute and outside the repo. `_rel_or_none` resolves `path.relative_to(root)` and returns `None` on `ValueError`/`OSError` (`drift.py:372-377`), and the harvester skips on `None` (`harvest.py:46`). So an out-of-tree `.py` is silently dropped, never rendered as an absolute path. Correct, but note the *silent drop*: if a future caller passes `--docs` at an external code root expecting its debt counted, it vanishes with no signal (`harvest.py:46`). Acceptable for v1 (harvest_debt takes no extra roots), flag if that surface widens.
- **`.py` filter is suffix-only — `detect()` already gates on real code classification.** `raw.endswith(".py")` (`harvest.py:42`) runs over `files["code"]` which `detect()` populates by its own classification, so a `.pyc`/binary blob is excluded upstream; the suffix check is a cheap narrowing, not the sole guard. A non-UTF-8 file mis-suffixed `.py` that slips through is caught fail-soft (`harvest.py:48-53`). No silent corruption path.
- **Marker-parse: true-comment-only guard is correct; one residual false-positive class.** `stripped.startswith(prefix)` (`harvest.py:81`) means a `# TODO:` *inside* a string body or trailing inline comment (`x = 1  # TODO: ...`) never matches — confirmed against the plan's stated scope (inline trailing is a known open question, plan.md:40). The residual: a line that is genuinely a comment but whose author wrote prose like `# TODO: handle in a string "with quotes"` is parsed verbatim into `ceiling`/`trigger` — that's intended capture, not corruption. No regex, no multi-line state, so no cross-line bleed. Clean.
- **`marker` token derivation is fragile to prefix-set drift.** `marker = prefix.strip("# :").upper()` (`harvest.py:94`) derives the bare token by stripping the leading `#`/space/colon chars off the matched `DEBT_PREFIXES` entry. `str.strip(chars)` strips *every* leading/trailing char in the set, so `"# TODO:"` → `"TODO"` correctly today — but it strips from BOTH ends and treats the arg as a char-set, so a future prefix containing those chars internally (none today) or with a trailing space would mis-token. Low risk given the frozen tuple at `python_rationale.py:15`, but the derivation is implicit; an explicit map prefix→token would be byte-safer. Flag, not block.
- **`; upgrade:` split is first-occurrence and null-safe.** `_UPGRADE_SEP in body` then `body.partition(_UPGRADE_SEP)` (`harvest.py:97-98`) splits on the FIRST separator, so a trigger containing `"; upgrade:"` again keeps the remainder intact (`harvest.py:30` comment matches code). Empty trigger after strip degrades to `no_trigger` (`harvest.py:101-111`) rather than emitting an empty-string trigger — preserves the `no_trigger == (trigger is None)` invariant (`models.py:27`). Only `# DEBT:` carries triggers; `TODO/FIXME/HACK` always `no_trigger` (`harvest.py:97` gate on `marker == "DEBT"`). Tally `M == no_trigger_count` (`cli/debt.py:101`, `models.py:59`) is consistent.
- **Write path is atomic and fail-loud, per the one mandatory data-access rule.** `_persist` routes through `write_text_atomic` (`cli/debt.py:135-138`) — tmp+replace, no half-file observable — and `mkdir(parents=True, exist_ok=True)` first. Unlike fail-soft harvest, a user-requested `--write` surfaces its `OSError` rather than swallowing (matches plan.md:35 and `conventions/data-access.md` atomic-write rule). `--json` to stdout still persists the *markdown* view (`cli/debt.py:62-65`) — one on-disk format, no dual-format drift. Clean.
- **Empty-ledger and empty-ceiling rendering never produce a malformed row.** Empty ledger short-circuits to a fixed no-debt message (`cli/debt.py:82-86`); an empty `ceiling` renders `(no detail)` so the dash never dangles (`cli/debt.py:113`). Trailing `.rstrip() + "\n"` (`cli/debt.py:103`) gives a single deterministic EOF newline. No data-shape edge leaks a broken line.

## Security

- **Verdict: clean — low surface, read-only harvest, no untrusted input crosses a privileged sink.** The whole feature is a pure read over `.py` files the host already trusts (its own source) plus an opt-in `--write` to a fixed in-repo path. No network, no shell, no eval, no deserialization. None of the four hypothesised threats lands as an exploitable bug; below is the adversarial walk with the residual hardening notes.
- **Symlink traversal during enumeration — NOT reachable on the feature's own path.** Enumeration is `detect(project_root)` (`harvest.py:37`) with **no `follow_symlinks` argument**, so it takes the default `follow_symlinks=False` (`detect.py:351`), and `os.walk(..., followlinks=follow_symlinks)` (`detect.py:414`) never descends symlinked dirs. A symlinked *file* could still be read, but `_rel_or_none` calls `p.resolve().relative_to(root)` (`drift.py:375`) — a symlink pointing **outside** `root` resolves out-of-tree and is dropped (`harvest.py:46`); one pointing **inside** root just reads an in-repo file. So no out-of-tree content is harvested via symlink. Residual: the harvester hard-wires the safe default but doesn't *assert* it — if a future refactor threads `follow_symlinks=True` into `harvest_debt`, a symlinked dir to `/etc` or `$HOME` would be walked and any in-resolved-tree `.py` read. Pin the default explicitly (or pass `follow_symlinks=False` at the call site) so the safety isn't one keyword-arg away. Low.
- **Absolute-path leak in ledger output — closed, with one residual.** Every row is repo-relative POSIX: `_rel_or_none` returns `path.resolve().relative_to(root).as_posix()` or `None` (`drift.py:372-377`), and a `None` rel is skipped before any row is built (`harvest.py:45-46`). The renderer only ever interpolates `row.rel_path` (`cli/debt.py:114`), never an absolute `Path`. So neither stdout nor `.context/debt.md` leaks a home dir or machine path — matches the data-access "no absolute paths" rule already verified by the data-integrity lens. Residual: the error/`leftover` path prints `{leftover}` (raw argv) to stderr (`cli/debt.py:52-55`); that's user-supplied flags echoed back, not a path leak, and goes to stderr not the persisted ledger — informational only.
- **Marker content written to `.context/debt.md` — content-injection is in-scope but impact is bounded to a local markdown file.** Marker text (`ceiling`/`trigger`) is captured verbatim from comment bodies (`harvest.py:99-100`) and interpolated into markdown **with no escaping** (`cli/debt.py:114-117`). An attacker who can land a `.py` comment in the repo can therefore inject arbitrary markdown — e.g. `# TODO: click [here](javascript:...)`, a fake `## heading`, an HTML/`<script>` blob, or a `]( ` that breaks row structure — into `debt.md`. Why this stays **low, not high**: (1) the write sink is a fixed in-repo path (`context_dir / "debt.md"`, `cli/debt.py:138`) via `write_text_atomic` — the marker text **cannot influence the path**, so no path-traversal/overwrite-elsewhere; (2) the threat actor already has source-commit access, i.e. a far stronger primitive than "write text into a sibling .md"; (3) it's inert markdown on disk, executed by nothing in this pipeline. The real exposure is *downstream*: if `debt.md` is later rendered in a browser/Markdown viewer that executes embedded HTML/JS, stored-XSS-style. Mitigation if that rendering ever happens: HTML-escape `<`/`>`/backticks (or neutralise `]`/newlines) in `_render_row` before interpolation, or wrap ceiling/trigger in inline code. Defense-in-depth, not a v1 blocker.
- **Huge-file / pathological-input DoS — present but low, bounded by trust + read-only single-pass.** `path.read_text(encoding="utf-8")` (`harvest.py:49`) slurps each whole file into memory with **no size cap**; a multi-GB `.py` (or one crafted with millions of `# TODO:` lines) inflates memory and the per-file row list. It's O(n) single-pass, `splitlines()` over the text (`harvest.py:63`), no quadratic blowup and no regex (so no catastrophic-backtracking ReDoS — `startswith` only, `harvest.py:81`). Decompression/zip-bomb is N/A (plain text read). Severity is low because the input is the host's *own* trusted source tree (you'd be DoS-ing your own `dummyindex` run), the process is a short-lived local CLI, and non-UTF-8/oversize-triggered `OSError`/`MemoryError`-adjacent failures on read are caught fail-soft (`harvest.py:50-53`) — though note a true `MemoryError` is **not** an `OSError`/`UnicodeDecodeError` and would propagate. Optional hardening: a size guard (skip files over e.g. a few MB) or streamed line read, since `detect()` already knows file sizes.
- **Path-set drift / unreadable files — fail-soft, no crash, no info leak.** Unreadable or non-UTF-8 files are skipped without raising (`harvest.py:48-53`), and the suffix gate (`harvest.py:42`) runs over `detect()`'s already-classified `files["code"]` (sensitive-file and ignore exclusions applied upstream in `detect`), so a secret-bearing `.env`-style file mis-suffixed wouldn't reach here as "code" unless `detect` classified it so. No secret-scanning is in scope for this feature and none is claimed.

## Product surface

- **Clean-repo run — prints `_No debt markers found in the repo's Python source._` under a `# debt ledger` heading (`cli/debt.py:82-86`); exits 0 — clear and correctly scoped to Python.** Gap: the message is the *only* output, with no `0 markers, 0 with no trigger.` tally line that every non-empty run ends on (`cli/debt.py:100-102`). A user scripting on the tally line (`grep "with no trigger"`) gets a match on a debt-laden repo but an empty grep on a clean one — the success signal *shape* differs between the two states, so a CI gate keyed on the tally silently passes a clean repo by absence rather than by a `0` value.
- **`--write` gives no on-success feedback — silent persist (`cli/debt.py:64-67`).** `--write` writes `.context/debt.md` then prints the *same ledger to stdout* as a no-flag run; nothing names the file that was written or confirms it. Gap: a user who ran `debt --write` to refresh the on-disk ledger has no stdout confirmation that the file was created/updated (no `wrote .context/debt.md` line) — they must independently `cat`/`ls` the path. Contrast the fail-loud write contract (`plan.md:35`): failure surfaces loudly, success is mute.
- **`--write` + `--json` is a quiet semantic trap — stdout is JSON, disk is markdown (`cli/debt.py:62-67`).** Running `debt --write --json` persists the *markdown* view to `.context/debt.md` while emitting JSON to stdout. This is a deliberate "on-disk ledger is always human-readable" decision (`spec.md:16`), but it's undiscoverable at the CLI: a user who wants a machine-readable file on disk (`debt --write --json` expecting JSON in `debt.md`) gets markdown instead, with no warning. Gap: no flag to persist JSON, and no stderr note that `--json` was ignored for the written artifact. The help text (`help.py:213-215`) states "--write also persists .context/debt.md" and "--json emits the stable structure" without resolving which format `--write` writes when both are passed.
- **No-trigger tally `N markers, M with no trigger.` (`cli/debt.py:100-102`) under-explains the feature's whole reason to exist.** The tally is a bare count; nothing in the rendered ledger tells the reader *why* "no trigger" is the rot-risk class (the framing lives only in `spec.md:9`/`plan.md:32`, not in user-facing output). A first-time user sees `12 markers, 9 with no trigger.` and has no in-band cue that the 9 are the actionable set, or that `# DEBT: ...; upgrade: ...` is the syntax to add a trigger. Gap: no legend/hint line and no separation of the no-trigger rows from triggered ones in the per-file listing — they interleave in line order (`cli/debt.py:90-97`), so the rot-risk subset can't be skimmed.
- **Python-only scope is a silent surprise — no JS/TS markers, and the CLI never says so (`harvest.py:42`).** A polyglot-repo user runs `context debt`, sees a count, and reasonably assumes it covers their `// TODO:` in `.ts`/`.js` — it doesn't (Python-only by design, `plan.md:31`). Gap: neither the ledger header nor the clean-repo message disambiguates "Python source" forcefully for a populated ledger — the `# debt ledger` heading (`cli/debt.py:88`) carries no "(Python only)" qualifier, so a low TODO count on a JS-heavy repo reads as "clean" when it's actually "unscanned". The clean-repo message *does* say "Python source" (`cli/debt.py:85`); the populated-ledger header does not. Help text says "over the repo's Python source" (`help.py:208`) but that's one indirection away from the live output.
- **Discoverability rests on `context --help` alone (`help.py:207-215`); no inverse/empty-arg hint.** `debt` is listed in the help block with flags, so `dummyindex context --help` surfaces it. Gap: there's no `debt --help` sub-help (an unknown flag just errors with exit 2, `cli/debt.py:51-56`, dumping the raw `leftover` list rather than a usage line), and the feature isn't cross-referenced from the drift signal it complements (`community-8`, `plan.md:27`) — a user looking at drift output gets no "see also: context debt" pointer, so the intent-vs-mtime pairing the design intends (`spec.md:5`) is undiscoverable from the drift side.
- **`[path]` positional is accepted but its effect on a multi-root invocation is unsurfaced (`cli/debt.py:38`, `help.py:207`).** `debt [path] [--root DIR]` takes a scope path via `parse_path_and_root`, but the help line gives no example of scoping the harvest to a subdir, and the rendered ledger header (`# debt ledger`) never echoes which root/path was scanned. Gap: running `debt src/` vs `debt` produces differently-scoped ledgers with identical headers — no provenance line states what was scanned, so a persisted `.context/debt.md` from a scoped run is indistinguishable from a full-repo one.
