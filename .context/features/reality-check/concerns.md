## Data integrity

- `confidence.py:56-60` — the two-file confidence mirror is NOT atomic together: `feature.json` is `_atomic_write`-replaced first, then `_mirror_confidence_to_index` does a separate read-modify-write of `INDEX.json`; a crash (or a raise inside the mirror) between the two `tmp.replace` calls leaves `feature.json=AMBIGUOUS` but `INDEX.json` at the old value — exactly the disagreement the plan claims is prevented (plan.md:48,115). Per-file atomicity ≠ cross-file atomicity. Fix: build both payloads first, write both `.tmp` files, then `replace` both back-to-back (still a tiny window, but no read/parse between), or record a journal/sentinel so a re-run reconciles — and document that INDEX is best-effort.
- `confidence.py:91-93` (and `:56-60`) — promote/demote are non-transactional in the same way: `del payload[DEMOTED_FROM_KEY]` + restore is committed to `feature.json`, then the INDEX mirror runs separately; a failure after line 91 loses the stash permanently while INDEX still shows AMBIGUOUS — the inverse is then unrecoverable (no stash to restore). Fix: mirror INDEX before popping the stash, or stage both writes as above.
- `confidence.py:108-111` — INDEX mirror silently no-ops when the feature_id is absent from `idx["features"]` (loop simply matches nothing) and `_atomic_write` rewrites the file unchanged; a demoted feature missing from INDEX leaves the two stores permanently disagreeing with zero signal. Fix: detect "no entry matched" and surface it (return False / log), don't swallow.
- `confidence.py:108-111` — `_mirror_confidence_to_index` updates EVERY matching entry but does not de-dup; if `INDEX.json` ever holds two rows with the same `feature_id` both get flipped, which may not be intended and masks a corrupt index. Low risk but worth an assert/first-match-only decision.
- `confidence.py:111`, `:56`, `:91`; `render.py:16` — JSON is dumped with `indent=2` but WITHOUT `sort_keys=True`, so demote then promote does not round-trip key order: demote appends `confidence_demoted_from` at the end of `feature.json`, promote deletes it, but any other tool that rewrites the file in sorted order will churn the diff; more importantly `_reality-check.json` (`render.py:16`) and `INDEX.json` preserve insertion order, so an upstream producer reordering keys produces noisy no-op diffs. Fix: pass `sort_keys=True` to every `json.dumps` on these persisted artefacts for stable serialization.
- `confidence.py:47` vs `:83` — the idempotency guard compares `prior == ConfidenceLevel.AMBIGUOUS` (enum member) while promote compares `payload.get("confidence") != ConfidenceLevel.AMBIGUOUS.value` (string); both are equivalent only because `ConfidenceLevel(str, Enum)` (enums.py:16) makes str-equality hold — verified, code wins, plan.md:115 is correct — but the divergence is a latent trap if the enum ever stops subclassing `str`. Fix: compare `.value` consistently on both sides.
- `confidence.py:50-55` — demote stashes `prior` only when `DEMOTED_FROM_KEY not in payload`, but the idempotency early-return at `:47-48` only catches an already-AMBIGUOUS feature; if `feature.json` arrives AMBIGUOUS *with no stash* (hand-edited, or a prior crash between the two writes above), demote returns at :48 and promote at :86 both no-op — the feature is wedged at AMBIGUOUS with no recovery path. Fix: on demote, if confidence is already AMBIGUOUS but no stash exists, treat it as un-stashable and log rather than silently leaving it stuck.
- `extract.py:47` — the claim-dedup key `(kind, subject.lower(), object.lower())` collapses distinct claims that differ only by case OR by dotted prefix is NOT the issue (prefix is kept here, stripped later in verify by `_bare_name`) — but two genuinely different prose claims `\`A\` calls \`b\`` and `\`a\` calls \`B\`` collapse to one verdict; more concretely, a `file:line` claim and a `calls` claim can never collide (kind differs) so that's safe, but case-collapse means `Foo:10` vs `foo:10` (`file:line` kind, `object` = line number) dedup only on line number + lowercased path — distinct files differing only in case map to one claim on case-insensitive checkouts. Low severity; flag that dedup is case-insensitive by design (extract.py:47) and confirm that's intended for `file:line` paths.
- `verify.py:67` / `_summarize` — idempotency of the *report* itself depends on claim ordering being stable; claims are appended per-doc in `_CANONICAL_DOCS` order (verify.py:41) then per-regex order (extract.py:61-68), and `re.finditer` is source-order — so ordering is deterministic and the report round-trips, GOOD. No issue; recorded as confirmation that `_reality-check.json` is sort-stable on re-run for identical inputs.

## Security

- `verify.py:157-202` (`_resolve_cited_path`) — **doc-controlled path traversal / arbitrary file read**. `path_str` comes straight from a doc citation parsed by `_FILE_LINE_RE` (`extract.py:32-34`), whose class `[\w./\-]+` permits `../`. The body does `on_disk = repo_root / path_str` (`verify.py:181`) and `repo_root / candidates[0]` (`verify.py:193`) with **no containment check** against `repo_root`. A `plan.md`/`concerns.md` line like `` `../../../../etc/cron.d/x:1` `` or `` `../../.git/config:1` `` resolves outside the repo; `_verify_claim` then `resolved.open("rb")` and counts lines (`verify.py:144`), so the verdict ("file has N lines" vs "file not found") is an **oracle that confirms existence and line count of arbitrary host files**. Since `.context/` docs and citations are authored by LLM council agents (untrusted, attacker-influenceable via prompt-injected source comments), this is a real read primitive. Load-bearing mitigation: **none** — plan.md never lists path-confinement as a property; it only claims "deterministic precedence." Fix: `resolved.resolve().relative_to(repo_root.resolve())` guard before any read, reject on `ValueError`.

- `verify.py:63` + `verify.py:376-384` (`_repo_root_from_meta`) — **`repo_root` is taken verbatim from unvalidated `meta.json["root"]`**. `reality_check_feature` passes `Path(_repo_root_from_meta(context_dir) or context_dir.parent)` as the join base for every citation read. A tampered/poisoned `meta.json` (a writable artefact in `.context/`) sets `root` to `/` or `/home/<user>`, turning the traversal above into a **whole-filesystem-relative** read oracle (e.g. citation `etc/shadow:1` against `root:/`). It is never canonicalised, never checked to be a real ancestor of `context_dir`. Load-bearing mitigation: **none** — plan.md "Decisions" lists meta as trusted input implicitly; no validation step exists. Fix: require `meta["root"]` to resolve to a real ancestor of `context_dir`, else fall back to `context_dir.parent`.

- `cli/reality_check.py:39` + `verify.py:30` — **feature-id path injection via `--feature`**. `feature_id = parsed.get("feature")` (no sanitisation of `..`/`/`) flows into `feat_dir = context_dir / "features" / feature_id` (`verify.py:30`) and `context_dir / "features" / feature_id` again in the CLI write path (`cli/reality_check.py:62-69`). `--feature ../../../tmp/evil` escapes the `features/` dir; combined with `write_report` / `demote_feature_on_contradiction` this is a **write** primitive (report JSON/MD + confidence mutation) outside the intended feature folder, not just a read. The `feat_dir.is_dir()` check (`verify.py:31`) gates reads but the CLI still writes to the traversed dir if it exists. Load-bearing mitigation: **none** — plan.md treats `feature_id` as a trusted key. Fix: reject `feature_id` containing `/`, `\`, or `..`, or assert `feat_dir.resolve()` is a direct child of `context_dir/"features"`.

- `verify.py:143-146` — **no size / symlink / file-type guard on the read target**. After resolution, `sum(1 for _ in resolved.open("rb"))` streams the *entire* file to count lines (`verify.py:144`). A citation pointing at a multi-GB file, a FIFO/device node, or a symlink into `/proc`/`/dev` (e.g. `../../../../dev/zero:1` if it had a matching name, or a doc-planted symlink under the repo) causes **unbounded read / hang / DoS** in what is advertised as a "read-mostly auditor." `resolved.is_file()` (`verify.py:141`) follows symlinks and is true for many special cases. Load-bearing mitigation: **none** — the `except OSError → ambiguous` (`verify.py:145`) catches errors but not a successful-but-enormous/blocking read. Fix: `os.lstat` reject symlinks, cap bytes read (stop after N lines / max size), reject non-regular files.

- `confidence.py:97-111` (write-back to `feature.json` + `INDEX.json`) — **integrity: a poisoned doc can silently demote a feature's confidence**. `--demote` flips confidence to AMBIGUOUS purely on `report.has_contradictions` (`cli/reality_check.py:65-67`), which an attacker-authored false claim in a doc can force (any unresolved symbol → `contradicted`). In a council loop this lets a single tampered doc degrade the trust signal of arbitrary features. Not a memory-safety issue but a **trust-signal tampering** path; the demote is "idempotent and reversible" per plan.md `verify.py`-side decisions, but reversibility doesn't prevent the false demote. Mitigation: acceptable for a local dev tool *only if* the upstream docs are trusted; flag that the threat model must state docs are trusted input, which the path-traversal findings above already violate.

## Product surface

- caller runs `reality-check --feature X --demote` on docs they've just fixed — clean report restores the stashed confidence via `promote_feature_on_clean` (`cli/reality_check.py:68-69` → `confidence.py:64-94`) — gap: this PROMOTE also writes `feature.json` + `INDEX.json` (`confidence.py:89-93`) but the printed report (`render.py:21-59`) never says confidence moved; the only signal is a discarded bool. `--demote` is documented as "demote on contradiction" (`cli/reality_check.py:11-14`) yet silently mutates the index on the clean path too — surprising side effect, invisible in output.
- caller runs it on a feature whose docs contain no `calls`/`uses`/`file:line`/`has_method` backtick shape — report says "0 concrete claim(s)… 0 contradicted", exit 0, no sections (`render.py:25-31`, `models.py:43-44`) — gap: a feature with zero checkable claims is indistinguishable from a fully-verified one. "all good" pass is really "nothing was checked"; the caller reads it as fact-checked when no fact was checked.
- caller writes a true architectural claim in plain prose ("the orchestrator loads artefacts once, then dispatches per claim") — extracted: nothing; only the four regexes in `extract.py:24-38` fire, all requiring backtick-fenced tokens in rigid subject/verb/object or file:line phrasings — gap: the vast majority of doc prose is invisible to the checker. "fact-check the docs against the AST" overstates coverage; it fact-checks a narrow citation grammar, not the prose.
- caller cites `` `models.py:7` `` and the file exists with ≥7 lines — verdict "verified" (`verify.py:143-152`) — gap: verification counts lines, never reads line 7's content. Any in-range number "verifies"; an edit that shifts the symbol off line 7 (but keeps the file long enough) still passes. The `file:line` verdict means "this file is at least N lines", not "the cited thing is at line N".
- caller writes `` class `Claim` has method `to_dict` `` — verdict "verified" iff both `Claim` and `to_dict` exist as symbols anywhere in the map (`verify.py:115-123`) — gap: no ownership check. `to_dict` is a method on several frozen dataclasses; a claim binding it to the wrong class still verifies. Conversely the "contradicted" branch fires when either name is merely absent from the map — a renamed class or a method indexed under a different name reads as a hard contradiction (false positive), which on `--demote` flips the feature to AMBIGUOUS over a naming mismatch, not a real lie.
- caller relies on the report to decide the council exit gate — exit code is `1 if report.has_contradictions else 0` (`cli/reality_check.py:75`), driven solely by `contradicted > 0` (`models.py:43-44`) — gap: ambiguous verdicts (external refs, missing call edges, multi-match basenames — `verify.py:99,110,198`) never affect exit code or trigger demotion. A doc that is entirely unverifiable-but-suspicious passes the gate exactly like a verified one.


## Cross-review notes

- **Strong consensus (security + product + database): the trust posture is implicit and unstated, yet four LLM/attacker-influenceable inputs reach filesystem I/O with no confinement.** Security enumerated all four — doc citation path traversal in `_resolve_cited_path` (`verify.py:157-202`), unvalidated `meta.json["root"]` as the join base (`verify.py:376-384`), `--feature` path injection into a *write* primitive (`cli/reality_check.py:39,62-69`; `verify.py:30`), and no size/symlink guard on read targets (`verify.py:143-146`). Product independently confirmed the over-trust from the user-surface side. Agreed top-priority fix: containment-check every resolved path against `repo_root.resolve()` (reject on `ValueError`), reject `feature_id` containing `/`/`\`/`..`, validate `meta["root"]` is a real ancestor, and reject non-regular/symlink targets with a byte cap. The plan must name docs/citations/meta/`--feature` as untrusted at a trust boundary.
- **Consensus (database + product + security): `--demote` silently *promotes* on a clean run and the mutation is invisible in the report.** DB framed the cross-file `feature.json`→`INDEX.json` mirror as non-atomic (`confidence.py:56-60,91-93`); product framed the same write as a surprising, unreported side effect (`render.py` prints no confidence delta); security framed it as a trust-signal tampering path (a poisoned doc can force the demote). Agreed: make the two-file write atomic-together (stage both `.tmp`, replace back-to-back), and have `render_report_md` state "confidence demoted/restored/unchanged."
- **Consensus (database + product): a zero-claim or all-ambiguous feature reads as a clean pass.** DB confirmed report ordering is deterministic (good), but product + the exit gate (`cli/reality_check.py:75`, `models.py:43-44`) show exit code keys only on `contradicted > 0` — ambiguous/unverifiable docs pass identically to verified ones, and "0 claims" is indistinguishable from "fully verified." Agreed: surface a "nothing checkable" headline and consider gating on ambiguous count.
- **Verified-not-a-bug (database, confirmed): demote/promote strict-inverse idempotency holds** because `ConfidenceLevel` subclasses `str` (`confidence.py:47` enum vs `:83` `.value` comparison) — the plan's claim is accurate; pin both comparisons to `.value` as a latent-trap guard, not a live fix.
