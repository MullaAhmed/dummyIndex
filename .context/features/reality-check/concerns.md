## Data integrity

- **Prose-rooted call/uses edges silently dropped — partial false-negative surface (`reality_check.py:474-493`).** `_load_call_edges` normalizes every node `label`, but module-level nodes in `symbol-graph.json` carry a docstring *as the label* (verified: 531 of 6084 calls/uses edges have a source label like `"Token-usage reporting over Claude Code transcripts…"`). After `clean()` these become multi-word strings that no parsed claim can match, so any legitimate `` `module` calls `X` `` claim whose subject is a docstring-labeled node resolves `ambiguous` instead of `verified`. Not a corruption (no false `contradicted`), but the plan's Open-question about label coupling (`plan.md:131-133`) understates it: the mismatch is already live for ~9% of edges, not a hypothetical future format drift.

- **Graph reads `label`, never `norm_label`, yet `norm_label` exists (`reality_check.py:478-484` vs node keys `['label','norm_label',…]`).** The graph builder already publishes a normalized field; the verifier re-implements its own `rstrip("()").lstrip(".").rsplit(".")` normalization over raw `label`. Two independent normalizers over the same data is exactly the silent-drift coupling the plan flags — a builder that populates `norm_label` correctly but changes `label` would desync verdicts with zero alert. Prefer reading `norm_label` when present.

- **Normalizer divergence: `rstrip("()")` (graph) vs `endswith("()")`+slice (claims) (`reality_check.py:481` vs `reality_check.py:411-412`).** Claim side strips a trailing `()` only as a suffix; graph side strips *all* trailing `(` and `)` chars greedily. For ordinary identifiers these agree, but they are not the same function — a label ending in a stray paren normalizes differently on each side, re-introducing match asymmetry. Two code paths claiming to "normalize the same way" (`plan.md:47-48`) should call one shared helper.

- **`has_method` verdict does not verify the method belongs to the class (`reality_check.py:265-273`).** Both `cls` and `method` are checked for existence in the flat `symbol_names` set independently; a `` class `Foo` has method `bar` `` claim is `verified` whenever *any* class named `Foo` and *any* symbol named `bar` exist anywhere in the repo — even if `bar` lives on an unrelated class. This is a false-positive (over-verification) on a name collision; the symbol-graph `method` relation (present in edges) is the correct evidence source and is not consulted.

- **`symbol_names` collisions make calls/uses over-permissive (`reality_check.py:243-258`, `_load_symbols:446-455`).** Verification keys on bare names in a flat `frozenset`; `_bare_name` discards the dotted owner. Two distinct `save` methods on different classes collapse to one name, so a call-edge between *either* pair satisfies the claim. Acceptable given the regex extracts bare names, but worth noting the verdict granularity is name-level, not symbol-level — a real claim about `A.save → B.save` can be "verified" by an unrelated `C.save → D.save` edge.

- **Write integrity is sound — atomic, schema-stamped, but bypasses the canonical helper (`reality_check.py:592-596` vs `data-access.md:5-9`).** `_atomic_write` correctly does tmp+`replace`, and reports carry `schema_version` (`reality_check.py:60,424`). However it is a private re-implementation of `write_text_atomic` (`atomic_io.py:11-24`) — the convention names that as the canonical helper and warns against per-module reimplementations. Functionally equivalent here (no EOF normalization, byte-faithful), but it sidesteps the hash-baseline contract the convention centralizes; route through `write_text_atomic`.

- **JSON outputs are not sorted-key — breaks byte-identical re-run guarantee (`reality_check.py:546`, `:640`, `:675`, `:695`).** `data-access.md:13` requires deterministic serialization (`sort_keys=True`); the reality-check writers emit `json.dumps(payload, indent=2)` *without* `sort_keys`. Report dicts are built in fixed order so they're stable, but the `feature.json`/`INDEX.json` mutations (demote/promote/mirror) round-trip an arbitrarily-ordered loaded dict and re-emit it unsorted — a key-order churn that can dirty diffs on every `--demote` run. The `_reality-check.json` artefact is also un-gitignored leaf output and should follow the sorted convention.

- **Demote/promote idempotency and inverse-pairing are correct (`reality_check.py:610-678`).** Verified: demote no-ops once `confidence == AMBIGUOUS` (`:631`) and only stashes a valid prior with no existing stash (`:634-639`); promote acts only on AMBIGUOUS + valid stash and pops it (`:667-674`). Inverse pairing holds. Minor inconsistency: demote compares `prior == ConfidenceLevel.AMBIGUOUS` (enum, `:631`) while promote compares `payload.get("confidence") != ConfidenceLevel.AMBIGUOUS.value` (string, `:667`); both happen to work via the `str, Enum` mixin (`enums.py:16`) but the asymmetry is fragile — if `ConfidenceLevel` ever drops the `str` mixin, line 631's comparison silently becomes always-False and demotion would re-stash a real prior over AMBIGUOUS. Pin both to `.value`.

- **`--demote` mirror is not transactional across two files (`reality_check.py:640-644`, `:675-677`).** `feature.json` is written atomically, then `_mirror_confidence_to_index` writes `INDEX.json` atomically — but the pair is not atomic together. A crash between the two leaves `feature.json` at AMBIGUOUS with `INDEX.json` still showing the old confidence (or vice-versa on promote). No relational DB to enforce this, but the gap means the two confidence mirrors can disagree; mirror failure is also silent (`:686-687` returns on missing index). Low likelihood, but it is a genuine cross-artefact consistency hole the plan's "mirror through `_mirror_confidence_to_index`" framing (`plan.md:67-69`) treats as a single guaranteed step.

- **Claim dedup key can collapse semantically distinct file:line claims (`reality_check.py:194`, `:214-215`).** `_extract_claims` dedupes on `(kind, subject.lower, object.lower)`; for `file:line` the object is the line *number*. `` `foo.py:42` `` cited in two docs with different surrounding `text` dedup to one claim — fine. But `Foo.bar` vs `foo.BAR` (calls) collapse via `.lower()`, and Python is case-sensitive, so a real claim about `Config` and one about `config` merge and only the first survives. Edge case, but the lowercase key trades a small false-negative risk for dedup.

## Security

- **Feature-id path traversal — arbitrary directory escape via `--feature` (`reality_check.py:148`, `cli/reality_check.py:39-45,62`).** `feature_id` flows from the CLI `--feature` value into `context_dir / "features" / feature_id` with zero sanitization (no `..`/`/`/absolute-path rejection, no `resolve()`-under-root containment check). `dummyindex context reality-check --feature ../../../../etc` resolves `feat_dir` outside `.context/`; `--feature /abs/path` (absolute RHS of `/`) discards the prefix entirely per `pathlib` semantics and lands at `/abs/path`. The `.is_dir()` guard (`:149`) only gates *existence*, not *containment*, so any readable directory on disk becomes a "feature folder" whose `plan.md`/`concerns.md`/legacy docs are read (`:159-167`). Same untrusted id reaches `features_dir / report.feature_id` in demote/promote (`:622`, `:659`) and `read_feature_files` (`dev_pick.py:333`) — a `--demote` run could then atomically rewrite a `feature.json`/`INDEX.json` *outside* the intended tree. Local CLI so blast radius is the invoking user, but it breaks the repo-root trust boundary the tool implies. Fix: reject `feature_id` containing `/`, `\`, `..`, or NUL and assert `feat_dir.resolve()` is relative-to `(context_dir/"features").resolve()` before any read/write.

- **Doc-driven file read outside repo root — citation path escape (`reality_check.py:331-335`, `:291-294`).** `_resolve_cited_path` joins an attacker/author-controlled citation string straight onto `repo_root`: `on_disk = repo_root / path_str` (`:331`) with `path_str` taken verbatim from a `` `…:NN` `` token in a doc (`_FILE_LINE_RE`, `:82-83`). The regex character class `[\w./\-]+` admits `../` sequences and the `\.` segment, so a doc line like `` `../../../../etc/passwd:1` `` makes `on_disk` escape the repo, `.is_file()` true, and the file gets opened and line-counted (`:294`, `resolved.open("rb")`). No path traversal guard, no `resolve()`-under-root check before the open. It only ever *counts lines* (the content isn't emitted into the report — line numbers, not bytes, surface), so this is an existence/line-count oracle rather than a content-exfil primitive, but it confirms reads occur on arbitrary host paths driven purely by doc text. Fix: containment-check `on_disk.resolve()` (and the `feat_dir / path_str` branch at `:334-335`) against `repo_root.resolve()` before `.is_file()`.

- **`repo_root` is attacker-influenced via `meta.json` `root` (`reality_check.py:181`, `:526-534`).** The repo root used for every citation resolution is read from `meta.json`'s `root` field with no validation (`_repo_root_from_meta`). A tampered `meta.json` (same-trust as the rest of `.context/`, but a supply-chain/checked-in-context vector) repoints `repo_root` to any absolute path, compounding the citation-escape above into reads anywhere the user can read. Treat `root` as a boundary value: resolve it and confirm it's an existing dir, and prefer `context_dir.parent` when it disagrees.

- **ReDoS in claim-extraction regexes — checked, CLEARED (`reality_check.py:74-88`).** `_CALL_RE`/`_USES_RE`/`_HAS_METHOD_RE`/`_FILE_LINE_RE` use only single, non-overlapping character-class quantifiers (`[\w.]*`, `[\w]*`, `[\w./\-]+`, `\d+`) with no nested quantifiers, no quantified alternation, and no backreferences — linear-time matching, no catastrophic backtracking. `re.finditer` over doc text (`:208-215`) is safe even on adversarial input. No fix needed; recorded so a future regex change (e.g. adding `(\w+)*` or `(a|a)*` shapes) is a known-sensitive edit.

- **Symbol-existence verdict is name-level, not provenance-checked — over-verification (`reality_check.py:243-273`).** Restating the data-integrity collision concern under a security lens: because verification keys on bare names in a flat `frozenset` (`has_method` at `:268`, calls/uses at `:243`), a malicious or careless doc can get a *false* `verified` for a `` class `Foo` has method `bar` `` or `` `A.x` calls `B.y` `` claim by exploiting any same-named symbol anywhere in the repo. This is an integrity weakness in the *attestation* the tool produces — its whole purpose is to certify grounding, and a name collision lets an unfounded claim pass certification. Not a host-compromise, but it undermines the trust signal `--demote`/`verified` carries; the symbol-graph `method`/owner relation is the sound evidence source and isn't consulted.

- **No size/symlink guard on read targets (`reality_check.py:164`, `:294`).** Doc reads use `errors="ignore"` and citation targets are line-counted with `resolved.open("rb")` with no size cap and no symlink rejection. Combined with the traversal above, a citation pointing at a symlink to a huge/pseudo file (e.g. a FIFO or `/dev`-style path on some systems) could hang or balloon memory during the `sum(1 for _ in resolved.open("rb"))` count (`:294`). Low likelihood on a normal repo; pair the containment check with `is_symlink()` rejection and an upper line/byte bound.

## Product surface

- **Promote is silent and unobservable from the CLI report (`cli/reality_check.py:65-69`, `reality_check.py:648-678`).** On a clean `--demote` run the tool calls `promote_feature_on_clean`, which may restore a stashed confidence (e.g. flip a feature back from AMBIGUOUS to HIGH) — but `render_report_md` (`reality_check.py:551-589`) prints only verified/contradicted/ambiguous counts. A user runs `reality-check --feature X --demote`, sees "0 contradicted", and has no signal that their feature's confidence was just *mutated upward* in `feature.json` + `INDEX.json`. The single most surprising side effect of `--demote` (a confidence write on a *clean* report) is invisible in the output. Gap: report should state "confidence restored HIGH→… / demoted …→AMBIGUOUS / unchanged".

- **`--demote` name implies one direction but does two (`cli/reality_check.py:11-14`, `:65-69`).** The flag is named `--demote` and the docstring frames it as "a report with contradictions flips confidence to AMBIGUOUS", yet the same flag also *promotes* (restores) on a clean run. A user wanting only the safety demotion (never an automatic re-promotion) has no way to opt out — the promote is bundled. Gap: an operator who manually set a confidence after a contradiction will see it silently overwritten by the stashed value on the next clean `--demote`.

- **Zero-claim feature reports as a clean pass — false "all good" (`reality_check.py:419-431`, `render_report_md:551-561`).** A feature whose `plan.md`/`concerns.md` contain no `` `X` calls `Y` ``/`` `f.py:NN` ``-shaped backtick claims yields `claims_total=0`, `contradicted=0`, exit `0`, and a report reading "0 concrete claim(s) … 0 verified". There is no distinction between "checked and grounded" and "nothing was checkable." A doc full of prose assertions with no machine-parseable claim shape passes reality-check trivially. Gap: the headline can't tell a reviewer the tool found nothing to verify vs. verified everything.

- **Only four claim shapes are checkable; most doc prose is invisible to the checker (`reality_check.py:74-88`).** The four regexes match only backtick-fenced `` `A` calls `B` ``, `` `A` uses `B` ``, `` `Cls` has method `m` ``, and `` `path:NN` ``. Prose like "the demote loop is idempotent", "verification loads the backbone once", or a citation written `reality_check.py:610` *without backticks* are never extracted — so a doc can be riddled with wrong narrative claims and still pass. The tool verifies *grounding of a narrow citation grammar*, not the doc's actual assertions; a reader trusting a clean report over-reads its scope. Gap: the report never states which claim-grammars it covers, so "verified" reads stronger than it is.

- **Ambiguous is the dominant bucket but the report gives it no triage priority (`reality_check.py:577-588`).** Per the prior Data-integrity findings (`concerns.md:3` — ~9% of edges docstring-labelled → ambiguous; `:11` name collisions; `:259-263` "no direct call edge" → ambiguous), a large share of legitimate claims land in Ambiguous. The report lists them flat with reason "—" or "both symbols exist but no direct call edge", advising "worth a manual look." For a feature with dozens of ambiguous claims this is an undifferentiated wall with no ranking of which are likely tool-limitation (indirect call) vs. real doc drift. Gap: actionability collapses when ambiguous count is high — the user can't tell noise from signal.

- **Contradicted `has_method` can be a false positive the user can't distinguish from a real one (`reality_check.py:265-273`).** A `` class `Foo` has method `bar` `` claim is `verified` if *any* `Foo` and *any* `bar` exist (prior `concerns.md:9` over-verification), but is `contradicted` with reason "method not in symbols" when the method genuinely exists under a different spelling or is dynamically added. The persona is told to "revise or remove" (`reality_check.py:567-570`) a claim that may be correct — pushing a doc edit that *removes a true statement* to satisfy the checker. Gap: a contradiction verdict carries no confidence qualifier, yet the remediation copy treats every contradiction as the doc being wrong.

- **`file:line` is checked against current line *count*, not content — citations rot silently after edits (`reality_check.py:291-302`).** A `` `foo.py:42` `` claim is `verified` whenever `foo.py` exists and has ≥42 lines; it does not check that line 42 still contains the cited symbol. After any edit that shifts or replaces line 42's content while keeping the file long enough, the citation stays "verified" while pointing at the wrong line. Gap: the strongest-sounding verdict (`verified` on a precise `file:line`) is the weakest guarantee — it's an in-bounds check, not a content check, and the report doesn't say so.

- **Legacy-doc claims attributed to docs the user may have already deleted (`reality_check.py:62-71`).** `_CANONICAL_DOCS` still scans five v0.14 essay docs (`architecture.md`…`product.md`). A re-councilled feature that kept a stale `security.md` around will have its claims extracted and possibly contradicted, with `source_file` pointing at a doc the persona no longer maintains — the remediation ("revise on next council pass") targets a file outside the current `plan.md`+`concerns.md` workflow. Gap: contradictions can be raised against orphaned docs, muddying which file the user must actually edit.
