## Data integrity

- **[RESOLVED — `outstanding-audit-fixes`]** `confidence.py:56-60` — the two-file confidence mirror was NOT atomic together: `feature.json` was `_atomic_write`-replaced first, then `_mirror_confidence_to_index` did a separate read-modify-write of `INDEX.json`; a crash between the two `tmp.replace` calls left `feature.json=AMBIGUOUS` but `INDEX.json` at the old value. **Fixed:** both payloads are now staged to `.tmp` siblings *before* either replace, then committed back-to-back in `_commit_confidence_change` (`confidence.py:160-203`). **Residual (documented best-effort, still open by design):** the two `Path.replace` calls are distinct syscalls, so a crash *between* them leaves INDEX.json lagging feature.json by one transition — reconciled on the next run (the mirror is idempotent). The full journal/sentinel crash-reconciliation was deliberately left out (spec.md "Deliberately left"): narrowed-window + correct ordering + best-effort documentation delivered instead.
- **[RESOLVED — `outstanding-audit-fixes`]** `confidence.py:91-93` (and `:56-60`) — promote/demote were non-transactional: `del payload[DEMOTED_FROM_KEY]` was committed to `feature.json`, then the INDEX mirror ran separately; a failure after the pop lost the stash permanently. **Fixed:** the stash is now popped only from the *staged* payload, and INDEX.json is staged before either replace (`_commit_confidence_change` stages INDEX first, then feature.json, then replaces back-to-back). A raise before the first replace leaves the on-disk feature.json — including its stash — untouched, so a failure can never strand the feature at AMBIGUOUS with no restore path.
- **[RESOLVED — `outstanding-audit-fixes`]** `confidence.py:108-111` — the INDEX mirror silently no-op'd when the feature_id was absent from `idx["features"]` and rewrote the file unchanged with zero signal. **Fixed:** `_mirror_confidence_to_index` now tracks a `matched` flag and returns `(index_path, index_tmp, matched)`; `_commit_confidence_change` propagates `matched` as its return — `True` if at least one entry matched, `False` when none did (or INDEX absent/unreadable) — surfacing a mirror that landed nowhere rather than swallowing it.
- **[RESIDUAL — open]** `confidence.py:205-220` — `_mirror_confidence_to_index` still updates EVERY matching entry without de-dup; if `INDEX.json` ever holds two rows with the same `feature_id` both get flipped. Low risk; an assert / first-match-only decision was out of scope for `outstanding-audit-fixes`.
- **[RESOLVED — `outstanding-audit-fixes`]** `confidence.py`, `render.py` — JSON was dumped `indent=2` but WITHOUT `sort_keys=True`, producing noisy no-op diffs on key reorder. **Fixed:** `feature.json` and `INDEX.json` (both staged in `_commit_confidence_change`/`_mirror_confidence_to_index`) and `_reality-check.json` (`render.py:18`) are now all dumped `sort_keys=True`. `render.py` also switched off the retired local `_atomic_write` onto the unified `write_text_atomic` (`render.py:7,19-22`).
- **[RESIDUAL — open]** `confidence.py:88` vs `:140` — the idempotency guard compares `prior == ConfidenceLevel.AMBIGUOUS` (enum member) while promote compares `payload.get("confidence") != ConfidenceLevel.AMBIGUOUS.value` (string); both are equivalent only because `ConfidenceLevel(str, Enum)` (enums.py:16) makes str-equality hold — verified, code wins — but the divergence is a latent trap if the enum ever stops subclassing `str`. Fix: compare `.value` consistently on both sides. Not addressed by `outstanding-audit-fixes` (the return type widened, the comparisons did not).
- **[RESIDUAL — open]** `confidence.py:90-94` — demote stashes `prior` only when `DEMOTED_FROM_KEY not in payload`, but the idempotency early-return at `:88` only catches an already-AMBIGUOUS feature; if `feature.json` arrives AMBIGUOUS *with no stash* (hand-edited, or a prior crash mid-mirror), demote and promote both no-op — the feature is wedged at AMBIGUOUS with no recovery path. Fix: on demote, if confidence is already AMBIGUOUS but no stash exists, treat it as un-stashable and log rather than silently leaving it stuck. Still open after `outstanding-audit-fixes` (the atomic-mirror fix narrows the *crash window* that can produce a stashless AMBIGUOUS but does not add the un-wedge recovery path).
- `extract.py:47` — the claim-dedup key `(kind, subject.lower(), object.lower())` collapses distinct claims that differ only by case OR by dotted prefix is NOT the issue (prefix is kept here, stripped later in verify by `_bare_name`) — but two genuinely different prose claims `\`A\` calls \`b\`` and `\`a\` calls \`B\`` collapse to one verdict; more concretely, a `file:line` claim and a `calls` claim can never collide (kind differs) so that's safe, but case-collapse means `Foo:10` vs `foo:10` (`file:line` kind, `object` = line number) dedup only on line number + lowercased path — distinct files differing only in case map to one claim on case-insensitive checkouts. Low severity; flag that dedup is case-insensitive by design (extract.py:47) and confirm that's intended for `file:line` paths.
- `verify.py:67` / `_summarize` — idempotency of the *report* itself depends on claim ordering being stable; claims are appended per-doc in `_CANONICAL_DOCS` order (verify.py:41) then per-regex order (extract.py:61-68), and `re.finditer` is source-order — so ordering is deterministic and the report round-trips, GOOD. No issue; recorded as confirmation that `_reality-check.json` is sort-stable on re-run for identical inputs.

## Security

- **[RESOLVED — `outstanding-audit-fixes`]** `verify.py:157-202` (`_resolve_cited_path`) — **doc-controlled path traversal / arbitrary file read**. `path_str` comes from a doc citation parsed by `_FILE_LINE_RE` (`extract.py:32-34`), whose class permits `../`; the resolver joined `repo_root / path_str` with **no containment check**, making the line-count verdict an oracle over arbitrary host files. **Fixed:** every concrete-path return branch of `_resolve_cited_path` now routes through `_confine` → `resolve_under_root` (re-exported from `pipeline/io`) against `repo_root` (or `feat_dir` for the feature-own-doc branch) — `verify.py:195-264`. An out-of-root path (absolute, `../` escape) yields a `("contradicted", "cited path resolves outside the repository root")` verdict and the file is never opened. Note `_FILE_LINE_RE` requires a `.ext` before `:line`, so extensionless paths like `/etc/passwd:1` are grammar-rejected upstream; the confinement test must use extension-bearing proof strings. **Residual (symlink-swap, documented):** the symlink is resolved *before* the confinement check — an in-root symlink is legitimate, an escaping one is contradicted — but a post-resolution symlink swap (TOCTOU between `resolve_under_root` and the `open`) remains a documented residual, accepted as out of scope for a local-dev auditor.

- **[RESOLVED — `outstanding-audit-fixes`]** `verify.py:63` + `_repo_root_from_meta` — **`repo_root` was taken verbatim from unvalidated `meta.json["root"]`** (`Path(_repo_root_from_meta(...) or context_dir.parent)`), so a poisoned `meta.json` setting `root` to `/` turned every citation into a whole-filesystem read oracle. **Fixed:** `reality_check_feature` now anchors `repo_root` via `_trusted_repo_root` (`verify.py:468-488`). The trusted anchor is the **resolved git toplevel** (`_git_toplevel` via `is_git_repo`/`resolve_git_dir`, `verify.py:437-455`), *not* "any ancestor of `context_dir`" (which admitted `/`). `meta.json["root"]` is honored only when it resolves to exactly that toplevel (resolved-equality, not string-equality); otherwise it falls back to the anchor, and with no git toplevel at all to the historical `context_dir.parent`. A nested-`.context/` fixture (subtree where `context_dir.parent` is not the repo root) with a real `.git` exercises the non-trivial case.

- **[RESOLVED — `outstanding-audit-fixes`]** `cli/reality_check.py:39` + `verify.py:30` — **feature-id path injection via `--feature`** turned an unsanitised id into a write primitive (`write_report` / confidence mutation) outside `features/`. **Fixed:** the id is rejected at the **CLI boundary** (`cli/reality_check.py:50-57`, non-zero exit, before any read or write) *and* inside the verifier via `_reject_unsafe_feature_id` (`verify.py:36-49`, raises `ValueError`) — both reject any `/`, `\`, `..`, or NUL. Defence in depth: neither entry point trusts the value.

- **[RESOLVED — `outstanding-audit-fixes`]** `verify.py:143-146` — **no size / symlink / file-type guard on the read target**: `sum(1 for _ in resolved.open("rb"))` streamed the entire file, so a multi-GB blob, FIFO/device, or symlink into `/proc` caused unbounded read / hang. **Fixed:** before the line-count read, the resolved path passes `is_safe_read_target(resolved, max_bytes=_MAX_CITED_FILE_BYTES)` (16 MiB, `verify.py:175-180`) — a symlink (`S_ISLNK`), non-regular file, or oversize blob yields `ambiguous` ("cited path is not a safe regular file to read") and is never streamed.

- **[PARTIALLY RESOLVED — `outstanding-audit-fixes`]** `confidence.py:97-111` — **a poisoned doc can silently demote a feature's confidence**. The *silent* half is fixed: `--demote` now prints the confidence delta (`demoted/restored/unchanged`) at the CLI off the widened `ConfidenceTransition | None` return (`cli/reality_check.py:76-89`, `confidence.py:42-54`), so the mutation is no longer invisible. **Residual (open, by threat model):** `--demote` still flips on `report.has_contradictions`, which an attacker-authored false claim can force; the underlying trust-signal-tampering exposure is now *gated by the path-confinement + trusted-root fixes above* (a poisoned doc can no longer also leak host files), but a false demote from an in-repo false claim is still possible and acceptable only if the upstream docs are treated as trusted input. The demote remains idempotent and reversible.

## Product surface

- **[RESOLVED — `outstanding-audit-fixes`]** caller runs `reality-check --feature X --demote` on docs they've just fixed — clean report restores the stashed confidence via `promote_feature_on_clean`. **Was:** this PROMOTE wrote `feature.json` + `INDEX.json` but the only signal was a discarded bool — a surprising, invisible side effect on the clean path. **Fixed:** `promote`/`demote` now return a `ConfidenceTransition | None`, and the CLI prints one stderr line per run — `demoted <prior>→AMBIGUOUS`, `restored AMBIGUOUS→<value>`, or `unchanged` (`cli/reality_check.py:76-89`). The mutation is no longer invisible. (The rendered Markdown report still does not embed the delta — it is surfaced at the CLI, which is where the `--demote` action lives.)
- caller runs it on a feature whose docs contain no `calls`/`uses`/`file:line`/`has_method` backtick shape — report says "0 concrete claim(s)… 0 contradicted", exit 0, no sections (`render.py:25-31`, `models.py:43-44`) — gap: a feature with zero checkable claims is indistinguishable from a fully-verified one. "all good" pass is really "nothing was checked"; the caller reads it as fact-checked when no fact was checked.
- caller writes a true architectural claim in plain prose ("the orchestrator loads artefacts once, then dispatches per claim") — extracted: nothing; only the four regexes in `extract.py:24-38` fire, all requiring backtick-fenced tokens in rigid subject/verb/object or file:line phrasings — gap: the vast majority of doc prose is invisible to the checker. "fact-check the docs against the AST" overstates coverage; it fact-checks a narrow citation grammar, not the prose.
- caller cites `` `models.py:7` `` and the file exists with ≥7 lines — verdict "verified" (`verify.py:143-152`) — gap: verification counts lines, never reads line 7's content. Any in-range number "verifies"; an edit that shifts the symbol off line 7 (but keeps the file long enough) still passes. The `file:line` verdict means "this file is at least N lines", not "the cited thing is at line N".
- caller writes `` class `Claim` has method `to_dict` `` — verdict "verified" iff both `Claim` and `to_dict` exist as symbols anywhere in the map (`verify.py:115-123`) — gap: no ownership check. `to_dict` is a method on several frozen dataclasses; a claim binding it to the wrong class still verifies. Conversely the "contradicted" branch fires when either name is merely absent from the map — a renamed class or a method indexed under a different name reads as a hard contradiction (false positive), which on `--demote` flips the feature to AMBIGUOUS over a naming mismatch, not a real lie.
- caller relies on the report to decide the council exit gate — exit code is `1 if report.has_contradictions else 0` (`cli/reality_check.py:75`), driven solely by `contradicted > 0` (`models.py:43-44`) — gap: ambiguous verdicts (external refs, missing call edges, multi-match basenames — `verify.py:99,110,198`) never affect exit code or trigger demotion. A doc that is entirely unverifiable-but-suspicious passes the gate exactly like a verified one.
