## Data integrity

There is no relational database in this feature. Its durable state is the
filesystem-backed `.context/config.json` intent ledger and Claude project JSON,
plus external Claude CLI materialisation.

- **High — concurrent config reconcilers can silently lose intent updates.** Both reconciliation helpers read a complete immutable `Config`, derive an appended `wired` tuple, and write the whole document (`dummyindex/context/domains/config.py:567-619`, `dummyindex/context/domains/config.py:622-659`). `write_config` uses the single fixed `config.json.tmp` path and an unguarded replace, with no lock or compare-and-swap (`dummyindex/context/domains/config.py:662-682`). The installer sequences those operations inside one process (`dummyindex/installer/install.py:634-643`), but two install/ingest processes can read the same baseline and commit different ledgers; the last whole-document replacement wins, while the shared temporary path can also be replaced or removed underneath the other writer. Use a unique same-directory temporary file for safe replacement and serialize the full read-modify-write transaction (or reject/retry when the source revision changed).

- **High — project settings merges have the same last-writer-wins race, including with the external Claude CLI.** Hook, marketplace, and plugin operations load the entire JSON object, mutate it in memory, and rewrite it (`dummyindex/context/claude_settings.py:64-112`, `dummyindex/context/claude_plugins.py:105-124`, `dummyindex/context/claude_plugins.py:142-155`). Their common writer again uses one fixed `.json.tmp` sidecar without a lock or revision check (`dummyindex/context/claude_settings.py:175-179`). The default-plugin transaction then invokes project-scoped Claude marketplace/install commands that are documented to target the same project settings (`dummyindex/context/default_plugins.py:613-642`) after in-process declaration (`dummyindex/installer/install.py:651-659`). Concurrent dummyindex runs, hook refreshes, or CLI settings changes can therefore drop unrelated keys even though each individual `replace` is atomic. Protect the complete load/merge/write cycle with a cross-process lock or optimistic revision check; a unique temp file alone fixes collision, not stale-snapshot loss.

- **Medium — the persisted `wired` ledger does not enforce unique target identity.** `Config.__post_init__` validates scope and the tri-state flag but not duplicate `wired` targets (`dummyindex/context/domains/config.py:160-192`), and `_parse_wired` accepts every individually valid entry without a uniqueness pass (`dummyindex/context/domains/config.py:356-375`). Reconciliation prevents only newly appended duplicates via a transient target set (`dummyindex/context/domains/config.py:606-618`, `dummyindex/context/domains/config.py:641-658`), so pre-existing duplicate or contradictory records survive indefinitely. Declaration iterates every record (`dummyindex/context/default_plugins.py:487-524`) while materialisation collapses plugin targets to a set (`dummyindex/context/default_plugins.py:670-676`), which can produce inconsistent classifications/reporting for the same identity. Reject duplicate targets on parse, or normalize them with an explicit deterministic precedence and migration.

## Security

1. **High — predictable config/settings temporary files let a repository symlink clobber an arbitrary user-writable file before the atomic replace.** Both persistence writers derive a fixed sibling name, call `write_text` on it, and only then rename it over the destination (`dummyindex/context/domains/config.py:662-682`, `dummyindex/context/claude_settings.py:175-179`). An attacker can commit `.claude/settings.json.tmp -> ../../victim` in a repo with no current managed hooks; install’s first hook write follows the symlink and truncates `victim` with JSON, then renames the symlink over `settings.json`. An enriched repo can analogously carry `.context/config.json.tmp` and a config state that triggers migration/reconciliation. **Load-bearing mitigation today:** malformed destination JSON fails closed before settings mutation, and the final rename is atomic, but neither prevents following the attacker-controlled temporary symlink during the preceding write (`dummyindex/context/claude_settings.py:39-61`, `dummyindex/context/hooks.py:376-430`, `dummyindex/installer/install.py:634-643`). Create unique sibling temps with exclusive/no-follow semantics and publish through the already-open verified directory.

2. **High — a repository-controlled `.claude` directory symlink redirects managed-hook and plugin-setting writes outside the repository.** Hook installation selects `<project>/.claude/settings.json` and writes it through the shared settings mutator before the plugin pass (`dummyindex/context/hooks.py:376-430`, `dummyindex/context/claude_settings.py:64-113`); `wire_default_plugins` later uses the same unbounded target for marketplaces and enabled plugins (`dummyindex/context/default_plugins.py:448-524`). A clone containing `.claude -> ../victim-config` therefore makes ordinary auto-init create or rewrite `victim-config/settings.json`, after which the project-scoped Claude CLI also operates through the redirected project path. **Load-bearing mitigation today:** malformed JSON and same-name marketplace conflicts fail closed (`dummyindex/context/default_plugins.py:357-406`), while installer symlink validation covers only selected skill/command destinations, not repository integration state (`dummyindex/installer/install.py:81-124`, `dummyindex/installer/install.py:512-532`, `dummyindex/installer/install.py:597-665`). Resolve every integration target against the project boundary and bind writes to verified no-follow directory handles.

3. **Medium — the default path executes whichever `claude` binary wins inherited `PATH`.** The availability probe runs bare `claude --version`, and marketplace/install calls use the same unresolved executable name (`dummyindex/context/default_plugins.py:602-604`, `dummyindex/context/default_plugins.py:613-642`). `default_runner` changes into the repository and inherits the process environment while invoking list argv (`dummyindex/context/default_plugins.py:565-581`). A repository paired with a launcher or direnv configuration that puts the repository first on `PATH` therefore gains code execution as soon as the default-plugin pass probes availability. **Load-bearing mitigation today:** list argv avoids shell interpolation, execution is timeout-bounded, immediate plugin targets are allowlisted, third-party refs are full SHAs, and reviewed code-running surfaces are disclosed before mutation (`dummyindex/context/default_plugins.py:149-223`, `dummyindex/installer/install.py:629-632`). None authenticates the executable, so resolve an approved absolute Claude CLI path before entering repository context and run it with a minimal explicit environment.

## Product surface

- **A first-time Claude user follows the README install command — neither the reviewed default identities nor the PATH-resolved external CLI execution are disclosed on that first-contact surface before installation starts — the user cannot make an informed pre-run opt-out decision or verify which executable will perform materialisation.** The README quickstart names only `dummyindex install --platform claude` (`README.md:12-22`, `README.md:69-93`), and `install --help` describes “default Claude plugins” generically (`dummyindex/installer/args.py:8-30`). Runtime trust lines do identify pinned third-party sources, reviewed surfaces, code execution, and the one-run opt-out, but appear only after skill placement and project initialization have begun (`dummyindex/context/default_plugins.py:169-223`, `dummyindex/installer/install.py:117-151`, `dummyindex/installer/install.py:597-665`). They do not identify the bare `claude` executable that is resolved from inherited `PATH` and launched from the repository for availability, marketplace, and install operations (`dummyindex/context/default_plugins.py:565-581`, `dummyindex/context/default_plugins.py:602-642`).

- **A project build, guidance write, hook install, settings merge, or plugin materialisation fails during `dummyindex install` — warnings or “auto-init skipped” are emitted, then the command still prints `Done` and exits successfully — callers cannot distinguish “skill files copied” from “project ready” through the final status or exit code.** `_auto_init_project` returns `False` for import/refresh/build failures but treats guidance, hook, plugin, and equipment failures as best-effort before returning `True` (`dummyindex/installer/install.py:337-465`); plugin failures are warning-only (`dummyindex/installer/install.py:634-665`). The installer prints the same `Done` footer for both outcomes and only explains the skip when the target is not a repository (`dummyindex/installer/install.py:164-193`), while the command dispatcher returns without propagating a structured partial-success result (`dummyindex/__main__.py:270-289`).

- **A user runs install while another install, hook refresh, or Claude project-settings write is in flight — each actor reads and later replaces the complete config or settings document without a lock or revision check, so one successful write can silently erase the other — neither command output nor the final `Done` footer reports a conflict or verifies that unrelated keys survived.** Config reconciliation performs whole-document read/derive/write transactions through one fixed temporary path (`dummyindex/context/domains/config.py:567-682`), while hook and plugin operations independently merge then replace the full `.claude/settings.json` snapshot (`dummyindex/context/claude_settings.py:64-112`, `dummyindex/context/claude_settings.py:175-179`, `dummyindex/context/claude_plugins.py:105-155`). Materialisation then invokes project-scoped Claude marketplace/install commands against the same repository (`dummyindex/context/default_plugins.py:613-642`). The installer sequences its own steps but has no cross-process coordination or final state audit (`dummyindex/installer/install.py:164-193`, `dummyindex/installer/install.py:634-665`).

- **A user opts out with `--no-default-plugins`, then later reinstalls or updates without remembering the flag — the CLI opt-out applies to one run and an absent persisted value defaults back to enabled — there is no CLI operation that records a durable opt-out or changes one default independently.** The public option explicitly says “for this run” (`dummyindex/installer/args.py:19-29`), and `resolve_enabled` treats a missing config value as enabled (`dummyindex/context/default_plugins.py:281-290`). Durable control exists through manual `.context/config.json` or Claude settings edits in the detailed guide, so the product surface exposes persistence as file-format knowledge rather than an install/config command.

- **A Claude user runs the documented uninstall after auto-init — the skill family and slash-command aliases are removed, while project guidance, executable hooks, and plugin declarations remain — there is no single supported teardown command for the installed project surface.** Uninstall help accurately warns that Claude guidance and hooks stay intact (`dummyindex/installer/args.py:32-46`), and the implementation removes skills/commands but does not invoke hook, project-guidance, or plugin cleanup (`dummyindex/installer/uninstall.py:19-144`). Hooks have a separate public `context hooks uninstall` verb, but `context bootstrap` only writes guidance and exposes no removal verb (`dummyindex/cli/hooks.py:1-90`, `dummyindex/cli/bootstrap.py:1-71`); uninstall output also reports only the first removed path. The result is an uneven teardown path in which hooks can continue running after the owning skill is gone and the remaining managed state must be discovered and handled separately.

## Cross-review notes

The Security review flagged two operator-facing primitives that compound the product-surface gaps above:

- **Sentinel-substring laundering meets one-directional uninstall.** Security §THREAT 1 shows `dummyindex install`/upgrade re-emits any co-located non-sentinel hook verbatim while preserving the managed entry (`claude_settings.py:95-103`). Combined with the Product-surface finding that `uninstall.py` never removes hooks at all, the operator impact is: a hook body that rode in (laundered or user-authored) on one upgrade is *never* cleaned by `dummyindex uninstall` — the only removal path (`hooks.uninstall`) is unreachable from the uninstall verb. An operator who "removed dummyindex to stop its hooks" still has the managed entry — and anything co-located in it — executing every session. The structured-key remediation (a `_dummyindex: true` field) should land together with wiring `installer/uninstall.py` to the existing `hooks.uninstall`.

- **Bare-`claude` PATH resolution fires on the silent default-enable path.** Security §THREAT 2 notes `_claude_available`/install resolve the bare name `claude` with `cwd=project_root` (`default_plugins.py:246,267,276`). Operator impact via the Product surface: because the superpowers default-install is silent and opt-out is invisible (finding §2), the subprocess that resolves `claude` from PATH runs on *every* default install/upgrade without the user electing into it — so the PATH-trust exposure is on the default path, not an opt-in one. Resolving to an absolute `claude` (or `shutil.which` with a sanitised PATH) and surfacing the materialisation step in the log would make the network/exec side-effect both safer and visible.
