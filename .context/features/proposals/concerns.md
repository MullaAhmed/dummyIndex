## Data integrity

- **Multi-file scaffold is NOT atomic across the 4 files** — `ensure_proposal` (`store.py:82-101`) does `mkdir` then four sequential `write_text_atomic` calls. Each file is individually atomic, but the *set* is not: a crash/exception after `proposal.json` + `spec.md` but before `checklist.md` leaves a half-scaffolded `.context/proposals/<slug>/` on disk. Downstream `waves.py` `parse_checklist` and `cli/status._proposals` (`status.py:143-153`) then read a directory missing `checklist.md`/`plan.md` with no guard. The plan claims "atomic multi-file proposal scaffolding" (`plan.md:11,39`) but per-file atomicity ≠ transactional scaffold. No cleanup/rollback of the partial dir on failure.

- **`--force` re-propose is destructive, not idempotent** — with `force=True`, `ensure_proposal` re-writes all four templates over an existing proposal (`store.py:79,82`, `target.mkdir(exist_ok=True)`). Any human/`/dummyindex-plan`-authored `spec.md`/`plan.md`/`checklist.md` prose, `## Wave N` headings, and `reused_symbols` (which the plan says become "build's ground truth", `plan.md:106`) are silently overwritten with empty placeholders. A re-propose to refresh consistency clobbers all downstream enrichment; `apply_consistency` is the safe idempotent path but `ensure_proposal --force` is not. No merge/preserve; no warning.

- **`reused_symbols` survives nothing on re-scan but is dropped on a hand-edited reload round-trip risk is low — however `apply_consistency` only carries `related_features`+`conventions` via `dataclasses.replace` (`store.py:124-128`)**, correctly preserving `status`/`reused_symbols`/`title` from the re-read `proposal.json`. This is correct (verified). No concern — flagging as checked.

- **No schema_version validation on read** — `read_proposal` (`store.py:104-108`) → `Proposal.from_dict` (`models.py:40-53`) never checks `schema_version`. The data-access convention mandates readers reject payloads with a newer `schema_version` than the build supports (`conventions/data-access.md:13`, citing `meta.py:60-66`). This domain violates that contract: a future `schema_version: 2` proposal.json loads silently, coercing/dropping unknown fields, risking silent data corruption when a newer-format proposal is read by an older build. `from_dict` is tolerant-by-design for missing keys (good) but has no upper-bound guard.

- **Slug collision after normalization is a silent overwrite vector** — `validate_slug` lowercases + strips (`store.py:44`). Two distinct user slugs `My-Feat` and `my-feat` normalize to the same `my-feat` dir. Without `--force` the second raises `ProposalExistsError` (safe), but the collision is only caught by case, not reported as a normalization collision — the user may not realize `My-Feat` mapped onto an existing `my-feat`. Minor: traversal guard itself is solid (`store.py:45-50`, charset frozenset excludes `/` `.`), verified against the data-access repo-relative-path discipline.

- **`read_proposal` raises bare `FileNotFoundError`, breaking the typed-exception contract** — `store.py:104-108` lets `path.read_text` throw raw `FileNotFoundError` rather than a `ProposalError` subclass (`errors.py:5-26` defines only `ProposalExistsError`/`ProposalSlugError`). `apply_consistency` (`store.py:123`) calls `read_proposal` immediately after `ensure_proposal`, so the scaffold path is safe, but any external caller (or a deleted-between-steps race) gets an untyped error the CLI's `except ProposalSlugError/ProposalExistsError` (`propose.py:62-67`) won't map — it escapes as a traceback, not a clean exit code. Plan flags this as an open question (`plan.md:135-136`); confirming it as a real integrity gap, not just stylistic.

- **`json.loads` of a corrupt `proposal.json` is unguarded** — `read_proposal` (`store.py:107`) calls `json.loads(path.read_text(...))` with no `JSONDecodeError` handling. A truncated/partial `proposal.json` (e.g. from the non-transactional scaffold above, or external corruption) surfaces a raw `json.JSONDecodeError`, again unmapped by `propose.py`'s except clauses. Validate-at-boundary (`conventions/data-access.md:25`) is honored for slugs but not for the deserialized JSON payload.

- **Checklist wave structure integrity is out-of-context here (correctly)** — scaffolded `checklist.md` is intentionally flat (`store.py:170-176`), no `## Wave N`; `waves.py` degrades flat→one-item-per-wave (`plan.md:84-91`). No integrity issue in *this* feature's seed. Verified the seed contract; wave-grouping ownership is downstream (`/dummyindex-plan`), so no concern lands on proposals.

## Security

- **`validate_slug` does no Unicode/NFKC normalization or null-byte check before the charset gate** — `store.py:42-51` only does `.strip().lower()` then a per-char `_SLUG_OK_CHARS` membership test (`store.py:30,45`). The ASCII path is solid: `/`, `.`, `\` are all outside the frozenset, so literal `../`, `..\\`, absolute `/etc/...`, and `a/../../b` are rejected — traversal into `.context/proposals/<slug>/` via raw separators is closed (verified). The gap is upstream: a slug containing a NUL (`add\x00export`) or non-ASCII homoglyph fails the charset test and raises `ProposalSlugError`, which is the *safe* outcome — but the lack of an explicit reject-before-lower means the threat model rests entirely on the frozenset being exhaustive. Any future widening of `_SLUG_OK_CHARS` (e.g. adding `.` for versioned slugs like `feat.v2`) silently reopens `..` traversal. Defense-in-depth missing: no post-validation assertion that `proposal_dir(...).resolve()` stays under `proposals_root(...).resolve()`. Attack path today: none exploitable; risk is a one-character charset edit turning `validate_slug` into a traversal primitive with no second guardrail. LOW exploitability / HIGH blast-radius-on-regression.

- **CLI re-validates the slug a second time on an unsanitized raw value at print time** — `cli/propose.py:72` calls `proposal_dir(context_dir, slug)` with the *raw* `slug` (not the `safe_slug` returned through the pipeline), so `validate_slug` runs again inside `proposal_dir` (`store.py:59-61`). This is defensively fine (same guard) but means the trust boundary is crossed twice with the raw string; if `ensure_proposal` and `proposal_dir` ever diverge on normalization (e.g. one strips, one doesn't), the printed path could point at a different dir than the one written. Minor TOCTOU-flavored inconsistency, not an exploit. LOW.

- **`title` is fully attacker-controlled and written verbatim into four on-disk markdown/JSON artifacts with zero sanitization** — `ensure_proposal` takes `title` straight from `--title` (`cli/propose.py:69 → store.py:67,83,92,95,98`) and interpolates it unescaped into `_spec_template`/`_plan_template`/`_checklist_template` (`store.py:146-176`, e.g. `f"# Spec — {title}\n\n"`) and into `proposal.json` via `json.dumps` (`store.py:88` — JSON encoding here is the *only* output-encoding in the feature). A title like `"x\n## Consistency\n<!-- dummyindex:consistency:end -->"` injects a forged consistency sentinel into `spec.md`; on the next `apply_consistency`, `_inject_consistency` (`store.py:206-215`) does `find(_CONSISTENCY_BEGIN)`/`find(_CONSISTENCY_END)` and will splice around the *attacker's* injected end-marker, corrupting or truncating the real block. Markdown/template injection into a doc later read by `/dummyindex-plan` and build agents = prompt-injection surface (the title becomes instructions an LLM reads as ground truth). No length cap on `title` either — a multi-MB title is written to disk unbounded. Remediation: strip control chars / cap length / escape or fence the title; never let raw `title` carry the sentinel substrings. MEDIUM — local-only input, but it is the feature's untrusted-data boundary and it reaches an LLM-consumed artifact.

- **`title` flows unescaped into the `query` retrieval domain** — `scan_consistency` passes raw `title` to `query(context_dir, title, top_k=...)` (`scan.py:30,37`). Within this feature it only affects which features rank (output is feature-id strings, re-emitted to stdout `cli/propose.py:75`), so there's no injection *here*; the risk is delegated to `query`'s own input handling (out of context for `proposals`, flagged for the `query` feature). Confirmed no path-building from `query` output in this feature. INFORMATIONAL.

- **Predictable `.tmp` sidecar in the scaffold write path is a symlink/clobber target** — every file is written via `write_text_atomic` (`store.py:18,86-98`) which does `path.with_suffix(suffix + ".tmp")` then `tmp.replace(path)` (`atomic_io.py:22-24`). The tmp name is fully predictable (`proposal.json.tmp`, `spec.md.tmp`, …) inside the proposal dir. On a shared/multi-user checkout an attacker who can write into `.context/proposals/<slug>/` could pre-create `proposal.json.tmp` as a symlink; `tmp.write_text` would then follow it and write through the link before `replace`. `replace` is atomic and the dir is normally single-user, so exploitability is LOW (requires pre-existing write access to the target dir, which already implies game-over), but the non-randomized tmp name and absence of `O_EXCL`/`O_NOFOLLOW`-style create is a hardening gap. Same `.tmp` predictability also means a stale `*.tmp` from a crashed prior run is silently overwritten, never cleaned. LOW.

- **Scaffold trusts `--force` to overwrite without authn/authz or audit** — `ensure_proposal(..., force=True)` (`store.py:79,82`) blows away an existing proposal dir's four files with no record of who/what/when (this is the destructive-overwrite vector already raised under Data integrity; here the security lens is the *absence of an audit trail* — no append-only log of propose/force events). For a local single-user CLI this is acceptable; called out so that if `propose` is ever exposed via a server/MCP seam, `--force` becomes an unauthenticated destructive write with no repudiation defense. INFORMATIONAL (deployment-context-dependent).

- **No `schema_version` upper-bound check on read = trust-boundary gap on deserialization** — `read_proposal` → `Proposal.from_dict` (`store.py:104-108`, `models.py:40-53`) deserializes `proposal.json` with no guard that `schema_version <= SCHEMA_VERSION` (`constants.py`, `SCHEMA_VERSION = 1`), violating the data-access convention's reject-newer-payload rule (`conventions/data-access.md:13`, identifier `meta.py:60-66` — confidence acceptable, convention path verified). Security framing: a malformed/hostile `proposal.json` (truncated from a non-atomic scaffold, or hand-crafted) reaches `json.loads` (`store.py:107`) with no `JSONDecodeError` handling and no field-bound validation — untrusted file content parsed without validate-at-boundary (`conventions/data-access.md:25`). Raw `json.JSONDecodeError`/`FileNotFoundError` escape past `propose.py`'s `except ProposalSlugError/ProposalExistsError` (`cli/propose.py:62-67`) as an uncaught traceback (overlaps the typed-exception integrity finding). Remediation: cap `schema_version`, wrap `json.loads` in a typed `ProposalError`, validate fields on load. MEDIUM.

## Product surface

- **Slug collision after normalization gives a misleading "already exists" error — no hint that case/whitespace folded** — user runs `propose --slug "My-Feat"` after a `my-feat` proposal exists; `validate_slug` strips+lowercases to `my-feat` (`store.py:44`), `ensure_proposal` sees the dir exists and raises `ProposalExistsError(safe_slug=my-feat, ...)` (`store.py:79-80`) — observed message is `proposal 'my-feat' already exists at ...` (`errors.py:13-14`). The user typed `My-Feat` but the error names `my-feat`, with no "your slug normalized to X" note — they may not connect the two and think a *different* proposal collided. Gap: the error echoes `safe_slug`, not the raw input, so the normalization is invisible; a one-line "normalized from 'My-Feat'" would close it.

- **Re-propose to refresh consistency is a destructive trap — the CLI surfaces no warning before clobbering enriched docs** — user has a `/dummyindex-plan`-enriched proposal (real spec prose, `## Wave N` headings, `reused_symbols`), wants to re-run the consistency scan, and reaches for `propose --slug X --title ... --force`. Observed: `ensure_proposal(force=True)` re-writes all four templates as empty placeholders (`store.py:79,82,92-98`) — spec/plan/checklist prose gone, replaced by `_spec_template`/`_plan_template`/`_checklist_template` stubs (`store.py:145-176`). The CLI prints only `context propose: <dir> (4 files)` (`propose.py:73`) — no "overwrote existing proposal" warning, no diff, no confirmation prompt. Gap: there is no consistency-only refresh path exposed at the CLI (`apply_consistency` is the safe idempotent op but `propose.run` always calls `ensure_proposal` first, `propose.py:60-70`); the user's only re-entry is the destructive one, and it's silent. This is the data-integrity `--force` finding viewed as a UX trap: the surface offers no safe "re-scan" verb.

- **`--force` re-write resets `status` back to `planned` — silently un-tracks in-progress work** — a proposal mid-build carries `status: in_progress`/`done` (`enums.py:10-12`, written by the build loop). `propose --force` constructs a fresh `Proposal(slug, title)` defaulting `status=planned` (`store.py:83`, `models.py:23`) and writes it over `proposal.json`. Observed: an in-progress proposal silently reverts to `planned`; `cli/status._proposals` (`status.py:143-153`) then lists it as not-started. Gap: `--force` is documented/perceived as "refresh the seed," but it also rewinds lifecycle state with no warning — surprising for anyone who used it to re-stamp consistency on an active proposal.

- **A vague feature title scaffolds successfully but silently yields a near-empty consistency block — no "couldn't ground this" signal** — user runs `propose --slug thing --title "improve stuff"`. `scan_consistency` calls `query(context_dir, title, ...)` (`scan.py:30,37`); a vague title token-overlaps poorly, so `related_features` is likely empty. Observed: the CLI prints `related features: (none detected)` (`propose.py:76-77`) and the spec's `## Consistency` block renders `_No related features detected by the consistency scan._` (`store.py:191`). Both are framed as a *neutral* outcome identical to "repo not indexed yet" (`scan.py:38-42` swallows `FileNotFoundError` → also empty). Gap: the user can't distinguish "your title was too vague to ground" from "the index is empty / unbuilt" — the empty result has two very different causes and the surface conflates them, offering no nudge to write a more specific title.

- **Checklist wave clarity is invisible at scaffold — the seed is flat and the CLI never says waves come later** — user reads the freshly scaffolded `checklist.md` and sees a single flat `- [ ] _First derived checklist item._` (`store.py:170-176`), no `## Wave N` headings. Nothing in the template body or CLI output explains that wave grouping is added downstream by `/dummyindex-plan` and that a flat list degrades to one-item-per-wave at build (`plan.md:84-91`, `waves.py`). Observed: the user scaffolding a proposal by hand has no in-surface cue that waves are a thing or how to author them — the only hint lives in `plan.md` docs, not at the point of use. Gap: the `_checklist_template` comment mentions deriving from plan+acceptance but says nothing about `## Wave N`; a user hand-building the checklist will likely produce a flat list and never learn parallel waves exist.

- **Required-flag errors are correct but `propose` advertises nothing — no usage/help on the subcommand** — omitting `--slug` or `--title` yields `error: --slug <slug> and --title <text> are both required` (`propose.py:43-48`), and an unknown flag yields `error: unknown argument for 'propose': '...'` (`propose.py:117`). Both exit 2. There's no `--help`/usage string for `propose` (the parser is hand-rolled, `_parse_propose_args` `propose.py:83-118`); a user who mistypes a flag gets the rejection but no synopsis of valid flags. Gap: minor, but the self-parsing CLI (`plan.md:126-127`) trades the shared parser's help affordance for a bespoke one that emits errors without ever showing the correct invocation.

- **"4 files" success line gives no next-step — user is left at a dead end after scaffold** — on success the CLI prints `context propose: <dir> (4 files)` + related features/conventions (`propose.py:73-79`) and exits 0. There's no "now run `/dummyindex-plan` to flesh out spec/plan" or "edit <dir>/spec.md" pointer. Observed: a user who scaffolded a proposal sees a path and a file count but no guidance toward the enrichment step the whole seed-don't-own design (`plan.md:128-129`) depends on. Gap: the seed is explicitly a starting point handed off to `/dummyindex-plan`/build, yet the success surface names neither — the handoff is undiscoverable from the command's own output.
