# Council — Stage 3 critiques (proposals)

## critic-database (data integrity)

## Data integrity

- **Multi-file scaffold is NOT atomic across the 4 files** — `ensure_proposal` (`store.py:82-101`) does `mkdir` then four sequential `write_text_atomic` calls. Each file is individually atomic, but the *set* is not: a crash/exception after `proposal.json` + `spec.md` but before `checklist.md` leaves a half-scaffolded `.context/proposals/<slug>/` on disk. Downstream `waves.py` `parse_checklist` and `cli/status._proposals` (`status.py:143-153`) then read a directory missing `checklist.md`/`plan.md` with no guard. The plan claims "atomic multi-file proposal scaffolding" (`plan.md:11,39`) but per-file atomicity ≠ transactional scaffold. No cleanup/rollback of the partial dir on failure.

- **`--force` re-propose is destructive, not idempotent** — with `force=True`, `ensure_proposal` re-writes all four templates over an existing proposal (`store.py:79,82`, `target.mkdir(exist_ok=True)`). Any human/`/dummyindex-plan`-authored `spec.md`/`plan.md`/`checklist.md` prose, `## Wave N` headings, and `reused_symbols` (which the plan says become "build's ground truth", `plan.md:106`) are silently overwritten with empty placeholders. A re-propose to refresh consistency clobbers all downstream enrichment; `apply_consistency` is the safe idempotent path but `ensure_proposal --force` is not. No merge/preserve; no warning.

- **`reused_symbols` round-trip on re-scan is correct (verified)** — `apply_consistency` only carries `related_features`+`conventions` via `dataclasses.replace` (`store.py:124-128`), correctly preserving `status`/`reused_symbols`/`title` from the re-read `proposal.json`. No concern — flagging as checked.

- **No schema_version validation on read** — `read_proposal` (`store.py:104-108`) → `Proposal.from_dict` (`models.py:40-53`) never checks `schema_version`. The data-access convention mandates readers reject payloads with a newer `schema_version` than the build supports (`conventions/data-access.md:13`, citing `meta.py:60-66`). This domain violates that contract: a future `schema_version: 2` proposal.json loads silently, coercing/dropping unknown fields, risking silent data corruption when a newer-format proposal is read by an older build. `from_dict` is tolerant-by-design for missing keys (good) but has no upper-bound guard.

- **Slug collision after normalization is a silent overwrite vector** — `validate_slug` lowercases + strips (`store.py:44`). Two distinct user slugs `My-Feat` and `my-feat` normalize to the same `my-feat` dir. Without `--force` the second raises `ProposalExistsError` (safe), but the collision is only caught by case, not reported as a normalization collision — the user may not realize `My-Feat` mapped onto an existing `my-feat`. Minor: traversal guard itself is solid (`store.py:45-50`, charset frozenset excludes `/` `.`), verified against the data-access repo-relative-path discipline.

- **`read_proposal` raises bare `FileNotFoundError`, breaking the typed-exception contract** — `store.py:104-108` lets `path.read_text` throw raw `FileNotFoundError` rather than a `ProposalError` subclass (`errors.py:5-26` defines only `ProposalExistsError`/`ProposalSlugError`). `apply_consistency` (`store.py:123`) calls `read_proposal` immediately after `ensure_proposal`, so the scaffold path is safe, but any external caller (or a deleted-between-steps race) gets an untyped error the CLI's `except ProposalSlugError/ProposalExistsError` (`propose.py:62-67`) won't map — it escapes as a traceback, not a clean exit code. Plan flags this as an open question (`plan.md:135-136`); confirming it as a real integrity gap, not just stylistic.

- **`json.loads` of a corrupt `proposal.json` is unguarded** — `read_proposal` (`store.py:107`) calls `json.loads(path.read_text(...))` with no `JSONDecodeError` handling. A truncated/partial `proposal.json` (e.g. from the non-transactional scaffold above, or external corruption) surfaces a raw `json.JSONDecodeError`, again unmapped by `propose.py`'s except clauses. Validate-at-boundary (`conventions/data-access.md:25`) is honored for slugs but not for the deserialized JSON payload.

- **Checklist wave structure integrity is out-of-context here (correctly)** — scaffolded `checklist.md` is intentionally flat (`store.py:170-176`), no `## Wave N`; `waves.py` degrades flat→one-item-per-wave (`plan.md:84-91`). No integrity issue in *this* feature's seed. Verified the seed contract; wave-grouping ownership is downstream (`/dummyindex-plan`), so no concern lands on proposals.

## Security

_Adversarial lens: slug validation (path traversal), scaffold file writes, trusting `--title` feature-request input. 7 threats; full prose in `concerns.md` `## Security`._

- **Slug charset is the *only* traversal guard (`store.py:42-51`)** — ASCII `../`, `..\\`, `/etc/...`, `a/../b` all rejected by `_SLUG_OK_CHARS` (`store.py:30`); verified closed today. Gap: no NFKC/null-byte reject-before-lower and no `resolve()`-under-`proposals_root` assertion — a one-char charset widening (e.g. adding `.`) silently reopens traversal. LOW exploit / HIGH regression blast-radius.
- **`title` written verbatim into 4 artifacts, zero sanitization (`store.py:67,83-98,146-176`)** — markdown/sentinel injection: a crafted title carrying `<!-- dummyindex:consistency:end -->` corrupts `_inject_consistency`'s splice (`store.py:206-215`); reaches `/dummyindex-plan`+build LLMs as ground truth = prompt-injection surface. No length cap. Only output-encoding is `json.dumps` (`store.py:88`). MEDIUM.
- **CLI passes raw (not safe) slug at print time (`cli/propose.py:72`)** — double trust-boundary crossing; benign while `proposal_dir` re-validates, but a normalization divergence would point the printed path off the written dir. LOW.
- **`title` → `query` unescaped (`scan.py:30,37`)** — no path-building from query output in this feature; risk delegated to `query`. INFORMATIONAL.
- **Predictable `.tmp` sidecar (`atomic_io.py:22-24`)** — `path.with_suffix(".tmp")` is symlink/clobber-target on a shared dir; LOW (requires pre-existing write to target dir). No `O_EXCL`/`O_NOFOLLOW`; stale tmp never cleaned.
- **`--force` overwrite has no authn/authz/audit (`store.py:79,82`)** — fine for local CLI; becomes unauthenticated destructive write with no repudiation defense if `propose` is ever exposed via server/MCP. INFORMATIONAL.
- **No `schema_version` upper-bound + unguarded `json.loads` on read (`store.py:104-108`, `models.py:40-53`)** — untrusted file content deserialized without validate-at-boundary, violating `conventions/data-access.md:13,25`; raw `JSONDecodeError`/`FileNotFoundError` escape `propose.py`'s except clauses (`cli/propose.py:62-67`). MEDIUM.

## critic-product (product surface)

- **Slug collision error names the normalized slug, not the user's input (`store.py:79-80`, `errors.py:13-14`)** — `propose --slug "My-Feat"` over an existing `my-feat` raises `proposal 'my-feat' already exists`; user typed `My-Feat`, never sees the case/whitespace fold. No "normalized from X" hint.
- **No safe consistency-refresh verb — re-propose is the only re-entry and it's a silent clobber (`propose.py:60-70`, `store.py:79-98`)** — to re-scan an enriched proposal a user reaches for `--force`, which rewrites all four files as empty templates (`store.py:145-176`); CLI prints only `(4 files)` (`propose.py:73`), no warning/diff/confirm. `apply_consistency` is the safe op but isn't exposed standalone.
- **`--force` resets `status` to `planned` (`store.py:83`, `models.py:23`)** — fresh `Proposal()` defaults `planned`, silently rewinding an `in_progress`/`done` proposal (`enums.py:10-12`); `status._proposals` then lists it as not-started (`status.py:143-153`). Lifecycle rewind is undocumented and unwarned.
- **Vague title scaffolds fine but yields an empty consistency block indistinguishable from "unindexed" (`scan.py:30,37-42`, `propose.py:76-77`, `store.py:191`)** — `--title "improve stuff"` token-overlaps poorly → empty `related_features`; the `(none detected)` / `_No related features detected_` output is identical to the `FileNotFoundError`-swallowed "repo not indexed" path. No nudge to write a more specific title.
- **Wave structure is invisible at scaffold (`store.py:170-176`)** — seed `checklist.md` is flat, no `## Wave N`; neither template body nor CLI output explains waves are added downstream by `/dummyindex-plan` and that flat degrades to one-item-per-wave (`plan.md:84-91`). A hand-builder never learns parallel waves exist.
- **No `--help`/usage for the self-parsing `propose` (`propose.py:83-118`)** — required-flag and unknown-flag errors are correct (`propose.py:43-48,117`) but emit no synopsis of valid flags; the bespoke parser drops the shared parser's help affordance (`plan.md:126-127`).
- **Success line is a dead end — no next-step pointer (`propose.py:73-79`)** — `(4 files)` + related features, exit 0, but no "run `/dummyindex-plan`" / "edit spec.md" guidance; the seed-don't-own handoff (`plan.md:128-129`) is undiscoverable from the command's own output.
