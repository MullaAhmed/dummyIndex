# critic-database — first pass

## Data integrity

- **High — scope, ownership, and byte-budget validation are vulnerable to a plan/apply race.** `_project_guidance_plan` resolves and validates the selected path, reads its existing owner, and computes its byte budget, then `bootstrap_project_agents_md` calls the shared writer as a separate operation; `_atomic_write` re-resolves a symlink target at write time without repeating the project-scope check. A concurrent replacement of the leaf or a parent symlink between those steps can invalidate the checked owner/budget or redirect the atomic replacement outside the validated scope. Bind validation to the final resolved target or revalidate immediately before the replace (`dummyindex/context/output/agents_md.py:110-161`, `dummyindex/context/output/agents_md.py:255-270`, `dummyindex/context/output/bootstrap.py:100-132`, `dummyindex/context/output/bootstrap.py:396-410`).
- **Medium — stale managed guidance can survive uninstall with no reported cleanup issue.** Discovery stops at depth 32 and returns early after 20,000 inspected files, but its return type carries only discovered names; `remove_project_agents_md` passes that partial tuple into cleanup, and `AgentsMdCleanupResult` can report neither truncation nor unscanned paths. Large/deep repositories can therefore receive an apparently clean uninstall while old fallback files still contain active dummyindex blocks. Return scan-completeness metadata as a cleanup issue or persist installed targets instead of reconstructing them from a bounded walk (`dummyindex/context/output/agents_md.py:68-107`, `dummyindex/context/output/agents_md.py:183-199`, `dummyindex/context/output/agents_md.py:273-327`, `dummyindex/context/output/agents_md.py:407-441`).
- **Medium — dual-host bootstrap is preflighted but not transactionally committed.** The `both` path validates Claude and Codex targets first, then writes Claude guidance before attempting Codex guidance; if the second write fails or filesystem state races after preflight, the command returns an error with the first file already changed and no rollback or structured partial-success result. Either stage both replacements and commit together or explicitly model/report the partial state so callers can repair it deterministically (`dummyindex/cli/bootstrap.py:41-69`).

# critic-security — first pass

## Security

- **High — a repository-local symlink race can turn a scoped guidance refresh into an out-of-project overwrite.** `dummyindex/context/output/agents_md.py:110-161`, `dummyindex/context/output/agents_md.py:255-270`, `dummyindex/context/output/bootstrap.py:82-132`, `dummyindex/context/output/bootstrap.py:396-410` — **Attack scenario:** an attacker with concurrent write access to the repository swaps the selected fallback leaf or one of its parent directories after `_project_guidance_plan` resolves and approves it but before `_atomic_write`; the writer resolves the then-current symlink target without reapplying the project boundary, so it can replace any external file writable by the user, potentially with content read from the earlier in-scope file. The cleanup path has the same validate-then-read/remove gap (`dummyindex/context/output/agents_md.py:407-440`, `dummyindex/context/output/bootstrap.py:135-180`). — **Mitigation:** bind mutation to an already validated parent directory handle (`dir_fd`/no-follow operations where available), reject changed component identities, and recheck the leaf immediately before the final replace; a plain earlier `Path.resolve()` is not a race-safe boundary.

- **High — trusted project configuration can nominate an arbitrary in-repository regular file as the managed-block target.** `dummyindex/codex_guidance.py:37-67`, `dummyindex/codex_guidance.py:146-181`, `dummyindex/codex_guidance.py:233-243`, `dummyindex/context/output/agents_md.py:223-240`, `dummyindex/context/output/agents_md.py:110-129` — **Attack scenario:** after a repository has a durable `trust_level = "trusted"` entry, a malicious collaborator commits `project_doc_fallback_filenames = [".git/config"]`, a workflow path, or an executable path; when no standard AGENTS candidate exists, bootstrap treats that existing regular file as guidance and prepends Markdown to it, while the atomic writer preserves its executable/mode bits (`dummyindex/context/output/bootstrap.py:396-410`). The containment check prevents an external write but does not prevent corruption of security-sensitive repository metadata or automation. — **Mitigation:** require explicit confirmation before first claiming a project-configured nonstandard target, reject VCS metadata/control paths and executable files by default, and persist the approved target so later project-config changes cannot silently retarget the writer.

- **Medium — global Codex instructions can be created or retained as group/world-writable persistent prompt input.** `dummyindex/codex_guidance.py:29-34`, `dummyindex/context/output/agents_md.py:164-180`, `dummyindex/context/output/bootstrap.py:396-441` — **Attack scenario:** `CODEX_HOME` points at a shared directory, or the installer runs with a permissive umask; `_atomic_write` creates the directory with normal `mkdir` permissions and the file from mode `0666`, and it preserves an existing target's mode without checking owner or write bits. Another local account or shared-group member can then inject instructions into global `AGENTS.md`/`AGENTS.override.md`, which Codex consumes in future sessions. — **Mitigation:** for user-global guidance, require a caller-owned non-group/world-writable Codex home and target, create the directory/file with owner-only write permission, and refuse or prominently warn on unsafe existing ownership/modes; ambient umask alone is not an adequate instruction-integrity control.

Evidence handling: no catalogued prose document is quoted or treated as authority in
these findings; current source controls the threat assessment.

# critic-product — first pass

## Product surface

- **Scenario:** a trusted repository's `.codex/config.toml` is malformed, unreadable, or contains an invalid fallback value — **Observed behavior:** config loading treats the layer as absent, invalid entries are dropped individually, and project bootstrap falls back to a lower layer or creates `AGENTS.md` while still reporting a successful managed-block write (`dummyindex/codex_guidance.py:37-87`, `dummyindex/codex_guidance.py:146-205`, `dummyindex/context/output/agents_md.py:142-160`, `dummyindex/context/output/agents_md.py:223-240`) — **Gap:** the caller receives no diagnostic that its configured target or byte budget was ignored, so a configuration typo is indistinguishable from an intentional default-selection decision.

- **Scenario:** a library caller invokes `preflight_project_agents_md`, then calls `bootstrap_project_agents_md` expecting to apply the checked decision — **Observed behavior:** preflight discards the rendered body, owner, budget, and filesystem snapshot and returns only a `Path`; bootstrap calls `_project_guidance_plan` again and can select a different target or fail under changed config/filesystem state (`dummyindex/context/output/agents_md.py:110-161`) — **Gap:** the public preflight is only a point-in-time validation probe, not an applyable plan, and its return type gives the caller no revision/token with which to detect that the decision changed.

- **Scenario:** `dummyindex context init <repo> --platform codex` encounters malformed markers, an unwritable target, or an invalid byte budget — **Observed behavior:** the index build remains committed, Codex guidance is reported only as `skipped (...)` on stderr, and the command returns `0` (`dummyindex/cli/init.py:180-208`, `dummyindex/cli/init.py:222-223`) — **Gap:** automation cannot distinguish a fully usable Codex setup from an index-only partial success by exit status or a structured result; it must parse human stderr.

- **Scenario:** `dummyindex context bootstrap <repo> --platform both` passes preflight, writes Claude guidance, then hits a late Codex I/O or race failure — **Observed behavior:** the command prints the Claude success, returns `3` on the Codex error, and leaves the Claude file changed because the two writes are sequential and have no rollback (`dummyindex/cli/bootstrap.py:41-69`) — **Gap:** one command has a partial durable result but exposes no machine-readable per-host outcome or repair/resume instruction.

- **Scenario:** Codex uninstall meets a malformed managed file or a stale fallback outside the bounded scan (depth 32, more than 20,000 files, or markers after 256 KiB) — **Observed behavior:** known cleanup errors are printed and skipped, scan truncation simply returns the paths found so far, skill removal continues, and uninstall still prints `skill removed`/`nothing to remove` and completes normally (`dummyindex/context/output/agents_md.py:183-199`, `dummyindex/context/output/agents_md.py:273-346`, `dummyindex/context/output/agents_md.py:407-441`, `dummyindex/installer/uninstall.py:19-24`, `dummyindex/installer/uninstall.py:133-144`, `dummyindex/installer/uninstall.py:165-214`) — **Gap:** callers receive neither a cleanup-completeness flag nor a non-success result, so they cannot reliably assert that persistent Codex instructions are gone without rescanning the repository themselves.

these findings. The final generated plan describes preflight as returning the plan,
but current source returns only a path and recomputes the plan during bootstrap;
code wins, and the conflict is recorded here.
