## Data integrity

Lens here = release-script version/tag/changelog integrity + test-fixture isolation/determinism. No relational DB; "data integrity" maps to durable on-disk artefacts (`pyproject.toml`, `CHANGELOG.md`, tag stream) and fixture cross-contamination. Findings are thin, as the plan predicts.

- **Version source-of-truth split from the tag stream (drift risk).** `decide_bump` reads commit history but `next_version` increments off `read_current_version()` parsed from `pyproject.toml` (`scripts/release.py:121-126,201`), while `last_tag()` selects the bump *range* from the `v*` git tags (`scripts/release.py:158-162`). The committed `pyproject` version and the latest `v*` tag are assumed in lock-step; nothing reconciles them. If a prior release pushed a tag but the `pyproject` version commit was lost/reverted (or vice-versa), the next run computes a wrong/duplicate version with no guard. Low-frequency but irreversible (tag + GitHub Release happen in the workflow). Consider asserting `read_current_version()` matches `last_tag()` stripped of `v` before bumping. Note: this is CI side-effect code, explicitly *not* unit-tested (`scripts/release.py:118`).

- **No idempotency / re-run guard on the mutating path.** `main()` (`scripts/release.py:189-208`) is not idempotent: a second run over the same commit range (e.g. workflow retry before the release commit lands) re-bumps from the *current* pyproject and prepends a second `CHANGELOG.md` entry (`prepend_changelog`, `scripts/release.py:141-149`) for a near-identical version. The pure core is fine; the risk is the orchestration assuming exactly-once. Acceptable if the workflow guarantees single-shot, but that guarantee lives outside this module and isn't asserted here.

- **`next_version` silently assumes a clean 3-part numeric semver.** `current.split(".")` with `int(p)` (`scripts/release.py:80`) will raise on any pre-release/build suffix (`0.28.0-rc1`, `0.28.0+meta`) — fails loud, which is correct, but there's no contract test pinning that pre-release strings are rejected rather than mis-parsed. Boundary input from `pyproject.toml` is trusted without validation beyond the regex capture.

- **Changelog prepend can double-stack blank lines but won't corrupt.** `prepend_changelog` branches on whether the file starts with `# changelog` (case-folded) (`scripts/release.py:146-149`); the else-branch re-emits `# Changelog` even if a differently-cased/spaced title existed, risking a duplicated top-level heading. Cosmetic, not integrity — the historical entries below are preserved (`rest`/`existing` retained). Flagging as low.

- **Fixture isolation is clean — no cross-contamination.** `tmp_repo` is a pure `tmp_path` alias (`tests/conftest.py:9-11`); `usage_corpus` builds entirely under `tmp_path/projects` and returns that root, mutating nothing shared (`tests/usage/conftest.py:50-145`). `SAMPLE_REPO` is a frozen anchor consumers `copytree` before mutating (`tests/paths.py:11-14`, convention in `.context/conventions/testing.md:31,52`). No autouse fixture writes outside `tmp_path`. The autouse `_no_real_plugin_install` only sets an env var via `monkeypatch` (auto-reverted) (`tests/conftest.py:14-21`) — no leak across tests.

- **Determinism of fixtures is sound.** `usage_corpus` hard-codes timestamps/uuids/token counts (`tests/usage/conftest.py:61-144`) so aggregates are exact and reproducible; no clock, RNG, or ordering dependence. The one time-sensitive release surface — `prepend_changelog` stamps `date.today()` (`scripts/release.py:142`) — is in the *untested* side-effect zone, so it can't make the suite flaky, but any future test of `prepend_changelog` must freeze the date or it will be non-deterministic. Pre-emptive flag only.

## Security

Adversarial lens: a malicious commit author / repo contributor (the release script runs in CI over *their* commit subjects and bodies), and a hostile *cloned/shared repo* whose gitignored `.context/cache/` content the statusline reads into a user's shell. No injection CVEs invented — these are the concrete paths.

- **Untrusted badge file read into PowerShell `Write-Host` on every prompt (clone-borne, LOW-MED).** `statusline.ps1:17-18` does `Get-Content -Raw '.context/cache/freshness-badge'` then `Write-Host -NoNewline $badge`. The badge is *normally* an internally-derived literal (`compute_badge` only ever emits `[ctx ✓]` or `[ctx: N drift]`, `dummyindex/context/drift.py:91-109`) written atomically by trusted code (`dummyindex/cli/plan_update.py:39-50`). But the statusline does **not** recompute — it prints **whatever bytes are on disk**, and that path is *gitignored scratch* (`plan_update.py:26-31`), so it is not integrity-protected by the commit history. Attack path: clone/unzip a repo (or a co-worker commits one despite gitignore, or a supply-chain step drops a file there) containing a crafted `.context/cache/freshness-badge`; on the victim's next prompt the statusline prints its raw contents to the terminal. `Write-Host` does **not** evaluate the string as PowerShell — there is no code-exec here — but the content reaches the terminal **unsanitised**, so ANSI/OSC escape sequences in the file are honoured by the emulator: cursor manipulation, screen clearing, title rewriting, or (on terminals with OSC-52 / hyperlink / DECRQSS-style reply quirks) clipboard writes and spoofed UI. Same exposure exists in the bash twin (`statusline.sh:16` raw `cat`). Concrete mitigation: before printing, strip non-printable/ESC bytes (`$badge -replace '[\x00-\x1f\x7f]',''`) and length-cap it; the expected payload is a ~12-char ASCII badge, so a strict allowlist is cheap and lossless.

- **Relative cache path → wrong-repo / cwd-confused read (LOW).** Both statuslines resolve `.context/cache/freshness-badge` **relative to the process cwd** (`statusline.ps1:17`, `statusline.sh:16`), not to the repo Claude Code reported. Whatever directory the shell happens to sit in when the prompt renders supplies the badge. An attacker who can influence cwd (e.g. a repo that drops a `.context/cache/freshness-badge` in a commonly-cd'd subdir, or a sibling checkout) gets their badge printed for an unrelated project, compounding the escape-sequence issue above. The Python hot-path resolves through `resolve_context_root` (`plan_update.py:54-60`), but the shell twins skip that and trust cwd. Low because the blast radius is just "print the wrong short string," but it's the delivery vector for the escape-injection finding.

- **Release script trusts commit subjects/bodies verbatim — `subprocess` is safe, but rendered output is not neutralised (LOW, no shell-injection).** `_git(...)` always uses **list-form** `subprocess.run(["git", *args], ...)` with `check=True`, no `shell=True` (`scripts/release.py:152-155`) — so a commit message can **not** inject shell commands; this is the correct pattern and worth affirming. The residual risk is *content*, not *execution*: `describe()`/`render_notes()` pass attacker-controlled commit subjects straight into `CHANGELOG.md`, `release-notes.md`, and `$GITHUB_OUTPUT` (`scripts/release.py:88-115,180-186,201-205`). A commit subject containing a newline-free but `GITHUB_OUTPUT`-significant payload is the one to watch: `emit_output` writes `key=value\n` lines to the `$GITHUB_OUTPUT` file (`scripts/release.py:184-186`), and `version` comes from `next_version` (numeric, safe) — but if any future caller routes a *commit-derived* string through `emit_output`, a `\n`-bearing value would forge additional output keys (classic GH Actions output-injection). Today only `released`/`version` flow through, both machine-generated, so this is **latent, not live** — flag so the neutral-value invariant is kept if `emit_output` ever carries `describe()` output. Markdown/`)`/backtick metacharacters in subjects also pass unescaped into the changelog (content spoofing in release notes), low severity.

- **`emit_output` value newline-injection guard is absent (LOW, latent).** `emit_output` (`scripts/release.py:180-186`) writes `f"{key}={value}\n"` with no check that `value` is single-line / free of `\n` or GitHub's heredoc delimiter syntax. Safe *today* because both values are constrained (`"true"/"false"`, numeric semver), but there's no assertion pinning that invariant. If the contract ever loosens, this is a GitHub-Actions output-forging primitive. Cheap fix: reject/replace `\n` in `value` (and the `<<EOF` heredoc form) at the boundary, or assert values match a strict pattern.

- **Fixture isolation holds under an adversarial read too (affirmation).** Re-checked from the security angle: no autouse fixture or anchor writes outside `tmp_path`; `SAMPLE_REPO`/`FIXTURES_DIR` are read-only anchors consumers `copytree` before mutating (`tests/paths.py:11-14`, `.context/conventions/testing.md`), and `_no_real_plugin_install` *prevents* the dangerous default — it forces `SKIP_INSTALL_ENV=1` so a test can never shell out to the real `claude` CLI by accident (`tests/conftest.py:14-21`). That autouse guard is itself a security control (blast-radius containment for the install path) and is correctly fail-closed: unit tests must *explicitly* inject a runner to bypass it. No tmp/predictable-path or symlink-follow issue in the corpus builder (`tests/usage/conftest.py:50-145`) — everything is rooted under pytest's `tmp_path`.

## Product surface

Lens: maintainer/contributor experience of the release driver, test fixtures, and statusline. Infra, so thin by design. Cite `path:range`.

- **Releasing locally has no dry-run / preview** — release maintainer wants to see "what would this release?" before pushing — `scripts/release.py:189-208` `main()` only runs the full mutating path (writes `pyproject.toml`, prepends `CHANGELOG.md`, writes `release-notes.md`); the pure core (`decide_bump`/`render_notes`, `scripts/release.py:56-115`) *is* side-effect-free and could back a `--dry-run` that prints the bump + rendered notes without writing, but no CLI flag exists (no `argparse`, no `sys.argv` read anywhere in the module). A maintainer debugging "why did CI cut a minor?" must read commits by eye or run the workflow. Gap: no preview entry point despite the core being trivially preview-able.

- **No confirmation / no guard on the irreversible step, by design but undocumented for the human** — the commit/push/`gh release create` live in the workflow (`.github/workflows/release.yml:48-62`) with `if: steps.prep.outputs.released == 'true'` as the only gate; there is no manual-approval environment, no `workflow_dispatch`, nothing a maintainer can hold. Every push to `main` that parses as feat/fix auto-ships to PyPI (via `publish.yml`). That's the intended contract (`release.yml:3` "direct push, no release PR") but the only place it's written is a workflow comment — a contributor reading `scripts/release.py` alone won't learn that their `feat:` commit auto-releases. Gap: the auto-publish blast radius is invisible from the script; no CONTRIBUTING note ties commit-type to "this ships."

- **Fixtures are undocumented except inline** — a contributor adding an extractor case wants to know what `tests/fixtures/sample_repo/` covers — there is **no README in `tests/fixtures/`** (confirmed: only `app.py`, `helpers.py`, `web/`; sole `tests/*.md` is `tests/eval/BASELINE.md`). The `web/app.ts` frozen TS fixture is referenced by the plan (`plan.md:16`) but its shape/intent is only discoverable by opening it and grepping consumers. By contrast `usage_corpus` *is* well self-documented — its docstring enumerates every hard path it exercises (`tests/usage/conftest.py:1-8`) — which sets the bar the sample_repo fixture misses. Gap: `sample_repo` has no manifest of what each file is meant to exercise; onboarding friction for extractor contributors.

- **`paths.py` anchors are the only documented "use this" but discovery is implicit** — a new test author wants the blessed way to reach fixtures — `tests/paths.py:1-14` docstring does tell them to import `REPO_ROOT`/`SAMPLE_REPO` instead of chaining `Path(__file__).parent.parent`, which is good. But nothing *enforces* or surfaces it (no lint, no mention in a testing README at the tests root); the convention lives in `.context/conventions/testing.md` and the module docstring only. Minor: relies on a contributor stumbling onto `paths.py` before re-rolling their own anchor.

- **Statusline wiring is copy-paste-fragile for the maintainer** — a Windows maintainer wants to install the badge — the only setup instruction is a comment with a hardcoded `C:\\path\\to\\statusline.ps1` placeholder (`statusline.ps1:13-14`); there's no installer command, no `dummyindex` subcommand that prints the correct absolute path, and the cold-path fallback (`dummyindex context statusline`, referenced `statusline.ps1:11`) is mentioned but not linked to the wiring. A maintainer must hand-edit `settings.json` with a path they resolve themselves. Gap: no generated/printed wiring snippet; manual absolute-path editing is the documented UX.

- **Silent "no release" is correct but opaque on a failed expectation** — a maintainer who *expected* a release sees only `print("No releasable commits since ... — skipping.")` (`scripts/release.py:197`) — it doesn't list the commit subjects it saw or why each was non-releasing (e.g. all `chore:`/`docs:`). With no `--dry-run` and no verbose mode, diagnosing "I pushed a feat but nothing shipped" means re-reading git log manually. Low, but it's the most likely real maintainer confusion and the message gives no actionable detail.
