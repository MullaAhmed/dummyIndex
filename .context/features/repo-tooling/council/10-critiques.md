## Data integrity

Lens here = release-script version/tag/changelog integrity + test-fixture isolation/determinism. No relational DB; "data integrity" maps to durable on-disk artefacts (`pyproject.toml`, `CHANGELOG.md`, tag stream) and fixture cross-contamination. Findings are thin, as the plan predicts.

- **Version source-of-truth split from the tag stream (drift risk).** `decide_bump` reads commit history but `next_version` increments off `read_current_version()` parsed from `pyproject.toml` (`scripts/release.py:121-126,201`), while `last_tag()` selects the bump *range* from the `v*` git tags (`scripts/release.py:158-162`). The committed `pyproject` version and the latest `v*` tag are assumed in lock-step; nothing reconciles them. If a prior release pushed a tag but the `pyproject` version commit was lost/reverted (or vice-versa), the next run computes a wrong/duplicate version with no guard. Low-frequency but irreversible (tag + GitHub Release happen in the workflow). Consider asserting `read_current_version()` matches `last_tag()` stripped of `v` before bumping. Note: this is CI side-effect code, explicitly *not* unit-tested (`scripts/release.py:118`).

- **No idempotency / re-run guard on the mutating path.** `main()` (`scripts/release.py:189-208`) is not idempotent: a second run over the same commit range (e.g. workflow retry before the release commit lands) re-bumps from the *current* pyproject and prepends a second `CHANGELOG.md` entry (`prepend_changelog`, `scripts/release.py:141-149`) for a near-identical version. The pure core is fine; the risk is the orchestration assuming exactly-once. Acceptable if the workflow guarantees single-shot, but that guarantee lives outside this module and isn't asserted here.

- **`next_version` silently assumes a clean 3-part numeric semver.** `current.split(".")` with `int(p)` (`scripts/release.py:80`) will raise on any pre-release/build suffix (`0.28.0-rc1`, `0.28.0+meta`) — fails loud, which is correct, but there's no contract test pinning that pre-release strings are rejected rather than mis-parsed. Boundary input from `pyproject.toml` is trusted without validation beyond the regex capture.

- **Changelog prepend can double-stack blank lines but won't corrupt.** `prepend_changelog` branches on whether the file starts with `# changelog` (case-folded) (`scripts/release.py:146-149`); the else-branch re-emits `# Changelog` even if a differently-cased/spaced title existed, risking a duplicated top-level heading. Cosmetic, not integrity — the historical entries below are preserved (`rest`/`existing` retained). Flagging as low.

- **Fixture isolation is clean — no cross-contamination.** `tmp_repo` is a pure `tmp_path` alias (`tests/conftest.py:9-11`); `usage_corpus` builds entirely under `tmp_path/projects` and returns that root, mutating nothing shared (`tests/usage/conftest.py:50-145`). `SAMPLE_REPO` is a frozen anchor consumers `copytree` before mutating (`tests/paths.py:11-14`, convention in `.context/conventions/testing.md:31,52`). No autouse fixture writes outside `tmp_path`. The autouse `_no_real_plugin_install` only sets an env var via `monkeypatch` (auto-reverted) (`tests/conftest.py:14-21`) — no leak across tests.

- **Determinism of fixtures is sound.** `usage_corpus` hard-codes timestamps/uuids/token counts (`tests/usage/conftest.py:61-144`) so aggregates are exact and reproducible; no clock, RNG, or ordering dependence. The one time-sensitive release surface — `prepend_changelog` stamps `date.today()` (`scripts/release.py:142`) — is in the *untested* side-effect zone, so it can't make the suite flaky, but any future test of `prepend_changelog` must freeze the date or it will be non-deterministic. Pre-emptive flag only.

## Security

Adversarial lens: malicious commit author (release script runs in CI over their commit subjects/bodies) + hostile cloned/shared repo whose gitignored `.context/cache/` the statusline reads into a shell. No invented CVEs.

- **Untrusted badge → PowerShell `Write-Host` every prompt (clone-borne, LOW-MED).** `statusline.ps1:17-18` reads `.context/cache/freshness-badge` raw and `Write-Host -NoNewline $badge`. Badge is normally an internal literal (`compute_badge` → `[ctx ✓]`/`[ctx: N drift]`, `dummyindex/context/drift.py:91-109`) written atomically by trusted code (`plan_update.py:39-50`), but the statusline does NOT recompute — it prints whatever bytes sit at that *gitignored scratch* path (`plan_update.py:26-31`), not integrity-protected by git. Attack: ship a repo with a crafted badge file; victim's next prompt prints it raw. `Write-Host` does not eval as PowerShell (no code-exec), but the unsanitised content reaches the terminal so ANSI/OSC escapes are honoured (cursor/title/screen manipulation, OSC-52 clipboard on some emulators). Same in bash twin (`statusline.sh:16` raw `cat`). Fix: strip `[\x00-\x1f\x7f]` + length-cap before printing; expected payload is ~12 ASCII chars.
- **Relative cache path → cwd-confused read (LOW).** Both twins resolve `.context/cache/freshness-badge` relative to process cwd (`statusline.ps1:17`, `statusline.sh:16`), not the repo Claude reported. The Python hot-path goes through `resolve_context_root` (`plan_update.py:54-60`); the shells skip it and trust cwd — delivery vector for the escape-injection finding above.
- **Release script: `subprocess` safe, rendered output not neutralised (LOW, no shell-injection).** `_git` uses list-form `subprocess.run(["git", *args], check=True)`, no `shell=True` (`scripts/release.py:152-155`) — commit messages can NOT inject shell commands; affirm this. Residual risk is content: `describe()`/`render_notes()` route attacker commit subjects verbatim into `CHANGELOG.md`/`release-notes.md` (`scripts/release.py:88-115,201-205`) — markdown/backtick spoofing, low.
- **`emit_output` newline-injection guard absent (LOW, latent).** `emit_output` writes `f"{key}={value}\n"` to `$GITHUB_OUTPUT` with no single-line/heredoc check (`scripts/release.py:180-186`). Safe today — only `released` (true/false) and numeric `version` flow through — but if a commit-derived string (`describe()`) is ever routed here, a `\n`-bearing value forges output keys (GH Actions output-injection). Flag to keep the neutral-value invariant; cheap fix: assert `value` matches a strict pattern / reject `\n` + `<<EOF`.
- **Fixture isolation holds adversarially (affirmation).** No autouse fixture/anchor writes outside `tmp_path`; `SAMPLE_REPO`/`FIXTURES_DIR` are read-only anchors `copytree`'d before mutation (`tests/paths.py:11-14`). `_no_real_plugin_install` is itself a security control — fail-closed, forces `SKIP_INSTALL_ENV=1` so no test shells out to the real `claude` CLI (`tests/conftest.py:14-21`). No predictable-tmp/symlink issue in the corpus builder (`tests/usage/conftest.py:50-145`).

## Product surface

Lens: maintainer/contributor experience of the release driver, test fixtures, and statusline. Infra, so thin by design. Cite `path:range`.

- **No local dry-run / preview** — `main()` (`scripts/release.py:189-208`) only runs the full mutating path; the pure core (`decide_bump`/`render_notes`, `scripts/release.py:56-115`) is side-effect-free and could back a `--dry-run`, but no CLI flag / `argparse` / `sys.argv` read exists. A maintainer debugging "why did CI cut a minor?" reads commits by eye or runs the workflow. Gap: no preview despite a trivially preview-able core.
- **No human confirmation on the irreversible step (by design, undocumented in the script)** — commit/push/`gh release create` live in `.github/workflows/release.yml:48-62`, gated only by `if: ... released == 'true'`; no manual-approval env, no `workflow_dispatch`. Every feat/fix push to `main` auto-ships to PyPI. Intended (`release.yml:3`) but stated only in a workflow comment — a contributor reading `scripts/release.py` alone never learns their `feat:` auto-releases. Gap: auto-publish blast radius invisible from the script.
- **`sample_repo` fixture undocumented** — no README in `tests/fixtures/` (only `app.py`, `helpers.py`, `web/`; sole `tests/*.md` is `tests/eval/BASELINE.md`). `web/app.ts` (`plan.md:16`) is discoverable only by opening it + grepping consumers. `usage_corpus` sets the bar with a path-by-path docstring (`tests/usage/conftest.py:1-8`) that `sample_repo` misses. Gap: no manifest of what each fixture file exercises.
- **`paths.py` anchors documented only in-module** — `tests/paths.py:1-14` docstring tells authors to import `REPO_ROOT`/`SAMPLE_REPO`, but nothing enforces/surfaces it (no lint, no tests-root README); convention lives in `.context/conventions/testing.md` + the docstring. Minor.
- **Statusline wiring is copy-paste-fragile** — only setup is a comment with a hardcoded `C:\\path\\to\\statusline.ps1` placeholder (`statusline.ps1:13-14`); no installer, no subcommand that prints the absolute path. Maintainer hand-edits `settings.json`. Gap: no generated wiring snippet.
- **Silent "no release" is opaque on a failed expectation** — `print("No releasable commits ... — skipping.")` (`scripts/release.py:197`) lists neither the commit subjects seen nor why each was non-releasing. With no `--dry-run`/verbose, "I pushed a feat but nothing shipped" means re-reading git log. Low but the likeliest real maintainer confusion.
