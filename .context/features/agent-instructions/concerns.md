## Data integrity

- **Atomicity holds, but via a local clone, not the canonical helper.** `_atomic_write` (`dummyindex/context/output/instructions.py:530-534`) does `mkdir(parents=True)` → write `path+".tmp"` → `tmp.replace(path)`, so a concurrent reader (statusline badge, drift hook) never sees a half-written doc — the one mandatory rule (`conventions/data-access.md:5-7`) is satisfied. The concern is *duplication*, not correctness: it byte-for-byte reimplements `write_text_atomic` (`domains/atomic_io.py:11-24`) instead of calling it, exactly the "don't bypass `write_text_atomic` with a plain reimplementation" pitfall (`conventions/data-access.md:27`). Verified equivalent (same `encoding="utf-8"`, no normalization), so no data hazard today — fix path is dedup: route the three `write_*` (`:505,:509,:526`) through `write_text_atomic` and delete `_atomic_write`. Low severity.
- **No Windows `PermissionError` fallback.** Unlike the cache writer's copy-then-delete guard (`pipeline/io/cache.py:98-105`), `_atomic_write`'s `tmp.replace(path)` (`:534`) can raise on Windows if a reader holds the target open. For these append-rarely, regenerated-wholesale docs the window is tiny and `build_all` is the sole writer, so impact is low — but it's a second reason the local copy is strictly worse than the shared helper, which is the natural place to add the guard once.
- **Idempotent / deterministic regeneration confirmed.** Output is fully content-determined: `generate_*` are pure string-returning (no disk read, no clock, no randomness), the static docs are frozen literals (`_HOW_TO_USE` `:24-109`, `_PLAYBOOK_BODIES` `:338-487`), and emission order is stable — `PLAYBOOK_IDS = tuple(sorted(_PLAYBOOK_BODIES))` (`:490`) and `_select_architecture_docs` sorts by `DOC_CONFIDENCE_ORDER` (`:316-332`). Re-running `build_all` over an unchanged index produces byte-identical files; `replace()` makes the swap atomic. No partial-write or duplicate-emission risk.
- **`KeyError`-on-miss is the right failure mode, not silent corruption.** `generate_playbook_md` rejects an unknown id with a `KeyError` listing valid ids (`:493-499`) rather than emitting an empty/placeholder doc — a regeneration over a bad id fails loud, never writes garbage. Good.

## Security

Adversarial lens: index-derived strings (dir names, file paths, source-doc titles) flow unsanitized into `architecture/overview.md`, a doc future agents read as authoritative navigation. Trust boundary = repo filesystem + committed `.md` files; output isn't browser-rendered, so the payload is markdown/prompt injection into an agent-facing doc, not XSS. 5 threats:

- **HIGH — source-doc title rendered verbatim.** `title = f" — {d.title}"` → `f"...(**{d.confidence}**{title})"` (`instructions.py:261-265`), `d.title` extracted from doc H1 (`domains/source_docs/catalog.py:144`, `domains/source_docs/models.py:14`). A `docs/architecture.md` with an instruction-shaped H1 is auto-selected by the `"architecture"` filename signal (`:306-313`) and high-confidence-sorted to the top (`:331`) of the "Documented architecture" list. Escape/strip control + markdown tokens, collapse to one line, cap length.
- **HIGH — markdown-link injection via `d.path`.** `f"- [`{d.path}`](../../{d.path}) ..."` (`:263`) uses `d.path` unescaped as both link text and href; filesystem-legal `)`/`](`/`../` in a path forge a trusted-looking target or break the link. Percent-encode href, escape link text, reject `..`/metachars.
- **MEDIUM — dir/file names unescaped in table + list.** `` `{s.name}/` `` in a 5-col table row (`:232`) and `` `{f.path}` `` in root list (`:244`); a POSIX-legal backtick/pipe/newline in a name corrupts the table or injects a row. Backtick/pipe-escape cells; reject newlines.
- **LOW — no bound on catalog/layout size.** `_select_architecture_docs` + loops emit one line per match unbounded (`:228-266`); doc-flooding buries the AST ground truth and burns agent context. Cap + elide.
- **NOTE — atomic write is not a content-trust control; no secrets emitted.** `_atomic_write` (`:530-534`) only prevents torn reads; the only sensitive property is integrity of agent-facing instructions (the injection findings). `HOW_TO_USE`'s detect-secrets guidance (`:87-102`) is recommended template text, not a control this feature enforces.

## Product surface

Lens: the generated `HOW_TO_USE.md` / `architecture/overview.md` / `playbooks/*.md` ARE the UX for every future agent. Scanned the shipped docs against the code that emits them. `scenario — observed behavior — gap`:

- **Agent opens `architecture/overview.md` to find documented architecture — every doc's confidence renders as `**DocConfidence.HIGH**` / `.MEDIUM` / `.LOW` (`overview.md:23-37`), the raw enum repr, not the intended `**high**` — the SHIPPED, COMMITTED doc is stale.** The code was fixed in HEAD~? commit `680aa70 fix(source-docs): render DocConfidence as value` (pins `__str__ = str.__str__` so `f"{d.confidence}"` now yields `high`), and `instructions.py:264` still interpolates the bare member — so *regenerating* would now print `**high**`, but `overview.md` was never regenerated after the fix. This is the canonical stale-after-changes failure: the curated/committed artefact disagrees with the code, and the leaked `DocConfidence.HIGH` text is exactly what an agent reads as the confidence label. The doc itself even says "Each doc carries a `confidence` from `source-docs/INDEX.md`" (`overview.md:21`) while showing a Python enum repr instead — actively misleading. Gap: `architecture/overview.md` is not regenerated by `rebuild --changed`'s normal path here (or the index predates the fix); no freshness signal tells an agent the confidence column is wrong.
- **Agent reads the role column to orient on the main package — `dummyindex/` (203 files, 910 symbols — the entire product) shows Role `_unknown_` (`overview.md:15`).** `_DIR_ROLE_HINTS` (`instructions.py:119-168`) has no entry for the repo's own top-level package name, and `_role_hint_for` returns `None`→`_unknown_` on miss (`:300-301`, by design per plan.md:34). The single largest, most important directory in the overview carries the least-useful hint, while `scripts/` and `tests/` get real roles. Gap: the heuristic role table is keyed on generic names (`src`/`lib`/`app`) and silently degrades for the namesake package — the overview's most load-bearing row is the least informative. Not a correctness bug (degradation is intentional), but a product-surface miss: the flagship dir reads as a blank.
- **Agent following `add-feature.md` hits "this might be an **update** (see `update.md` if present)" (`playbooks/add-feature.md:6`) — `update.md` is NOT present** (`.context/playbooks/` ships only the five fixed ids: add-endpoint, add-feature, add-migration, fix-bug, refactor). The reference is guarded by "if present", so it's not a broken link per se, but the only place the word "update" is offered as a recipe points at a playbook the generator can never emit (`PLAYBOOK_IDS` is frozen at five, `instructions.py:490`; plan.md:43 flags this as an open question). Gap: playbook coverage hole — the doc names an `update` flow with no playbook behind it and no extension seam to add one; an agent told "this is an update" has nowhere to go.
- **Agent needs a playbook for any task outside the five canned kinds (add a CLI command, wire a hook, add a convention, write a skill, do a release) — `HOW_TO_USE.md:22` promises "`playbooks/` — task-specific recipes" but the set is hardcoded to five generic web/CRUD shapes** (`add-endpoint`/`add-migration` assume HTTP routers + Alembic/Prisma — `add-endpoint.md`, `add-migration.md`). For THIS repo (a Python CLI + Claude-skill toolchain, no HTTP layer, no DB), `add-endpoint` and `add-migration` are dead weight while the repo's actual recurring tasks (add a `context <verb>` subcommand, add a council stage, ship a skill) have zero recipe. Gap: playbooks are stack-generic, not project-tuned; two of five don't apply to the repo they ship in, and the repo's real workflows are uncovered. The plan acknowledges no extension seam (plan.md:43).
- **Agent trusts the language list to know what it's working in — `overview.md:7` says "Languages: powershell, python, typescript" and lists `dummyindex/` as `powershell, python` (`:15`).** Only 2 files match powershell across the whole map (verified: `grep -c powershell map/files.json` → 2), yet PowerShell is surfaced as a top-level stack language and as a `dummyindex/` language alongside Python. Gap: no proportionality / threshold on the language rollup — a 2-file incidental `.ps1` (likely a hook installer) is presented with equal billing to the 200+ Python files, over-stating the stack to an orienting agent. Minor, but the Stack section is the first thing read and it mis-frames the project as multi-stack.
- **Agent regenerates the index and expects `overview.md` to refresh — but the freshness/staleness contract for THIS doc is unstated in the doc itself.** `HOW_TO_USE.md:55-56` explains rebuild-vs-reconcile for `map/`/`tree.json` and feature docs, but `architecture/overview.md` (a generated, AST-derived doc that CAN go stale, as the `DocConfidence` leak proves) is never named in the "two layers, two rules" split (`:82`). An agent can't tell whether a stale-looking overview is a `rebuild --changed` fix or a reconcile fix. Gap: the navigation/maintenance guidance omits the one generated narrative doc that most visibly drifts; missing a "regenerate the overview via rebuild" pointer.
