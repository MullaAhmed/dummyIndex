## Data integrity

- **Backup-then-gitignore ordering window: the snapshot lands in the worktree BEFORE it is ignored.** `backup_docs` copies every doc into `root/.context/_doc_backups/<utc>/` and writes the manifest (`safety.py:93-116`) and only *afterwards* calls `_ensure_backup_ignored` (`safety.py:120`). If `.context/.gitignore` does not yet carry `_doc_backups/`, there is a window where the certified-clean tree is now dirty with the backup itself â€” defeating the precondition the reversibility gate just established. plan.md:46 asserts the ignore "so the snapshot can't dirty the very tree the guard just certified clean," but the code writes the snapshot first and ignores second. Reverse the order (ensure-ignored before the copy loop) so the gitignore is in place before any backup byte hits the tree.

- **Backup copies and the manifest bypass `write_text_atomic` â€” the one mandatory persistence rule.** `data-access.md:5-7` ("Atomic writes (the one mandatory rule)") requires every persisted artifact be tmp+`replace`; the manifest is written with a plain `write_text(... )` (`safety.py:114-116`) and each doc via `shutil.copy2` (`safety.py:105`), neither atomic. A crash mid-`copy2` or mid-manifest-write leaves a truncated backup file or half-written `manifest.json` that `restore_backup` will then read (`safety.py:138`) or copy back as authoritative original content (`safety.py:164`) â€” silent corruption of the very safety net. Restore *reads* atomically (tmp+`replace`, `safety.py:162-165`) but the backup that feeds it does not, so the chain is only as strong as its non-atomic write half. The `.gitignore` append at `safety.py:189-195` is likewise a non-atomic read-modify-write.

- **No verification that the backup actually captured every in-scope doc â€” partial backup passes silently at backup time.** `backup_docs` skips any `src` that is not a file (`safety.py:97-98`) and any path resolving outside root (`safety.py:100-102`) with a bare `continue`, recording only what it `captured` (`safety.py:106,111`). A doc that vanishes or errors between discovery and copy is dropped from the manifest with no count-mismatch check against the input `files`. The CLI then reports `len(backup.files)` as success (`cli/doc_reorg.py:88`) â€” a short backup reads as complete. The `skipped` honesty surfaces only at *restore* time (`safety.py:158-159`), i.e. after the destructive session has already run; there is no backup-time assertion that captured count == in-scope count before the user is told it is safe to proceed.

- **Atomicity of the multi-file restore is per-file, not all-or-nothing â€” a mid-restore failure leaves a half-restored tree.** Each entry is tmp+`replace` and a failed entry unlinks its `.tmp` then raises `BackupError` (`safety.py:161-168`), aborting the loop. Files already `replace`d before the failure stay restored; files after it do not â€” the tree is left in a mixed old/new state with no rollback of the partial restore. plan.md:41 frames this gate as "even a mid-restore failure leaves the tree clean," which is true only for stray `.tmp` debris, NOT for restore completeness. Because the starting tree was clean, `git restore` is still a full undo, but the *restore command itself* is not atomic across files and reports nothing about how far it got before raising. Consider catching per-file errors into `skipped` and continuing, then surfacing the partial set, rather than aborting on the first `OSError`.

- **`created_since` is recomputed by re-running discovery, so an upstream scope change silently mis-classifies rollback files.** restore_backup derives `created_since = current_docs âˆ’ backed_up` by re-calling `discover_doc_files` (`safety.py:171-175`), which transitively depends on `source_docs.discovery.discover_default_doc_paths` (`discovery.py:12-14,29`). plan.md:20 flags that a change to default-doc discovery "silently widens/narrows reorg scope." Concretely: if discovery's scope *narrowed* between backup and restore, a doc that genuinely existed at backup time but is no longer discoverable is absent from `current`, so it is neither restored-as-missing nor reported â€” and if it *widened*, an unrelated pre-existing doc now in scope is mis-reported as `created_since` (a reorg artifact the user may `git clean -fd` per `cli/doc_reorg.py:113-119`), risking deletion of a file the reorg never created. The manifest is the only trustworthy record of backup-time scope; `created_since` should be computed against a re-walk that does not re-consult mutable upstream discovery, or the manifest should record enough to make this set difference scope-stable.

- **No-data-loss guarantee leans entirely on a clean-tree precondition the CLI guard does not enforce on the destructive paths.** The library `backup`/`restore` domain calls never invoke `require_clean_tree` themselves â€” the gate is a separate CLI `GUARD` action (`cli/doc_reorg.py:51-66`) the user must run first, and it re-implements the clean/dirty/unknown branching inline instead of routing through `require_clean_tree` (`safety.py:57-75`), per plan.md:54. Nothing forces `backup` (let alone the session's subsequent in-place edits) to have passed the guard. If a caller runs `backup` on a dirty tree, the backup snapshots already-modified docs as if they were pristine originals, and a later `restore` faithfully recreates the *dirty* state, not the committed baseline â€” the "given a clean starting tree" invariant (plan.md:7) is assumed, never asserted, on the path that actually writes. The reversibility promise is only as good as the user remembering to run `guard` separately.

- **`git status --porcelain` includes untracked files, so an untracked backup dir or unrelated untracked file blocks the guard â€” but tampering with `.git` is undetectable.** `git_is_clean` treats any non-empty porcelain output as dirty (`safety.py:54`), which correctly catches a stray pre-existing `_doc_backups/` if it were ever un-ignored, but the guarantee that "git is an authoritative undo" (plan.md:7, safety.py:5-7) assumes the user has not also discarded git history out-of-band; that is an accepted boundary, not a defect â€” noted for completeness, no action required.
## Security

- **Restore path-escape gate is symlink-blind: `.resolve()` is run, the membership check passes, then the write follows a symlink out of the repo (`safety.py:149-168`).** The gate computes `dest = (root / rel).resolve()` then asserts `dest.relative_to(root)`. `Path.resolve()` collapses symlinks, so the *check* sees the real target â€” but if any **parent directory** of `root/rel` is a symlink that still resolves *under* `root` at check time, the check passes, yet `dest.parent.mkdir(...)` (`safety.py:161`) and `tmp.replace(dest)` (`safety.py:165`) operate on that path and write *through* the symlink to wherever it points. Concrete path: attacker (or a prior reorg step) plants `docs/` as a symlink to `/etc` resolving outside root â†’ rejected; but plants `<root>/sub` â†’ `<root>/real` (in-repo symlink) and a manifest entry `sub/x.md`; `resolve()` yields `<root>/real/x.md`, `relative_to(root)` passes, and the restore clobbers `<root>/real/x.md` â€” a file the manifest's author chose, not the doc that was backed up. The gate proves the *resolved* path is in-repo; it never proves no symlink was *traversed*, so an in-repo symlink redirects the write to an attacker-chosen in-repo target. `lstat`/`O_NOFOLLOW`-style refusal (reject if any component of `root/rel` is a symlink) is the missing control. Code wins over plan.md:40 which frames this gate as a complete "path-escape" defence â€” it stops `../` and absolute paths, not symlink redirection.

- **Backup follows symlinks on the source side and silently drops out-of-repo docs, so a reorg edit later mutates a file no backup protects (`safety.py:95-106`).** `src.resolve().relative_to(root)` rejects (via bare `continue`) any in-scope doc whose real path is outside the repo â€” e.g. `docs/shared.md` symlinked to `~/secrets/shared.md`. That doc is dropped from the manifest with no record, yet it is still a "doc in scope" the running session will edit in place through the symlink. Result: the session destructively rewrites an out-of-repo file (`~/secrets/shared.md`) that the backup deliberately never captured and that `git restore` cannot undo (it is not in the worktree) â€” a permanent, un-reversible edit to an arbitrary file outside the repo, exactly the reversibility promise the package claims to guarantee (plan.md:7). The drop is silent (no `skipped`/warning at backup time), so the operator is told "N docs backed up" and proceeds believing the edit is reversible.

- **Destructive backup/restore run with NO clean-tree assertion â€” the reversibility gate is opt-in and the CLI never opts in (`cli/doc_reorg.py:51-101`, `safety.py:57-75`).** `require_clean_tree` is defined but called *nowhere* in the CLI (grep confirms zero call sites); the `GUARD` action re-implements `git_is_clean` inline (`cli/doc_reorg.py:51-66`) as a separate, advisory, read-only command. `BACKUP` (`cli/doc_reorg.py:78-90`) and `RESTORE` (`cli/doc_reorg.py:97-101`) invoke the domain directly without ever gating on a clean tree. Attack/failure path: run `backup` on a dirty tree â†’ the snapshot captures already-modified docs as "originals"; the session then reorganises; a later `restore` faithfully reproduces the *dirty* pre-reorg state, not the committed baseline â€” and because no guard ever fired, the user was never warned the undo is now poisoned. The "given a clean starting tree" invariant (plan.md:7) is *assumed by the writing paths, asserted by none of them*; the guarantee depends entirely on the operator remembering to run a separate `guard` first.

- **`created_since` is fed to the user as a `git clean -fd` target while being derived from mutable upstream discovery â€” a scope-widen turns rollback advice into a data-loss instruction (`safety.py:171-175`, `cli/doc_reorg.py:113-119`).** `restore_backup` computes `created_since = current_docs âˆ’ backed_up` by re-walking `discover_doc_files` â†’ `discover_default_doc_paths` (`discovery.py:29`, verified in symbols.json). The CLI then literally tells the operator these files "the reorg created" should be dropped with `git clean -fd` (`cli/doc_reorg.py:113-119`). If discovery's default scope *widened* between backup and restore (a dependency the feature explicitly flags as able to "silently widen/narrow reorg scope," plan.md:20), a pre-existing, un-backed-up, possibly-untracked file now appears in `current`, is mis-classified as `created_since`, and the user is instructed to `git clean -fd` it â€” destroying a file the reorg never touched. The set difference trusts live discovery, not the manifest's frozen scope; the irreversible action (`git clean -fd`) is gated on the least trustworthy input.

- **Non-atomic backup writes corrupt the safety net itself; a crash mid-snapshot yields a truncated manifest/copy that `restore_backup` later treats as authoritative (`safety.py:105,114-116,138`).** `data-access.md`'s "atomic writes (the one mandatory rule)" is bypassed on the backup side: docs via `shutil.copy2` (non-atomic, `safety.py:105`) and the manifest via plain `write_text` (`safety.py:114-116`). A crash/kill mid-`copy2` leaves a truncated backup file that restore copies back as "original" content; a half-written `manifest.json` either fails JSON parse (`safety.py:138-140`, recoverable) or â€” worse â€” truncates at a record boundary, silently shrinking backup scope so restore reads as complete while missing docs. The undo chain is only as atomic as its weakest write, and the *producing* half is not atomic. Not remotely exploitable, but it converts a power loss / OOM-kill during backup into silent doc loss that the operator cannot distinguish from a successful snapshot.

## Product surface (critic-product, stage 3)

- **No preview/dry-run before the destructive session â€” the user cannot see what will be edited.** The four verbs are guard/list/backup/restore (`cli/doc_reorg.py:3-7`); `list` prints in-scope docs (`cli/doc_reorg.py:68-76`) but there is no command that shows the *proposed* reorg (diff of moves/renames) before the session starts Edit-ing. The user backs up blind, then the session rewrites under per-file Edit confirms with no upfront "here is the plan" surface. Gap: the only preview is the post-hoc `list` of scope, not the intended changes.

- **`backup` reports success with zero clean-tree gating, so the user is told "N doc(s) backed up" on a dirty tree and reasonably believes the undo is sound.** `BACKUP` calls `backup_docs` directly and prints `len(backup.files)` (`cli/doc_reorg.py:78-90`) without ever invoking the guard or `require_clean_tree` (`safety.py:57-75`). The success line `doc-reorg backup: N doc(s) -> <dir>` (`cli/doc_reorg.py:88`) carries no "tree was dirty" caveat. Gap: the clean-tree precondition is the operator's to remember; the success message actively reassures even when the precondition was never met.

- **Undo is discoverable only at backup time, never at restore-failure or session start; the restore string scrolls away.** The exact restore command is printed once, on the backup line (`cli/doc_reorg.py:89`). If the operator loses that scrollback there is no `doc-reorg latest`/`list-backups` to rediscover the `<utc>` dir â€” they must hand-navigate `.context/_doc_backups/`. Gap: undo discoverability hinges on retained terminal output of a single line.

- **The clean-tree gate is advisory and decoupled â€” nothing chains guardâ†’backupâ†’edit, so the gate is trivially skippable.** `GUARD` is a standalone read-only exit-code command (`cli/doc_reorg.py:51-66`); `BACKUP`/`RESTORE` never call it. A user who runs `backup` then edits without ever running `guard` gets no warning. plan.md:54 itself flags "Guard bypasses `require_clean_tree`." Gap: messaging frames guard as *the* pre-check (`cli/doc_reorg.py:4`) but no path enforces or even nudges it.

- **`--allow-dirty` exists in the library but is invisible at the CLI â€” the documented escape hatch is unreachable from the product surface.** `require_clean_tree(..., allow_dirty=True)` is the "single deliberate escape hatch" (plan.md:45, `safety.py:57,63`), yet the CLI never calls `require_clean_tree` and parses no `--allow-dirty` flag (`cli/doc_reorg.py:51-66`). Gap: a user on an intentionally-dirty tree has no sanctioned override; their only option is to ignore the advisory guard entirely.

- **`restore` hands the user a raw `git clean -fd` instruction for `created_since` files â€” a destructive command the tool tells them to run, with no dry-run nudge.** On restore the CLI prints "drop them with `git clean -fd` for a full rollback" (`cli/doc_reorg.py:113-119`). `git clean -fd` deletes untracked files repo-wide, and the `created_since` set is derived from mutable upstream discovery. Gap: no suggestion of `git clean -nd` (dry-run) first or scoping the clean to the listed paths.

- **Per-file Edit confirmation fatigue is unbounded and unbatched.** Destructive edits happen in-session via Edit "so the user confirms each" (`cli/doc_reorg.py:9-11`); `list` shows the count (`cli/doc_reorg.py:75`) but the flow offers no batching/progress surface. Gap: dozens of individual confirms with no "M of N" feedback invite rubber-stamping â€” defeating the per-file safety the design leans on.

- **What the user loses: no rename/move tracking â€” restore cannot tell a rename from a delete+create.** `restore_backup` recreates manifest paths and reports anything new as `created_since` (`safety.py:171-182`, plan.md:31). A rename `a.md`â†’`b.md` restores `a.md` (original content) and lists `b.md` as `created_since` to `git clean`; the user loses the *new* content of `b.md` unless they notice. Gap: the surface presents renames as "created file, drop it," with no signal that dropping it discards the reorg's actual output.

## Cross-review notes (critic-product, stage 3)

- **Operator impact of the symlink-clobber finding (Security): a restore can silently write to a doc the operator never inspected.** The in-repo-symlink-traversal write (`safety.py:149-168`) means `doc-reorg restore` may clobber `<root>/real/x.md` while the success line reports "restored N doc(s)" (`cli/doc_reorg.py:105`) naming the *manifest* path, not the real target. The operator's only feedback is a reassuring count, so the redirection is invisible at the moment they expect verification.

- **Operator impact of the advisory-only clean-tree gate (Security): the reversibility promise is presented as a guarantee but enforced as a suggestion.** Help text bills the net as "refuse-on-dirty, backup, and honest restore" (`cli/doc_reorg.py:10-11`) and guard as "the reorg's pre-check" (`cli/doc_reorg.py:4`), but no destructive path calls the guard (`cli/doc_reorg.py:78-101`). The operator reads a refuse-on-dirty guarantee, gets advisory behavior, and the gap surfaces only after the destructive session has run.
