### critic-database (data integrity) — stage 3

## Data integrity

- **Negative control is genuinely non-vacuous — verified at the token level.** `query`'s scorer matches via *substring* (`tok in fid_l` / `name_l` / basename / symbol name), not exact equality (`query.py:266-309`). I confirmed the negative-control tokens `('kubernetes','pod','scheduling','across','availability','zones')` share no substring with `community`, the sample names, basenames (`app`/`web`/`helpers`), or any symbol name, so `_score_feature` returns `None` for every feature and `query` returns `()` (`query.py:303-305,211-220`). The triple assertion (target absent, rank `None`, `matches == ()`) at `test_retrieval_eval.py:194-206` is therefore the strongest form and holds. Clean on the core non-vacuity claim.

- **Flaky-gate risk: `total_estimated_tokens` is `used` (only emitted matches), not all top_k — and budget truncation can silently drop a ranked match before hit@3 is counted.** The gate computes `rank` from `result.matches` (`test_retrieval_eval.py:138`), but `matches` is built under a token budget: the loop breaks with `truncated=True` when `remaining < 80`, and `_build_match` may return `None` and be skipped (`query.py:223-237`). With the default `budget_tokens` and 3 tiny sample features this won't fire today, but the gate reads the *post-budget* `matches`, so a future budget/`estimate_tokens` change (`query.py:163-166`) could drop the rank-3 feature out of `matches` and trip `T_HIT` as a *retrieval* regression when the cause is budgeting. The eval passes no explicit `budget_tokens`, so it inherits `_DEFAULT_BUDGET_TOKENS` — a coupling worth a one-line comment, not a bug.

- **Baseline/floor margin arithmetic is internally consistent and correctly tied to N=12.** `BASELINE.md:64-75` derives `11/12≈0.917 > T_HIT=0.90` (one miss tolerated) and `10/12≈0.833 < 0.90` (two misses trip), matching `T_HIT=0.90`/`T_MRR=0.85` at `test_retrieval_eval.py:46-47`. The fixture file has exactly 12 positives + 1 negative = 13 entries (`retrieval_fixtures.json:1-68`), satisfying the `>= 12` guard (`:127`) and the single-negative assertion (`:80`). The margins are **hard-coded to N=12**: the `>= 12` guard lets the corpus *grow* without touching the floors, but per `BASELINE.md:69-75` the floor magnitudes are calibrated to exactly 12 — at N=24 one miss is `23/24≈0.958` and the 0.90 floor silently tolerates **two** misses. Documented as an open question (`plan.md:104-105`) but the live coupling is that `>= 12` admits growth that invalidates the margin without any tripwire.

- **Fixture-quality guard mirrors the scorer's vocab but not its substring semantics — a latent false-negative.** `_feature_vocab` builds the overlap set from `feature.json` `files` basenames + `map/symbols.json` symbol names tokenized via the public `tokenize` (`test_retrieval_eval.py:216-245`), and the guard asserts `question_tokens & vocab` is non-empty (`:259-261`) — set *equality* of tokens. But the actual scorer hits on *substring* (`tok in name_l`, `query.py:267-276`). A question token that substring-matches a symbol (e.g. `app` in `make_app`) scores in `query` yet the guard, comparing whole tokens, could deem a future fixture "unanswerable" or vice-versa. Today both `make_app`→`make`,`app` and the questions tokenize cleanly so they agree, but the guard is a *stricter* proxy than the scorer; it can reject a fixture the scorer would actually answer. Acceptable (fails safe toward fixture quality), worth a comment.

- **`_feature_vocab` silently no-ops when `map/symbols.json` is absent or a feature carries no `files`/`members`.** `payload.get("files", [])` and the `symbols_json.exists()` guard (`test_retrieval_eval.py:229,237`) mean a feature with empty `files` and empty `members` yields an empty `vocab`, making the overlap assertion (`:261`) fail with a clear message — good. But if `symbols.json` is missing entirely the symbol vocab is silently skipped (`:237`), so the guard could pass on basename-only overlap while the *scorer* (which also reads symbols) had richer signal — no integrity hole, but the guard's coverage degrades silently rather than asserting the index shape it depends on (`feature.json` `members` ↔ `symbols.json` `symbol_id`/`node_id` join at `:240-242`).

- **`_path_in_citations` trailing-segment match is sound but unanchored on the left.** `cpath == expected_path or cpath.endswith("/" + expected_path)` (`test_retrieval_eval.py:107`) correctly requires a `/`-delimited trailing segment, so `app.py` will not spuriously match `webapp.py`. No path-collision risk among the three frozen ids (`app.py`/`web/app.ts`/`helpers.py`). Clean — but note `expected_path` for `community-1` is `web/app.ts` and citations are repo-relative (`:104`); if `build_all` ever emitted absolute or `./`-prefixed citation paths this exact/endswith check would silently stop hitting and the gate would report a *retrieval* miss. Frozen-index assumption, fine for now.

### critic-security (security) — stage 3

## Security

- **Test-only surface, no untrusted input — clean.** No network, subprocess, shell, or code execution in the eval path. Fixtures load via `json.loads(read_text(...))` (`test_retrieval_eval.py:71`) — never `pickle`/`yaml.load`/`eval`/`exec` — over an in-repo file (`retrieval_fixtures.json:1-68`) anchored to the module dir (`:50`); no path-injection seam. `runner.py`/`query.py` parse via tree-sitter + `json.loads` only (grep for exec/subprocess/pickle/import is empty).

- **Fixture corpus fully trusted, version-controlled; no deserialization risk.** Fields read individually (`fx["question"]`, `fx["expected_feature_id"]`, `:134-136`); none is fed to a constructor, format-evaluated, or opened as a path. `expected_path` is only compared (`==`/`endswith`, `:107`), never opened — a malicious `../` value would just fail to match, not traverse.

- **Writes confined to pytest temp dirs.** `tmp_path_factory.mktemp` copy + `cache_root` under the same temp base (`:63-67`); nothing touches the real repo, `SAMPLE_REPO`, or live `.context/`. Fresh `dest` for `copytree` — no clobber/symlink-follow hazard.

- **Eval-masking-a-regression is correctness, not security.** The vacuous-gate / budget-truncation risks (Data-integrity section; `:127,138-141`, `query.py:223-237`) have no attacker-controlled path — fixtures + sample repo are in-tree. Negative control (`:182-206`) is the right defense. Not double-counted as a security threat.

- **Verdict: clean.** Zero security threats; no remediation required.

### critic-product (product surface) — stage 3

## Product surface

- **A contributor who trips the gate cannot tell *which* fixture regressed from the failure message.** The two gate assertions emit only the aggregate number and a pointer to `BASELINE.md` — `f"hit-rate@3 {hit_rate_at_3:.4f} below floor {T_HIT} — retrieval regression (see tests/eval/BASELINE.md)"` (`test_retrieval_eval.py:166-173`). Nothing in the message names the offending question, its `expected_feature_id`, or the rank it landed at. The per-fixture data needed to say "`community-2`/`helpers.py` dropped from rank 1 to rank ∞ for question X" *is* computed in the loop (`rank`, `expected_id`, `fx['question']`, `:134-141`) but is discarded before the assertion fires — observed behavior: the maintainer gets "0.833 below 0.90" and must re-run with manual instrumentation to localize. **Gap:** no per-fixture miss table accumulated and surfaced in the assertion (or printed on failure), so the first debugging step is always "add a print and re-run."

- **The recorded metrics only print under `-s`/`-rA`, so the default-mode failure shows the floor breach without the diagnostic context that was printed.** The metrics block uses bare `print()` (`test_retrieval_eval.py:158-163`); pytest captures stdout and only replays it for *failed* tests in the report — but the per-positive breakdown (which fixture hit, at what rank) is never printed at all, only the three aggregates. Observed: on a green run the contributor sees nothing unless they pass `-s`; on a red run they see the three aggregates but still not the per-fixture culprit. **Gap:** the comment at `:122-123` claims "pytest surfaces them ... on failure," which is true for the aggregates but oversells actionability — the printed payload still doesn't answer "which fixture?"

- **Path-in-citations failures are conflated with retrieval misses in the contributor's mental model even though their messages differ.** A hit whose `expected_path` is absent from citations raises a *detailed* assertion naming the id, the path, the actual citations, and the question (`test_retrieval_eval.py:146-150`) — genuinely actionable. But this fires *inside* the same gate test as the floor breach, and a reader of `BASELINE.md`/`plan.md` is told the gate is "MRR + hit@3" (`plan.md:76-78`); the citation assertion is a *third* failure mode not surfaced in the feature's contract narrative. **Gap:** a contributor who hits the citation assertion has no doc path explaining it is a *path-stability* failure (e.g. `build_all` changed citation path shape, `:104-108`) distinct from a scoring regression — they will read "retrieval regression" framing and look in the wrong place (the scorer, not the citation/path emitter).

- **The baseline-update workflow is prose-only and manual, with no command or check that the two sources of truth stayed in sync.** Re-observation is "run the harness, update `BASELINE.md` and the constants" described only in prose (`BASELINE.md:1-9`, `plan.md:101-102`); the observed numbers live in a Markdown table (`BASELINE.md:38-40`) while the enforced floors live as Python constants (`test_retrieval_eval.py:46-47`), with the linkage maintained by a hand-written comment (`:42-44`). Observed behavior: a contributor who legitimately improves scoring has no `dummyindex`/`pytest` affordance that prints "new observed numbers; here are the floor values to paste" — they must hand-transcribe from `-s` output into both files. **Gap:** no `--update-baseline` mode or even a single emitted line formatted for copy-paste; nothing fails if `BASELINE.md`'s table and the constants drift apart, so the "rationale of record" can silently discontradict the enforced floor.

- **Nothing tells a contributor that legitimate scoring *improvement* needs a baseline bump — the floors only ratchet on regression, never on progress.** `plan.md:101-103` flags this as an open question ("nothing detects that `query()` scoring legitimately improved, raising the achievable baseline above the current floors"). Observed behavior: if a refactor raises true hit@3 to 1.0 and a later regression brings it back to 0.95, the gate stays green (0.95 ≥ 0.90) even though it is a real regression from the *improved* baseline. **Gap:** the gate is a one-sided floor with no "you are now well above your floor — consider re-baselining" signal, so the margin silently widens into slack over time and the gate's sensitivity decays without any contributor-visible prompt.

- **The `>= 12` corpus-growth guard is a foot-gun: it invites adding fixtures while the floors silently become too loose, and the failure (if the count drops) is cryptic.** `assert len(positives) >= 12, "spec requires ≥12 positive fixtures"` (`test_retrieval_eval.py:127`) actively *permits* growth, but per the data-integrity finding above and `BASELINE.md:66-75` the `0.90`/`0.85` magnitudes are calibrated to exactly 12. Observed behavior: a contributor adds three good fixtures (N=15), the suite stays green, and they have no signal that the one-miss tolerance just changed from "1 miss" toward "1.5 misses" — the margin doc (`BASELINE.md:66-75`) still says "with 12 positives." **Gap:** the guard's message frames 12 as a *minimum* (encouraging growth) while the floors treat it as an *exact* calibration point; there is no assertion or note coupling "fixture count changed" to "re-derive the floors," so the most natural contribution (more fixtures) quietly erodes the gate.
