## Security

- **[RESOLVED — `b157a1c`, branch `fix/outstanding-audit-fixes`]** `cache.py` + `extract/__init__.py` — **cache poisoning / unvalidated deserialization of graph content.** `load_cached` returned the parsed `{hash}.json` verbatim into the committed graph with zero schema validation. **Mitigated (defense in depth):** (1) **cache-dir confinement** is now the load-bearing containment — the ambient `DUMMYINDEX_CACHE_DIR` is confined under the repo root (see next item), so an attacker can no longer redirect cache reads/writes to a poisoned out-of-tree store; (2) `load_cached` now applies an **accept-superset schema guard** (`_is_valid_cache_payload`) that returns a MISS (re-extract, never merge) on a non-dict payload, non-list `nodes`/`edges`, or a node lacking a string `id`. The guard is explicitly **anti-corruption, not anti-poisoning** — a well-formed-but-adversarial graph still passes the shape check; the real containment is the dir confinement. _Residual (open, documented): in-repo self-poisoning of one's own cache via a SHA256-colliding file remains theoretically possible since content-addressing only prevents forging another file's result; the schema guard bounds shape, not adversarial content. Accepted scope per the proposal._
- **[RESOLVED — `b157a1c`, branch `fix/outstanding-audit-fixes`]** `cache.py` — **`DUMMYINDEX_CACHE_DIR` was `.resolve()`'d but otherwise unvalidated.** **Fixed (cache trust-channel split):** `cache_dir()` now has two channels with different trust levels. (1) The **trusted in-process** override (`_TRUSTED_CACHE_DIR`, set by `cache_dir_override` → `set_trusted_cache_dir`, `context/build/common.py`) is honored unconditionally — it never carries untrusted input and no longer flows through the env var (and no longer mutates the user's environment). (2) The **ambient** `DUMMYINDEX_CACHE_DIR` env var is now confined to the repo root via `resolve_under_root`: an out-of-repo / `../` / absolute-join value is **silently ignored**, falling back to `<root>/.context/cache/` (never raises — `test_cache_env_var_is_restored`). **Opt-out policy recorded:** `DUMMYINDEX_CACHE_DIR` is **preserved, not deprecated** — but now confined. _Note: the original concerns framing ("`cache_dir_override` survives via `cache_root`") was inverted — the override set the env var, which won over the threaded root; the audit corrected this and the fix re-routes the override off the env var entirely._
- **[RESOLVED — `b157a1c`, branch `fix/outstanding-audit-fixes`] (walk-time; read-time residual documented)** `extract/__init__.py` — **symlink escape outside repo root under `follow_symlinks=True`.** A crafted repo with `evil -> /home/user/.ssh` was walked and its files handed to extractors / `_read_text_safely`, disclosing out-of-tree content into the committed graph. **Fixed:** `collect_files` now computes a `containment_root` (explicit `root`, else `target`) and **filters at leaf emission** — a leaf whose `os.path.realpath` escapes the root via `resolve_under_root` is dropped before either read sink (the cache-hash read in `cache.py` or the extractor read in `generic.py`) ever touches it. _Residual (open, by design): this is **walk-time** containment. A post-enumeration symlink swap (TOCTOU) — replacing a vetted leaf with an escaping symlink between emission and read — is a documented residual; a true read-time guard would live in `generic.py` and is deferred to a later task. `is_safe_read_target` (`paths.py`) exists as the primitive for such a read-time guard but is not yet wired into the extractor read path._
- `references.py:106-125` — **ReDoS in the single-pass matcher: NOT exploitable; mitigation is load-bearing.** Every needle (rel-paths + fallback basenames derived from arbitrary repo paths) is `re.escape`'d and the alternation is wrapped only in a **zero-width lookahead** `(?=(...))` with no outer `+`/`*`/`{}` quantifier. Escaping kills attacker metacharacters; the lookahead-without-quantifier shape gives no catastrophic-backtracking structure even with hundreds of long common-prefix needles. Verified empirically: 500 × 200-char prefix-colliding needles over a 100 KB text scanned in ~0.1 s. The escape + lookahead-only construction is the actual defense — keep both.
- `io/git.py` (whole module) — **subprocess injection: not present (and a genuine hardening win).** All git "detection" is pure-filesystem (`is_git_repo`, `resolve_git_dir`, `submodule_paths` read `.git`, `commondir`, `.gitmodules` via configparser); **no `subprocess`/`os.system`/shell call exists**, so there is no command-injection surface from crafted branch names, `gitdir:` pointers, or `.gitmodules` `path=` values. Residual: `resolve_git_dir`/`submodule_paths` return `.resolve()`'d paths from attacker-controlled `gitdir:`/`commondir`/`path` content that point **anywhere on disk** (consumed by hook-scrubbing callers outside this feature) — out of scope here but worth a containment check at the consuming call site, not in this module.
- **[RESOLVED — `b157a1c`, branch `fix/outstanding-audit-fixes`]** `extract/resolve.py` (re-noted, plan Open-Q) — the Python/Java cross-file resolvers re-read+re-parsed every `.py`/`.java` from disk on a fully-cached run, so untrusted bytes were touched a **second** time independent of cache state. **Fixed (P2):** the resolver reads now route through `read_source_bytes` inside the same `build_read_cache()` block as the per-file pass, so within one build a file's bytes are read once and reused — the resolver no longer issues an independent disk read. _Residual: walk-time symlink containment (above) is the surviving boundary; the resolver no longer adds a second untrusted-byte read._

## Product surface

- Empty repo / no source files — `extract([])` sets `root = Path(".")`, runs zero per-file work, and returns `{"nodes": [], "edges": [], "input_tokens": 0, "output_tokens": 0}` (`extract/__init__.py:136-137,244-249`); `cluster()` short-circuits on an empty graph to `{}` (`cluster.py:73`). No error, no "nothing to index" message — caller pointing at an empty/wrong dir gets a silent empty graph, not a warning. Gap: no signal distinguishing "empty repo" from "wrong path".

- Repo with only unsupported extensions — every path hits `extractor is None` and is `continue`d (`extract/__init__.py:161-162`); same silent empty result. A caller with, say, an all-Haskell/HTML repo gets an empty graph and no list of skipped files. Gap: no "0 of N files were extractable" notice.

- Sub-100-file repo — progress printing is gated on `total >= _PROGRESS_INTERVAL` (100) (`extract/__init__.py:155,171`). On a 60-file repo nothing prints during AST extraction. For most small repos the caller sees no progress at all during the structural pass — matches the prompt's "no progress on sub-100-file repos." Gap: no per-stage heartbeat for the common small-repo case.

- Huge repo — extraction is a single serial `for` loop over all paths (`extract/__init__.py:154-170`), no parallelism, no batching, no memory cap; the full `all_nodes`/`all_edges` lists are held in memory and the whole graph is serialised at once (`graph.py:42-43`). No handler caps graph size or streams output. Gap: no degradation strategy for very large repos beyond the 100-file progress counter; capability of "scales to large repos" is hinted by the progress meter but not delivered by the loop.

- **[RESOLVED — `b157a1c`, branch `fix/outstanding-audit-fixes`]** graspologic-vs-Louvain partition divergence — committed `community` was only cross-machine-stable when graspologic presence was constant. **Fixed (GATE strategy (b)):** the committed path is now **always NetworkX-Louvain** (`_louvain_partition`, pure-Python, always present); Leiden (`_leiden_partition`) is split off-path and never feeds committed bytes. Two callers — with or without graspologic — now produce identical `symbol-graph.json` communities. The determinism guarantee is now cross-machine, not just per-backend. (See the matching Data-integrity and Cross-review entries.)

- **[RESOLVED — `b157a1c`, branch `fix/outstanding-audit-fixes`]** `cluster()` docstring vs behavior — docstrings claimed "Run Leiden" / "best quality" with no mention of the fallback. **Fixed:** the module + `cluster`/`_partition`/`_split_community`/`_louvain_partition` docstrings now state that the committed `community` field is produced by NetworkX-Louvain only (GATE strategy (b)) and that Leiden is off the committed path. Docs now match code.

- Full-repo re-hash on every `--changed` — `rebuild_changed` (`dummyindex/context/build/incremental.py:64`) runs `detect()` then `_hash_files` over *every* tracked code+doc file to diff against the manifest (`incremental.py:99-133,330-344`), before any cache reuse. So even a one-line edit re-reads and SHA256s the entire repo from disk each `--changed`. The per-file *extraction* cache then skips unchanged AST work, but the hashing cost is unavoidably O(repo). Gap: hidden full-disk-read cost on every incremental run; "incremental" describes the rebuild, not the change-detection.

- **[RESOLVED — `b157a1c`, branch `fix/outstanding-audit-fixes`] (P2)** Files read multiple times per build — a `.py` was read for the structural pass (or cache hit), again by `_resolve_cross_file_imports`, again by the textual-reference pass, plus the `--changed` hash — up to three+ touches. **Fixed:** all in-build source reads route through one `read_source_bytes` seam memoized by `build_read_cache()`; `extract` threads the read bytes out via `file_bytes` so the textual-reference pass (`_text_for_scan` in `references.py`) decodes those instead of re-reading. Each source file is now read **at most twice** across the whole build (the in-build read, plus the separately-counted `--changed` incremental hash in `incremental.py` which is excluded from the in-build count). _Residual (open, separate concern below): the `--changed` full-repo re-hash is a distinct O(repo) cost, not addressed by P2._

- Cache is never garbage-collected — entries are content-addressed `<sha256>.json` and only ever written (`cache.py:80-108`); editing a file leaves the old hash's entry on disk forever. Over many edits `.context/cache/` grows unbounded with dead entries. Gap: no eviction/prune surface; caller on a churny repo accumulates stale cache files with no command to clean them.

- Cache key ignores path — `file_hash` hashes content only, `root` is vestigial (`cache.py:20-40`). Two byte-identical files at different paths share one cache entry, which is fine for reuse, but `.md` frontmatter is stripped before hashing (`cache.py:10-17`) so a Markdown edit that only changes frontmatter is a cache hit and a no-op — intended, but a caller who edits front-matter and expects a re-index will see "no source files changed." Gap: silent no-op on frontmatter-only doc edits is correct-by-design but surprising.

- Parse failure is silent per-file — a file that fails to parse returns an `error` result (not cached, contributes no nodes/edges) and the loop continues (`extract/__init__.py:167-170`); cross-file resolution failures degrade to a logged WARNING (`extract/__init__.py:205-206,213-214`). Caller running the indexer gets a graph silently missing the un-parseable files with no summary of which files were dropped. Gap: no end-of-run report of skipped/errored files; partial graphs look complete.

- tree-sitter too old / missing — `_check_tree_sitter_version` raises a clear `ImportError`/`RuntimeError` with the exact pip command before any work (`extract/__init__.py:61-75,132`). This is the one boundary that fails loud and actionable — good. No gap.

- Bare `rebuild` on a curated index — refuses with exit 2 and points at `--full` / `--changed` (`rebuild.py:76-86`); `--full` warns it discards taxonomy (`rebuild.py:38-45`). Clear, scriptable guardrails. No gap, but the warning is stderr-only — a caller piping stdout won't see why nodes/communities changed.
