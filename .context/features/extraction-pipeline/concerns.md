## Data integrity

- `pipeline/extract/__init__.py:190-197` — the `id_remap` pass mutates `n["id"]`/`e["source"]`/`e["target"]` IN PLACE on dicts returned directly by `load_cached` (`cache.py:75` `json.loads`), violating the repo's no-mutation convention; safe only because each `load_cached` returns a fresh parse — fix by building new dicts (`{**n, "id": ...}`) so the cache result is never aliased/mutated.
- `pipeline/extract/__init__.py:216-221` — `global_label_to_nid` is keyed on the lowercased, paren/dot-stripped label, so two distinct symbols sharing a normalized label (e.g. `Foo` in two files, or `bar()` vs `.bar()`) collapse to ONE nid (last node in iteration order wins); `calls` edges then silently resolve to the wrong target — deterministic but incorrect — disambiguate by stem/source_file or skip resolution when the label is non-unique.
- `pipeline/extract/__init__.py:154-170` vs `plan.md:39` — on a fully-cached re-run the per-file pass is skipped, but `_resolve_cross_file_imports` (`resolve.py:78-79`) and `_resolve_cross_file_java_imports` (`resolve.py:189`) unconditionally `path.read_bytes()`+re-parse every `.py`/`.java` from disk; if a file changed on disk after its cache entry was written, the resolver sees NEW bytes while the structural nodes are STALE — cross-file edges can reference nids that no longer match the cached labels. Cache the resolver's parse on the same content hash, or resolve against the cached result instead of re-reading disk.
- `analysis/cluster.py:34-47` vs docstrings `cluster.py:64,30` — Leiden(graspologic)/Louvain seeds (`_RANDOM_SEED=42`) and the `(-len, sorted(c)[0])` tiebreak (`cluster.py:109`) DO hold and make community IDs stable on EITHER backend, but the two backends produce DIFFERENT partitions, so `symbol-graph.json`'s `community` field is only byte-stable when graspologic presence is itself constant across machines — pin the backend (require graspologic, or always use Louvain for the committed artefact) or exclude `community` from the committed bytes.
- `export/graph.py:36` — `data["hyperedges"]` is copied by reference from `G.graph["hyperedges"]` and never sorted, unlike nodes/links (`graph.py:40-41`); if any producer appends hyperedges in graph-construction order, the committed JSON is non-deterministic on that key — sort it before dump with a stable key.
- `io/cache.py:97` — `save_cached` writes `json.dumps(result)` with no `sort_keys`/`ensure_ascii` control, so cache-entry bytes are insertion-order-dependent; harmless for committed artefacts (cache isn't committed) but means the `<hash>.json` content can't be content-diffed across versions — add `sort_keys=True` for cache hygiene only.
- `io/cache.py:10-17` — `_body_content` strips `.md` frontmatter via `text.find("\n---", 3)`, which matches the FIRST `\n---` anywhere in the body (e.g. a horizontal-rule `---` inside prose), not just the closing fence; a doc with an early body `---` gets its head silently truncated from the hash → false cache hits across genuinely different content — anchor the close to a line-only `^---\s*$` match.
- `dummyindex/pipeline/build/structure.py:354` / `dummyindex/export/common.py:7` — `_filter_cross_edges` defaults missing `confidence` to the `ConfidenceLevel.EXTRACTED` enum while cache-roundtripped edges carry the bare string `"EXTRACTED"`; both serialize identically and `_CONFIDENCE_SCORE_DEFAULTS.get` works under `(str,Enum)` equality, so this is NOT a live bug — noting it only as a type-drift seam to keep an eye on if the enum ever stops subclassing `str`.

## Security

- `cache.py:60-77` + `extract/__init__.py:163-178` — **cache poisoning / unvalidated deserialization of graph content.** `load_cached` returns the parsed `{hash}.json` verbatim and `extract` does `per_file.append(cached)` → `result.get("nodes"/"edges")` straight into the committed graph with **zero schema validation** (no key/type/shape check on nodes/edges, no clamp on list size). Attack: an untrusted repo ships a file whose SHA256 equals an entry it also drops at `.context/cache/<that-hash>.json` (or in a shared `DUMMYINDEX_CACHE_DIR`); on the next run the poisoned entry is loaded *instead of* re-extracting that file, injecting attacker-chosen node ids / edges / `source_file` strings into `symbol-graph.json` and every downstream view that keys off them. Content-addressing prevents forging *another* file's result, but not self-poisoning of one's own cache. Blast radius is graph/index corruption + misleading provenance, not code execution (output is JSON, never `eval`/`pickle` — confirmed deserialization is uniformly `json.loads`). Load-bearing mitigation: **none** — nothing validates the cached payload before it joins the graph.
- `cache.py:43-57` — **`DUMMYINDEX_CACHE_DIR` is `.resolve()`'d but otherwise unvalidated, then `mkdir(parents=True)`.** Any absolute path the process can reach becomes a cache root and is created if absent; entries are then read back (see above). A poisoned env var (set by an untrusted wrapper / leaked into a subagent's environment) silently redirects both reads and writes to attacker-controlled storage, enabling the poisoning above without touching the repo tree. Mitigation: only that the value is `resolve()`'d (no traversal-string left), and `cache_dir_override` (`context/build/common.py:129-141`) sets it to a controlled in-repo target on the trusted path — but the env var is honoured unconditionally when a caller sets it, with no allow-listing under the repo root. Effective mitigation against an attacker-set value: **none**.
- `extract/__init__.py:268-293` — **symlink escape outside repo root under `follow_symlinks=True`.** The guard at 279-283 only rejects *self-referential* loops (`parent_real == real` / prefix), not symlinks whose target lives **outside** `target`/root. A crafted repo containing `evil -> /home/user/.ssh` (or `/etc`) is walked, and matching files are handed to extractors + `_read_text_safely` (`references.py:164-176`), which read up to 2 MB of arbitrary out-of-tree file content into node/edge text and `offset:` references in the committed graph — an information-disclosure path (secrets/keys surfaced into `.context/`). Default `follow_symlinks=False` (rglob) does not chase symlinked dirs, so this is gated on the caller opting in. Load-bearing mitigation when enabled: **none** (no `realpath`-under-root containment check on leaf targets).
- `references.py:106-125` — **ReDoS in the single-pass matcher: NOT exploitable; mitigation is load-bearing.** Every needle (rel-paths + fallback basenames derived from arbitrary repo paths) is `re.escape`'d and the alternation is wrapped only in a **zero-width lookahead** `(?=(...))` with no outer `+`/`*`/`{}` quantifier. Escaping kills attacker metacharacters; the lookahead-without-quantifier shape gives no catastrophic-backtracking structure even with hundreds of long common-prefix needles. Verified empirically: 500 × 200-char prefix-colliding needles over a 100 KB text scanned in ~0.1 s. The escape + lookahead-only construction is the actual defense — keep both.
- `io/git.py` (whole module) — **subprocess injection: not present (and a genuine hardening win).** All git "detection" is pure-filesystem (`is_git_repo`, `resolve_git_dir`, `submodule_paths` read `.git`, `commondir`, `.gitmodules` via configparser); **no `subprocess`/`os.system`/shell call exists**, so there is no command-injection surface from crafted branch names, `gitdir:` pointers, or `.gitmodules` `path=` values. Residual: `resolve_git_dir`/`submodule_paths` return `.resolve()`'d paths from attacker-controlled `gitdir:`/`commondir`/`path` content that point **anywhere on disk** (consumed by hook-scrubbing callers outside this feature) — out of scope here but worth a containment check at the consuming call site, not in this module.
- `extract/resolve.py:78-79` (re-noted, plan Open-Q) — Python cross-file resolver re-reads + re-parses every `.py` from disk on a fully-cached run, bypassing the cache. Not a vuln, but it means the symlink/out-of-tree read surface above is exercised a **second** time for `.py`, independent of cache state — relevant when reasoning about how many times untrusted bytes are touched.

## Product surface

- Empty repo / no source files — `extract([])` sets `root = Path(".")`, runs zero per-file work, and returns `{"nodes": [], "edges": [], "input_tokens": 0, "output_tokens": 0}` (`extract/__init__.py:136-137,244-249`); `cluster()` short-circuits on an empty graph to `{}` (`cluster.py:73`). No error, no "nothing to index" message — caller pointing at an empty/wrong dir gets a silent empty graph, not a warning. Gap: no signal distinguishing "empty repo" from "wrong path".

- Repo with only unsupported extensions — every path hits `extractor is None` and is `continue`d (`extract/__init__.py:161-162`); same silent empty result. A caller with, say, an all-Haskell/HTML repo gets an empty graph and no list of skipped files. Gap: no "0 of N files were extractable" notice.

- Sub-100-file repo — progress printing is gated on `total >= _PROGRESS_INTERVAL` (100) (`extract/__init__.py:155,171`). On a 60-file repo nothing prints during AST extraction. For most small repos the caller sees no progress at all during the structural pass — matches the prompt's "no progress on sub-100-file repos." Gap: no per-stage heartbeat for the common small-repo case.

- Huge repo — extraction is a single serial `for` loop over all paths (`extract/__init__.py:154-170`), no parallelism, no batching, no memory cap; the full `all_nodes`/`all_edges` lists are held in memory and the whole graph is serialised at once (`graph.py:42-43`). No handler caps graph size or streams output. Gap: no degradation strategy for very large repos beyond the 100-file progress counter; capability of "scales to large repos" is hinted by the progress meter but not delivered by the loop.

- graspologic absent → Louvain fallback — `_partition` catches `ImportError` and runs `nx.community.louvain_communities` (`cluster.py:46-56`). Both paths seed `_RANDOM_SEED=42` and re-index by `(-size, smallest-member)` (`cluster.py:109`), so IDs are stable, but the *partition itself* differs between Leiden and Louvain. Two callers — one with graspologic, one without — produce different `symbol-graph.json` communities for the same repo. Gap: the determinism guarantee is per-machine-given-same-backend, not cross-machine; nothing tells the caller which backend ran or that committing the file from a Louvain machine will diff against a Leiden teammate's. Plan flags this exactly as an open question (`plan.md:38`).

- `cluster()` docstring vs behavior — docstrings say "Run Leiden community detection" / "best quality" (`cluster.py:64,28`) with no mention of the Louvain fallback in the public `cluster()` summary. A caller reading the docstring believes Leiden always runs. Gap: documented capability ("Leiden") overstates what an un-extra'd install delivers (Louvain). Code wins; flag as doc conflict.

- Full-repo re-hash on every `--changed` — `rebuild_changed` (`dummyindex/context/build/incremental.py:64`) runs `detect()` then `_hash_files` over *every* tracked code+doc file to diff against the manifest (`incremental.py:99-133,330-344`), before any cache reuse. So even a one-line edit re-reads and SHA256s the entire repo from disk each `--changed`. The per-file *extraction* cache then skips unchanged AST work, but the hashing cost is unavoidably O(repo). Gap: hidden full-disk-read cost on every incremental run; "incremental" describes the rebuild, not the change-detection.

- Files read multiple times per build — a `.py` file is read once for the structural pass (or its cache hit) and then `_resolve_cross_file_imports` re-reads + re-parses it from disk a second time (`resolve.py:77-80`), independent of the cache. On a fully-cached re-run the structural pass is skipped but every `.py` is still re-read and re-parsed by the resolver. Add the incremental hash read and a `.py` is touched up to three times per `--changed`. Gap: cache reuse doesn't extend to cross-file resolution; plan flags this (`plan.md:39`).

- Cache is never garbage-collected — entries are content-addressed `<sha256>.json` and only ever written (`cache.py:80-108`); editing a file leaves the old hash's entry on disk forever. Over many edits `.context/cache/` grows unbounded with dead entries. Gap: no eviction/prune surface; caller on a churny repo accumulates stale cache files with no command to clean them.

- Cache key ignores path — `file_hash` hashes content only, `root` is vestigial (`cache.py:20-40`). Two byte-identical files at different paths share one cache entry, which is fine for reuse, but `.md` frontmatter is stripped before hashing (`cache.py:10-17`) so a Markdown edit that only changes frontmatter is a cache hit and a no-op — intended, but a caller who edits front-matter and expects a re-index will see "no source files changed." Gap: silent no-op on frontmatter-only doc edits is correct-by-design but surprising.

- Parse failure is silent per-file — a file that fails to parse returns an `error` result (not cached, contributes no nodes/edges) and the loop continues (`extract/__init__.py:167-170`); cross-file resolution failures degrade to a logged WARNING (`extract/__init__.py:205-206,213-214`). Caller running the indexer gets a graph silently missing the un-parseable files with no summary of which files were dropped. Gap: no end-of-run report of skipped/errored files; partial graphs look complete.

- tree-sitter too old / missing — `_check_tree_sitter_version` raises a clear `ImportError`/`RuntimeError` with the exact pip command before any work (`extract/__init__.py:61-75,132`). This is the one boundary that fails loud and actionable — good. No gap.

- Bare `rebuild` on a curated index — refuses with exit 2 and points at `--full` / `--changed` (`rebuild.py:76-86`); `--full` warns it discards taxonomy (`rebuild.py:38-45`). Clear, scriptable guardrails. No gap, but the warning is stderr-only — a caller piping stdout won't see why nodes/communities changed.
