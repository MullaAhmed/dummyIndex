## Data integrity

- **Cache round-trip drops the confidence_score for already-cached edges (cache-state-dependent non-determinism in committed output).** Edges are built with `confidence` = a `ConfidenceLevel` enum member (`generic.py:476,521,559,586,607,635`; `__init__.py:237`, `resolve.py`). `save_cached` JSON-dumps the result (`cache.py:97`); on the next run `load_cached` returns it via `json.loads` (`cache.py:75`), so `confidence` is now a plain `str` (`"EXTRACTED"`/`"INFERRED"`), not the enum. `to_json` backfills `confidence_score` via `_CONFIDENCE_SCORE_DEFAULTS.get(conf, 1.0)` (`graph.py:34-35`) where the dict is keyed by **enum members** (`common.py:7`). Verified empirically: `_CONFIDENCE_SCORE_DEFAULTS.get("extracted")` → `None`, `.get("EXTRACTED")` → `None`, `ConfidenceLevel.EXTRACTED == "extracted"` is `False` and hashes differ. Net effect: a fresh (cache-miss) extract emits `confidence_score=1.0/0.5/0.2`; an identical cache-hit run silently falls back to the `1.0` default for **every** edge regardless of true confidence. Same content → two different `symbol-graph.json` payloads depending on whether `.context/cache/{hash}.json` exists. Contradicts plan's "every stage re-runnable to byte-identical output" (`plan.md:46`) and data-access.md's byte-identical re-run contract (`data-access.md:13`). Fix: coerce `conf` through `ConfidenceLevel(conf)` (or key the defaults dict by `.value`/string) before the `.get`. Note: enum `.value` is uppercase `"EXTRACTED"` (`enums.py:16`) — any string-keying must match that exact casing.

- **`to_json` emits nodes/links in NetworkX graph-internal order, not a sorted order — re-run byte-stability depends on insertion order being identical, which holds only as long as the upstream `all_nodes`/`all_edges` list order is itself fully deterministic.** `node_link_data` (`graph.py:26`) preserves `G.nodes()` / edge iteration order, and `json.dump(... indent=2)` is written with **no `sort_keys` and no pre-sort** (`graph.py:37-38`) — unlike every other artifact writer in the repo, which sorts entries for byte-identical re-runs (`data-access.md:13`, `maps.py:134`, `manifest.py:189`, `meta.py:96`). The graph happens to be stable today because `all_nodes`/`all_edges` are appended in `paths` order (`__init__.py:174-178`) and the global raw-call pass iterates `per_file` deterministically (`__init__.py:224`). But there is no sort guarding it: any future change to file-collection order, a set-based intermediate, or dict-insertion-order drift in `global_label_to_nid` (`__init__.py:216-221`) would silently reorder the committed `symbol-graph.json` with no test catching it. Recommend an explicit sort of nodes (by `id`) and links (by `source,target,relation`) in `to_json`, matching the convention the rest of the store already enforces.

- **`extract`'s id-remap rewrites node/edge ids in place by mutating the cached dicts returned from `load_cached` (mutation of shared cache payloads; violates the repo's immutability rule).** `load_cached` returns a fresh `json.loads` dict each call (`cache.py:75`), but `extract` then mutates `n["id"]` and `e["source"]/e["target"]` directly on those dicts (`__init__.py:190-197`), and `build_from_json` further mutates edges/nodes in place (`build/__init__.py:75-76,88-91,102-103`). Within a single run this is benign (each cache load is a fresh object), but it means `per_file` results cannot be safely reused or re-iterated after this point, and it conflicts with the global immutability convention ("NEVER mutate existing ones"). Lower severity than the two above — flagging as a latent-aliasing/contract concern, not an active corruption.

- **Cross-file resolution failures are swallowed, so a partial-extraction run emits a *structurally valid but semantically incomplete* graph with no marker distinguishing it from a complete one.** `_resolve_cross_file_imports` and the Java resolver are wrapped in bare `except Exception` that only logs a warning (`__init__.py:205-206,213-214`); the run still returns and writes `symbol-graph.json`. Combined with per-file parse errors being skipped *without caching* (`__init__.py:168`) and git detection never raising (`git.py:11-14`), a degraded extraction is indistinguishable downstream from a clean one — no `partial`/`degraded` flag on the emitted artifact, no row-level completeness signal. The plan documents this as intentional "resilience over strictness" (`plan.md:106-110`), so this is a *contract* concern: consumers reading `symbol-graph.json` cannot tell a complete graph from one missing all cross-file INFERRED edges. Recommend stamping a `degraded: true` / failed-stage marker into the artifact when any resolution pass is swallowed.

- **`.md` cache key hashes only the post-frontmatter body, so two distinct files with identical bodies but different frontmatter collide on the same cache entry — and a body-identical edit that only touches frontmatter serves a stale extraction.** `file_hash` strips YAML frontmatter for `.md` before hashing (`cache.py:10-17,39`). This is deliberate (avoid cache busting on metadata-only edits) but has a real collision edge: if extraction output ever depends on frontmatter (e.g. a title/tag surfaced as a node label), the content-addressable key no longer uniquely identifies the extraction input, and `load_cached` (`cache.py:60-77`) returns the wrong/stale payload. Also note the frontmatter delimiter scan is naive — `text.find("\n---", 3)` (`cache.py:14`) matches the first `\n---` anywhere in the body (e.g. a Markdown horizontal rule or a `---` inside a code fence), which can mis-slice the body and produce a hash that doesn't correspond to the intended content boundary. Bounded blast radius (md-only, and only if extraction reads frontmatter), but it breaks the "same content → same entry, different content → different entry" invariant the cache contract rests on (`data-access.md:17`, `plan.md:103-105`).

- **`_split_community` runs its second pass over a whole-graph subgraph slice rather than the community's induced/connected component, and re-indexing collapses to size-descending order without a stable tiebreak for equal-size communities.** `_split_community(G, nodes)` builds `G.subgraph(nodes)` from the *full* graph (`cluster.py:109`), which the plan itself flags as untested at scale against isolate-assigned singletons (`plan.md:120-122`). Separately, the final re-index sorts communities by `len` descending (`cluster.py:103`) with **no secondary key**: two communities of equal size get an order determined by Python's stable sort over `raw.values()` insertion order, which is partition-dependent. Leiden (graspologic) is non-seeded here (`cluster.py:31-41` — no `random_seed` passed), unlike the Louvain fallback which is seeded `42` (`cluster.py:48`). So on a graspologic-equipped machine, community ids (and therefore each node's `community` field in `symbol-graph.json`, `graph.py:30`) are **not** guaranteed stable run-to-run, and differ from a Louvain-only machine — a determinism gap in committed output that depends on which optional dep is installed. Recommend: pass a fixed `random_seed` to graspologic `leiden`, and add a deterministic tiebreak (e.g. min node-id) to the size sort at `cluster.py:103`.

## Security

Adversarial lens: this pipeline parses **arbitrary untrusted repo source as DATA**. An attacker who can land a file in a target tree (PR, dependency, cloned repo) controls the parser input, the walk surface, and the cache key.

- **Unbounded recursion over attacker-controlled AST → stack-overflow DoS on a single crafted file.** `_extract_generic`'s `walk` recurses per child node (`generic.py:82,222,337,339`) and `walk_calls` likewise (`generic.py:365`), with no depth cap and no `sys.setrecursionlimit` guard anywhere in the pipeline (grep: zero hits). A deeply-nested but tiny source file (e.g. thousands of nested parens / brackets / blocks — `((((((…))))))`) yields a tree-sitter tree whose depth maps 1:1 to Python call depth. Attack path: drop one ~10 KB pathological file in any indexed dir → `RecursionError` (or, with a raised limit, a native C-stack segfault) aborts extraction of the **whole batch** mid-run. The per-file `try/except` at `generic.py:40-46` wraps only `parser.parse`, not the recursive walk, so the `RecursionError` propagates out of `_extract_generic` and is **not** caught by `extract`'s per-file loop (`__init__.py:167` calls the extractor with no guard). One file kills the index. Fix: convert `walk`/`walk_calls` to an explicit work-stack, or bound depth and wrap the walk in the same `try/except` that already guards parse.

- **No file-size ceiling on the extraction read path → trivial OOM from one huge file.** `_extract_generic` does `source = path.read_bytes()` (`generic.py:42`) with no `st_size` pre-check, and `collect_files` enumerates every matching extension with no size filter (`__init__.py:255-293`). The only size guard in the codebase lives in a *different* feature — `build/references.py:100-102` ("empty string … if too large") — and is never applied here (code wins over any doc claiming size-bounded reads). Attack path: a multi-GB `.py`/`.c`/`.ts` file (or a git-LFS-style placeholder expanded to a huge blob) is fully read into memory, then tree-sitter builds an in-memory AST several× that size, then the recursive walk allocates a node dict per AST node — peak RSS is a large multiple of file size. A single 2 GB source file OOM-kills the process. Fix: stat-gate in `collect_files`/`_extract_generic` and skip-with-marker over a configurable max (mirror `references.py`'s existing guard).

- **`follow_symlinks=True` walk only blocks *self-referential* symlink loops, not escape via symlinks pointing outside the target tree.** `collect_files(follow_symlinks=True)` uses `os.walk(..., followlinks=True)` (`__init__.py:278`) and its loop guard at `__init__.py:279-284` clears `dirnames` only when the resolved link target is an *ancestor of itself* (`parent_real == real or parent_real.startswith(real + os.sep)`). A symlink that points *sideways/upward* to an unrelated absolute path (e.g. `evil -> /` or `evil -> /home/user/.ssh`) is **not** caught — `realpath` of `/home/user/.ssh` is not an ancestor of the link's own parent, so the guard passes and `os.walk` descends into it. Attack path: a repo ships `docs/leak -> /etc` (or `-> ~`); running with `follow_symlinks=True` walks outside the intended root, reads source-extension files from arbitrary filesystem locations, and emits their `source_file` absolute paths and parsed identifiers into `symbol-graph.json` — an information-disclosure / scope-escape primitive. The default path (`follow_symlinks=False`, `rglob` at `__init__.py:270-276`) does not follow links and is safe; the risk is gated on the opt-in flag, but the flag's own guard is advertised (by its existence) as loop-safe while silently permitting traversal escape. Fix: after `realpath`, assert the target stays within `realpath(target_root)` and prune otherwise; reject links whose resolved target is outside the root regardless of loop status.

- **Content-only, path-independent cache key enables cross-file cache poisoning and a same-content collision channel.** `file_hash` hashes file *contents only*, deliberately dropping every path component (`cache.py:20-40`, docstring 21-33), and `load_cached`/`save_cached` key the on-disk entry purely on that SHA256 (`cache.py:60-77,80-108`). Consequence 1 — **collision/confusion across languages and roles:** two files with byte-identical content but different *paths/suffixes* (a `.py` and a copied `.txt`-renamed-`.py`, or the same vendored file in two locations) map to one cache entry; whichever extractor ran first wins, and the second file silently serves the first's nodes/edges — `source_file` in the cached payload then points at the *wrong* file. Consequence 2 — **poisoning via `DUMMYINDEX_CACHE_DIR`:** the cache dir is taken from an env var with no validation (`cache.py:51-53`); a shared or attacker-influenced cache directory lets a pre-seeded `{hash}.json` (the hash is computable offline from public file content) be loaded as trusted extraction output for any victim file with that content — `load_cached` returns it verbatim with no integrity/provenance check beyond filename==hash (`cache.py:74-77`), and the result flows straight into the committed graph. Consequence 3 — **`.md` body-only hashing** (`cache.py:10-17,39`) further widens collisions: distinct `.md` files differing only in frontmatter share a key (already noted under Data integrity for staleness; the security framing is that frontmatter — a place attackers put titles/tags — is excluded from the integrity key). Fix: bind the cache key to the (root-relative) path and suffix in addition to content, validate/confine `DUMMYINDEX_CACHE_DIR` to within the project root, and treat a cache hit whose stored `source_file` disagrees with the requesting path as a miss.

- **Cache payload is loaded as fully-trusted JSON and its ids flow unsanitised into in-place graph mutation.** `load_cached` returns whatever `json.loads` yields from `{hash}.json` (`cache.py:74-77`) and `extract` immediately mutates and trusts its `id`/`source`/`target` fields during the id-remap (`__init__.py:190-197`) and global raw-call join (`__init__.py:216-242`), with no schema validation that the cached dict has the expected shape. A poisoned or corrupted cache entry (see prior bullet, or a partially-written file despite the tmp+replace at `cache.py:95-99`) can inject arbitrary node ids / edge targets / `source_location` strings into `symbol-graph.json`, which every downstream `.context/` consumer treats as ground truth. This compounds the poisoning vector: the trust boundary (untrusted on-disk cache) is crossed with zero validation. Fix: validate the loaded payload against the node/edge schema at `load_cached` (reject on mismatch → treat as miss), per the repo's "never trust external data … schema-based validation at boundaries" rule.

- **(Scope note — not a feature finding) git access here is subprocess-free, so there is no argv-injection surface in this feature.** The plan and `git.py` are explicit that repo detection is *pure filesystem* (`git.py:1-14,24-59`) — no `subprocess`, no shell, no argv built from repo content. The only `git` *subprocess* in the codebase is `context/build/git_delta.py:144` (build-lifecycle, a different feature) and `context/default_plugins.py:211`; neither is on the extraction path. Confirmed via grep: no `subprocess`/`shell=True`/`os.system` under `dummyindex/pipeline/`. The residual filesystem-git risk is benign-by-design: `resolve_git_dir` follows an attacker-writable `.git` file's `gitdir:`/`commondir` pointer to an arbitrary path (`git.py:55-59,145-161`) and `.resolve()`s it, but it only *returns* the path for callers that scrub legacy hooks — extraction never executes anything from it, so within this feature it is information-level only. Flag for the consuming feature (hook-scrub), not here.

## Product surface

- **Unsupported / unrecognized file silently vanishes — no "skipped N files (unsupported)" summary.** — A path whose suffix isn't in `_DISPATCH` hits `extractor is None` and `continue`s with zero output (`pipeline/extract/__init__.py:161-162`); a per-file parse failure returns an `error` payload and is dropped from `all_nodes`/`all_edges` with no surfaced count (`__init__.py:167-170`, plan `plan.md:106-110`). — A user pointing ingest at a Haskell/Elm/Swift-light repo gets a graph that's empty or thin with no message telling them "0/0 files matched a supported language" — indistinguishable from "ran fine, nothing to index." No final tally of matched/unmatched/errored files is printed.

- **`.ex/.jl/.dart/.v/.vue` are extractable by single-file target but invisible to directory ingest — silent, asymmetric coverage.** — `_DISPATCH` registers `.ex`, `.jl`, `.dart`, `.v`, `.vue` extractors (`__init__.py:105-111`) but `collect_files`'s `_EXTENSIONS` walk set omits all of them (`__init__.py:255-260`); the plan flags this exact gap (`plan.md:116-119`). — `dummyindex` on a directory silently skips every Elixir/Julia/Dart/Verilog/Vue file, yet `dummyindex path/to/file.ex` extracts it fine. A user who confirms one file works will reasonably assume the whole repo is covered; it isn't, and nothing says so.

- **Huge-repo progress is coarse (every 100 files) and stdout-`print`-only — no ETA, no per-phase signal, nothing on a 99-file repo.** — Progress prints only when `total >= _PROGRESS_INTERVAL` and `i % 100 == 0` (`__init__.py:154-156,171-172`, `_PROGRESS_INTERVAL = 100` at `:115`); below 100 files there is zero feedback, and the cross-file resolution + global raw-call passes (`__init__.py:199-242`) and clustering (`analysis/cluster.py:59-104`) emit no progress at all. — On a 90-file repo the user sees nothing until it's done; on a 50k-file repo they get a percent counter for the AST pass but then a silent stall through resolution and Leiden with no "this phase can take a while" hint and no ETA. The plan advertises a "staged pipeline" (`plan.md:43-46`) but only one of the three stages reports.

- **Cache hit vs miss is completely invisible — a user can't tell a warm fast run from a cold full re-parse.** — `load_cached` returning a payload just `append`s and `continue`s with no log (`__init__.py:163-166`); `save_cached` on a miss is equally silent (`__init__.py:168-169`). No "N cached / M extracted" summary anywhere. — A user wondering why a re-run was instant (or why it was slow) gets no signal that the content-addressable cache (`pipeline/io/cache.py:43-108`) was hit or skipped, and no way to confirm caching is working other than timing it by hand. There is also no surfaced cache location or size.

- **"Byte-identical re-run" is the headline contract, but a cache round-trip silently changes committed output — the user can't see it and isn't warned.** — Plan promises "every stage re-runnable to byte-identical output" (`plan.md:46`), yet the Data-integrity section above documents that a cache-hit run flattens every edge's `confidence_score` to the `1.0` default (`export/graph.py:34-35`, cache returns plain strings not enum members). — From the product surface this is the worst kind of surprise: the feature's central promise is contradicted by its own caching, with no marker on `symbol-graph.json` and no warning. A user diffing two "identical" runs sees spurious churn and has no diagnostic pointing at the cache as the cause. Fix is in Data-integrity; the product impact is loss of trust in the determinism claim.

- **Partial / degraded extraction produces a structurally valid graph with no `degraded` flag and no user-facing summary of what was lost.** — Cross-file (Python + Java) resolver failures are swallowed with only a `logging.warning` (`__init__.py:205-206,213-214`) — invisible unless the user has configured logging output — and the run still writes a normal `symbol-graph.json`. — A user gets a graph missing all cross-file INFERRED edges with no banner, no exit-code change, and no `degraded: true` stamp distinguishing it from a clean run (also raised under Data integrity as a contract concern). For ingest UX specifically: the final report should say "cross-file resolution failed, graph is structure-only" — it says nothing.

- **No surfaced file-size or huge-file handling — a single multi-GB file stalls/OOMs the run with no progress and no skip notice.** — `_extract_generic` does `path.read_bytes()` with no `st_size` gate (`generic.py:42`) and `collect_files` applies no size filter (`__init__.py:255-293`); the only size guard lives in a different feature (`context/build/references.py:100-102`). — Operationally (see Security OOM finding), one giant generated/vendored file freezes the AST pass with the progress counter stuck on the prior multiple-of-100 and no "skipping large file X (NN MB)" message — the user can't tell ingest is hung vs. working.

- **Incremental-vs-full is opaque at this seam: `extract` always walks the full path list; "incremental" lives entirely in the build layer.** — `extract` iterates every path it's handed (`__init__.py:154`) and relies solely on the content cache for speed; the `rebuild --changed` incrementality is decided upstream by callers (`context/build/maps.py:111`, `runner.py:114`, `tree.py:68`, `enriched_refresh.py:105`) via `detect()`'s file list, not here. — A user running a "changed-only" rebuild gets no per-feature confirmation of which files were re-extracted vs cache-served; conversely a full rebuild re-walks (cheaply, via cache) every file with no statement that it's doing a full pass. The CLAUDE.md distinction between `rebuild --changed` (deterministic backbone) and full rebuild is real at the build layer but this pipeline emits no signal letting the user verify which mode actually ran.

## Cross-review notes

Operator-impact framing of the three Security flags, for whoever runs ingest in CI or on shared infra:

- **Symlink-escape (Security: `follow_symlinks=True` escape, `__init__.py:278-284`)** surfaces as a *silent scope/output surprise* to the operator: enabling the documented `follow_symlinks` flag on a repo containing `docs/x -> /etc` (or `-> ~`) pulls absolute paths from outside the tree straight into `symbol-graph.json`'s `source_file` fields with no warning that the walk left the root. An operator reviewing the emitted graph has no signal that off-tree files were indexed.

- **Cache-poisoning (Security: content-only key + unvalidated `DUMMYINDEX_CACHE_DIR`, `cache.py:51-53,74-77`)** is a *cache-behavior surprise* with operator blast radius: a shared/CI cache dir means a pre-seeded `{hash}.json` is served verbatim as trusted extraction with no provenance check — and (tying to the Product-surface "cache hit is invisible" bullet) the operator gets *no log line* distinguishing a poisoned/foreign cache hit from a legitimate one. Confining the cache dir to the project root and printing cache provenance would close both the security and the UX gap at once.

- **OOM (Security: no file-size ceiling, `generic.py:42`)** is the operator-facing reliability cliff already noted under Product surface: one oversized file OOM-kills the whole batch mid-run with the progress counter frozen, so in a CI log it reads as an unexplained hang/kill rather than "skipped file X (too large)". A stat-gate with a skip-marker turns a silent process death into a visible, recoverable skip.
