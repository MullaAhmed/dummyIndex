## Data integrity

- **Local `_atomic_write` diverges from the mandatory convention helper.** `writers.py:109-113` reimplements tmp+`replace` instead of routing through `write_text_atomic` (`domains/atomic_io.py:11-24`), so it inherits none of the convention's guarantees: no `cache/`-style Windows `PermissionError` copy-then-delete fallback (cf. `pipeline/io/cache.py:98-105`), and on Windows `tmp.replace(path)` can raise when a reader holds `INDEX.json`, leaving the `.tmp` orphan and the prior `INDEX` stale with no signal. Code wins over plan's "per the data-access convention" claim (`plan.md:85`) — the shape matches but the resilience does not.
- **Two-file write is not atomic as a unit.** `write_catalog` does two independent `_atomic_write` calls (`writers.py:21-22`); each file is individually atomic but the pair is not transactional. A crash or exception between line 21 and 22 leaves `INDEX.json` updated and `INDEX.md` stale (or vice-versa) — downstream consumers reading one as ground truth for the other (`output/docs.py` reads the catalog) can act on mismatched confidence. No cross-file integrity check or shared sha gate.
- **`mtime`-derived age signal is non-robust to clock skew / copied files.** `age_delta = newest_code_mtime - stat.st_mtime` (`catalog.py:158`) trusts filesystem `st_mtime` absolutely. A `git clone`/checkout/`cp`/rsync resets doc mtime to "now," making a genuinely stale doc look `fresh` and grade `high`; conversely a doc on a clock-skewed mount or restored-from-backup can show a future or ancient mtime and mis-bucket. The `fresh` floor at `-1s` (`catalog.py:27`) means even 1s of NTP jitter flips `recent`↔`fresh` — both map to `high` so harmless here, but the underlying signal has no skew tolerance and no content-based corroboration (sha256 is computed but never used for staleness).
- **Single scalar `newest_code_mtime` makes age coarse and brittle.** Age judges every doc against the one latest-touched file in the whole repo (`catalog.py:104,158`; plan.md:93 concedes this). One unrelated edit (e.g. a formatting sweep) refreshes `newest_code_mtime` for all docs, so a stale doc about an untouched subsystem still shows a small `age_delta` and can hold `high`. Determinism is preserved but the freshness verdict is only as good as that scalar.
- **`unknown` age → `high` is a deliberate mass-promotion with no stale-catalog backstop.** On a first build (or any run where `newest_code_mtime is None`), every low-broken-ratio doc grades `high` (`catalog.py:86-87,155-156`). Correct per the plan's first-build rationale (`plan.md:82`), but there is no separate "catalog itself is stale" detector — nothing flags that the catalog was built against a now-changed AST. Staleness detection is delegated entirely to the external cache manifest (`data-access.md:19-21`), which does not cover the source-docs `INDEX` files specifically.
- **`broken_ratio` float persisted unrounded in the entry; rounding only at the dict boundary.** `DocEntry.broken_ratio` stores raw `len(broken)/len(refs)` (`catalog.py:152,182`); the plan says `to_dict` rounds to 4 places (`plan.md:54`). The grading thresholds (`_classify_confidence`, `catalog.py:76,84`) compare the *unrounded* value against `0.10`/`0.40`, while the serialized JSON shows the rounded value — a doc graded off `ratio=0.10000001 > 0.10` can display `0.1000`, a small but real audit-trail divergence between the stored signal and the verdict it drove. Deterministic, but not self-consistent on inspection.
- **Confidence ordering relies on a lookup with a silent default.** Sort uses `DOC_CONFIDENCE_ORDER.get(e.confidence, 3)` (`catalog.py:191`); any confidence value not in the map silently sorts last rather than erroring. Today the set is closed (high/medium/low via `DocConfidence`), so byte-identical re-runs hold — but a future enum member would be absorbed without a contract failure, eroding the "council can trust it" determinism guarantee (`plan.md:79`).
- **JSON written with `sort_keys=False`.** `writers.py:21` serializes with `sort_keys=False`, unlike the convention's `meta.py:96` which adds `sort_keys=True` for byte-identical re-runs (`data-access.md:13`). Determinism here rests entirely on `to_dict` emitting keys in a stable insertion order; it holds for frozen-dataclass `to_dict`, but it is a weaker guarantee than the convention's explicit sort and would break silently if `to_dict` key order ever shifts.

## Security

Adversarial lens: this engine reads arbitrary prose docs, including external trees the user points at with `--docs PATH`, and trusts their bytes/metadata. The dominant risks are resource-exhaustion DoS from untrusted file content and symlink-mediated read of files outside the intended doc roots. Backtracking is **not** a real threat here — empirically confirmed below — so it is recorded as a non-finding rather than padded into one.

- **Unbounded full-file read of arbitrary `--docs` content — memory-exhaustion DoS (High).** `build_doc_catalog` does `data = p.read_bytes()` (`catalog.py:138`) and `extract_doc_text` does `path.read_text(... errors="ignore")` (`refs.py:153`) with **no `size_bytes`/`st_size` pre-check**, even though `stat()` is taken one line later (`catalog.py:139`) and `size_bytes` is already a model field (`models.py:17`). Attack path: `dummyindex ... --docs /path/with/huge.md` (or a multi-GB sparse/real `.txt` planted in any walked doc root) → every file under that root is slurped whole into RAM, then a second full copy is materialised as `text`, then `sha256(data)` hashes the whole buffer (`catalog.py:175`). A single oversized doc, or `/dev/zero`-fed regular file, OOM-kills the indexer. No streaming, no cap, no per-file budget. Remediation: skip or truncate files whose `stat().st_size` exceeds a constant cap (e.g. a `_MAX_DOC_BYTES` in `constants.py`) **before** `read_bytes`, and hash via a streamed/`hashlib` chunk loop.
- **Decompression-bomb DoS via office/PDF converters (High).** `.docx`/`.xlsx`/`.pdf` are dispatched to third-party parsers with no resource guard: `xlsx_to_markdown` calls `openpyxl.load_workbook` (`detect.py:162`), `docx_to_markdown` calls `python-docx`'s `Document` (`detect.py:123`), `extract_pdf_text` calls `pypdf.PdfReader` (`detect.py:107-110`). `.docx`/`.xlsx` are ZIP containers — a ~tens-of-KB zip-bomb spreadsheet expands to gigabytes of cell strings, and `xlsx_to_markdown` accumulates **every** cell of **every** sheet into an in-memory `rows`/`sections` list (`detect.py:166-180`) with no row/cell cap; a crafted PDF with millions of glyph ops makes `page.extract_text()` (`detect.py:111`) burn unbounded CPU. The blanket `except Exception` (`detect.py:115,154,185`) only catches *raised* errors — it does **not** bound memory or wall-clock before the bomb detonates. Attack path: `--docs DIR` containing a single malicious `.xlsx`. Remediation: gate office/PDF parsing behind the same size cap, cap rows/cells/pages processed, and consider running converters in a subprocess with an RLIMIT/timeout.
- **Symlinked doc *files* in a `--docs` root are read through to their target — path-confinement bypass (Medium).** `walk_doc_files` uses `os.walk(root)` with the default `followlinks=False` (`common.py:34`), which stops directory-symlink descent — but `os.walk` still *yields* symlinked **files**, and each is `p.resolve()`d (`common.py:42`) and later `read_bytes()`/`read_text()`d (`catalog.py:138`, `refs.py:153`). Attack path: a `--docs` root (e.g. a shared/cloned docs folder) containing `notes.md -> /etc/passwd` or `-> ~/.ssh/id_rsa`; the symlink target's **bytes flow into `INDEX.json`** — its `sha256`, `size_bytes`, harvested headings/title, and any backticked tokens become catalog content, and its `abs_path`/`path` is the *resolved* target outside the repo. This is read-amplification: a doc tree an attacker can write into exfiltrates arbitrary readable files into a committed artifact. The repo's own deep walker defaults `follow_symlinks=False` and prunes self-referential link loops (`detect.py:352,414-421`), but the doc-catalog `walk_doc_files` has **no symlink check at all**. Remediation: in `walk_doc_files`, skip entries where `p.is_symlink()` before `resolve`, or after resolving assert the target stays under the intended root (and apply the same to in-repo discovery's root `iterdir`, `discovery.py:60-62`).
- **`--docs` roots are explicitly de-confined — external absolute paths are by-design, but the catalog does not flag them as untrusted (Medium).** `_relpath_or_abs` marks anything outside `repo_root` as `is_external=True` with an absolute `path` (`catalog.py:91-95`), and `_attribute_source_root` attributes it to the `--docs` root (`catalog.py:203-219`). That is correct provenance, but `is_external` is a *display* flag only — external-doc bytes feed the **same** confidence grading and, downstream, `generate_project_md` can lift an external README's first paragraph into `PROJECT.md` (per spec `output/docs.py:122-194`). A doc the repo owner never wrote (planted in a sibling `--docs` dir) can therefore become the project's self-description and seed `extra_names`/title tokens the grader trusts. Threat is content-trust, not traversal: the tool treats external prose with the same authority as in-repo prose. Remediation: never source `PROJECT.md` description from `is_external` docs, and visibly tier external docs below in-repo ones in confidence.
- **`_attribute_source_root` compares a resolved doc path against an *unresolved* `repo_root` in one branch (Low).** `build_doc_catalog` resolves `repo_root` up front (`catalog.py:124`) and passes that resolved value, so today the `doc_path.relative_to(repo_root)` check (`catalog.py:216`) is sound; but the helper re-`.resolve()`s only `extra_doc_roots` (`catalog.py:211`), not `repo_root`, so a future caller passing an unresolved/symlinked `repo_root` would mis-attribute in-repo docs as external (empty `source_root`). Defensive, low impact today — flagged because the resolution contract is implicit, not enforced at the helper boundary.
- **Doc content is trusted as ground-truth signal, but the trust is bounded — informational.** Backticked tokens harvested from prose (`extract_code_refs`, `refs.py:191-205`) and JSON keys (`harvest_json_keys`, `keys.py:13-47`) only ever *widen* the known-name set used to mark refs as non-broken; they never execute, never index into the AST, and `harvest_json_keys` is capped at `limit=5000` keys (`keys.py:13,28`). So a hostile doc can at worst *suppress* its own `broken_refs` (grade itself falsely `high`) by citing plausible tokens — a self-serving confidence inflation, not code execution or cross-doc tampering. Worth noting because the council consumes `confidence` as evidence: a doc can lie its way to `high`. No fix required beyond awareness; the `broken_count ≥ 4` floor and ratio thresholds (`catalog.py:76-87`) limit the blast radius.

- **Non-finding (regex): no catastrophic backtracking.** `_INLINE_CODE_RE`, `_CODE_FENCE_RE` (`re.DOTALL`), and the shape regexes (`_DOTTED_RE`, `_FILE_PATH_RE`, etc., `refs.py:13-21`) were measured against adversarial inputs: per-token regexes are anchored and bounded to ≤160 chars (`refs.py:13,96`), and the whole-document `fence`/`inline` passes ran linearly (~0.03s on 3 MB of pathological backticks). The harvest regexes are not a ReDoS vector — recorded so it is not re-raised. The real doc-DoS lives in unbounded I/O and the office/PDF converters above, not the regex layer.

## Product surface

- **No docs at all (greenfield repo) — catalog still builds, `INDEX.md` shows a self-recovery hint — minor gap.** With zero discovered docs, `_render_catalog_md` emits `_No documents discovered. Pass --docs PATH..._` (`writers.py:46-51`) — good UX. But `INDEX.json` is still written with `docs: []` and `default_discovery_used: true`, and `output/docs.py:_render_doc_section` short-circuits to a "no docs" line (`docs.py:255-261` region). Gap: a user who *expects* docs (e.g. has a `docs/` dir of `.adoc`/`.org` files) gets a silent empty catalog with no hint that the extension set is closed — `.adoc`, `.org`, `.tex`, `.ipynb` are simply not in `_DOC_EXTENSIONS` (`discovery.py:27-32`) and never reported as skipped.
- **Many docs — `INDEX.md` table is uncapped, but `PROJECT.md` section caps at 5 with no "+N more" — inconsistent capping.** `_render_catalog_md` renders *every* doc as a table row (`writers.py:56-62`), so a 500-doc monorepo gets a 500-row table — fine as a catalog. But the curated `PROJECT.md` "notable" list silently truncates to `[:5]` (`docs.py:274`) with no "+N more" affordance, unlike the low-conf broken-ref list which *does* show `_… +{more} more_` (`writers.py:85-86`). Gap: a reader of `PROJECT.md` cannot tell 5 high-conf docs from 50; the cap is invisible.
- **Confidence-grading surprise: `unknown` age + low-broken-ratio → `high`, displayed identically to a genuinely-fresh `high` — trust ambiguity.** On a first build (`newest_code_mtime is None`), every clean doc grades `high` (`catalog.py:86-87,155-156`) and renders `**high**` with age column `unknown` (`writers.py:61`). A user scanning the confidence column sees `high` and trusts it, but the `unknown` age means *the freshness dimension was never evaluated* — the `high` rests solely on broken-ratio. Gap: `high` conflates "verified fresh AND clean" with "clean, freshness-unknown"; the age column carries the caveat but the bolded confidence verb dominates the eye. No visual tier between "high (fresh)" and "high (unknown age)".
- **Confidence-grading surprise: a doc with ZERO refs always grades on age alone — a prose-only README can be `high` while saying nothing checkable.** `broken_ratio = 0.0` when `refs` is empty (`catalog.py:152`), so a doc with no backticked tokens sails to `high` if recent/unknown (`_classify_confidence`, `catalog.py:84-87`). The `INDEX.md` broken-refs column shows `—` (`writers.py:57`) — correct — but the `high` badge implies "safe to quote" (`docs.py:269`) when in fact *nothing was cross-checked*. Gap: "0 refs, high confidence" reads as endorsement; it's actually "unfalsifiable." A `referenced_count == 0` doc should arguably be tiered separately, not lumped with genuinely-verified `high`.
- **Broken-ref false-positive surface: refs are matched against a single global name-set, so a doc citing another sub-project's symbol is flagged broken even when correct.** `find_broken_refs` matches each token against `symbol_names ∪ file_paths ∪ extra_names ∪ whitelist` (`catalog.py:146-151`); there is no per-doc scoping. Gap (UX): in a monorepo, a `--docs ../other-service/README.md` citing `OtherService.handle` is reported `broken` and can drag that external doc to `low` (`writers.py:65-93` lists it under "Low-confidence docs" with the legit symbol shown as "no longer in the AST") — a confusing false signal, since the symbol *does* exist, just not in *this* repo's AST. The remediation (`extra_names` / `--docs` co-indexing) is not surfaced to the user.
- **Staleness signal clarity: the `INDEX.md` "X day(s) before the newest code change" line only renders for `stale`/`old` buckets — `aging` docs get no day-count.** `writers.py:88-92` gates the human-readable day delta on `age_bucket in ("stale","old")`. An `aging` doc (31–90 days, `catalog.py:29`) shows only the bucket word `aging` in the table with no day count anywhere. Gap: the most actionable staleness number (how many days behind) is hidden for the middle bucket; a user must open `INDEX.json` to get `age_delta_seconds`. Also the phrase "before the newest code change" is computed against the single global `newest_code_mtime` (prior Data-integrity §6) — so the day-count can read alarmingly large because of one unrelated recent edit, not because *this* doc's subsystem changed.
- **Staleness signal clarity: the catalog itself carries no "I am stale" banner — `generated_at` is present but never compared to current source.** `DocCatalog.generated_at` is an ISO timestamp (`catalog.py:195`) but neither `INDEX.md` nor `_render_doc_section` surfaces "this catalog was built N days ago / against a since-changed AST." The drift hook (`plan-update`, `help.py:51`) reports *feature-doc* drift via the cache manifest, which "does not cover the source-docs `INDEX` files specifically" (prior Data-integrity §5). Gap: a user reading `source-docs/INDEX.md` cannot tell whether the confidence verdicts were computed against today's code or last month's; the `_ADVISORY_BANNER` (`writers.py:31`) is generic, not freshness-aware.
- **`--docs` external-root UX: missing paths warn-and-skip silently to stderr, so a typo'd root yields a confidently-wrong (smaller) catalog with no nonzero exit.** `resolve_doc_paths` prints `warning: --docs path not found, skipping` to stderr and drops the path (`common.py:173-178`), continuing the build. Gap: in a CI/non-interactive context the warning scrolls past, the run exits 0, and the catalog silently omits the intended docs — the user believes their `--docs ./design-docs` was ingested. No summary line ("ingested 2 of 3 --docs roots") and no opt-in strict mode. Compounding: a `--docs` pointing at a *file* (not dir) vs dir is not distinguished in the warning.
- **`--docs` external-root UX: external docs are de-confined by design but the only at-a-glance signal is the absolute `path`; the confidence column does NOT tier them below in-repo docs.** `is_external=True` docs render with an absolute path and a non-clickable abs link (`writers.py:104-106`), and they appear in the SAME confidence-sorted table interleaved with in-repo docs (`catalog.py:191`, `writers.py:56`). Gap: an external `high` doc sorts *above* an in-repo `medium` doc, implying the external (less-trusted-provenance) doc is the stronger evidence. Prior Security §19 flagged the trust dimension; the product gap is that the *sort* actively elevates external docs and the table has no "external" column to warn the reader.
- **`--docs` is repeatable and dedup'd by resolved path, but two roots whose trees overlap (symlink/junction or nested) silently double-attribute by first-match order — confusing `source_root` in `INDEX.json`.** `_attribute_source_root` returns the *first* `extra_doc_roots` entry the doc lives under (`catalog.py:209-214`), iteration order = user CLI order. Gap: pass `--docs /a --docs /a/sub` and a doc under `/a/sub` attributes to `/a` (whichever is listed first), not the more-specific root — a minor provenance surprise with no user-facing explanation of the precedence rule.

## Cross-review notes

- **Security §16/§17 (unbounded read / decompression bomb) has a direct product-surface consequence: there is no size column or skip-report, so a user cannot SEE which docs were huge.** Even before remediation lands, the product gap is that `INDEX.md` shows no `size_bytes` column (it is a model field, `models.py:17`) — so when the indexer OOMs or stalls on a `--docs` tree, the user gets no catalog row pointing at the offending file; the run just dies. A `_MAX_DOC_BYTES` cap (Security remediation) should *also* emit a visible "skipped (too large): path (N MB)" row so the product surface explains the omission rather than silently dropping the file.
- **Security §18 (symlinked doc files read through to target) is invisible at the product surface — the resolved target path lands in `path`/`abs_path` with no "via symlink" marker.** When `notes.md -> /etc/passwd` is ingested, `INDEX.md` shows the *resolved* `/etc/passwd` as the doc path (`catalog.py:164`, `_relpath_or_abs` resolves first) with confidence/headings — a reader auditing the catalog has no signal that this entry arrived via a symlink in their `--docs` tree. Beyond the security fix (skip symlinks), the product surface should mark or omit symlinked entries so the catalog is self-explaining.
- **Security §19 partially already mitigated in code — flag for the security critic.** Prior Security §19 warns an external README's first paragraph can seed `PROJECT.md`. The downstream `_description_from_catalog` already excludes externals: its candidate filter is `not d.is_external and d.confidence != DocConfidence.LOW` for READMEs and `not d.is_external and d.confidence == DocConfidence.HIGH` for the title fallback (`output/docs.py:206,213`). So the *description* path is already confined to in-repo docs — code wins over the §19 "can become the project's self-description" claim for the description specifically. The residual §19 risk (external bytes still feed `extra_names`/title tokens that *widen* the trusted name-set and so can suppress their own broken-refs) stands. Recommend §19 be narrowed accordingly.
